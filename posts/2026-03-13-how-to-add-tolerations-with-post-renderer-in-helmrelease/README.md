# How to Add Tolerations with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderer, Tolerations, Taints, Scheduling

Description: Learn how to add tolerations to Helm chart workloads using Flux post-renderers so pods can be scheduled on tainted nodes.

---

## Introduction

Kubernetes taints and tolerations work together to control which pods can be scheduled on which nodes. Taints are applied to nodes to repel pods that do not have matching tolerations. This mechanism is essential for dedicating nodes to specific workloads, isolating critical systems, and managing heterogeneous clusters with specialized hardware.

When deploying applications through Helm charts, you need your pods to tolerate any taints on the nodes where they should run. While some charts expose toleration configuration through values, many do not, or they do not apply tolerations to all generated workloads consistently.

Flux post-renderers solve this by allowing you to inject tolerations into any workload resource generated by a Helm chart, ensuring your pods can be scheduled on tainted nodes without chart modifications.

## Prerequisites

To follow this guide, you need:

- A Kubernetes cluster with Flux CD installed
- Nodes with taints applied (or plans to add them)
- A HelmRelease managed by Flux
- kubectl access to the cluster

## Understanding Taints and Tolerations

A taint on a node consists of a key, value, and effect. The effect determines what happens to pods without matching tolerations:

- `NoSchedule`: Pods without matching tolerations are not scheduled on the node
- `PreferNoSchedule`: The scheduler tries to avoid placing pods without tolerations, but it is not guaranteed
- `NoExecute`: Pods without matching tolerations are evicted from the node

A toleration on a pod specifies that it can tolerate a specific taint, allowing it to be scheduled on (or remain on) the tainted node.

## Adding a Basic Toleration

To add a toleration so pods can run on dedicated nodes:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    tolerations:
                      - key: dedicated
                        operator: Equal
                        value: application
                        effect: NoSchedule
```

This tolerates the taint `dedicated=application:NoSchedule`, allowing the pods to be scheduled on nodes with that taint.

## Adding Multiple Tolerations

Pods may need to tolerate multiple taints, especially in clusters with layered node management:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    tolerations:
                      - key: dedicated
                        operator: Equal
                        value: application
                        effect: NoSchedule
                      - key: environment
                        operator: Equal
                        value: production
                        effect: NoSchedule
                      - key: node.kubernetes.io/not-ready
                        operator: Exists
                        effect: NoExecute
                        tolerationSeconds: 300
```

The `tolerationSeconds` field specifies how long the pod can remain on a node after the taint is added before being evicted.

## Tolerating All Taints on a Node

To schedule a pod on any node regardless of taints, use an empty key with the Exists operator:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: monitoring-agent
  namespace: monitoring
spec:
  interval: 5m
  chart:
    spec:
      chart: monitoring-agent
      version: "1.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: DaemonSet
            patch: |
              apiVersion: apps/v1
              kind: DaemonSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    tolerations:
                      - operator: Exists
```

This is common for DaemonSets that need to run on every node, including control plane nodes and dedicated workload nodes.

## GPU Node Tolerations

GPU nodes are often tainted to prevent non-GPU workloads from consuming resources:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: ml-inference
  namespace: ml
spec:
  interval: 5m
  chart:
    spec:
      chart: ml-inference
      version: "1.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: inference-server
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: inference-server
              spec:
                template:
                  spec:
                    nodeSelector:
                      accelerator: nvidia-gpu
                    tolerations:
                      - key: nvidia.com/gpu
                        operator: Exists
                        effect: NoSchedule
                      - key: dedicated
                        operator: Equal
                        value: gpu
                        effect: NoSchedule
```

This combines node selectors with tolerations to both target GPU nodes and tolerate their taints.

## Adding Tolerations to All Workloads in a Chart

For charts with multiple workload types, apply tolerations across all of them:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-platform
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-platform
      version: "3.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    tolerations:
                      - key: dedicated
                        operator: Equal
                        value: platform
                        effect: NoSchedule
          - target:
              kind: StatefulSet
            patch: |
              apiVersion: apps/v1
              kind: StatefulSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    tolerations:
                      - key: dedicated
                        operator: Equal
                        value: platform
                        effect: NoSchedule
          - target:
              kind: Job
            patch: |
              apiVersion: batch/v1
              kind: Job
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    tolerations:
                      - key: dedicated
                        operator: Equal
                        value: platform
                        effect: NoSchedule
```

This ensures all Deployments, StatefulSets, and Jobs in the chart can run on nodes tainted for the platform workload.

## Verifying Tolerations

After deploying, check that tolerations are applied:

```bash
kubectl get deployment my-app -n production -o jsonpath='{.spec.template.spec.tolerations}' | jq .
```

Verify pods are running on tainted nodes:

```bash
kubectl get pods -n production -l app.kubernetes.io/name=my-app -o wide
kubectl describe node <node-name> | grep Taints
```

## Conclusion

Flux post-renderers make it easy to add tolerations to any Helm chart workload, ensuring pods can be scheduled on tainted nodes. Whether you need to tolerate dedicated node taints, GPU node taints, or system taints, strategic merge patches provide a clean way to inject tolerations without chart modifications. Always combine tolerations with node selectors to ensure pods land on the right nodes, and use `tolerationSeconds` with NoExecute taints to control eviction behavior during node issues.
