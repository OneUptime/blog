# How to Patch Talos Machine Configs with talosctl machineconfig patch

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, talosctl, Machine Configuration, Patching, DevOps

Description: A detailed walkthrough of using the talosctl machineconfig patch command to modify Talos Linux configurations offline before applying them to your cluster.

---

The `talosctl machineconfig patch` command is a tool for modifying Talos Linux machine configurations offline, before they ever touch a running node. Unlike `talosctl apply-config --patch`, which patches and applies in one step, `machineconfig patch` lets you preview changes, validate them, and build configuration pipelines. This guide covers the command in depth with practical examples.

## What machineconfig patch Does

The command takes an existing machine configuration file, applies one or more patches to it, and outputs the resulting configuration. It does not communicate with any nodes or make any changes to running systems. This makes it safe to use in testing, CI/CD pipelines, and configuration review workflows.

The basic syntax is:

```bash
talosctl machineconfig patch <config-file> --patch <patch> [--output <output-file>]
```

## Basic Usage

Start with a simple example. Suppose you have a `controlplane.yaml` generated by `talosctl gen config` and you want to change the cluster name:

```bash
# Patch the config and write to a new file
talosctl machineconfig patch controlplane.yaml \
    --patch '[{"op": "replace", "path": "/cluster/clusterName", "value": "production-cluster"}]' \
    -o patched-controlplane.yaml
```

Or using a strategic merge patch from a file:

```bash
# Create the patch file
cat > cluster-name-patch.yaml <<'EOF'
cluster:
  clusterName: production-cluster
EOF

# Apply it
talosctl machineconfig patch controlplane.yaml \
    --patch @cluster-name-patch.yaml \
    -o patched-controlplane.yaml
```

## Strategic Merge Patches vs JSON Patches

The `machineconfig patch` command supports both patch formats, just like other Talos patching operations.

Strategic merge patches use the same YAML structure as the machine configuration. They are simpler and more readable:

```yaml
# strategic-merge-patch.yaml
machine:
  network:
    hostname: production-cp-1
  kubelet:
    nodeLabels:
      environment: production
```

JSON patches (RFC 6902) use explicit operations and are more precise:

```json
[
    {"op": "replace", "path": "/machine/network/hostname", "value": "production-cp-1"},
    {"op": "add", "path": "/machine/kubelet/nodeLabels/environment", "value": "production"}
]
```

The strategic merge approach is generally easier to write and read. Use JSON patches when you need to remove fields, replace entire lists, or when the merge behavior of strategic patches does not give you the result you need.

## Applying Multiple Patches

You can chain multiple patches together. They are applied in order:

```bash
# Apply multiple patches in sequence
talosctl machineconfig patch controlplane.yaml \
    --patch @patches/common.yaml \
    --patch @patches/environment-prod.yaml \
    --patch @patches/node-cp-1.yaml \
    -o final-cp-1.yaml
```

Each patch is applied to the result of the previous one, so later patches can override earlier ones. This is exactly the layering approach that works well for managing configurations across environments.

## Practical Examples

### Adding Container Registry Mirrors

```yaml
# registry-mirrors.yaml
machine:
  registries:
    mirrors:
      docker.io:
        endpoints:
          - https://registry-mirror.internal:5000
      gcr.io:
        endpoints:
          - https://gcr-mirror.internal:5000
```

```bash
talosctl machineconfig patch worker.yaml \
    --patch @registry-mirrors.yaml \
    -o worker-with-mirrors.yaml
```

### Configuring Disk Encryption

```yaml
# disk-encryption.yaml
machine:
  systemDiskEncryption:
    state:
      provider: luks2
      keys:
        - nodeID: {}
          slot: 0
    ephemeral:
      provider: luks2
      keys:
        - nodeID: {}
          slot: 0
```

```bash
talosctl machineconfig patch controlplane.yaml \
    --patch @disk-encryption.yaml \
    -o encrypted-controlplane.yaml
```

### Setting Up Cluster Networking for Cilium

```yaml
# cilium-network.yaml
cluster:
  network:
    cni:
      name: none  # Disable default CNI, we will install Cilium
    podSubnets:
      - 10.244.0.0/16
    serviceSubnets:
      - 10.96.0.0/12
  proxy:
    disabled: true  # Cilium replaces kube-proxy
machine:
  kubelet:
    extraArgs:
      rotate-server-certificates: "true"
```

```bash
talosctl machineconfig patch controlplane.yaml \
    --patch @cilium-network.yaml \
    -o controlplane-cilium.yaml
```

### Adding Extra Kernel Arguments

```yaml
# kernel-args.yaml
machine:
  install:
    extraKernelArgs:
      - net.core.somaxconn=65535
      - vm.max_map_count=262144
```

```bash
talosctl machineconfig patch worker.yaml \
    --patch @kernel-args.yaml \
    -o worker-tuned.yaml
```

## Using machineconfig patch in CI/CD

The offline nature of `machineconfig patch` makes it perfect for CI/CD pipelines. Here is a practical workflow:

```bash
#!/bin/bash
# build-configs.sh
# CI/CD script to generate and validate Talos configurations

set -euo pipefail

CLUSTER="production"
ENDPOINT="https://api.production.internal:6443"
SECRETS="secrets.yaml"

# Step 1: Generate base configs
talosctl gen config "$CLUSTER" "$ENDPOINT" \
    --from-secrets "$SECRETS" \
    -o base/

# Step 2: Apply common patches
for role in controlplane worker; do
    talosctl machineconfig patch "base/${role}.yaml" \
        --patch @patches/common/ntp.yaml \
        --patch @patches/common/dns.yaml \
        --patch @patches/common/registry.yaml \
        -o "staged/${role}.yaml"
done

# Step 3: Apply role-specific patches
talosctl machineconfig patch staged/controlplane.yaml \
    --patch @patches/controlplane/etcd.yaml \
    --patch @patches/controlplane/api-server.yaml \
    -o staged/controlplane.yaml

talosctl machineconfig patch staged/worker.yaml \
    --patch @patches/worker/kubelet.yaml \
    -o staged/worker.yaml

# Step 4: Generate per-node configs
declare -A CP_NODES=(
    ["cp-1"]="10.0.1.10"
    ["cp-2"]="10.0.1.11"
    ["cp-3"]="10.0.1.12"
)

for node in "${!CP_NODES[@]}"; do
    talosctl machineconfig patch staged/controlplane.yaml \
        --patch @"patches/nodes/${node}.yaml" \
        -o "final/${node}.yaml"

    # Step 5: Validate each config
    talosctl validate --config "final/${node}.yaml" --mode metal
    echo "Validated config for $node"
done

echo "All configurations generated and validated successfully"
```

## Comparing Configurations

After patching, you might want to see what changed. Since the output is YAML, standard diff tools work well:

```bash
# See what a patch changes
talosctl machineconfig patch controlplane.yaml \
    --patch @my-patch.yaml \
    -o patched.yaml

diff controlplane.yaml patched.yaml
```

For a more readable diff, you can use `dyff` which understands YAML structure:

```bash
# Using dyff for YAML-aware diffing
dyff between controlplane.yaml patched.yaml
```

## Extracting Current Config and Patching

You can combine `machineconfig patch` with the ability to extract configs from running nodes:

```bash
# Get the current config from a running node
talosctl get machineconfig --nodes 10.0.1.10 -o yaml > current-cp-1.yaml

# Apply a patch to see what the result would be
talosctl machineconfig patch current-cp-1.yaml \
    --patch @my-update.yaml \
    -o proposed-cp-1.yaml

# Review the differences
diff current-cp-1.yaml proposed-cp-1.yaml

# If satisfied, apply to the node
talosctl apply-config --nodes 10.0.1.10 --file proposed-cp-1.yaml
```

This workflow gives you a preview of changes before applying them, which is valuable for production systems where you want to minimize surprises.

## Handling Patch Conflicts

When patches conflict, the last patch wins. But sometimes you get unexpected results when patches target overlapping parts of the configuration. For example:

```yaml
# patch-a.yaml - Sets two nameservers
machine:
  network:
    nameservers:
      - 8.8.8.8
      - 8.8.4.4

# patch-b.yaml - Sets one nameserver
machine:
  network:
    nameservers:
      - 1.1.1.1
```

If you apply both patches, the final result has only `1.1.1.1` because `nameservers` is a list that gets replaced entirely by the strategic merge patch. To keep all three nameservers, you would need to include all of them in the second patch.

Understanding this behavior is critical for building reliable patch pipelines. Always verify the output when stacking patches.

## Output Options

The `-o` flag controls where the patched configuration is written:

```bash
# Write to a specific file
talosctl machineconfig patch input.yaml --patch @p.yaml -o output.yaml

# Write to stdout (useful for piping)
talosctl machineconfig patch input.yaml --patch @p.yaml -o -

# Pipe to validation
talosctl machineconfig patch input.yaml --patch @p.yaml -o - | \
    talosctl validate --config /dev/stdin --mode metal
```

The stdout option is particularly useful in pipelines where you want to chain operations without creating intermediate files.

## Conclusion

The `talosctl machineconfig patch` command is an essential tool for anyone managing Talos Linux configurations. Its offline nature makes it safe for experimentation and perfect for automation. By building your configuration management around this command, you get a repeatable process that can be reviewed, tested, and version-controlled. Use it to build layered configurations, preview changes before applying them, and integrate Talos config management into your existing CI/CD workflows. The key is to think of your configurations as code and treat the patch command as a build step in your infrastructure pipeline.
