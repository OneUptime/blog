# How to Version Control Istio Configuration Files

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Version Control, Git, Configuration, DevOps

Description: How to organize and version control Istio configuration files with Git for better collaboration, auditability, and rollback support.

---

If your Istio configuration lives in a shared wiki, someone's laptop, or a collection of `kubectl apply` commands in a Slack channel, you are one bad edit away from a production outage with no easy way to figure out what changed. Version controlling your Istio configuration with Git is the foundation of reliable mesh management.

Here is how to structure, organize, and manage your Istio configuration files under version control.

## What to Version Control

Version control everything that defines how your mesh behaves:

- **IstioOperator** - The mesh installation configuration
- **VirtualService** - Routing rules
- **DestinationRule** - Traffic policies, circuit breaking, load balancing
- **Gateway** - Ingress and egress gateway definitions
- **AuthorizationPolicy** - Access control rules
- **PeerAuthentication** - mTLS settings
- **RequestAuthentication** - JWT validation rules
- **Telemetry** - Metrics, tracing, and logging configuration
- **Sidecar** - Sidecar scope configuration
- **ServiceEntry** - External service definitions
- **EnvoyFilter** - Advanced proxy configuration
- **WasmPlugin** - Wasm plugin definitions

Things you should NOT version control:

- TLS certificates and private keys (use external secret management)
- Short-lived debug configurations (use annotations instead)
- Anything auto-generated by Istio itself

## Repository Organization

There are two main approaches to organizing Istio configuration in Git:

### Approach 1: Centralized Istio Repository

All Istio configuration lives in a single repository:

```text
istio-config/
  README.md
  mesh/
    istio-operator.yaml
    peer-authentication.yaml
    telemetry.yaml
  gateways/
    ingress/
      gateway.yaml
      certificates.yaml
    egress/
      gateway.yaml
  services/
    api-gateway/
      virtual-service.yaml
      destination-rule.yaml
      authorization-policy.yaml
    payment-service/
      virtual-service.yaml
      destination-rule.yaml
      authorization-policy.yaml
    user-service/
      virtual-service.yaml
      destination-rule.yaml
  environments/
    production/
      kustomization.yaml
    staging/
      kustomization.yaml
```

This works well when a dedicated platform team manages the mesh configuration.

### Approach 2: Configuration Alongside Application Code

Each application repository contains its own Istio configuration:

```text
payment-service/
  src/
  Dockerfile
  k8s/
    deployment.yaml
    service.yaml
  istio/
    virtual-service.yaml
    destination-rule.yaml
    authorization-policy.yaml
```

This works well when development teams own their service mesh configuration. The tradeoff is that mesh-wide configuration (like PeerAuthentication and Telemetry) still needs a central home.

### Approach 3: Hybrid

Use a central repository for mesh-wide configuration and let each team manage their service-specific configuration:

```text
# Central repo: istio-platform-config
mesh/
  istio-operator.yaml
  peer-authentication.yaml
  telemetry.yaml
  gateways/

# Per-team repos: each team's application repo
payment-service/
  istio/
    virtual-service.yaml
    destination-rule.yaml
```

This is the most common pattern in practice. The platform team controls the mesh-wide settings, and application teams control their own routing and policies.

## File Naming Conventions

Be consistent with naming. Here is a convention that works:

```text
# Resource type as the filename
virtual-service.yaml
destination-rule.yaml
authorization-policy.yaml

# Or if you have multiple resources of the same type
virtual-service-api.yaml
virtual-service-web.yaml

# Avoid
config.yaml           # Too vague
my-service.yaml       # What kind of resource?
test.yaml             # Is this active?
```

## One Resource Per File

Put each Istio resource in its own file. Do not combine multiple resources in a single file with `---` separators:

```yaml
# Bad - multiple resources in one file
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: my-service
# ...
---
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: my-service
# ...
```

```yaml
# Good - separate files
# virtual-service.yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: my-service
# ...
```

```yaml
# destination-rule.yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: my-service
# ...
```

One resource per file makes git diffs cleaner, code reviews easier, and reduces merge conflicts.

## Using Kustomize for Environment Management

Kustomize lets you maintain a base configuration and apply environment-specific patches:

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - virtual-service.yaml
  - destination-rule.yaml
  - authorization-policy.yaml
```

```yaml
# environments/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
patches:
  - target:
      kind: VirtualService
      name: my-service
    patch: |
      - op: replace
        path: /spec/http/0/timeout
        value: 10s
  - target:
      kind: DestinationRule
      name: my-service
    patch: |
      - op: replace
        path: /spec/trafficPolicy/connectionPool/tcp/maxConnections
        value: 200
```

## Branching Strategy

Use a standard Git branching strategy:

```text
main (production)
  |
  |--- feature/add-canary-routing
  |--- feature/update-auth-policy
  |--- fix/timeout-configuration
```

Every change goes through a pull request with:

1. A description of what is changing and why
2. CI validation (see below)
3. At least one reviewer who understands Istio configuration
4. A link to any related monitoring dashboards or runbooks

## Pull Request Template

Create a PR template for Istio configuration changes:

```markdown
## Change Description
<!-- What is changing and why? -->

## Affected Services
<!-- Which services will be affected by this change? -->

## Risk Assessment
<!-- What could go wrong? How would we detect it? -->

## Rollback Plan
<!-- How to roll back if something goes wrong -->

## Validation
- [ ] `istioctl analyze` passes
- [ ] YAML syntax is valid
- [ ] Tested in staging environment
- [ ] Relevant dashboards bookmarked for monitoring
```

## CI Validation Pipeline

Automate validation on every pull request:

```yaml
name: Validate Istio Config
on:
  pull_request:
    paths:
      - 'istio/**'
      - 'services/**/istio/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install istioctl
        run: |
          curl -L https://istio.io/downloadIstio | ISTIO_VERSION=1.24.0 sh -
          sudo mv istio-*/bin/istioctl /usr/local/bin/

      - name: YAML lint
        run: |
          pip install yamllint
          yamllint -d relaxed istio/

      - name: Istio analysis
        run: istioctl analyze -R istio/

      - name: Dry run
        run: |
          find istio/ -name '*.yaml' -exec kubectl apply --dry-run=client -f {} \;
```

## Tagging Releases

Tag your configuration versions so you can reference specific states:

```bash
git tag -a v1.5.0 -m "Add canary routing for payment service"
git push origin v1.5.0
```

Use tags as deployment references in your GitOps tool:

```yaml
# ArgoCD Application
spec:
  source:
    targetRevision: v1.5.0
```

## Handling Configuration Drift

Sometimes someone makes a manual change in the cluster that does not match what is in Git. Detect this:

```bash
# Export current config and diff against git
kubectl get virtualservices -n production -o yaml > current-vs.yaml
diff current-vs.yaml istio/services/production/virtual-services/
```

Better yet, use a GitOps controller that automatically detects and reconciles drift.

## Documenting Configuration

Add comments to your YAML files explaining the "why":

```yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: payment-service
  namespace: production
spec:
  host: payment-service
  trafficPolicy:
    connectionPool:
      tcp:
        # Limit to 50 because the payment gateway enforces
        # a connection limit of 100 and we have 2 replicas
        maxConnections: 50
    outlierDetection:
      # Payment service has known slow responses during
      # batch processing (every hour). Do not eject too aggressively.
      consecutive5xxErrors: 10
      interval: 30s
      baseEjectionTime: 60s
```

Version controlling Istio configuration is not optional for production. It is the minimum bar for operating a service mesh responsibly. Start simple, be consistent with your structure, and invest in CI validation early.
