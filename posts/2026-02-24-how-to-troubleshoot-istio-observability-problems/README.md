# How to Troubleshoot Istio Observability Problems

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Observability, Metrics, Tracing, Troubleshooting, Prometheus

Description: Fix common Istio observability issues including missing metrics, broken distributed traces, incomplete telemetry data, and Prometheus scraping failures.

---

One of the best things about Istio is the observability you get for free - metrics, distributed traces, and access logs without changing a single line of application code. But when that telemetry pipeline breaks, you are flying blind. Missing metrics mean your dashboards show nothing. Broken traces mean you cannot debug latency issues. And the root cause is often some small misconfiguration that is easy to overlook.

This guide walks through how to diagnose and fix the most common Istio observability problems.

## Missing Metrics in Prometheus

The most frequent observability complaint is "my Istio metrics are gone." Start by checking if the metrics are actually being generated by the proxy.

Check if the Envoy sidecar is producing metrics:

```bash
# Port-forward to the sidecar's admin interface
kubectl port-forward <pod-name> -n production 15000:15000 &

# Check if stats are being generated
curl -s localhost:15000/stats/prometheus | grep istio_requests_total

# Check the specific metric you are looking for
curl -s localhost:15000/stats/prometheus | grep istio_request_duration_milliseconds
```

If the metrics exist on the proxy but are not showing up in Prometheus, the issue is with scraping.

Check Prometheus scrape targets:

```bash
# Port-forward to Prometheus
kubectl port-forward svc/prometheus -n istio-system 9090:9090 &

# Check targets in the Prometheus UI
# Navigate to http://localhost:9090/targets
```

Look for targets that are down or have errors. Common scrape issues include:

**Pod annotations missing**: Prometheus needs to know where to scrape. Check that your pods have the right annotations:

```bash
kubectl get pod <pod-name> -n production -o jsonpath='{.metadata.annotations}' | jq .
```

The Istio sidecar should automatically add these annotations:

```yaml
prometheus.io/scrape: "true"
prometheus.io/port: "15020"
prometheus.io/path: "/stats/prometheus"
```

If they are missing, the sidecar injection might not be working correctly.

## Telemetry API Configuration

Istio uses the Telemetry API to configure what metrics are generated. Check if there is a Telemetry resource that might be disabling metrics:

```bash
# Check for Telemetry resources
kubectl get telemetry --all-namespaces

# See the details
kubectl get telemetry -n production -o yaml
kubectl get telemetry -n istio-system -o yaml
```

A misconfigured Telemetry resource can disable metrics entirely:

```yaml
# This would disable all metrics - check if something like this exists
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: disable-metrics
  namespace: production
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: ALL_METRICS
          disabled: true
```

If you find something like this, either delete it or fix the configuration:

```yaml
# Correct configuration to enable standard metrics
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: default
  namespace: production
spec:
  metrics:
    - providers:
        - name: prometheus
```

## Missing or Broken Distributed Traces

Distributed tracing requires your application to propagate trace headers. Istio generates span data, but it cannot stitch traces together across services unless the application forwards certain headers.

The headers that must be propagated are:

- `x-request-id`
- `x-b3-traceid`
- `x-b3-spanid`
- `x-b3-parentspanid`
- `x-b3-sampled`
- `x-b3-flags`
- `traceparent`
- `tracestate`

If you are using W3C trace context (the default in newer Istio versions), `traceparent` and `tracestate` are the critical ones.

Check if trace headers are being generated:

```bash
# Make a request and check the headers
kubectl exec <pod-name> -c istio-proxy -n production -- \
  curl -v http://orders-service:8080/health 2>&1 | grep -i "x-b3\|traceparent"
```

Check if the tracing backend is receiving data:

```bash
# If using Jaeger
kubectl port-forward svc/jaeger-query -n istio-system 16686:16686 &

# If using Zipkin
kubectl port-forward svc/zipkin -n istio-system 9411:9411 &
```

If traces show up but are fragmented (each service shows as a separate trace instead of connected spans), the application is not propagating headers. You need to add header propagation to your application code.

## Sampling Rate Issues

If traces are appearing for some requests but not others, check the sampling rate:

```bash
# Check mesh config for tracing settings
kubectl get configmap istio -n istio-system -o yaml | grep -A 10 "tracing"
```

The default sampling rate might be too low:

```yaml
# In the mesh config
meshConfig:
  defaultConfig:
    tracing:
      sampling: 100.0  # 100% sampling for debugging
```

For production, you typically want a lower rate (1-10%) to avoid overwhelming your tracing backend. But for debugging, crank it up to 100%:

```bash
# Update sampling rate temporarily
kubectl get configmap istio -n istio-system -o yaml | \
  sed 's/sampling: .*/sampling: 100.0/' | \
  kubectl apply -f -
```

Remember to set it back to a reasonable value after debugging.

## Access Log Issues

If access logs are not showing up, check whether they are enabled:

```bash
# Check mesh config for access log settings
kubectl get configmap istio -n istio-system -o yaml | grep accessLog
```

Enable access logs if they are not configured:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    accessLogFile: /dev/stdout
    accessLogEncoding: JSON
```

Or use `istioctl`:

```bash
istioctl install --set meshConfig.accessLogFile=/dev/stdout --set meshConfig.accessLogEncoding=JSON
```

To enable access logs for specific workloads only (to avoid the volume of mesh-wide logging), use the Telemetry API:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: enable-access-logs
  namespace: production
spec:
  selector:
    matchLabels:
      app: orders-service
  accessLogging:
    - providers:
        - name: envoy
```

## Custom Metrics Not Appearing

If you added custom metrics through the Telemetry API and they are not showing up, verify the configuration:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: custom-metrics
  namespace: production
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT
          tagOverrides:
            request_host:
              operation: UPSERT
              value: request.host
```

Check that the provider name matches what is configured in the mesh config:

```bash
kubectl get configmap istio -n istio-system -o yaml | grep -A 20 "extensionProviders"
```

## Debugging the Telemetry Pipeline

When you cannot figure out where the telemetry pipeline is broken, trace the data flow step by step:

1. Check metrics at the proxy level:

```bash
curl -s localhost:15000/stats/prometheus | head -50
```

2. Check that the metrics endpoint is accessible:

```bash
kubectl exec <pod-name> -c istio-proxy -n production -- \
  curl -s localhost:15090/stats/prometheus | grep istio_requests_total | head -5
```

3. Check Prometheus scrape config:

```bash
kubectl get configmap prometheus -n istio-system -o yaml | grep -A 5 "istio-mesh"
```

4. Verify data in Prometheus:

```bash
# Check if any Istio metrics exist at all
curl -s "http://localhost:9090/api/v1/label/__name__/values" | jq '.data[]' | grep istio
```

5. Check Grafana data source connection:

```bash
kubectl logs -n istio-system deployment/grafana | grep "error\|failed"
```

## Kiali Dashboard Issues

If Kiali is not showing the service graph correctly, it usually means it cannot reach Prometheus:

```bash
# Check Kiali logs
kubectl logs -n istio-system deployment/kiali | grep "error"

# Verify Kiali's Prometheus configuration
kubectl get configmap kiali -n istio-system -o yaml | grep prometheus_url
```

Make sure the Prometheus URL in Kiali's configuration is correct and reachable from within the cluster.

## Summary

Istio observability problems usually fall into three categories: metrics not being generated (check proxy stats and Telemetry resources), metrics not being collected (check Prometheus scraping and pod annotations), and tracing not working (check header propagation and sampling rates). Start from the source (the Envoy proxy) and work your way through the pipeline to find where data stops flowing. Most of the time, it is a small configuration issue that is easy to fix once you find it.
