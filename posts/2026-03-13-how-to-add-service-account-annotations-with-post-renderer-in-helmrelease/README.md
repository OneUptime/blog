# How to Add Service Account Annotations with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderer, ServiceAccount, Annotations, IAM

Description: Learn how to add annotations to Kubernetes ServiceAccounts generated by Helm charts using Flux post-renderers for IAM role binding.

---

## Introduction

ServiceAccount annotations are a critical integration point between Kubernetes and cloud provider IAM systems. On AWS, the `eks.amazonaws.com/role-arn` annotation links a ServiceAccount to an IAM role through IRSA (IAM Roles for Service Accounts). On GCP, the `iam.gke.io/gcp-service-account` annotation connects to Workload Identity. On Azure, similar annotations enable Azure AD Pod Identity.

While some Helm charts provide values to configure ServiceAccount annotations, many charts either do not generate a ServiceAccount at all or do not expose annotation configuration. Flux post-renderers allow you to add these annotations to any ServiceAccount generated by a Helm chart, enabling cloud IAM integration without chart modifications.

This guide covers how to use post-renderers to add ServiceAccount annotations for various cloud providers and use cases.

## Prerequisites

Before you begin, ensure you have:

- A Kubernetes cluster with Flux CD installed (EKS, GKE, or AKS)
- IAM roles or service accounts configured in your cloud provider
- A HelmRelease that creates a ServiceAccount
- kubectl access to the cluster

## Adding AWS IRSA Annotations

To link a Kubernetes ServiceAccount to an AWS IAM role using IRSA:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ServiceAccount
              name: my-app
            patch: |
              apiVersion: v1
              kind: ServiceAccount
              metadata:
                name: my-app
                annotations:
                  eks.amazonaws.com/role-arn: "arn:aws:iam::123456789012:role/my-app-production-role"
```

With this annotation, pods using this ServiceAccount will automatically receive AWS credentials for the specified IAM role through the IRSA mechanism.

## Adding GCP Workload Identity Annotations

For Google Kubernetes Engine with Workload Identity:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ServiceAccount
              name: my-app
            patch: |
              apiVersion: v1
              kind: ServiceAccount
              metadata:
                name: my-app
                annotations:
                  iam.gke.io/gcp-service-account: "my-app@my-project.iam.gserviceaccount.com"
```

This connects the Kubernetes ServiceAccount to a GCP service account through Workload Identity.

## Adding Azure Workload Identity Annotations

For Azure Kubernetes Service with Workload Identity, you need two patches: an annotation on the ServiceAccount with the client ID, and a label on the Deployment pod template to enable the mutating admission webhook:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ServiceAccount
              name: my-app
            patch: |
              apiVersion: v1
              kind: ServiceAccount
              metadata:
                name: my-app
                annotations:
                  azure.workload.identity/client-id: "00000000-0000-0000-0000-000000000000"
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  metadata:
                    labels:
                      azure.workload.identity/use: "true"
```

Azure Workload Identity requires the `azure.workload.identity/client-id` annotation on the ServiceAccount and the `azure.workload.identity/use: "true"` label on the Pod template (not the ServiceAccount) to trigger the mutating admission webhook that injects OIDC token volumes.

## Multiple Annotations for Cross-Service Access

Applications often need access to multiple cloud services. You can add multiple annotations:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: data-pipeline
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: data-pipeline
      version: "1.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ServiceAccount
              name: data-pipeline
            patch: |
              apiVersion: v1
              kind: ServiceAccount
              metadata:
                name: data-pipeline
                annotations:
                  eks.amazonaws.com/role-arn: "arn:aws:iam::123456789012:role/data-pipeline-role"
                  eks.amazonaws.com/sts-regional-endpoints: "true"
```

The `sts-regional-endpoints` annotation ensures STS calls use regional endpoints for lower latency.

## Adding Vault Agent Injection Annotations

HashiCorp Vault Agent Injector uses annotations on **Pod templates** (not ServiceAccounts) to trigger sidecar injection. Use a post-renderer patch targeting the Deployment:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  metadata:
                    annotations:
                      vault.hashicorp.com/agent-inject: "true"
                      vault.hashicorp.com/role: "my-app-role"
```

This configures HashiCorp Vault Agent injection by annotating the pod template, which is where the Vault mutating webhook looks for its configuration.

## Verifying ServiceAccount Annotations

After deploying, verify the annotations are applied:

```bash
kubectl get serviceaccount my-app -n production -o yaml
```

Check that pods are using the correct ServiceAccount:

```bash
kubectl get pods -n production -l app.kubernetes.io/name=my-app -o jsonpath='{.items[0].spec.serviceAccountName}'
```

## Conclusion

Flux post-renderers provide a clean way to add annotations to ServiceAccounts generated by Helm charts. This is essential for integrating with cloud provider IAM systems like AWS IRSA, GCP Workload Identity, and Azure Workload Identity. By adding annotations through post-renderers instead of chart values, you maintain separation between chart management and infrastructure configuration, making it easy to adjust IAM bindings without modifying or forking Helm charts. Always verify the annotations after deployment and test that the resulting pods have the expected cloud permissions.
