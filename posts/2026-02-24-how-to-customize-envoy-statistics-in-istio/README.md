# How to Customize Envoy Statistics in Istio

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Envoy, Statistics, Monitoring, Prometheus, Service Mesh

Description: A practical guide to customizing Envoy proxy statistics in Istio for better observability and targeted metric collection.

---

Istio ships with a decent set of default metrics out of the box, but the reality is that every team needs something different from their observability stack. Maybe you need connection pool metrics for debugging connection exhaustion. Maybe you need to strip out stats you are never going to look at. Or maybe you want to add custom dimensions to existing metrics so your dashboards can slice data by business-relevant attributes.

This guide covers the practical steps to customize Envoy statistics in Istio so your metrics actually match what your team needs.

## Default Statistics in Istio

Before customizing anything, it helps to know what you get for free. Istio configures Envoy to emit a standard set of metrics through its Prometheus integration. The key ones include:

- `istio_requests_total` - Total request count with response code, source, and destination labels
- `istio_request_duration_milliseconds` - Request duration histogram
- `istio_request_bytes` - Request body size distribution
- `istio_response_bytes` - Response body size distribution
- `istio_tcp_sent_bytes_total` - Total bytes sent for TCP connections
- `istio_tcp_received_bytes_total` - Total bytes received for TCP connections
- `istio_tcp_connections_opened_total` - TCP connections opened
- `istio_tcp_connections_closed_total` - TCP connections closed

These are the "standard" Istio metrics generated by the built-in Envoy stats filter. They are separate from the raw Envoy stats (like `cluster.*` and `listener.*`), which are much more granular.

## Adding Custom Metric Dimensions

The most common customization is adding extra labels to the standard Istio metrics. You do this with the Telemetry API.

Say you want to add the `request.url_path` as a dimension on your request count metric:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: custom-dimensions
  namespace: istio-system
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT
          tagOverrides:
            request_url_path:
              operation: UPSERT
              value: "request.url_path"
```

Be very careful with this one. URL paths can have extremely high cardinality, which will blow up your Prometheus storage. A better approach is to use a more controlled attribute:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: safe-dimensions
  namespace: production
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT
          tagOverrides:
            request_protocol:
              operation: UPSERT
              value: "request.protocol"
            upstream_cluster:
              operation: UPSERT
              value: "upstream_peer.name"
```

## Removing Default Dimensions

Sometimes the default labels are too many. Every unique combination of label values creates a new time series. If you have lots of services and response codes, the cardinality grows fast.

You can remove labels you do not need:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: reduce-cardinality
  namespace: istio-system
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT
          tagOverrides:
            connection_security_policy:
              operation: REMOVE
            request_protocol:
              operation: REMOVE
```

Removing `connection_security_policy` is common because in most meshes, everything is mTLS anyway, so that label adds cardinality without information.

## Disabling Specific Metrics Entirely

If you do not care about TCP metrics (for example, in an HTTP-only mesh), you can disable them:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: disable-tcp-metrics
  namespace: istio-system
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: TCP_OPENED_CONNECTIONS
          disabled: true
        - match:
            metric: TCP_CLOSED_CONNECTIONS
          disabled: true
        - match:
            metric: TCP_SENT_BYTES
          disabled: true
        - match:
            metric: TCP_RECEIVED_BYTES
          disabled: true
```

This reduces the total number of time series and lowers the CPU and memory footprint on each proxy.

## Including Raw Envoy Stats

Beyond the standard Istio metrics, Envoy has hundreds of internal stats about connection pools, circuit breakers, DNS resolution, load balancing, and more. By default, most of these are not exposed through the Prometheus endpoint.

To include them, configure `proxyStatsMatcher` in your mesh config:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      proxyStatsMatcher:
        inclusionPrefixes:
          - "cluster.outbound"
          - "upstream_cx"
          - "upstream_rq"
          - "http.inbound"
        inclusionRegexps:
          - ".*circuit_breakers.*"
```

After applying this and restarting your workloads, you will see additional metrics on the `/stats/prometheus` endpoint of each proxy.

## Creating Custom Stats with EnvoyFilter

For advanced use cases, you can create entirely new metrics using Envoy's stats configuration through an EnvoyFilter:

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: custom-stat-tags
  namespace: istio-system
spec:
  configPatches:
    - applyTo: BOOTSTRAP
      patch:
        operation: MERGE
        value:
          stats_config:
            stats_tags:
              - tag_name: "app_version"
                regex: "(.*)\\.app_version\\.(.*)"
              - tag_name: "custom_route"
                fixed_value: "default"
```

This adds custom tag extraction rules at the Envoy bootstrap level. These tags get applied to all stats that match the regex pattern.

## Controlling Stats at the Namespace Level

One of the nice things about the Telemetry API is that it supports namespace-scoped configuration. You can have different stats settings for different namespaces:

```yaml
# Production gets minimal stats for performance
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: prod-stats
  namespace: production
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: ALL_METRICS
          tagOverrides:
            source_principal:
              operation: REMOVE
---
# Staging gets extra detail for debugging
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: staging-stats
  namespace: staging
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT
          tagOverrides:
            grpc_response_status:
              operation: UPSERT
              value: "response.grpc_status"
```

## Targeting Specific Workloads

You can even target individual workloads using a selector:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: debug-payment-service
  namespace: production
spec:
  selector:
    matchLabels:
      app: payment-service
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_DURATION
          tagOverrides:
            upstream_cluster_name:
              operation: UPSERT
              value: "upstream_peer.name"
```

## Verifying Your Customizations

After applying any changes, always verify that the metrics look correct.

Check the proxy configuration:

```bash
istioctl proxy-config bootstrap deploy/my-app -n production -o json | jq '.bootstrap.statsConfig'
```

Check actual metrics from the proxy:

```bash
kubectl exec deploy/my-app -n production -c istio-proxy -- curl -s localhost:15000/stats/prometheus | head -50
```

And make sure your Prometheus is actually scraping the new metrics:

```bash
kubectl port-forward svc/prometheus -n istio-system 9090:9090
```

Then query for your new metrics in the Prometheus UI.

## Practical Tips

Here are some things that have saved me time when working with Envoy stats customization:

1. **Always test in a non-production namespace first** - Stats changes can have performance implications
2. **Watch your proxy memory** - Each additional stat consumes memory in the Envoy process
3. **Use `istioctl analyze`** to catch configuration issues before they hit your pods
4. **Remember that some changes need pod restarts** - Bootstrap-level config changes require proxy restart
5. **Keep cardinality in mind** - A single high-cardinality label can create millions of time series

Getting your Envoy statistics right is about finding the balance between visibility and overhead. Start with the defaults, figure out what questions you cannot answer, and add targeted customizations to fill those gaps.
