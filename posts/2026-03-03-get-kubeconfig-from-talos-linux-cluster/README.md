# How to Get Your kubeconfig from a Talos Linux Cluster

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, kubeconfig, kubectl, Kubernetes, Cluster Access

Description: Learn the different ways to retrieve and manage your kubeconfig file from a Talos Linux Kubernetes cluster.

---

After bootstrapping your Talos Linux cluster, the next thing you need is a kubeconfig file. This file contains the credentials and endpoint information that `kubectl` uses to communicate with the Kubernetes API server. Without it, kubectl has no idea how to reach your cluster.

In a traditional Kubernetes setup, you might find the kubeconfig at `/etc/kubernetes/admin.conf` on the control plane node. But Talos does not give you shell access, so you cannot just copy the file. Instead, you use `talosctl` to retrieve it.

## The Basic Command

The simplest way to get your kubeconfig:

```bash
# Make sure your talosctl config is set
export TALOSCONFIG="$(pwd)/talosconfig"

# Retrieve the kubeconfig
talosctl kubeconfig --nodes 192.168.1.101
```

This command does two things:

1. Connects to the specified node and requests a kubeconfig
2. Merges it into your default kubeconfig file at `~/.kube/config`

After running this, you can immediately use `kubectl`:

```bash
# Verify access
kubectl get nodes
kubectl get pods -n kube-system
```

## Saving to a Specific File

If you do not want the kubeconfig merged into your default file, save it separately:

```bash
# Save to a specific file
talosctl kubeconfig ./my-cluster-kubeconfig --nodes 192.168.1.101

# Use it with kubectl
kubectl --kubeconfig=./my-cluster-kubeconfig get nodes

# Or set the environment variable
export KUBECONFIG=./my-cluster-kubeconfig
kubectl get nodes
```

This is handy when you manage multiple clusters and want to keep their configurations separate.

## Saving to a Directory

You can also pass a directory path, and Talos will write the kubeconfig as a `config` file inside that directory:

```bash
# Save to a specific directory
mkdir -p ~/.kube/clusters/my-cluster
talosctl kubeconfig ~/.kube/clusters/my-cluster --nodes 192.168.1.101

# The file is created at:
# ~/.kube/clusters/my-cluster/config
```

## Understanding the Generated kubeconfig

Let us look at what Talos generates. The kubeconfig contains several pieces:

```yaml
apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority-data: <base64-encoded-CA-cert>
    server: https://192.168.1.100:6443
  name: my-cluster
contexts:
- context:
    cluster: my-cluster
    user: admin@my-cluster
  name: admin@my-cluster
current-context: admin@my-cluster
users:
- name: admin@my-cluster
  user:
    client-certificate-data: <base64-encoded-client-cert>
    client-key-data: <base64-encoded-client-key>
```

The key elements are:

- **server** - The Kubernetes API endpoint (the one you specified when generating the cluster config)
- **certificate-authority-data** - The CA certificate that the API server uses
- **client-certificate-data** and **client-key-data** - Your admin client credentials

The admin kubeconfig generated by Talos has full cluster-admin privileges. Treat it like a root password.

## Merging Multiple Cluster kubeconfigs

If you work with multiple clusters, you can merge kubeconfigs:

```bash
# Retrieve kubeconfig for cluster-a (merges into ~/.kube/config)
talosctl kubeconfig --nodes 10.0.1.100 --talosconfig ./cluster-a/talosconfig

# Retrieve kubeconfig for cluster-b (also merges)
talosctl kubeconfig --nodes 10.0.2.100 --talosconfig ./cluster-b/talosconfig

# List available contexts
kubectl config get-contexts

# Switch between clusters
kubectl config use-context admin@cluster-a
kubectl config use-context admin@cluster-b
```

## Forcing a Refresh

The kubeconfig includes client certificates that have an expiration date. By default, the admin certificate generated by Talos is valid for one year. When it expires, you need to fetch a new one:

```bash
# Force overwrite the existing kubeconfig entry
talosctl kubeconfig --nodes 192.168.1.101 --force
```

The `--force` flag overwrites the existing entry in your kubeconfig file with fresh credentials.

## Specifying a Different Endpoint in kubeconfig

Sometimes the endpoint that nodes use internally differs from what your workstation should use. For example, the cluster might use a private VIP, but you access it through a public IP or bastion host.

You can modify the kubeconfig after retrieval:

```bash
# Retrieve the kubeconfig
talosctl kubeconfig --nodes 192.168.1.101

# Update the server URL in the kubeconfig
kubectl config set-cluster my-cluster \
  --server=https://public-api.example.com:6443
```

## Using kubeconfig with CI/CD

In CI/CD pipelines, you typically do not have talosctl configured. Instead, you can generate the kubeconfig once and store it as a secret:

```bash
# Generate kubeconfig and output to stdout
talosctl kubeconfig /dev/stdout --nodes 192.168.1.101 --talosconfig ./talosconfig

# In CI, you might do:
# Store the kubeconfig content as a CI secret, then:
echo "${KUBECONFIG_CONTENT}" > /tmp/kubeconfig
export KUBECONFIG=/tmp/kubeconfig
kubectl get nodes
```

Be careful with this approach in production. The admin kubeconfig has full access to the cluster. Consider creating service accounts with limited RBAC permissions for CI/CD instead.

## Creating Limited kubeconfig Files

For day-to-day work, you may not want to use the admin kubeconfig. Create a more limited one using Kubernetes RBAC:

```bash
# Create a namespace-scoped service account
kubectl create namespace dev-team
kubectl create serviceaccount dev-user -n dev-team

# Create a role and binding
kubectl create role dev-role -n dev-team \
  --verb=get,list,watch,create,update,delete \
  --resource=pods,services,deployments,configmaps

kubectl create rolebinding dev-binding -n dev-team \
  --role=dev-role \
  --serviceaccount=dev-team:dev-user

# Create a token for the service account
kubectl create token dev-user -n dev-team --duration=8760h
```

Then build a kubeconfig manually using that token. This gives developers access only to what they need, following the principle of least privilege.

## Troubleshooting kubeconfig Issues

### "Unable to connect to the server"

This means kubectl cannot reach the Kubernetes API. Check:

```bash
# Verify the server URL in your kubeconfig
kubectl config view --minify | grep server

# Test connectivity to that address
curl -k https://192.168.1.100:6443/healthz
```

If the URL is unreachable, your VIP, load balancer, or DNS might be misconfigured.

### "Unauthorized" or Certificate Errors

Your client certificate may have expired or the CA does not match:

```bash
# Check the certificate expiry
kubectl config view --raw | grep client-certificate-data | awk '{print $2}' | base64 -d | openssl x509 -noout -dates

# If expired, retrieve a fresh kubeconfig
talosctl kubeconfig --nodes 192.168.1.101 --force
```

### Wrong Cluster Context

If kubectl is talking to the wrong cluster:

```bash
# See which context is active
kubectl config current-context

# List all contexts
kubectl config get-contexts

# Switch to the correct context
kubectl config use-context admin@my-cluster
```

The kubeconfig is your gateway to the cluster. Keep the admin kubeconfig secure, use limited service accounts for automation, and remember that `talosctl kubeconfig` is always available to regenerate fresh credentials when needed.
