# How to Optimize Istio Resource Costs

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Cost Optimization, Service Mesh, Kubernetes, Resources

Description: Practical strategies to reduce the CPU, memory, and infrastructure costs of running Istio service mesh in production Kubernetes clusters.

---

Istio adds real costs to your Kubernetes cluster. Every pod gets a sidecar proxy that consumes CPU and memory. The control plane needs its own resources. Metrics, logs, and traces generate storage costs. If you are not intentional about optimization, you can easily spend 30-40% more on compute just because of the mesh.

The good news is that most of these costs can be significantly reduced without sacrificing the features you actually use. Here is how.

## Understanding Where Costs Come From

Before optimizing, you need to know where the money goes. Istio's resource costs break down into four areas:

1. **Sidecar proxies**: The biggest cost. Every pod in the mesh gets an Envoy sidecar that consumes CPU and memory.
2. **Control plane (istiod)**: Relatively small compared to sidecars, but still meaningful.
3. **Ingress/egress gateways**: Additional pods that handle external traffic.
4. **Observability storage**: Metrics, logs, and traces generated by the mesh.

For a cluster with 100 pods in the mesh, the sidecar cost dominates. If each sidecar requests 100m CPU and 128Mi memory, that is 10 CPU cores and 12.8 GB of memory just for sidecars. On cloud providers, that could easily be $200-400/month depending on your instance types.

## Step 1: Right-Size Sidecar Requests

The default sidecar resource requests are conservative. Istio sets them high to avoid OOM kills, but most workloads do not need that much.

Check actual sidecar usage first:

```bash
kubectl top pods -n your-namespace --containers | grep istio-proxy
```

Or use a Prometheus query to get the 95th percentile of sidecar CPU usage:

```promql
quantile(0.95, rate(container_cpu_usage_seconds_total{container="istio-proxy"}[5m]))
```

Once you know actual usage, set appropriate requests:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  values:
    global:
      proxy:
        resources:
          requests:
            cpu: 25m
            memory: 64Mi
          limits:
            cpu: 200m
            memory: 128Mi
```

For low-traffic services, you can go even lower. Some teams run sidecars with 10m CPU and 32Mi memory requests without issues.

## Step 2: Reduce Envoy Concurrency

By default, Envoy uses 2 worker threads. For most services, 1 thread is plenty:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      concurrency: 1
```

Each worker thread consumes memory for its connection pools and listeners. Dropping from 2 to 1 can save 20-30% of sidecar memory usage.

For high-traffic services that genuinely need more throughput, override this per-pod:

```yaml
apiVersion: v1
kind: Pod
metadata:
  annotations:
    proxy.istio.io/config: |
      concurrency: 2
```

## Step 3: Scope Sidecars with Sidecar Resources

This is one of the highest-impact optimizations. By default, every sidecar receives configuration for every service in the mesh. If you have 200 services, each sidecar holds routing rules, endpoints, and cluster configuration for all 200.

Use the `Sidecar` resource to limit visibility:

```yaml
apiVersion: networking.istio.io/v1beta1
kind: Sidecar
metadata:
  name: default
  namespace: your-namespace
spec:
  egress:
  - hosts:
    - "./*"
    - "istio-system/*"
```

This cuts sidecar memory usage dramatically. In a cluster with 200 services across 20 namespaces, a namespace with 10 services goes from knowing about 200 services to knowing about 10 plus the istio-system services.

The memory savings can be 50-70% per sidecar depending on your cluster size.

## Step 4: Disable Unused Features

Istio comes with many features enabled by default. Turn off what you do not use:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    accessLogFile: ""
    enableTracing: false
    enablePrometheusMerge: true
    defaultConfig:
      proxyMetadata:
        ISTIO_META_DNS_CAPTURE: "false"
        ISTIO_META_DNS_AUTO_ALLOCATE: "false"
```

Disabling access logging alone can save significant CPU on high-traffic services. Each access log entry requires serialization and write operations, which add up fast.

If you do need tracing, keep the sampling rate low:

```yaml
meshConfig:
  enableTracing: true
  defaultConfig:
    tracing:
      sampling: 1.0
```

One percent sampling gives you plenty of traces for debugging without the cost of 100% sampling.

## Step 5: Optimize the Control Plane

The control plane does not need to be large for most clusters:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  components:
    pilot:
      k8s:
        resources:
          requests:
            cpu: 200m
            memory: 512Mi
          limits:
            cpu: "1"
            memory: 2Gi
        hpaSpec:
          minReplicas: 1
          maxReplicas: 3
```

For clusters under 100 services, a single istiod replica with 200m CPU and 512Mi memory is usually sufficient. The HPA will scale up if needed.

## Step 6: Use Discovery Selectors

If you only use Istio in certain namespaces, tell istiod to stop watching the rest:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    discoverySelectors:
    - matchLabels:
        istio-injection: enabled
```

This reduces the number of Kubernetes API watches istiod maintains, lowering its CPU and memory usage. It also reduces the configuration that gets pushed to sidecars.

## Step 7: Consider Ambient Mode

Istio ambient mode eliminates sidecar proxies entirely for L4 traffic management. Instead of a per-pod sidecar, it uses a per-node ztunnel daemon and optional per-service waypoint proxies.

The cost savings can be dramatic:

- No per-pod sidecar memory and CPU overhead
- Reduced total proxy instances (one per node instead of one per pod)
- L7 processing only for services that need it

```bash
istioctl install --set profile=ambient -y
```

Label namespaces for ambient mode instead of sidecar injection:

```bash
kubectl label namespace your-namespace istio.io/dataplane-mode=ambient
```

For workloads that only need mTLS and L4 traffic management, ambient mode can reduce proxy costs by 80-90%.

## Step 8: Clean Up Unused Resources

Over time, clusters accumulate stale Istio resources. VirtualServices that point to deleted services, DestinationRules for decomissioned backends, ServiceEntries for APIs you no longer use. Each of these adds configuration that istiod processes and pushes to sidecars.

Periodically audit your Istio resources:

```bash
# Find VirtualServices referencing services that do not exist
for vs in $(kubectl get virtualservices -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\n"}{end}'); do
  ns=$(echo $vs | cut -d/ -f1)
  name=$(echo $vs | cut -d/ -f2)
  echo "Checking VirtualService $ns/$name"
done
```

## Measuring Your Savings

After making these changes, track the impact:

```promql
# Total sidecar CPU usage across the cluster
sum(rate(container_cpu_usage_seconds_total{container="istio-proxy"}[5m]))

# Total sidecar memory usage
sum(container_memory_working_set_bytes{container="istio-proxy"})

# Control plane CPU usage
sum(rate(container_cpu_usage_seconds_total{container="discovery"}[5m]))
```

Set up Grafana dashboards with these queries and monitor the trend over time. Most teams see a 40-60% reduction in Istio resource costs after implementing these optimizations.

## Summary

Istio cost optimization is mostly about two things: right-sizing sidecar resources based on actual usage, and reducing the amount of configuration each sidecar carries. Start with sidecar scoping and resource tuning, which give the biggest bang for your effort. Then consider ambient mode for workloads that do not need L7 features. Every optimization compounds, and the savings grow as your cluster grows.
