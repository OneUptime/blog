# How to Rotate Flux CD Git Credentials

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, GitOps, Kubernetes, Security, Git Credentials, Token Rotation, HTTPS

Description: Learn how to rotate HTTPS Git credentials (personal access tokens and app tokens) used by Flux CD for Git repository access.

---

When Flux CD accesses Git repositories over HTTPS, it uses credentials stored in Kubernetes Secrets. These credentials (personal access tokens, app installation tokens, or basic auth) should be rotated regularly to minimize the risk of credential compromise. This guide covers rotating HTTPS-based Git credentials for Flux CD.

## Types of Git Credentials in Flux

Flux supports several HTTPS credential types:

- **Personal Access Tokens (PAT)**: User-scoped tokens from GitHub, GitLab, or Bitbucket.
- **GitHub App Installation Tokens**: Short-lived tokens generated by a GitHub App.
- **Basic Auth**: Username and password (or token) combination.

## Step 1: Identify Current Git Credentials

Find which secrets are used for Git authentication:

```bash
# List all GitRepository sources and their secret references
flux get sources git -A -o json | jq -r '.[] | "\(.namespace)/\(.name): \(.spec.secretRef.name // "none")"'

# Check the current secret structure
kubectl get secret flux-system -n flux-system -o jsonpath='{.data}' | jq -r 'keys[]'

# Verify the credential type (username/password for HTTPS)
kubectl get secret flux-system -n flux-system -o jsonpath='{.data.username}' | base64 -d 2>/dev/null && echo " (HTTPS auth)"
```

## Step 2: Generate a New Token

Generate a new token from your Git provider.

For GitHub Personal Access Token:

```bash
# Create a new fine-grained PAT using GitHub CLI
# Required permissions: Contents (read/write), Metadata (read)
gh auth token  # Shows current token - create a new one via GitHub UI

# Or use the GitHub API to create a fine-grained token
# Navigate to: GitHub > Settings > Developer settings > Fine-grained tokens > Generate new token
```

For GitLab Project Access Token:

```bash
# Create a project access token via GitLab API
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_ADMIN_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "flux-git-token-'$(date +%Y%m%d)'",
    "scopes": ["read_repository", "write_repository"],
    "expires_at": "'$(date -d "+90 days" +%Y-%m-%d)'"
  }' \
  "https://gitlab.com/api/v4/projects/$PROJECT_ID/access_tokens"
```

## Step 3: Update the Kubernetes Secret

Update the Flux secret with the new credentials:

```bash
# For GitHub HTTPS authentication
kubectl create secret generic flux-system \
  --from-literal=username=git \
  --from-literal=password=ghp_NEW_TOKEN_HERE \
  --namespace=flux-system \
  --dry-run=client -o yaml | kubectl apply -f -

# For GitLab HTTPS authentication
kubectl create secret generic flux-system \
  --from-literal=username=oauth2 \
  --from-literal=password=glpat-NEW_TOKEN_HERE \
  --namespace=flux-system \
  --dry-run=client -o yaml | kubectl apply -f -

# For Bitbucket HTTPS authentication
kubectl create secret generic flux-system \
  --from-literal=username=x-token-auth \
  --from-literal=password=NEW_APP_PASSWORD_HERE \
  --namespace=flux-system \
  --dry-run=client -o yaml | kubectl apply -f -
```

## Step 4: Update Credentials for Multiple GitRepositories

If you have multiple GitRepository sources using different credentials:

```bash
#!/bin/bash
# rotate-git-credentials.sh
# Rotate Git credentials for all Flux GitRepository sources

# Define the new token
NEW_TOKEN="ghp_NEW_TOKEN_HERE"

# Find all secrets referenced by GitRepository resources
SECRETS=$(kubectl get gitrepositories -A -o json | \
  jq -r '.items[] | select(.spec.secretRef != null) | "\(.metadata.namespace)/\(.spec.secretRef.name)"' | sort -u)

for SECRET_REF in $SECRETS; do
  NAMESPACE=$(echo "$SECRET_REF" | cut -d/ -f1)
  SECRET_NAME=$(echo "$SECRET_REF" | cut -d/ -f2)

  echo "Updating secret: $NAMESPACE/$SECRET_NAME"
  kubectl create secret generic "$SECRET_NAME" \
    --from-literal=username=git \
    --from-literal=password="$NEW_TOKEN" \
    --namespace="$NAMESPACE" \
    --dry-run=client -o yaml | kubectl apply -f -
done
```

## Step 5: Verify the New Credentials Work

Trigger a reconciliation and verify the credentials are accepted:

```bash
# Force reconciliation of all Git sources
flux reconcile source git flux-system

# Check the status
flux get sources git -A

# Look for authentication errors in source-controller logs
kubectl logs -n flux-system deployment/source-controller --tail=30 | grep -i "auth\|401\|403\|token"

# Verify the source is ready
kubectl get gitrepository flux-system -n flux-system -o jsonpath='{.status.conditions[?(@.type=="Ready")]}'
```

## Step 6: Revoke the Old Token

After confirming the new token works, revoke the old one:

```bash
# For GitHub: Revoke via Settings > Developer settings > Personal access tokens
# Or use the API:
curl -X DELETE \
  -H "Authorization: Bearer $OLD_TOKEN" \
  "https://api.github.com/applications/$CLIENT_ID/token" \
  -d '{"access_token":"'"$OLD_TOKEN"'"}'

# For GitLab: Revoke via API
curl --request DELETE \
  --header "PRIVATE-TOKEN: $GITLAB_ADMIN_TOKEN" \
  "https://gitlab.com/api/v4/projects/$PROJECT_ID/access_tokens/$TOKEN_ID"
```

## Using GitHub App Tokens for Automatic Rotation

GitHub App installation tokens are short-lived (1 hour) and automatically rotated. This is the recommended approach for production:

```yaml
# gitrepository-github-app.yaml
# GitRepository using GitHub App credentials
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 5m
  url: https://github.com/myorg/fleet-repo.git
  secretRef:
    name: github-app-credentials
  ref:
    branch: main
```

Set up the GitHub App secret:

```bash
# Create a secret with GitHub App credentials
# The source-controller will use these to generate short-lived installation tokens
kubectl create secret generic github-app-credentials \
  --from-literal=githubAppID=12345 \
  --from-file=githubAppPrivateKey=github-app-key.pem \
  --from-literal=githubAppInstallationID=67890 \
  --namespace=flux-system
```

## Setting Up Automated Rotation with CronJob

Automate credential rotation with a Kubernetes CronJob:

```yaml
# cronjob-rotate-credentials.yaml
# CronJob to rotate Git credentials every 60 days
apiVersion: batch/v1
kind: CronJob
metadata:
  name: rotate-flux-git-credentials
  namespace: flux-system
spec:
  schedule: "0 0 1 */2 *"  # First day of every other month
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: credential-rotator
          containers:
            - name: rotator
              image: bitnami/kubectl:latest
              command:
                - /bin/bash
                - -c
                - |
                  # This is a placeholder - in production, integrate with your
                  # secrets manager (Vault, AWS Secrets Manager, etc.)
                  echo "Credential rotation triggered"
                  # Fetch new token from secrets manager
                  # Update Kubernetes secret
                  # Trigger reconciliation
          restartPolicy: OnFailure
```

## Best Practices

1. **Use GitHub Apps over PATs**: GitHub App tokens are short-lived and automatically rotated.
2. **Set token expiration**: Always create tokens with an expiration date.
3. **Use minimal scopes**: Grant only the permissions needed (read access for most use cases).
4. **Automate rotation**: Set up automated rotation with a CronJob or external secret manager.
5. **Use External Secrets Operator**: Integrate with Vault or cloud secret managers for automatic credential sync.
6. **Rotate immediately on compromise**: If credentials are suspected of being compromised, rotate them immediately and check audit logs.

Regular Git credential rotation is essential for maintaining the security of your Flux CD GitOps pipeline. By automating the process and using short-lived tokens where possible, you minimize the risk of credential-based attacks.
