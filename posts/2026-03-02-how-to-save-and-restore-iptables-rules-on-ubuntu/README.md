# How to Save and Restore iptables Rules on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, iptables, Firewall, Networking, Sysadmin

Description: Persist iptables firewall rules across reboots on Ubuntu using iptables-save, iptables-restore, and the iptables-persistent package with automation examples.

---

iptables rules are loaded into the running kernel and are lost on reboot by default. This is a common source of confusion when rules work perfectly until the server restarts and suddenly nothing is blocked anymore. Several approaches exist for making iptables rules persistent, and choosing between them depends on whether you're managing rules manually or through automation.

## Understanding the Problem

iptables rules live in kernel memory. When you run `sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT`, the rule is immediately active but will disappear when the system reboots. The kernel doesn't automatically reload any iptables configuration on startup.

The solution is to save rules to a file and restore them on boot. Ubuntu provides a few mechanisms for this.

## Saving Rules with iptables-save

The `iptables-save` command dumps the current iptables ruleset in a format that `iptables-restore` can read:

```bash
# View current rules in save format
sudo iptables-save

# Save IPv4 rules to a file
sudo iptables-save > /etc/iptables/rules.v4

# Save IPv6 rules to a file
sudo ip6tables-save > /etc/iptables/rules.v6

# Create the directory if it doesn't exist
sudo mkdir -p /etc/iptables
```

The saved file looks like this:

```text
# Generated by iptables-save v1.8.7 on Mon Mar  2 14:30:00 2026
*filter
:INPUT ACCEPT [0:0]
:FORWARD DROP [0:0]
:OUTPUT ACCEPT [0:0]
-A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT
-A INPUT -p tcp --dport 22 -j ACCEPT
-A INPUT -p tcp --dport 80 -j ACCEPT
-A INPUT -p tcp --dport 443 -j ACCEPT
-A INPUT -j DROP
COMMIT
# Completed on Mon Mar  2 14:30:00 2026
```

## Restoring Rules with iptables-restore

```bash
# Restore IPv4 rules from a file
sudo iptables-restore < /etc/iptables/rules.v4

# Restore IPv6 rules
sudo ip6tables-restore < /etc/iptables/rules.v6

# Test restore without applying (check syntax)
sudo iptables-restore --test < /etc/iptables/rules.v4
```

## Method 1: Using iptables-persistent

The `iptables-persistent` package automates save and restore through systemd:

```bash
sudo apt update
sudo apt install -y iptables-persistent
```

During installation, you'll be prompted:
- "Save current IPv4 rules?" - Yes
- "Save current IPv6 rules?" - Yes

This saves current rules to `/etc/iptables/rules.v4` and `/etc/iptables/rules.v6` and creates a systemd service that loads them at boot.

### Managing Rules with iptables-persistent

After making changes to your iptables rules, save them:

```bash
# Add a rule
sudo iptables -A INPUT -p tcp --dport 8080 -j ACCEPT

# Save the updated rules
sudo netfilter-persistent save

# Or use iptables-save directly
sudo iptables-save > /etc/iptables/rules.v4

# Reload rules from file (without reboot)
sudo netfilter-persistent reload
```

```bash
# Check the service status
sudo systemctl status netfilter-persistent
```

### Checking the Service

```bash
# Verify it starts at boot
sudo systemctl is-enabled netfilter-persistent

# Enable if not already
sudo systemctl enable netfilter-persistent
```

## Method 2: Using a systemd Service

If you prefer not to use `iptables-persistent`, create a custom systemd service:

```bash
sudo nano /etc/systemd/system/iptables-restore.service
```

```ini
[Unit]
Description=Restore iptables rules
Before=network-pre.target
Wants=network-pre.target
DefaultDependencies=no

[Service]
Type=oneshot
ExecStart=/sbin/iptables-restore /etc/iptables/rules.v4
ExecStart=/sbin/ip6tables-restore /etc/iptables/rules.v6
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable iptables-restore
sudo systemctl start iptables-restore
```

## Method 3: Using if-pre-up.d Scripts

An older but still functional approach is to restore rules as part of the network interface configuration:

```bash
sudo nano /etc/network/if-pre-up.d/iptables
```

```bash
#!/bin/sh
# Restore iptables rules before the network interface comes up

RULES=/etc/iptables/rules.v4
RULES6=/etc/iptables/rules.v6

if [ -f "$RULES" ]; then
    iptables-restore < "$RULES"
fi

if [ -f "$RULES6" ]; then
    ip6tables-restore < "$RULES6"
fi
```

```bash
sudo chmod +x /etc/network/if-pre-up.d/iptables
```

## Saving Rules Atomically

Manually running `iptables-save` after every change is error-prone. A wrapper script ensures you don't forget:

```bash
sudo nano /usr/local/bin/ipt-save
```

```bash
#!/bin/bash
# Save iptables rules after making changes
# Usage: sudo ipt-save

RULES_V4="/etc/iptables/rules.v4"
RULES_V6="/etc/iptables/rules.v6"
BACKUP_DIR="/etc/iptables/backups"

# Create backup directory
mkdir -p "$BACKUP_DIR"

# Back up existing rules with timestamp
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
if [ -f "$RULES_V4" ]; then
    cp "$RULES_V4" "$BACKUP_DIR/rules.v4.$TIMESTAMP"
fi

# Save current rules
iptables-save > "$RULES_V4"
ip6tables-save > "$RULES_V6"

echo "Rules saved:"
echo "  IPv4: $RULES_V4"
echo "  IPv6: $RULES_V6"
echo "  Backup: $BACKUP_DIR/rules.v4.$TIMESTAMP"
echo ""
echo "Current rule count:"
iptables -L --line-numbers | grep "^[0-9]" | wc -l
```

```bash
sudo chmod +x /usr/local/bin/ipt-save
```

Now after making rule changes:

```bash
# Make your rule changes
sudo iptables -A INPUT -p tcp --dport 9090 -j ACCEPT

# Save them
sudo ipt-save
```

## Working with Named Rulesets

For environments with different rule sets (e.g., development vs. production), use named files:

```bash
# Save current rules as a named set
sudo iptables-save > /etc/iptables/rules.v4.production

# Switch to a different ruleset
sudo iptables-restore < /etc/iptables/rules.v4.minimal

# Save as the active ruleset
sudo iptables-save > /etc/iptables/rules.v4
```

## A Complete Ruleset Example

Create a baseline ruleset file to restore on a new server:

```bash
sudo nano /etc/iptables/rules.v4.baseline
```

```text
*filter
:INPUT DROP [0:0]
:FORWARD DROP [0:0]
:OUTPUT ACCEPT [0:0]

# Allow loopback
-A INPUT -i lo -j ACCEPT
-A OUTPUT -o lo -j ACCEPT

# Allow established and related connections
-A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow ICMP (ping)
-A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# Allow SSH
-A INPUT -p tcp --dport 22 -m state --state NEW -j ACCEPT

# Allow HTTP and HTTPS
-A INPUT -p tcp --dport 80 -m state --state NEW -j ACCEPT
-A INPUT -p tcp --dport 443 -m state --state NEW -j ACCEPT

# Log and drop everything else
-A INPUT -m limit --limit 5/min -j LOG --log-prefix "iptables INPUT DROP: "
-A INPUT -j DROP

COMMIT
```

Apply and save:

```bash
# Restore the baseline ruleset
sudo iptables-restore < /etc/iptables/rules.v4.baseline

# Test SSH still works before saving
ssh -o ConnectTimeout=5 localhost true && echo "SSH test passed"

# Save as active rules
sudo iptables-save > /etc/iptables/rules.v4
```

## Verifying Rules Are Loaded at Boot

After a reboot, verify rules were restored:

```bash
sudo reboot
# After reboot...

# Verify rules are loaded
sudo iptables -L INPUT -n --line-numbers

# Check the service ran
sudo journalctl -u netfilter-persistent

# Verify rules match saved file
sudo iptables-save | diff - /etc/iptables/rules.v4
```

If there's no output from `diff`, the live rules match the saved rules exactly.

## Rolling Back to Previous Rules

The backup system in the `ipt-save` script makes rollback straightforward:

```bash
# List backups
ls /etc/iptables/backups/

# Roll back to a specific backup
sudo iptables-restore < /etc/iptables/backups/rules.v4.20260302_140000

# Verify the rollback
sudo iptables -L INPUT --line-numbers

# If it looks correct, save it as the active ruleset
sudo ipt-save
```

Persistent iptables rules are a prerequisite for any server where you're managing the firewall directly rather than through UFW. The `iptables-persistent` package is the simplest approach on Ubuntu and handles the systemd integration automatically, making it the recommended starting point for most deployments.
