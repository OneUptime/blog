# How to Set Up etcd Encryption in Talos Linux

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, etcd, Encryption, Kubernetes Security, Secrets Management

Description: A complete guide to setting up etcd encryption at rest in Talos Linux to protect sensitive Kubernetes data like secrets and configmaps.

---

Every Kubernetes cluster stores sensitive data in etcd. This includes secrets, configmaps, service account tokens, and other objects that may contain credentials, API keys, or private certificates. By default, this data is stored in plain text within etcd. Anyone with access to the etcd data files or etcd API can read everything. That is why encrypting etcd data at rest is one of the most important security measures you can take for your cluster.

Talos Linux makes it possible to configure Kubernetes secret encryption through its machine configuration. In this guide, we will cover how to set up etcd encryption in Talos Linux and verify that your sensitive data is actually encrypted.

## Why etcd Encryption Matters

Consider what lives inside etcd in a typical Kubernetes cluster: database passwords, TLS certificates, OAuth tokens, cloud provider credentials, and more. If an attacker gains access to an etcd backup or the underlying storage, they can extract all of this data without needing kubectl access or any RBAC permissions.

Encryption at rest ensures that even if someone gets physical access to the etcd data files, they cannot read the contents without the encryption keys.

## Configuring Encryption in Talos Machine Config

Talos Linux supports Kubernetes secret encryption through the `cluster.secretboxEncryptionSecret` field in the machine configuration. This uses the secretbox encryption provider, which is a well-tested encryption mechanism supported by Kubernetes.

When you generate a Talos configuration, encryption is actually enabled by default:

```bash
# Generate configuration - encryption is included automatically
talosctl gen config my-cluster https://10.0.0.1:6443

# The generated controlplane.yaml includes encryption settings
```

If you look at the generated configuration, you will find:

```yaml
cluster:
  secretboxEncryptionSecret: <auto-generated-base64-secret>
```

This secret is used to encrypt Kubernetes secrets at rest in etcd.

## Setting a Custom Encryption Secret

If you want to use your own encryption key rather than the auto-generated one, you can create a base64-encoded secret and specify it in your configuration:

```bash
# Generate a random 32-byte encryption key
ENCRYPTION_KEY=$(head -c 32 /dev/urandom | base64)

echo "Your encryption key: $ENCRYPTION_KEY"
```

Then add it to your Talos machine configuration:

```yaml
cluster:
  secretboxEncryptionSecret: "<your-base64-encoded-key>"
```

Keep this key safe. If you lose it and need to rebuild the cluster from an etcd backup, you will not be able to decrypt the data without this key.

## Understanding the Encryption Configuration

Behind the scenes, Talos translates the `secretboxEncryptionSecret` into a Kubernetes EncryptionConfiguration that gets passed to the kube-apiserver. The resulting configuration looks something like this:

```yaml
# This is what Talos generates internally for the API server
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - secretbox:
          keys:
            - name: secretbox-1
              secret: <your-encryption-key>
      - identity: {}
```

The `identity` provider at the end is a fallback that allows reading unencrypted data. This is important during migration because existing secrets that were stored before encryption was enabled can still be read.

## Applying the Configuration

Apply your machine configuration to all control plane nodes:

```bash
# Apply to each control plane node
talosctl apply-config --nodes 10.0.0.2 --file controlplane.yaml
talosctl apply-config --nodes 10.0.0.3 --file controlplane.yaml
talosctl apply-config --nodes 10.0.0.4 --file controlplane.yaml
```

After applying, the kube-apiserver will restart with the encryption configuration. All new secrets written to etcd will be encrypted.

## Encrypting Existing Secrets

When you enable encryption on a cluster that already has secrets stored in etcd, the existing secrets remain unencrypted until they are rewritten. To encrypt all existing secrets, you need to read and rewrite every secret:

```bash
# Re-encrypt all secrets in all namespaces
kubectl get secrets --all-namespaces -o json | kubectl replace -f -
```

This command reads every secret and writes it back, which triggers encryption with the new key. You can verify the encryption status afterward.

## Verifying Encryption

To confirm that secrets are actually encrypted in etcd, you can use `talosctl` to read raw data from etcd and check that it is not in plain text:

```bash
# Read a specific key from etcd directly
talosctl etcd get /registry/secrets/default/my-secret --nodes 10.0.0.2
```

If encryption is working, the output will be binary data rather than readable YAML or JSON. You should see something starting with `k8s:enc:secretbox:v1:` followed by encrypted bytes.

You can also create a test secret and verify it:

```bash
# Create a test secret
kubectl create secret generic test-encryption \
  --from-literal=mykey=myvalue

# Check it through the API (should be readable)
kubectl get secret test-encryption -o jsonpath='{.data.mykey}' | base64 -d
# Output: myvalue

# Check it directly in etcd (should be encrypted)
talosctl etcd get /registry/secrets/default/test-encryption --nodes 10.0.0.2
# Output: binary/encrypted data

# Clean up
kubectl delete secret test-encryption
```

## Key Rotation

Periodically rotating your encryption keys is a security best practice. To rotate keys in Talos Linux, you follow a multi-step process:

First, generate a new encryption key:

```bash
NEW_KEY=$(head -c 32 /dev/urandom | base64)
```

Then update the Talos machine configuration with the new key:

```yaml
cluster:
  secretboxEncryptionSecret: "<new-base64-encoded-key>"
```

Apply the updated configuration to all control plane nodes:

```bash
talosctl patch machineconfig --nodes 10.0.0.2,10.0.0.3,10.0.0.4 \
  --patch '{"cluster": {"secretboxEncryptionSecret": "'"$NEW_KEY"'"}}'
```

After the API server restarts with the new key, re-encrypt all existing secrets:

```bash
kubectl get secrets --all-namespaces -o json | kubectl replace -f -
```

## Backup Considerations

When you have etcd encryption enabled, your backup strategy must account for the encryption keys. An etcd backup without the corresponding encryption key is useless because you cannot decrypt the data.

Here are some practical recommendations:

Store the encryption key separately from etcd backups. If both are in the same location and that location is compromised, encryption provides no benefit.

```bash
# Take an etcd snapshot
talosctl etcd snapshot db.snapshot --nodes 10.0.0.2

# Store the encryption key securely (use a proper secrets manager)
echo "$ENCRYPTION_KEY" > encryption-key.txt
# Move this to a secure vault - do not leave it on disk
```

Document which encryption key corresponds to which backup. If you rotate keys, old backups need the old key to be restored.

## Troubleshooting

If secrets are not being encrypted, check the kube-apiserver logs:

```bash
# Check API server logs for encryption-related messages
talosctl logs kube-apiserver --nodes 10.0.0.2 | grep -i encrypt
```

Common issues include malformed base64 encoding of the encryption key and mismatched keys across control plane nodes. Make sure all control plane nodes have the same encryption configuration.

## Conclusion

Setting up etcd encryption in Talos Linux is straightforward because Talos handles much of the complexity for you. The key steps are ensuring your machine configuration includes the encryption secret, applying it consistently across all control plane nodes, re-encrypting existing secrets, and maintaining a solid backup strategy that includes the encryption keys. With these measures in place, your sensitive Kubernetes data is protected even if someone gains access to the underlying storage.
