# How to Set Up mTLS Between VMs and Kubernetes in Istio

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, MTLS, Virtual Machines, Kubernetes, Security

Description: How to configure mutual TLS authentication between VM-based workloads and Kubernetes pods within an Istio service mesh for end-to-end encryption.

---

One of the biggest benefits of adding VMs to an Istio mesh is getting automatic mTLS between VM workloads and Kubernetes services. Instead of managing TLS certificates manually on each VM, Istio handles certificate issuance, rotation, and enforcement. The VM sidecar gets a certificate from Istiod, and every connection between the VM and Kubernetes pods is encrypted and mutually authenticated.

This post explains how mTLS works for VMs, how to configure it, and how to troubleshoot when it does not work.

## How VM mTLS Works

The mTLS flow for VMs follows the same pattern as Kubernetes:

1. The VM's pilot agent connects to Istiod using a bootstrap JWT token
2. Istiod validates the token and issues an X.509 certificate to the VM's sidecar
3. The certificate contains a SPIFFE identity: `spiffe://cluster.local/ns/<namespace>/sa/<service-account>`
4. When the VM communicates with a Kubernetes service (or vice versa), both sidecars perform a TLS handshake
5. Each side validates the other's certificate against the shared root CA
6. If validation passes, the connection is established with encryption

The key requirement is that the VM's certificate chains back to the same root CA as the Kubernetes cluster's certificates. This is handled automatically when you use the bootstrap files generated by `istioctl`.

## Initial Certificate Bootstrap

When the VM sidecar first starts, it needs to authenticate with Istiod to get its first certificate. This uses a JWT token that was generated during the `istioctl x workload entry configure` step.

The token is stored at `/var/run/secrets/tokens/istio-token` on the VM. It is a short-lived token that Istiod validates against the Kubernetes token review API.

The root certificate at `/etc/certs/root-cert.pem` lets the sidecar verify Istiod's TLS certificate during the initial connection. This prevents man-in-the-middle attacks during bootstrap.

## Certificate Rotation

After the initial certificate is issued, the pilot agent automatically handles rotation. Istio workload certificates have a short lifetime (24 hours by default). The pilot agent requests a new certificate before the current one expires:

```bash
# Check the current certificate on the VM
sudo /usr/local/bin/pilot-agent request GET /certs
```

Or check the certificate details:

```bash
# On the VM
openssl s_client -connect localhost:15000 2>/dev/null </dev/null | openssl x509 -text -noout
```

You can also check from the Kubernetes side:

```bash
istioctl proxy-config secret <vm-workload-name> -n vm-apps
```

## Configuring PeerAuthentication for VMs

PeerAuthentication policies apply equally to VMs and Kubernetes workloads. To enforce strict mTLS for your VM namespace:

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: strict-mtls
  namespace: vm-apps
spec:
  mtls:
    mode: STRICT
```

This ensures that all traffic to workloads in the `vm-apps` namespace (both VM and Kubernetes) must use mTLS.

### Permissive Mode During Migration

When first onboarding VMs, you might want permissive mode to avoid breaking existing non-mesh traffic:

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: permissive-mtls
  namespace: vm-apps
spec:
  mtls:
    mode: PERMISSIVE
```

In permissive mode, the sidecar accepts both plaintext and mTLS connections. This lets you gradually migrate VM traffic into the mesh.

### Port-Level mTLS Configuration

If your VM runs multiple services on different ports, and some should not use mTLS:

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: mixed-mtls
  namespace: vm-apps
spec:
  selector:
    matchLabels:
      app: legacy-db
  mtls:
    mode: STRICT
  portLevelMtls:
    5432:
      mode: STRICT
    9090:
      mode: PERMISSIVE
```

Port 5432 (database) requires mTLS, while port 9090 (metrics endpoint) accepts plaintext.

## DestinationRule for mTLS to VMs

When Kubernetes workloads call VM services, you might need a DestinationRule to ensure mTLS is used:

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: legacy-db
  namespace: vm-apps
spec:
  host: legacy-db.vm-apps.svc.cluster.local
  trafficPolicy:
    tls:
      mode: ISTIO_MUTUAL
```

`ISTIO_MUTUAL` tells the client sidecar to use Istio's built-in mTLS with automatic certificate management. This is different from `MUTUAL` which requires you to specify certificate paths manually.

In most cases, if you have mesh-wide mTLS enabled, you do not need explicit DestinationRules for mTLS. But they can be useful for:
- Overriding a permissive mesh default for specific services
- Configuring mTLS for ServiceEntry-based services

## VM Identity and Authorization

The VM's identity is determined by the service account specified in the WorkloadGroup or WorkloadEntry:

```yaml
apiVersion: networking.istio.io/v1beta1
kind: WorkloadEntry
metadata:
  name: legacy-db-vm1
  namespace: vm-apps
spec:
  address: 10.0.1.10
  labels:
    app: legacy-db
  serviceAccount: legacy-db-sa
```

The SPIFFE ID for this VM will be:

```text
spiffe://cluster.local/ns/vm-apps/sa/legacy-db-sa
```

This identity is used in authorization policies:

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-from-k8s-frontend
  namespace: vm-apps
spec:
  selector:
    matchLabels:
      app: legacy-db
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/web/sa/frontend"
    to:
    - operation:
        ports:
        - "5432"
```

This policy allows the frontend service (running in Kubernetes) to connect to the legacy-db VM on port 5432. Any other source is denied.

## Verifying mTLS Between VM and Kubernetes

### Check from the Kubernetes side

```bash
# Verify mTLS is being used
istioctl authn tls-check deployment/sleep.sample legacy-db.vm-apps.svc.cluster.local

# Check the connection in proxy config
istioctl proxy-config clusters deployment/sleep -n sample | grep legacy-db
```

### Check from the VM

On the VM, examine the Envoy admin interface:

```bash
# Check active certificates
curl localhost:15000/certs

# Check listeners
curl localhost:15000/listeners

# Check cluster endpoints
curl localhost:15000/clusters | grep -A5 "reviews.bookinfo"
```

### Traffic test

```bash
# From Kubernetes to VM
kubectl exec -n sample -c sleep deployment/sleep -- \
  curl -sS legacy-db.vm-apps:5432

# From VM to Kubernetes (on the VM)
curl -sS reviews.bookinfo:9080/reviews/1
```

Both requests should succeed if mTLS is configured correctly.

## Troubleshooting mTLS Issues

### Token Expired

If the bootstrap JWT token expires before the VM sidecar starts, Istiod will reject the certificate request:

```bash
sudo journalctl -u istio | grep "authentication"
```

Fix: Regenerate the bootstrap files and copy the new token to the VM.

### Root CA Mismatch

If the VM's root-cert.pem does not match the mesh's root CA:

```bash
# On the VM
openssl x509 -in /etc/certs/root-cert.pem -fingerprint -noout

# On Kubernetes
kubectl get secret cacerts -n istio-system -o jsonpath='{.data.root-cert\.pem}' | base64 -d | openssl x509 -fingerprint -noout
```

Fingerprints must match.

### Connection Reset

If connections between VM and Kubernetes are reset:

```bash
# Check if mTLS is the issue by temporarily switching to permissive
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: debug-permissive
  namespace: vm-apps
spec:
  mtls:
    mode: PERMISSIVE
EOF
```

If connections work in permissive mode but not strict mode, the issue is certificate trust. Check the full certificate chain on the VM sidecar.

### Check Sidecar Proxy Logs

Enable debug logging on the VM:

```bash
curl -X POST localhost:15000/logging?connection=debug
curl -X POST localhost:15000/logging?http=debug
```

Then check logs:

```bash
sudo journalctl -u istio -f
```

Look for TLS handshake errors or certificate validation failures.

## Summary

mTLS between VMs and Kubernetes in Istio works automatically once the VM sidecar is properly bootstrapped with the right root CA and token. The sidecar gets certificates from Istiod, handles rotation, and enforces mTLS on all traffic. Use PeerAuthentication for enforcement policies, DestinationRules for client-side mTLS configuration, and AuthorizationPolicies for fine-grained access control based on the VM's SPIFFE identity. The most common issues are expired bootstrap tokens and root CA mismatches, both of which are easy to diagnose and fix.
