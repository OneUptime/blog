# How to Debug Envoy Proxy Configuration with istioctl

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Envoy, Istioctl, Debugging, Kubernetes, Service Mesh

Description: Learn how to debug Envoy proxy configurations in Istio using istioctl commands to inspect listeners, routes, clusters, and endpoints.

---

When something goes wrong in an Istio mesh, the problem almost always comes down to what Envoy is doing. Envoy is the data plane proxy that sits beside every pod, and its configuration is generated by Istiod based on your Istio resources. If traffic isn't flowing correctly, you need to look at what Envoy actually received - not just what you think you configured.

istioctl has a bunch of commands specifically built for poking around in Envoy's configuration. Here's how to use them.

## Understanding the Envoy Configuration Model

Before jumping into commands, it helps to understand how Envoy organizes its config. There are four main pieces:

- **Listeners** - These define the ports and protocols Envoy listens on. Each listener has filter chains that process incoming connections.
- **Routes** - These map incoming requests to upstream clusters based on path, headers, and other match criteria.
- **Clusters** - These represent upstream services. A cluster has a name, a load balancing policy, and connection settings.
- **Endpoints** - These are the actual IP addresses and ports backing each cluster.

When you create a VirtualService in Istio, it gets translated into listener and route configuration. A DestinationRule becomes cluster settings. Service endpoints from Kubernetes become Envoy endpoints. If any of these translations go wrong, traffic breaks.

## Dumping the Full Envoy Config

The quickest way to get everything is:

```bash
istioctl proxy-config all productpage-v1-abc123.default -o json
```

This dumps the entire Envoy configuration for a given pod. It's a lot of output, so you'll usually want to focus on one piece at a time. But it's useful for saving to a file and searching through:

```bash
istioctl proxy-config all productpage-v1-abc123.default -o json > envoy-full-config.json
```

## Debugging Listeners

Listeners are where traffic enters the proxy. Check them with:

```bash
istioctl proxy-config listeners productpage-v1-abc123.default
```

You'll see a table like:

```text
ADDRESS       PORT  MATCH                        DESTINATION
0.0.0.0       15006 ALL                          Inline Route: /*
0.0.0.0       15006 Addr: *:15006                Non-HTTP/Non-TCP
0.0.0.0       15001 ALL                          PassthroughCluster
0.0.0.0       9080  Trans: raw_buffer; App: http/1.1,h2c Route: 9080
```

Port 15006 is the inbound listener (traffic coming into the pod). Port 15001 is the outbound listener (traffic leaving the pod to other services). The other ports are service-specific listeners.

If you expect a listener on a certain port and don't see it, check that the corresponding Kubernetes Service exists and has that port defined. Envoy only creates listeners for services that Istiod knows about.

For more detail:

```bash
istioctl proxy-config listeners productpage-v1-abc123.default -o json
```

The JSON output shows filter chains, TLS context, and all the gory details of how each listener processes connections.

## Debugging Routes

Routes control where requests go after they hit a listener:

```bash
istioctl proxy-config routes productpage-v1-abc123.default
```

Output:

```text
NAME           DOMAINS                          MATCH     VIRTUAL SERVICE
9080           reviews, reviews.default + 1     /*        reviews-route.default
9080           ratings, ratings.default + 1     /*
```

This shows that requests on port 9080 for the `reviews` service are handled by a VirtualService called `reviews-route`. The `ratings` service has no VirtualService, so it uses the default routing.

To see the full route details including match conditions, retries, and timeouts:

```bash
istioctl proxy-config routes productpage-v1-abc123.default --name 9080 -o json
```

This is where you can verify that your VirtualService weight-based routing, header matches, or URI rewrites are actually reflected in Envoy's config.

## Debugging Clusters

Clusters represent the upstream destinations:

```bash
istioctl proxy-config clusters productpage-v1-abc123.default
```

```text
SERVICE FQDN                         PORT   SUBSET   DIRECTION    TYPE     DESTINATION RULE
reviews.default.svc.cluster.local    9080   v1       outbound     EDS      reviews-dr.default
reviews.default.svc.cluster.local    9080   v2       outbound     EDS      reviews-dr.default
reviews.default.svc.cluster.local    9080   v3       outbound     EDS      reviews-dr.default
```

The `SUBSET` column shows subsets from your DestinationRule. `TYPE` of `EDS` means endpoints are dynamically discovered from Istiod. If you see `ORIGINAL_DST`, it means traffic goes to the original destination without load balancing.

To check connection pool settings, circuit breakers, and outlier detection:

```bash
istioctl proxy-config clusters productpage-v1-abc123.default \
  --fqdn reviews.default.svc.cluster.local -o json
```

Look for `circuitBreakers`, `outlierDetection`, and `connectTimeout` in the JSON output to verify your DestinationRule settings are applied.

## Debugging Endpoints

Endpoints are the actual pod IPs that back a cluster:

```bash
istioctl proxy-config endpoints productpage-v1-abc123.default
```

Filter by a specific service:

```bash
istioctl proxy-config endpoints productpage-v1-abc123.default \
  --cluster "outbound|9080|v1|reviews.default.svc.cluster.local"
```

```text
ENDPOINT             STATUS      OUTLIER CHECK   CLUSTER
10.244.0.15:9080     HEALTHY     OK              outbound|9080|v1|reviews.default.svc.cluster.local
10.244.0.16:9080     HEALTHY     OK              outbound|9080|v1|reviews.default.svc.cluster.local
```

If an endpoint shows `UNHEALTHY` or the outlier check shows something other than `OK`, that's a strong signal. Unhealthy endpoints get removed from the load balancing pool, which can cause failures if all endpoints are unhealthy.

## Comparing Proxy Config with Expected Config

Sometimes the issue is that Istiod and Envoy disagree. Use `proxy-status` to detect this:

```bash
istioctl proxy-status
```

If a proxy shows `STALE`, you can compare what Istiod thinks the config should be versus what the proxy actually has:

```bash
istioctl proxy-status productpage-v1-abc123.default
```

This shows a diff of the configurations. Any discrepancy could indicate network issues between the sidecar and Istiod, or resource constraints causing slow config propagation.

## Common Debugging Patterns

**Traffic returns 503 errors.** Start with endpoints. If the target cluster has no healthy endpoints, you'll get 503s. Check that the service selector matches the pod labels and that health checks are passing.

```bash
istioctl proxy-config endpoints sleep-xyz789.default \
  --cluster "outbound|9080||reviews.default.svc.cluster.local" -o json
```

**Traffic goes to the wrong version.** Check routes to make sure your VirtualService weight or match rules are reflected:

```bash
istioctl proxy-config routes productpage-v1-abc123.default --name 9080 -o json | \
  grep -A 20 "weightedClusters"
```

**Connection resets or timeouts.** Look at the cluster configuration for circuit breaker settings that might be too aggressive:

```bash
istioctl proxy-config clusters productpage-v1-abc123.default \
  --fqdn reviews.default.svc.cluster.local -o json | \
  grep -A 10 "circuitBreakers"
```

**mTLS handshake failures.** Check if the listener expects TLS but the client isn't sending it (or vice versa):

```bash
istioctl proxy-config listeners productpage-v1-abc123.default -o json | \
  grep -A 5 "transportSocket"
```

## Using the Envoy Admin Interface

For real-time debugging, you can access Envoy's admin interface directly:

```bash
istioctl dashboard envoy productpage-v1-abc123.default
```

This opens the admin page where you can see active connections, recent stats, config dumps, and even toggle logging levels on the fly. The `/stats` endpoint is particularly useful for seeing error counts and latencies.

You can also hit the admin API through kubectl:

```bash
kubectl exec productpage-v1-abc123 -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "upstream_cx"
```

## Final Thoughts

Debugging Envoy configuration is a skill that pays off quickly. Once you understand the listener-route-cluster-endpoint model, most Istio problems become straightforward to diagnose. The istioctl `proxy-config` commands give you direct visibility into what Envoy is doing, which is always more reliable than guessing based on your YAML files.

Build the habit of checking proxy-config whenever traffic doesn't behave as expected. It's the fastest path to finding the root cause.
