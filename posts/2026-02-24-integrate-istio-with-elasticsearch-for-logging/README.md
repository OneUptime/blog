# How to Integrate Istio with Elasticsearch for Logging

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Elasticsearch, Logging, Observability, Kubernetes

Description: How to send Istio service mesh access logs and telemetry data to Elasticsearch for centralized logging and analysis.

---

When you are running Istio in production, the amount of log data generated by Envoy sidecars can be overwhelming. Every request between services produces access log entries, and debugging issues without centralized logging is painful. Elasticsearch gives you a powerful search and analytics engine to store, search, and visualize all that log data. Combined with Kibana, you get a solid logging stack for your mesh.

## What Logs Does Istio Generate

Istio produces several types of log data:

- **Envoy access logs** - Every request passing through the sidecar proxy gets logged with details like response code, latency, upstream cluster, and request headers
- **Istio control plane logs** - istiod produces logs about configuration distribution, certificate management, and service discovery
- **Envoy application logs** - The proxy itself logs operational events like connection errors, TLS handshake failures, and configuration updates

The access logs are the most valuable for operational visibility because they give you a complete picture of traffic flowing through the mesh.

## Setting Up Elasticsearch

Deploy Elasticsearch on Kubernetes using the ECK (Elastic Cloud on Kubernetes) operator:

```bash
kubectl create -f https://download.elastic.co/downloads/eck/2.11.0/crds.yaml
kubectl apply -f https://download.elastic.co/downloads/eck/2.11.0/operator.yaml
```

Create an Elasticsearch cluster:

```yaml
apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: mesh-logs
  namespace: elastic-system
spec:
  version: 8.12.0
  nodeSets:
  - name: default
    count: 3
    config:
      node.store.allow_mmap: false
    podTemplate:
      spec:
        containers:
        - name: elasticsearch
          resources:
            requests:
              memory: 2Gi
              cpu: 1
            limits:
              memory: 4Gi
    volumeClaimTemplates:
    - metadata:
        name: elasticsearch-data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 50Gi
```

Deploy Kibana for visualization:

```yaml
apiVersion: kibana.k8s.elastic.co/v1
kind: Kibana
metadata:
  name: mesh-kibana
  namespace: elastic-system
spec:
  version: 8.12.0
  count: 1
  elasticsearchRef:
    name: mesh-logs
```

Get the Elasticsearch password:

```bash
kubectl get secret mesh-logs-es-elastic-user -n elastic-system -o jsonpath='{.data.elastic}' | base64 -d
```

## Configuring Istio Access Logs

Enable Envoy access logging in Istio. The access log format determines what data gets captured:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    accessLogFile: /dev/stdout
    accessLogEncoding: JSON
    accessLogFormat: |
      {
        "timestamp": "%START_TIME%",
        "method": "%REQ(:METHOD)%",
        "path": "%REQ(X-ENVOY-ORIGINAL-PATH?:PATH)%",
        "protocol": "%PROTOCOL%",
        "response_code": "%RESPONSE_CODE%",
        "response_flags": "%RESPONSE_FLAGS%",
        "bytes_received": "%BYTES_RECEIVED%",
        "bytes_sent": "%BYTES_SENT%",
        "duration": "%DURATION%",
        "upstream_service_time": "%RESP(X-ENVOY-UPSTREAM-SERVICE-TIME)%",
        "upstream_host": "%UPSTREAM_HOST%",
        "upstream_cluster": "%UPSTREAM_CLUSTER%",
        "source_app": "%REQ(X-FORWARDED-FOR)%",
        "destination_service": "%REQ(:AUTHORITY)%",
        "request_id": "%REQ(X-REQUEST-ID)%",
        "user_agent": "%REQ(USER-AGENT)%"
      }
```

Using JSON encoding makes it much easier to parse and index in Elasticsearch compared to the default text format.

## Using the Telemetry API for Access Logging

The newer approach is to use Istio's Telemetry API:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: mesh-logging
  namespace: istio-system
spec:
  accessLogging:
  - providers:
    - name: envoy
    filter:
      expression: "response.code >= 400 || connection.mtls == false"
```

The filter expression lets you control which requests get logged. In this example, only error responses and non-mTLS connections are logged, which reduces log volume significantly.

## Shipping Logs to Elasticsearch

You need a log collector to get logs from the Envoy sidecars into Elasticsearch. Filebeat is the most common choice for Elasticsearch:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: filebeat
  namespace: elastic-system
spec:
  selector:
    matchLabels:
      app: filebeat
  template:
    metadata:
      labels:
        app: filebeat
    spec:
      serviceAccountName: filebeat
      containers:
      - name: filebeat
        image: docker.elastic.co/beats/filebeat:8.12.0
        args: ["-c", "/etc/filebeat/filebeat.yml", "-e"]
        env:
        - name: ELASTICSEARCH_HOST
          value: "mesh-logs-es-http.elastic-system.svc"
        - name: ELASTICSEARCH_PORT
          value: "9200"
        - name: ELASTICSEARCH_PASSWORD
          valueFrom:
            secretKeyRef:
              name: mesh-logs-es-elastic-user
              key: elastic
        volumeMounts:
        - name: config
          mountPath: /etc/filebeat
          readOnly: true
        - name: varlog
          mountPath: /var/log
          readOnly: true
        - name: containerlog
          mountPath: /var/lib/docker/containers
          readOnly: true
      volumes:
      - name: config
        configMap:
          name: filebeat-config
      - name: varlog
        hostPath:
          path: /var/log
      - name: containerlog
        hostPath:
          path: /var/lib/docker/containers
```

Filebeat configuration to collect Istio sidecar logs:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: filebeat-config
  namespace: elastic-system
data:
  filebeat.yml: |
    filebeat.autodiscover:
      providers:
      - type: kubernetes
        node: ${NODE_NAME}
        hints.enabled: true
        templates:
        - condition:
            contains:
              kubernetes.container.name: istio-proxy
          config:
          - type: container
            paths:
            - /var/log/containers/*${data.kubernetes.container.id}.log
            processors:
            - decode_json_fields:
                fields: ["message"]
                target: "istio"
                overwrite_keys: true

    output.elasticsearch:
      hosts: ["https://${ELASTICSEARCH_HOST}:${ELASTICSEARCH_PORT}"]
      username: "elastic"
      password: "${ELASTICSEARCH_PASSWORD}"
      ssl.verification_mode: none
      index: "istio-access-logs-%{+yyyy.MM.dd}"

    setup.template:
      name: "istio-access-logs"
      pattern: "istio-access-logs-*"
```

## Creating Kibana Dashboards

Once logs are flowing into Elasticsearch, create an index pattern in Kibana:

1. Open Kibana and go to Stack Management > Index Patterns
2. Create a pattern for `istio-access-logs-*`
3. Select `@timestamp` as the time field

Build useful dashboards with these visualizations:

**Request Volume Over Time** - A line chart showing requests per second grouped by destination service.

**Error Rate** - A visualization filtering on `response_code >= 400` grouped by service and path.

**Latency Distribution** - A histogram of the `duration` field to see response time distribution.

**Top Error Paths** - A table showing the most common request paths with 5xx errors.

## Index Lifecycle Management

Istio access logs generate a lot of data. Configure ILM to manage retention:

```bash
curl -X PUT "https://elasticsearch:9200/_ilm/policy/istio-logs-policy" -H 'Content-Type: application/json' -d'
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_age": "1d",
            "max_size": "50gb"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": {
            "number_of_shards": 1
          }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}'
```

This keeps hot data for 7 days, warm data for 30 days, and then deletes it.

## Searching and Analyzing Logs

With structured JSON logs in Elasticsearch, you can run powerful queries. Find all failed requests to a specific service:

```json
{
  "query": {
    "bool": {
      "must": [
        {"match": {"istio.destination_service": "payment-service"}},
        {"range": {"istio.response_code": {"gte": 500}}}
      ],
      "filter": {
        "range": {
          "@timestamp": {"gte": "now-1h"}
        }
      }
    }
  }
}
```

Find requests with high latency:

```json
{
  "query": {
    "bool": {
      "must": [
        {"range": {"istio.duration": {"gte": 5000}}}
      ]
    }
  },
  "sort": [{"istio.duration": "desc"}]
}
```

Sending Istio logs to Elasticsearch gives your team the ability to search, analyze, and visualize mesh traffic patterns at scale. The JSON access log format combined with Elasticsearch's query capabilities makes it straightforward to track down issues, identify slow services, and understand traffic patterns across your entire mesh.
