# How to Configure Kubernetes Provider with Kubeconfig

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Kubernetes, Kubeconfig, Provider, DevOps, Infrastructure as Code

Description: Learn how to configure the Terraform Kubernetes provider using kubeconfig files for authentication, including multi-cluster setups and context switching patterns.

---

The Terraform Kubernetes provider needs to know how to connect to your cluster. The most common way to provide this information is through a kubeconfig file - the same file that `kubectl` uses. This guide covers how to configure the Kubernetes provider with kubeconfig, handle multiple clusters, and avoid common pitfalls.

## How Kubeconfig Authentication Works

A kubeconfig file contains three key pieces of information:

- **Clusters** - API server endpoints and CA certificates
- **Users** - Authentication credentials (tokens, certificates, or exec-based auth)
- **Contexts** - Combinations of cluster + user + namespace

When you configure the Kubernetes provider to use a kubeconfig file, Terraform reads these values and uses them to authenticate API calls, just like kubectl does.

## Basic Kubeconfig Configuration

The simplest setup points the provider at your kubeconfig file:

```hcl
# versions.tf
terraform {
  required_providers {
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = "~> 2.35"
    }
  }
}

# provider.tf
provider "kubernetes" {
  config_path = "~/.kube/config"
}
```

This uses the current context from your kubeconfig file. Whatever cluster `kubectl` targets is what Terraform targets.

## Specifying a Context

If your kubeconfig has multiple contexts, you can select a specific one:

```hcl
provider "kubernetes" {
  config_path    = "~/.kube/config"
  config_context = "production-cluster"
}
```

To see your available contexts:

```bash
# List all contexts in your kubeconfig
kubectl config get-contexts

# Output:
# CURRENT   NAME                 CLUSTER              AUTHINFO             NAMESPACE
# *         production-cluster   prod-cluster         prod-admin           default
#           staging-cluster      staging-cluster      staging-admin        default
#           dev-cluster          dev-cluster          dev-admin            default
```

## Using a Custom Kubeconfig Path

You might keep separate kubeconfig files for different environments instead of merging everything into `~/.kube/config`:

```hcl
variable "environment" {
  description = "Target environment"
  type        = string
}

provider "kubernetes" {
  config_path = "${path.module}/kubeconfigs/${var.environment}.yaml"
}
```

With the directory structure:

```text
project/
  kubeconfigs/
    production.yaml
    staging.yaml
    development.yaml
  main.tf
  provider.tf
```

## Multi-Cluster Configuration

To manage resources across multiple clusters, use provider aliases:

```hcl
# Production cluster
provider "kubernetes" {
  config_path    = "~/.kube/config"
  config_context = "production-cluster"
}

# Staging cluster
provider "kubernetes" {
  alias          = "staging"
  config_path    = "~/.kube/config"
  config_context = "staging-cluster"
}

# Deploy to production
resource "kubernetes_namespace" "app_prod" {
  metadata {
    name = "my-app"
    labels = {
      environment = "production"
    }
  }
}

# Deploy to staging
resource "kubernetes_namespace" "app_staging" {
  provider = kubernetes.staging

  metadata {
    name = "my-app"
    labels = {
      environment = "staging"
    }
  }
}
```

## EKS Kubeconfig Setup

For Amazon EKS, generate the kubeconfig with the AWS CLI:

```bash
# Generate kubeconfig for an EKS cluster
aws eks update-kubeconfig \
  --name my-eks-cluster \
  --region us-east-1 \
  --kubeconfig ./eks-kubeconfig.yaml

# For a specific IAM role
aws eks update-kubeconfig \
  --name my-eks-cluster \
  --region us-east-1 \
  --role-arn arn:aws:iam::123456789012:role/EKSAdminRole \
  --kubeconfig ./eks-kubeconfig.yaml
```

```hcl
provider "kubernetes" {
  config_path = "${path.module}/eks-kubeconfig.yaml"
}
```

EKS kubeconfigs typically use exec-based authentication, calling `aws eks get-token` to generate short-lived tokens. The kubeconfig looks something like this:

```yaml
# eks-kubeconfig.yaml (generated by aws eks update-kubeconfig)
apiVersion: v1
clusters:
- cluster:
    certificate-authority-data: LS0tLS1C...
    server: https://ABCDEF1234567890.gr7.us-east-1.eks.amazonaws.com
  name: arn:aws:eks:us-east-1:123456789012:cluster/my-eks-cluster
contexts:
- context:
    cluster: arn:aws:eks:us-east-1:123456789012:cluster/my-eks-cluster
    user: arn:aws:eks:us-east-1:123456789012:cluster/my-eks-cluster
  name: arn:aws:eks:us-east-1:123456789012:cluster/my-eks-cluster
current-context: arn:aws:eks:us-east-1:123456789012:cluster/my-eks-cluster
users:
- name: arn:aws:eks:us-east-1:123456789012:cluster/my-eks-cluster
  user:
    exec:
      apiVersion: client.authentication.k8s.io/v1beta1
      command: aws
      args:
        - --region
        - us-east-1
        - eks
        - get-token
        - --cluster-name
        - my-eks-cluster
```

## GKE Kubeconfig Setup

For Google Kubernetes Engine:

```bash
# Generate kubeconfig for a GKE cluster
gcloud container clusters get-credentials my-gke-cluster \
  --region us-central1 \
  --project my-project

# Write to a specific file
KUBECONFIG=./gke-kubeconfig.yaml gcloud container clusters get-credentials my-gke-cluster \
  --region us-central1 \
  --project my-project
```

```hcl
provider "kubernetes" {
  config_path = "${path.module}/gke-kubeconfig.yaml"
}
```

For Azure Kubernetes Service:

```bash
# Generate kubeconfig for an AKS cluster
az aks get-credentials \

  --name my-aks-cluster \
  --file ./aks-kubeconfig.yaml
```

```hcl
provider "kubernetes" {
  config_path = "${path.module}/aks-kubeconfig.yaml"

```

## Inline Kubeconfig Content

Instead of referencing a file path, you can pass the kubeconfig content directly. This is useful when the kubeconfig is stored in a secret manager or generated dynamically:

```hcl
variable "kubeconfig_content" {
  description = "Raw kubeconfig YAML content"
  type        = string
  sensitive   = true
}

provider "kubernetes" {
  config_path = null  # Must be null when using config_paths

  # Parse the kubeconfig content
  host                   = yamldecode(var.kubeconfig_content).clusters[0].cluster.server
  cluster_ca_certificate = base64decode(yamldecode(var.kubeconfig_content).clusters[0].cluster["certificate-authority-data"])
  token                  = yamldecode(var.kubeconfig_content).users[0].user.token
}
```

A cleaner approach is to write the content to a temporary file:

```hcl
resource "local_file" "kubeconfig" {
  content  = var.kubeconfig_content
  filename = "${path.module}/.kubeconfig"
}

provider "kubernetes" {
  config_path = local_file.kubeconfig.filename
}
```

## Combining with Helm Provider

If you use both the Kubernetes and Helm providers, they can share the same kubeconfig:

```hcl
provider "kubernetes" {
  config_path    = "~/.kube/config"
  config_context = "production-cluster"
}

provider "helm" {
  kubernetes {
    config_path    = "~/.kube/config"
    config_context = "production-cluster"
  }
}

# Now you can use both providers
resource "kubernetes_namespace" "monitoring" {
  metadata {
    name = "monitoring"
  }
}

resource "helm_release" "prometheus" {
  name       = "prometheus"
  namespace  = kubernetes_namespace.monitoring.metadata[0].name
  repository = "https://prometheus-community.github.io/helm-charts"
  chart      = "kube-prometheus-stack"
  version    = "65.0.0"
}
```

## Handling Token Expiry

Kubeconfig files that use exec-based authentication (EKS, GKE) generate short-lived tokens. Terraform handles this well because it calls the exec command on each API request. However, kubeconfigs with static tokens can expire:

```hcl
# For clusters using static tokens, consider using exec instead
provider "kubernetes" {
  host                   = var.cluster_endpoint
  cluster_ca_certificate = base64decode(var.cluster_ca_cert)

  exec {
    api_version = "client.authentication.k8s.io/v1beta1"
    command     = "aws"
    args        = ["eks", "get-token", "--cluster-name", var.cluster_name]
  }
}
```

This is functionally equivalent to what the EKS kubeconfig does, but defined directly in Terraform instead of in a kubeconfig file.

## Troubleshooting

### "Unable to load kubeconfig"

Check that the file path is correct and the file exists:

```bash
ls -la ~/.kube/config
# Or the custom path you specified
```

### "Unauthorized" Errors

The credentials in your kubeconfig may have expired:

```bash
# For EKS, verify AWS credentials work
aws sts get-caller-identity

# For GKE, refresh credentials
gcloud auth application-default login

# For AKS, refresh credentials
az aks get-credentials --resource-group my-rg --name my-cluster --overwrite-existing
```

### "Context not found"

The specified context does not exist in the kubeconfig file:

```bash
kubectl config get-contexts --kubeconfig=/path/to/your/kubeconfig
```

### Provider Configuration Conflicts

Do not mix `config_path` with explicit `host`/`token`/`cluster_ca_certificate` settings. Use one approach or the other:

```hcl
# WRONG - mixing both approaches
provider "kubernetes" {
  config_path = "~/.kube/config"
  host        = "https://my-cluster.example.com"  # Conflicts with config_path
}

# RIGHT - use only config_path
provider "kubernetes" {
  config_path    = "~/.kube/config"
  config_context = "my-cluster"
}

# RIGHT - use only explicit config
provider "kubernetes" {
  host                   = "https://my-cluster.example.com"
  cluster_ca_certificate = base64decode(var.ca_cert)
  token                  = var.token
}
```

## Summary

Kubeconfig files are the most familiar way to configure the Terraform Kubernetes provider, especially for developers who already work with kubectl. Point the provider at your kubeconfig with `config_path`, select a context with `config_context`, and use aliases for multi-cluster setups. For CI/CD, generate kubeconfigs using the cloud provider CLIs (aws, gcloud, az) and keep them as ephemeral as possible to avoid stale credentials.
