# How to Configure Flux CD DNS Lookups

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, GitOps, Kubernetes, DNS, Networking, Configuration

Description: Learn how to configure DNS resolution for Flux CD controllers to work with custom DNS servers, private registries, and split-horizon DNS environments.

---

## Why Configure DNS for Flux CD?

Flux CD controllers need to resolve hostnames when connecting to Git repositories, Helm registries, OCI registries, and notification endpoints. In many enterprise environments, DNS configuration is non-trivial:

- **Private registries** may only be resolvable through internal DNS servers.
- **Split-horizon DNS** requires different resolution depending on the source.
- **Air-gapped environments** use custom DNS infrastructure.
- **Service mesh setups** may intercept or modify DNS behavior.

This guide covers how to configure DNS lookups for Flux CD controllers at both the pod level and the cluster level.

## Prerequisites

- A Kubernetes cluster (v1.20+)
- `flux` CLI installed (v2.0+)
- `kubectl` configured to access your cluster
- Understanding of your cluster's DNS architecture

## Understanding Flux CD DNS Behavior

Flux controllers are standard Kubernetes pods, so they use the cluster's DNS configuration by default. This means they rely on:

1. The cluster DNS service (typically CoreDNS)
2. The pod's `/etc/resolv.conf` generated by kubelet
3. Any DNS policies set on the pod spec

## Method 1: Configure Pod DNS Policy

You can override the DNS policy for Flux controllers using Kustomize patches. This is useful when controllers need to resolve names that are not available through the default cluster DNS.

First, export the Flux installation manifests:

```bash
# Export Flux manifests for customization
flux install --export > flux-install.yaml
```

Create a Kustomize overlay that sets a custom DNS policy and DNS config for the source-controller (the component most likely to need custom DNS):

```yaml
# kustomization.yaml
# Patches Flux controllers with custom DNS configuration
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - flux-install.yaml
patches:
  # Add custom DNS configuration to source-controller
  - target:
      kind: Deployment
      name: source-controller
      namespace: flux-system
    patch: |
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: source-controller
      spec:
        template:
          spec:
            # Use custom DNS settings alongside cluster DNS
            dnsPolicy: "None"
            dnsConfig:
              nameservers:
                - 10.0.0.53
                - 172.20.0.10
              searches:
                - flux-system.svc.cluster.local
                - svc.cluster.local
                - cluster.local
                - internal.company.com
              options:
                - name: ndots
                  value: "5"
                - name: timeout
                  value: "3"
                - name: attempts
                  value: "3"
  # Apply the same DNS config to helm-controller
  - target:
      kind: Deployment
      name: helm-controller
      namespace: flux-system
    patch: |
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: helm-controller
      spec:
        template:
          spec:
            dnsPolicy: "None"
            dnsConfig:
              nameservers:
                - 10.0.0.53
                - 172.20.0.10
              searches:
                - flux-system.svc.cluster.local
                - svc.cluster.local
                - cluster.local
                - internal.company.com
              options:
                - name: ndots
                  value: "5"
```

Apply the configuration:

```bash
# Apply Flux with custom DNS configuration
kubectl apply -k .
```

## Method 2: Configure CoreDNS for Custom Domains

Instead of patching Flux pods directly, you can configure the cluster's CoreDNS to forward specific domains to custom DNS servers. This approach affects all pods in the cluster, not just Flux.

Edit the CoreDNS ConfigMap:

```bash
# Edit CoreDNS configuration
kubectl edit configmap coredns -n kube-system
```

Add a forward rule for your internal domains:

```yaml
# CoreDNS Corefile configuration with custom domain forwarding
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
            lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods insecure
            fallthrough in-addr.arpa ip6.arpa
            ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf {
            max_concurrent 1000
        }
        cache 30
        loop
        reload
        loadbalance
    }
    # Forward internal.company.com to the corporate DNS server
    internal.company.com:53 {
        errors
        cache 30
        forward . 10.0.0.53 10.0.0.54 {
            policy round_robin
        }
    }
    # Forward registry.private.io to a specific DNS server
    registry.private.io:53 {
        errors
        cache 30
        forward . 10.0.0.53
    }
```

After editing, restart CoreDNS:

```bash
# Restart CoreDNS to apply changes
kubectl rollout restart deployment/coredns -n kube-system
```

## Method 3: Use HostAliases for Static Hostname Resolution

For cases where you need to map specific hostnames to IP addresses (similar to /etc/hosts entries), use hostAliases in the pod spec:

```yaml
# kustomization.yaml
# Add hostAliases to source-controller for static DNS entries
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - flux-install.yaml
patches:
  - target:
      kind: Deployment
      name: source-controller
      namespace: flux-system
    patch: |
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: source-controller
      spec:
        template:
          spec:
            hostAliases:
              # Map internal Git server to its IP address
              - ip: "10.1.2.3"
                hostnames:
                  - "git.internal.company.com"
              # Map internal Helm registry
              - ip: "10.1.2.4"
                hostnames:
                  - "helm.internal.company.com"
                  - "registry.internal.company.com"
```

## Method 4: Environment Variable Configuration

Some DNS-related behaviors can be configured through environment variables on the Flux controller containers:

```yaml
# Patch to add DNS-related environment variables to source-controller
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - flux-install.yaml
patches:
  - target:
      kind: Deployment
      name: source-controller
      namespace: flux-system
    patch: |
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: source-controller
      spec:
        template:
          spec:
            containers:
              - name: manager
                env:
                  # Set custom DNS resolver timeout
                  - name: GODEBUG
                    value: "netdns=go"
                  # Force pure Go DNS resolver instead of cgo
                  - name: RES_OPTIONS
                    value: "timeout:2 attempts:3"
```

## Verifying DNS Resolution

After applying DNS configuration changes, verify that Flux controllers can resolve the required hostnames:

```bash
# Exec into the source-controller pod to test DNS resolution
kubectl exec -it -n flux-system deployment/source-controller -- \
  nslookup git.internal.company.com

# If nslookup is not available, check the resolv.conf
kubectl exec -it -n flux-system deployment/source-controller -- \
  cat /etc/resolv.conf

# Check source-controller logs for DNS-related errors
kubectl logs -n flux-system deployment/source-controller | grep -i "dns\|resolve\|lookup"
```

## Troubleshooting DNS Issues

If Flux controllers cannot resolve hostnames, follow these steps:

```bash
# Check if the Flux source is failing due to DNS
flux get sources git --all-namespaces

# Look for DNS timeout or resolution errors in controller logs
kubectl logs -n flux-system deployment/source-controller --tail=50

# Test DNS resolution from within the cluster using a debug pod
kubectl run dns-test --rm -it --image=busybox:1.36 --restart=Never -- \
  nslookup git.internal.company.com

# Verify CoreDNS is running and healthy
kubectl get pods -n kube-system -l k8s-app=kube-dns
```

Common DNS error messages and their meanings:

- **"dial tcp: lookup hostname: no such host"**: The hostname cannot be resolved. Check DNS configuration.
- **"dial tcp: lookup hostname: i/o timeout"**: DNS server is unreachable. Check network policies and firewall rules.
- **"dial tcp: lookup hostname: server misbehaving"**: DNS server returned an invalid response. Check the upstream DNS server.

## Summary

Configuring DNS for Flux CD depends on your environment. For most setups, configuring CoreDNS with custom forwarders is the cleanest approach since it benefits all cluster workloads. For Flux-specific needs, use Kustomize patches to set dnsPolicy, dnsConfig, or hostAliases on individual controller deployments. Always verify DNS resolution from within the controller pods to confirm your configuration works correctly.
