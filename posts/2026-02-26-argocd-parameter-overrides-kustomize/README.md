# How to Use Parameter Overrides with Kustomize in ArgoCD

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: ArgoCD, GitOps, Kubernetes, Kustomize, Configuration Management

Description: Learn how to use ArgoCD parameter overrides with Kustomize to dynamically adjust image tags, replicas, namespaces, and patches without modifying your Git repository.

---

ArgoCD lets you override Kustomize parameters directly in the Application spec, through the CLI, or via the UI. This gives you the ability to make quick adjustments - like changing an image tag or namespace - without committing changes to your Git repository. While the GitOps purist approach is to always commit changes to Git, parameter overrides are useful for testing, emergency fixes, and CI/CD integrations that update image tags automatically.

## How Kustomize Overrides Work in ArgoCD

When ArgoCD processes a Kustomize application, it runs `kustomize build` against the specified path. Parameter overrides inject additional Kustomize directives into this build process. Under the hood, ArgoCD modifies the `kustomization.yaml` in memory before building, without changing the file on disk or in Git.

The key override types for Kustomize in ArgoCD are:

- **Image overrides**: Change the container image repository, tag, or both
- **Namespace override**: Change the target namespace
- **Name prefix/suffix**: Add prefixes or suffixes to all resource names
- **Common labels**: Add labels to all resources
- **Common annotations**: Add annotations to all resources

## Image Overrides in the Application Spec

The most common Kustomize override is the image tag. This is how CI/CD pipelines typically update the deployed version:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-staging
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/app-config.git
    path: overlays/staging
    targetRevision: main
    kustomize:
      images:
        - myregistry/my-app:v1.2.3           # Override the image tag
        - myregistry/sidecar-proxy:v2.0.1     # Override sidecar image
  destination:
    server: https://kubernetes.default.svc
    namespace: staging
```

The `kustomize.images` field follows the same syntax as `kustomize edit set image`. You can use several formats:

```yaml
kustomize:
  images:
    # Set tag only (image name must match existing image in manifests)
    - myregistry/my-app:v1.2.3

    # Change both registry and tag
    - myregistry/my-app=newregistry/my-app:v1.2.3

    # Set digest instead of tag
    - myregistry/my-app@sha256:abc123def456...

    # Change image name entirely
    - old-image=new-image:tag
```

## Namespace Override

Override the target namespace without modifying the kustomization:

```yaml
source:
  repoURL: https://github.com/myorg/app-config.git
  path: overlays/staging
  kustomize:
    namespace: staging-v2   # Override the namespace set in kustomization.yaml
```

This is equivalent to adding `namespace: staging-v2` to the kustomization.yaml.

## Name Prefix and Suffix

Add a prefix or suffix to all resource names generated by Kustomize:

```yaml
source:
  repoURL: https://github.com/myorg/app-config.git
  path: overlays/staging
  kustomize:
    namePrefix: v2-          # All resources get "v2-" prefix
    nameSuffix: -canary       # All resources get "-canary" suffix
```

A deployment named `my-app` would become `v2-my-app-canary`. This is useful for running multiple versions of the same application in the same namespace during canary deployments or A/B testing.

## Common Labels and Annotations

Inject labels or annotations into every resource:

```yaml
source:
  repoURL: https://github.com/myorg/app-config.git
  path: overlays/staging
  kustomize:
    commonLabels:
      team: backend
      cost-center: engineering
    commonAnnotations:
      deployed-by: argocd
      last-updated: "2026-02-26"
```

Be careful with `commonLabels` on existing resources. Kustomize adds labels to selector fields, which are immutable on Deployments. If you add or change `commonLabels` on an already-deployed application, the sync will fail because Kubernetes does not allow changing selectors on existing Deployments. You would need to delete and recreate the Deployment.

## Kustomize Version and Build Options

You can also control how ArgoCD runs Kustomize:

```yaml
source:
  repoURL: https://github.com/myorg/app-config.git
  path: overlays/staging
  kustomize:
    version: v5.3.0                    # Use a specific Kustomize version
    commonLabels:
      app: my-app
    images:
      - myregistry/my-app:v1.2.3
```

## Overriding via the ArgoCD CLI

The CLI provides commands to set Kustomize overrides on existing applications:

```bash
# Override the image tag
argocd app set my-app-staging \
  --kustomize-image myregistry/my-app:v1.2.3

# Override multiple images
argocd app set my-app-staging \
  --kustomize-image myregistry/my-app:v1.2.3 \
  --kustomize-image myregistry/sidecar:v2.0.0

# Override the namespace
argocd app set my-app-staging \
  --kustomize-namespace staging-v2

# Override name prefix
argocd app set my-app-staging \
  --kustomize-name-prefix "v2-"

# Override name suffix
argocd app set my-app-staging \
  --kustomize-name-suffix "-canary"

# Add common labels
argocd app set my-app-staging \
  --kustomize-common-label "team=backend"

# Add common annotations
argocd app set my-app-staging \
  --kustomize-common-annotation "deployed-by=ci"
```

## Overriding via the ArgoCD API

For programmatic access from CI/CD pipelines, use the ArgoCD API:

```bash
# Update image using the ArgoCD API
curl -X PATCH \
  "https://argocd.example.com/api/v1/applications/my-app-staging" \
  -H "Authorization: Bearer $ARGOCD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "spec": {
      "source": {
        "kustomize": {
          "images": ["myregistry/my-app:v1.2.3"]
        }
      }
    }
  }'
```

## CI/CD Integration Pattern

A common pattern is having your CI pipeline update the Kustomize image override after building a new container image:

```yaml
# .github/workflows/deploy.yaml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.version }}
    steps:
      - name: Build and push image
        run: |
          docker build -t myregistry/my-app:${{ github.sha }} .
          docker push myregistry/my-app:${{ github.sha }}

  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Update ArgoCD application image
        run: |
          argocd login $ARGOCD_SERVER --auth-token $ARGOCD_TOKEN
          argocd app set my-app-staging \
            --kustomize-image myregistry/my-app:${{ github.sha }}
          argocd app sync my-app-staging --async
```

## Viewing Current Overrides

Check what overrides are currently set on an application:

```bash
# Show application details including source overrides
argocd app get my-app-staging

# Show just the source configuration
argocd app get my-app-staging -o yaml | yq '.spec.source.kustomize'

# Show the computed parameters
argocd app get my-app-staging --show-params
```

## Override Precedence

When both the kustomization.yaml in Git and the ArgoCD Application spec define the same parameter, the ArgoCD override takes precedence. For example, if your kustomization.yaml has:

```yaml
images:
  - name: myregistry/my-app
    newTag: v1.0.0
```

And your ArgoCD Application has:

```yaml
kustomize:
  images:
    - myregistry/my-app:v1.2.3
```

The deployed image will be `myregistry/my-app:v1.2.3`.

## When to Use Overrides vs Git Commits

Use parameter overrides when:
- CI/CD updates image tags automatically after builds
- You need to test a change quickly without going through the full Git commit workflow
- Emergency hotfixes that need to deploy immediately
- Temporary debugging (changing log levels, adding debug containers)

Use Git commits when:
- Making permanent configuration changes
- Changing resource limits, replica counts, or other operational parameters
- Adding or removing resources
- Any change that should be tracked in your Git history and auditable

The ideal pattern is using overrides for image tag updates (since those change frequently and the CI build is the source of truth for what image to deploy) while committing everything else to Git. For more on structuring your ArgoCD configuration, see [How to Structure Your Git Repository for ArgoCD](https://oneuptime.com/blog/post/2026-02-26-argocd-git-repository-structure/view).

## Resetting Overrides

To remove overrides and revert to the values defined in Git:

```bash
# Remove all Kustomize overrides
argocd app unset my-app-staging --kustomize-image myregistry/my-app

# Remove namespace override
argocd app unset my-app-staging --kustomize-namespace

# Remove name prefix
argocd app unset my-app-staging --kustomize-name-prefix
```

After removing overrides, ArgoCD will use whatever is defined in the kustomization.yaml in your Git repository.

Parameter overrides in ArgoCD give you a pragmatic escape hatch from strict GitOps when you need it, while still maintaining the benefits of declarative configuration management. Use them judiciously for the right use cases, and keep the permanent state of your system in Git.
