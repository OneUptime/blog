# How to Configure Confd for Automated Configuration Management on RHEL

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: RHEL, Confd, Configuration Management, Etcd, Automation, Linux

Description: Learn how to install and configure confd on RHEL to automatically generate configuration files from backend data stores like etcd, Consul, and environment variables.

---

Confd is a lightweight configuration management tool that keeps local configuration files in sync with data stored in a backend like etcd, Consul, Redis, or even environment variables. When values change in the backend, confd regenerates the configuration files and optionally reloads services. This guide walks through setting up confd on RHEL.

## How Confd Works

Confd operates with two types of files:

- **Template resources** (.toml files) - define which keys to watch, where to write output, and what command to run on changes
- **Templates** (.tmpl files) - Go templates that reference the watched keys

Confd checks the backend at a configurable interval, and when it detects changes, it renders the templates and optionally runs a reload command.

## Prerequisites

- RHEL with root or sudo access
- A backend data store (etcd, Consul, or Redis)
- Basic familiarity with Go template syntax

## Installing Confd

Download the latest binary:

```bash
# Download confd
CONFD_VERSION="0.16.0"
curl -LO https://github.com/kelseyhightower/confd/releases/download/v${CONFD_VERSION}/confd-${CONFD_VERSION}-linux-amd64
sudo mv confd-${CONFD_VERSION}-linux-amd64 /usr/local/bin/confd
sudo chmod +x /usr/local/bin/confd
```

Verify the installation:

```bash
# Check confd version
confd --version
```

## Setting Up the Directory Structure

Create the required directories:

```bash
# Create confd configuration directories
sudo mkdir -p /etc/confd/{conf.d,templates}
```

## Configuring Confd with etcd

First, make sure etcd is running:

```bash
# Install and start etcd
sudo dnf install -y etcd
sudo systemctl enable --now etcd
```

Set some test values:

```bash
# Populate etcd with configuration values
etcdctl put /myapp/database/host "db.example.com"
etcdctl put /myapp/database/port "5432"
etcdctl put /myapp/database/name "production"
etcdctl put /myapp/database/user "appuser"
etcdctl put /myapp/cache/host "redis.example.com"
etcdctl put /myapp/cache/port "6379"
etcdctl put /myapp/log_level "info"
```

## Creating Template Resources

Template resources tell confd what to watch and where to write:

```bash
# Create a template resource for the app config
sudo tee /etc/confd/conf.d/myapp.toml > /dev/null << 'EOF'
[template]
src = "myapp.conf.tmpl"
dest = "/opt/myapp/config.conf"
keys = [
  "/myapp/database/host",
  "/myapp/database/port",
  "/myapp/database/name",
  "/myapp/database/user",
  "/myapp/cache/host",
  "/myapp/cache/port",
  "/myapp/log_level",
]
reload_cmd = "systemctl restart myapp"
check_cmd = "/opt/myapp/validate-config {{ .src }}"
owner = "myapp"
mode = "0640"
EOF
```

## Creating Templates

Create the template file that confd will render:

```bash
# Create the application configuration template
sudo tee /etc/confd/templates/myapp.conf.tmpl > /dev/null << 'TEMPLATE'
# Application Configuration
# Auto-generated by confd - do not edit manually
# Last updated: {{ now }}

[database]
host = {{ getv "/myapp/database/host" }}
port = {{ getv "/myapp/database/port" }}
name = {{ getv "/myapp/database/name" }}
user = {{ getv "/myapp/database/user" }}

[cache]
host = {{ getv "/myapp/cache/host" }}
port = {{ getv "/myapp/cache/port" }}

[logging]
level = {{ getv "/myapp/log_level" }}
TEMPLATE
```

## Running Confd

Test the template rendering once:

```bash
# Run confd once to generate the initial config
confd -onetime -backend etcd -node http://127.0.0.1:2379
```

Check the generated file:

```bash
# View the rendered configuration
cat /opt/myapp/config.conf
```

Run confd in watch mode:

```bash
# Run confd as a daemon watching for changes
confd -watch -backend etcd -node http://127.0.0.1:2379
```

## Creating a Systemd Service

```bash
# Create the systemd unit file
sudo tee /etc/systemd/system/confd.service > /dev/null << 'EOF'
[Unit]
Description=Confd Configuration Management
After=network-online.target etcd.service
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/local/bin/confd -watch -backend etcd -node http://127.0.0.1:2379
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF
```

```bash
# Enable and start confd
sudo systemctl daemon-reload
sudo systemctl enable --now confd
```

## Nginx Configuration Template

Generate Nginx configuration dynamically:

```bash
# Template resource for Nginx
sudo tee /etc/confd/conf.d/nginx.toml > /dev/null << 'EOF'
[template]
src = "nginx.conf.tmpl"
dest = "/etc/nginx/conf.d/app.conf"
keys = [
  "/nginx/upstream",
  "/nginx/server_name",
  "/nginx/ssl_cert",
  "/nginx/ssl_key",
]
reload_cmd = "nginx -t && systemctl reload nginx"
EOF
```

```bash
# Nginx template
sudo tee /etc/confd/templates/nginx.conf.tmpl > /dev/null << 'TEMPLATE'
upstream app_backend {
    {{range gets "/nginx/upstream/*"}}
    server {{ .Value }};
    {{end}}
}

server {
    listen 443 ssl;
    server_name {{ getv "/nginx/server_name" }};

    ssl_certificate {{ getv "/nginx/ssl_cert" }};
    ssl_certificate_key {{ getv "/nginx/ssl_key" }};

    location / {
        proxy_pass http://app_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
TEMPLATE
```

## Using Environment Variables as a Backend

For simpler setups, confd can read from environment variables:

```bash
# Set environment variables
export MYAPP_DB_HOST="localhost"
export MYAPP_DB_PORT="5432"
```

```bash
# Template resource using env backend
sudo tee /etc/confd/conf.d/env-config.toml > /dev/null << 'EOF'
[template]
src = "env-config.tmpl"
dest = "/opt/myapp/env.conf"
keys = [
  "/myapp/db/host",
  "/myapp/db/port",
]
EOF
```

```bash
# Run with environment backend
confd -onetime -backend env
```

## Using Consul as a Backend

```bash
# Run confd with Consul backend
confd -watch -backend consul -node http://127.0.0.1:8500
```

The template syntax stays the same. The keys map to Consul KV paths.

## Template Functions

Confd provides several useful template functions:

```bash
# Get a value with a default
{{ getv "/key" "default_value" }}

# Get all values under a prefix
{{ range gets "/prefix/*" }}
{{ .Key }} = {{ .Value }}
{{ end }}

# Get all keys under a prefix
{{ range ls "/prefix" }}
{{ . }}
{{ end }}

# Check if a key exists
{{ if exists "/key" }}
value = {{ getv "/key" }}
{{ end }}

# Join values
{{ $servers := getvs "/servers/*" }}
servers = {{ join $servers "," }}

# Convert to JSON
{{ getv "/config/json_data" | json }}
```

## Monitoring Confd

Check confd logs:

```bash
# View confd logs
sudo journalctl -u confd -f
```

```bash
# Run with debug logging
confd -watch -backend etcd -node http://127.0.0.1:2379 -log-level debug
```

## Conclusion

Confd on RHEL provides a simple, reliable way to keep configuration files synchronized with your data store. Whether you use etcd, Consul, or environment variables as the backend, confd watches for changes and regenerates configuration files automatically, removing the need for manual configuration updates and reducing the risk of configuration drift across your infrastructure.
