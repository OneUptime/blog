# How to Set Up Custom CRI Configuration in Talos Linux

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, CRI, containerd, Custom Configuration, Kubernetes

Description: A hands-on guide to creating custom Container Runtime Interface configurations in Talos Linux for specialized workload requirements.

---

The Container Runtime Interface (CRI) is the protocol that connects the kubelet to the container runtime. In Talos Linux, containerd implements the CRI, and Talos generates a default CRI configuration that works for most use cases. However, specialized workloads often need custom CRI settings - things like alternative runtimes, custom sandbox images, specific snapshotter configurations, or modified networking behavior.

This guide shows you how to create custom CRI configurations in Talos Linux using configuration fragments and machine files.

## CRI Configuration Architecture in Talos

Talos generates the containerd configuration from its machine configuration. The base configuration lives at `/etc/containerd/config.toml`. You can extend it by placing configuration fragments in `/var/cri/conf.d/`. These fragments get merged into the final configuration that containerd uses.

The merging order is:
1. Base configuration generated by Talos
2. Configuration fragments from `/var/cri/conf.d/` in alphabetical order

Later fragments can override earlier settings, giving you fine-grained control.

## Creating Your First Custom CRI Configuration

The simplest custom CRI configuration modifies basic CRI plugin settings:

```yaml
machine:
  files:
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            # Custom sandbox image
            sandbox_image = "registry.k8s.io/pause:3.10"
            # Increase max concurrent image downloads
            max_concurrent_downloads = 10
            # Set maximum log line size
            max_container_log_line_size = 32768
            # Disable apparmor for containers
            disable_apparmor = false
      path: /var/cri/conf.d/20-cri-custom.toml
      op: create
```

Apply this configuration and containerd will use these settings for all CRI operations.

## Custom Runtime Configuration

One of the most powerful customizations is adding alternative container runtimes. This lets pods choose different isolation levels:

```yaml
machine:
  files:
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            [plugins."io.containerd.grpc.v1.cri".containerd]
              default_runtime_name = "runc"
              [plugins."io.containerd.grpc.v1.cri".containerd.runtimes]
                # Standard runc runtime
                [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
                  runtime_type = "io.containerd.runc.v2"
                  [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
                    SystemdCgroup = true
                # gVisor runtime for sandboxed workloads
                [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.gvisor]
                  runtime_type = "io.containerd.runsc.v1"
                # Spin runtime for WebAssembly workloads
                [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.spin]
                  runtime_type = "io.containerd.spin.v2"
      path: /var/cri/conf.d/20-runtimes.toml
      op: create
```

Each runtime needs a corresponding Kubernetes RuntimeClass:

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: gvisor
---
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: spin
handler: spin
```

## Custom CNI Configuration

The CRI plugin manages CNI (Container Network Interface) configuration. You can customize the CNI settings:

```yaml
machine:
  files:
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            [plugins."io.containerd.grpc.v1.cri".cni]
              bin_dir = "/opt/cni/bin"
              conf_dir = "/etc/cni/net.d"
              max_conf_num = 1
              conf_template = ""
      path: /var/cri/conf.d/20-cni.toml
      op: create
```

The `max_conf_num` setting limits how many CNI configuration files are loaded. Setting it to 1 ensures only the highest-priority configuration file is used, which prevents conflicts between multiple CNI plugins.

## Custom Image Service Configuration

Configure how the CRI handles images:

```yaml
machine:
  files:
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            [plugins."io.containerd.grpc.v1.cri".containerd]
              # Use overlayfs snapshotter
              snapshotter = "overlayfs"
              # Discard unpacked layers to save disk space
              discard_unpacked_layers = true
            [plugins."io.containerd.grpc.v1.cri".registry]
              [plugins."io.containerd.grpc.v1.cri".registry.mirrors]
                [plugins."io.containerd.grpc.v1.cri".registry.mirrors."docker.io"]
                  endpoint = ["https://mirror.example.com", "https://registry-1.docker.io"]
      path: /var/cri/conf.d/20-image-service.toml
      op: create
```

Note that for registry mirrors, the recommended approach in Talos is to use `machine.registries.mirrors` instead of configuring them in the CRI configuration. Talos translates those settings into the appropriate containerd configuration automatically.

## Pod Sandbox Configuration

Customize how pod sandboxes are created:

```yaml
machine:
  files:
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            sandbox_image = "registry.k8s.io/pause:3.10"
            [plugins."io.containerd.grpc.v1.cri".containerd]
              [plugins."io.containerd.grpc.v1.cri".containerd.runtimes]
                [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
                  runtime_type = "io.containerd.runc.v2"
                  sandbox_mode = "podsandbox"
                  [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
                    SystemdCgroup = true
      path: /var/cri/conf.d/20-sandbox.toml
      op: create
```

## Streaming Configuration

The CRI streaming server handles interactive operations like `kubectl exec` and `kubectl logs`:

```yaml
machine:
  files:
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            stream_server_address = "127.0.0.1"
            stream_server_port = "0"
            stream_idle_timeout = "4h"
            enable_tls_streaming = false
      path: /var/cri/conf.d/20-streaming.toml
      op: create
```

The `stream_idle_timeout` setting controls how long idle exec/attach sessions are kept alive. The default is 4 hours. Increase this for long-running debug sessions or decrease it to free up resources faster.

## Applying Custom CRI Configuration

Apply the configuration and verify it:

```bash
# Apply the machine configuration
talosctl apply-config --nodes 10.0.0.5 --file worker.yaml

# Verify containerd accepted the configuration
talosctl service containerd --nodes 10.0.0.5

# Read the final merged configuration
talosctl read /etc/containerd/config.toml --nodes 10.0.0.5

# Check for errors in containerd logs
talosctl logs containerd --nodes 10.0.0.5 | grep -i error
```

## Testing Custom Runtime Classes

After configuring custom runtimes, test them:

```bash
# Test the default runtime
kubectl run test-default --image=nginx:alpine --restart=Never
kubectl exec test-default -- uname -r

# Test gVisor runtime
kubectl run test-gvisor --image=nginx:alpine --restart=Never \
  --overrides='{"spec":{"runtimeClassName":"gvisor"}}'
kubectl exec test-gvisor -- uname -r
# Should show a different kernel version (gVisor's)

# Clean up
kubectl delete pod test-default test-gvisor
```

## Combining Multiple Configuration Fragments

You can split your customizations across multiple files for better organization:

```yaml
machine:
  files:
    # Runtime configuration
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            [plugins."io.containerd.grpc.v1.cri".containerd]
              [plugins."io.containerd.grpc.v1.cri".containerd.runtimes]
                [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.gvisor]
                  runtime_type = "io.containerd.runsc.v1"
      path: /var/cri/conf.d/20-runtimes.toml
      op: create
    # Performance settings
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            max_concurrent_downloads = 10
            max_container_log_line_size = 32768
      path: /var/cri/conf.d/30-performance.toml
      op: create
    # Sandbox settings
    - content: |
        [plugins]
          [plugins."io.containerd.grpc.v1.cri"]
            sandbox_image = "registry.k8s.io/pause:3.10"
      path: /var/cri/conf.d/40-sandbox.toml
      op: create
```

## Troubleshooting Custom CRI Configuration

When things go wrong, check these areas:

```bash
# Verify the configuration fragments exist
talosctl list /var/cri/conf.d/ --nodes 10.0.0.5

# Read a specific fragment
talosctl read /var/cri/conf.d/20-cri-custom.toml --nodes 10.0.0.5

# Check if containerd started successfully
talosctl service containerd --nodes 10.0.0.5

# Look for TOML parsing errors
talosctl logs containerd --nodes 10.0.0.5 | grep -i "toml\|parse\|config"

# Check kubelet can connect to CRI
talosctl logs kubelet --nodes 10.0.0.5 | grep -i "cri\|runtime"
```

The most common issue is TOML syntax errors. Make sure your brackets are properly nested and strings are quoted correctly.

## Conclusion

Custom CRI configuration in Talos Linux gives you control over how containers are created and managed at the node level. The configuration fragment approach keeps your changes modular and upgrade-safe. Focus on the customizations that matter for your workloads - alternative runtimes for security, performance tuning for throughput, and sandbox configuration for compliance. Test each change on a single node before rolling it out across the cluster.
