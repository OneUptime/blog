# How to Configure Provider Filesystem Mirror in Terraform

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Providers, Filesystem Mirror, Air-Gapped, Offline, Infrastructure as Code, DevOps

Description: Learn how to configure a Terraform provider filesystem mirror for air-gapped environments, CI/CD pipelines, and local development with step-by-step setup and maintenance procedures.

---

A filesystem mirror is a directory on the local filesystem that contains Terraform provider packages. When configured, Terraform reads provider binaries directly from this directory instead of downloading them from the internet. This is the go-to solution for air-gapped environments, offline development, and CI/CD pipelines that need deterministic, network-independent initialization.

This guide covers how to create, configure, and maintain a filesystem mirror from start to finish.

## Filesystem Mirror Directory Structure

Terraform expects the mirror directory to follow a specific layout. There are two supported formats: packed and unpacked.

### Packed Layout (Recommended)

The packed layout stores providers as zip files with JSON metadata:

```
/opt/terraform/providers/
  registry.terraform.io/
    hashicorp/
      aws/
        index.json
        5.30.0.json
        terraform-provider-aws_5.30.0_linux_amd64.zip
        terraform-provider-aws_5.30.0_darwin_arm64.zip
      random/
        index.json
        3.6.0.json
        terraform-provider-random_3.6.0_linux_amd64.zip
        terraform-provider-random_3.6.0_darwin_arm64.zip
```

The `index.json` file lists available versions:

```json
{
  "versions": {
    "5.30.0": {}
  }
}
```

The version-specific JSON file (e.g., `5.30.0.json`) lists available platforms and their hashes:

```json
{
  "archives": {
    "linux_amd64": {
      "url": "terraform-provider-aws_5.30.0_linux_amd64.zip",
      "hashes": [
        "zh:abc123..."
      ]
    },
    "darwin_arm64": {
      "url": "terraform-provider-aws_5.30.0_darwin_arm64.zip",
      "hashes": [
        "zh:def456..."
      ]
    }
  }
}
```

### Unpacked Layout

The unpacked layout stores provider binaries directly in platform-specific directories:

```
/opt/terraform/providers/
  registry.terraform.io/
    hashicorp/
      aws/
        5.30.0/
          linux_amd64/
            terraform-provider-aws_v5.30.0_x5
          darwin_arm64/
            terraform-provider-aws_v5.30.0_x5
```

The packed layout is generated by `terraform providers mirror` and is generally preferred because it includes hash verification metadata.

## Creating a Filesystem Mirror

### Using terraform providers mirror

The easiest way to create a mirror:

```bash
# Create the mirror directory
sudo mkdir -p /opt/terraform/providers

# Generate the mirror from your project
cd /path/to/your/terraform/project
terraform providers mirror /opt/terraform/providers

# For specific platforms
terraform providers mirror \
  -platform=linux_amd64 \
  -platform=darwin_arm64 \
  /opt/terraform/providers
```

### From Multiple Projects

If you have several Terraform projects that use different providers, mirror all of them:

```bash
#!/bin/bash
# create-mirror.sh - Create a comprehensive provider mirror

set -euo pipefail

MIRROR_DIR="/opt/terraform/providers"
PLATFORMS="linux_amd64 darwin_arm64"

# List of project directories
PROJECTS=(
  "/code/infrastructure/networking"
  "/code/infrastructure/database"
  "/code/infrastructure/compute"
  "/code/infrastructure/monitoring"
)

mkdir -p "$MIRROR_DIR"

for project in "${PROJECTS[@]}"; do
  if [ -d "$project" ]; then
    echo "Mirroring providers for: $project"
    cd "$project"

    for platform in $PLATFORMS; do
      terraform providers mirror -platform="$platform" "$MIRROR_DIR"
    done
  else
    echo "WARNING: Project directory not found: $project"
  fi
done

echo ""
echo "Mirror created at: $MIRROR_DIR"
echo "Contents:"
find "$MIRROR_DIR" -name "*.zip" -exec basename {} \; | sort -u
```

### Manual Download

For environments where you cannot run Terraform at all, download provider packages manually from releases.hashicorp.com:

```bash
#!/bin/bash
# manual-download.sh - Download provider packages manually

MIRROR_DIR="/opt/terraform/providers/registry.terraform.io"
PROVIDER="hashicorp/aws"
VERSION="5.30.0"
PLATFORMS=("linux_amd64" "darwin_arm64")

PROVIDER_DIR="$MIRROR_DIR/$PROVIDER"
mkdir -p "$PROVIDER_DIR"

for platform in "${PLATFORMS[@]}"; do
  OS=$(echo "$platform" | cut -d_ -f1)
  ARCH=$(echo "$platform" | cut -d_ -f2)

  FILENAME="terraform-provider-aws_${VERSION}_${OS}_${ARCH}.zip"
  URL="https://releases.hashicorp.com/terraform-provider-aws/${VERSION}/${FILENAME}"

  echo "Downloading: $FILENAME"
  curl -LO "$URL" -o "$PROVIDER_DIR/$FILENAME"

  # Download the SHA256 sums for verification
  curl -LO "https://releases.hashicorp.com/terraform-provider-aws/${VERSION}/terraform-provider-aws_${VERSION}_SHA256SUMS"
done

# Create index.json
cat > "$PROVIDER_DIR/index.json" <<EOF
{
  "versions": {
    "$VERSION": {}
  }
}
EOF

# Create version.json (simplified - use terraform providers mirror for proper hashes)
echo "NOTE: For proper hash verification, use 'terraform providers mirror' instead"
```

## Configuring Terraform to Use the Mirror

### CLI Configuration File

The primary way to configure a filesystem mirror:

```hcl
# ~/.terraformrc (Linux/Mac)
# %APPDATA%/terraform.rc (Windows)

provider_installation {
  filesystem_mirror {
    path    = "/opt/terraform/providers"
    include = ["registry.terraform.io/*/*"]
  }
}
```

### Environment Variable

You can also specify the config file path via environment variable:

```bash
# Set the config file location
export TF_CLI_CONFIG_FILE="/etc/terraform/terraformrc"

# Useful for CI/CD where you want a system-wide configuration
```

### Multiple Mirror Sources

Combine filesystem mirrors with other installation methods:

```hcl
# ~/.terraformrc - Multiple sources with fallback
provider_installation {
  # Check the local filesystem mirror first
  filesystem_mirror {
    path    = "/opt/terraform/providers"
    include = ["registry.terraform.io/hashicorp/*"]
  }

  # For non-HashiCorp providers, use a network mirror
  network_mirror {
    url     = "https://terraform-mirror.company.internal/"
    include = ["registry.terraform.io/DataDog/*"]
  }

  # Fall back to direct download for everything else
  direct {
    exclude = [
      "registry.terraform.io/hashicorp/*",
      "registry.terraform.io/DataDog/*"
    ]
  }
}
```

### Fully Air-Gapped Configuration

For environments with zero network access:

```hcl
# ~/.terraformrc - Air-gapped, no network access
provider_installation {
  filesystem_mirror {
    path    = "/opt/terraform/providers"
    include = ["*/*/*"]
  }
  # No 'direct' block - no network access will be attempted
}
```

## CI/CD Pipeline Integration

### GitHub Actions with Filesystem Mirror

```yaml
# .github/workflows/terraform.yml
name: Terraform

on:
  push:
    branches: [main]

jobs:
  apply:
    runs-on: self-hosted  # Uses internal runner with mirror pre-installed
    steps:
      - uses: actions/checkout@v4

      - name: Configure Terraform Mirror
        run: |
          mkdir -p ~/.terraform.d
          cat > ~/.terraformrc <<'EOF'
          provider_installation {
            filesystem_mirror {
              path    = "/opt/terraform/providers"
              include = ["*/*/*"]
            }
          }
          EOF

      - name: Terraform Init
        run: terraform init

      - name: Terraform Apply
        run: terraform apply -auto-approve
```

### Pre-populating Runner Cache

For cloud-hosted runners that need offline providers:

```yaml
# .github/workflows/terraform.yml
jobs:
  apply:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Download cached providers from artifact storage
      - name: Download Provider Cache
        uses: actions/cache@v4
        with:
          path: /tmp/terraform-providers
          key: terraform-providers-${{ hashFiles('**/.terraform.lock.hcl') }}

      - name: Configure Mirror
        run: |
          cat > ~/.terraformrc <<'EOF'
          provider_installation {
            filesystem_mirror {
              path    = "/tmp/terraform-providers"
              include = ["*/*/*"]
            }
            direct {}
          }
          EOF

      - name: Terraform Init
        run: |
          terraform init
          # Populate the cache for future runs
          terraform providers mirror \
            -platform=linux_amd64 \
            /tmp/terraform-providers
```

## Maintaining the Mirror

### Adding New Provider Versions

When you need a newer version of a provider:

```bash
# Update the version constraint in your configuration
# Then re-mirror

cd /path/to/project
# Edit versions.tf to update the constraint

terraform providers mirror \
  -platform=linux_amd64 \
  -platform=darwin_arm64 \
  /opt/terraform/providers

# The new version is added alongside existing ones
```

### Removing Old Versions

Clean up old provider versions to save disk space:

```bash
#!/bin/bash
# cleanup-mirror.sh - Remove old provider versions, keep the latest N

set -euo pipefail

MIRROR_DIR="/opt/terraform/providers/registry.terraform.io"
KEEP_VERSIONS=3

find "$MIRROR_DIR" -name "index.json" | while read -r index_file; do
  PROVIDER_DIR=$(dirname "$index_file")
  PROVIDER_NAME=$(echo "$PROVIDER_DIR" | sed "s|$MIRROR_DIR/||")

  # Get all versions sorted
  VERSIONS=$(jq -r '.versions | keys[]' "$index_file" | sort -V)
  VERSION_COUNT=$(echo "$VERSIONS" | wc -l)

  if [ "$VERSION_COUNT" -gt "$KEEP_VERSIONS" ]; then
    # Calculate how many to remove
    REMOVE_COUNT=$((VERSION_COUNT - KEEP_VERSIONS))

    echo "Provider $PROVIDER_NAME: $VERSION_COUNT versions, removing $REMOVE_COUNT"

    echo "$VERSIONS" | head -n "$REMOVE_COUNT" | while read -r version; do
      echo "  Removing version $version"
      rm -f "$PROVIDER_DIR"/*"_${version}_"*.zip
      rm -f "$PROVIDER_DIR/${version}.json"
    done

    # Update index.json to remove old versions
    KEEP_PATTERN=$(echo "$VERSIONS" | tail -n "$KEEP_VERSIONS" | jq -R -s 'split("\n") | map(select(length > 0))')
    jq --argjson keep "$KEEP_PATTERN" \
      '.versions |= with_entries(select(.key | IN($keep[])))' \
      "$index_file" > "${index_file}.tmp"
    mv "${index_file}.tmp" "$index_file"
  fi
done
```

### Verifying Mirror Completeness

Before deploying to an air-gapped environment, verify the mirror has everything needed:

```bash
#!/bin/bash
# verify-mirror.sh - Ensure the mirror has all required providers

set -euo pipefail

MIRROR_DIR="/opt/terraform/providers"
PROJECT_DIR=$1

echo "Verifying mirror completeness for: $PROJECT_DIR"

cd "$PROJECT_DIR"

# Set up temp config to use only the filesystem mirror
TMPRC=$(mktemp)
cat > "$TMPRC" <<EOF
provider_installation {
  filesystem_mirror {
    path    = "$MIRROR_DIR"
    include = ["*/*/*"]
  }
}
EOF

export TF_CLI_CONFIG_FILE="$TMPRC"

# Try to initialize - if it fails, something is missing
if terraform init -backend=false 2>&1; then
  echo "SUCCESS: All required providers are available in the mirror"
else
  echo "FAILURE: Some providers are missing from the mirror"
  echo "Check the error output above for details"
  exit 1
fi

rm "$TMPRC"
```

## Disk Space Considerations

Provider binaries are large. The AWS provider alone is over 300 MB per platform. Plan your storage accordingly:

```bash
# Check mirror size
du -sh /opt/terraform/providers

# Check per-provider sizes
du -sh /opt/terraform/providers/registry.terraform.io/hashicorp/*

# Example output:
# 1.2G  /opt/terraform/providers/registry.terraform.io/hashicorp/aws
# 450M  /opt/terraform/providers/registry.terraform.io/hashicorp/google
# 380M  /opt/terraform/providers/registry.terraform.io/hashicorp/azurerm
# 25M   /opt/terraform/providers/registry.terraform.io/hashicorp/kubernetes
# 15M   /opt/terraform/providers/registry.terraform.io/hashicorp/random
```

For a typical set of providers across two platforms, budget 3-5 GB of disk space. With multiple versions and more platforms, this can grow to 10-20 GB.

## Best Practices

1. **Use `terraform providers mirror` to generate the mirror.** It creates the correct directory structure and hash metadata automatically.
2. **Mirror for all target platforms.** If your team uses Mac but CI/CD runs Linux, include both.
3. **Test the mirror in isolation** before deploying to air-gapped environments. Set up a test config with only the filesystem mirror and verify `terraform init` succeeds.
4. **Automate mirror creation** as part of your release or deployment pipeline.
5. **Version the mirror contents.** Keep a manifest of what providers and versions are included.
6. **Clean up old versions** periodically to manage disk space.
7. **Store the mirror on fast storage.** Provider zips are large, and `terraform init` reads them during every initialization.
8. **Document the mirror location** in your team's runbooks so everyone knows where to find it and how to update it.

A well-maintained filesystem mirror makes Terraform operations predictable and independent of external services. For air-gapped environments, it is not optional - it is the only way Terraform can function.
