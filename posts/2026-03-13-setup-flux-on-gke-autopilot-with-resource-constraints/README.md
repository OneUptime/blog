# How to Set Up Flux on GKE Autopilot with Resource Constraints

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, Kubernetes, GitOps, GKE, GKE Autopilot, Google Cloud, Resource Constraints

Description: Configure Flux CD controllers to work within GKE Autopilot's resource constraints and node auto-provisioning for a fully managed GitOps setup.

---

## Introduction

GKE Autopilot is Google's fully managed Kubernetes mode where Google controls node provisioning, scaling, and patching. This hands-off model is excellent for reducing operational overhead, but it comes with strict resource requirements: every pod must declare both requests and limits, certain privilege escalations are blocked, and Autopilot enforces minimum CPU and memory values per pod class.

Flux CD's controllers are standard Kubernetes Deployments, so they run on Autopilot without modification — with one important caveat. The default Flux resource requests set during `flux bootstrap` are below Autopilot's minimum thresholds, causing controllers to be automatically adjusted or rejected. This guide shows how to patch the Flux controllers with Autopilot-compliant resource settings and bootstrap Flux correctly from the start.

Whether you are setting up a new Autopilot cluster or migrating an existing Flux installation, the techniques here will give you a stable, fully reconciling Flux installation that coexists cleanly with Autopilot's automation.

## Prerequisites

- A GKE Autopilot cluster (`gcloud container clusters create-auto my-autopilot-cluster --region us-central1`)
- `flux` CLI v2.x installed
- `kubectl` configured for the Autopilot cluster
- GitHub personal access token or SSH key for Flux bootstrap
- Kustomize knowledge for patching controller resources

## Step 1: Understand GKE Autopilot Resource Minimums

GKE Autopilot enforces minimum resource requests per container. As of GKE 1.30+:

- **Standard workload class**: minimum 250m CPU, 512Mi memory per container
- **Balanced workload class**: minimum 500m CPU, 512Mi memory
- Host network pods, privileged containers, and `hostPath` volumes are not permitted

Flux controllers default to much lower requests (e.g., 100m CPU), so they must be patched.

## Step 2: Create the Flux Namespace and Kustomize Patch Files

Before bootstrapping, prepare a directory with patches that Flux will apply to itself after installation.

```bash
# Create the staging directory for Flux patches
mkdir -p flux-system-patches
```

```yaml
# flux-system-patches/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - gotk-components.yaml   # Generated by flux bootstrap
  - gotk-sync.yaml         # Generated by flux bootstrap
patches:
  - path: patch-autopilot-resources.yaml
    target:
      kind: Deployment
      labelSelector: "app.kubernetes.io/part-of=flux"
```

```yaml
# flux-system-patches/patch-autopilot-resources.yaml
# Patch all Flux controller Deployments to meet Autopilot minimums
apiVersion: apps/v1
kind: Deployment
metadata:
  name: not-used       # Required field; target selector overrides this
  namespace: flux-system
spec:
  template:
    spec:
      containers:
        - name: manager
          resources:
            requests:
              cpu: "250m"      # Autopilot standard class minimum
              memory: "512Mi"
            limits:
              cpu: "1000m"
              memory: "1Gi"
```

## Step 3: Bootstrap Flux with the Custom Path

```bash
# Bootstrap Flux pointing to the directory that contains your patches
flux bootstrap github \
  --owner=your-org \
  --repository=fleet-infra \
  --branch=main \
  --path=clusters/autopilot-cluster \
  --personal \
  --components-extra=image-reflector-controller,image-automation-controller

# Flux will commit gotk-components.yaml and gotk-sync.yaml to your repo
# Then it will apply the Kustomization that patches its own Deployments
```

## Step 4: Verify Controllers are Running on Autopilot

```bash
# Watch controllers reach Running state (Autopilot may take ~2 minutes
# to provision nodes for the new workloads)
kubectl get pods -n flux-system --watch

# Confirm resource requests were applied correctly
kubectl get deployment -n flux-system source-controller \
  -o jsonpath='{.spec.template.spec.containers[0].resources}'

# Check Autopilot did not further modify the resources
kubectl describe pod -n flux-system -l app=source-controller | grep -A 10 "Requests:"
```

## Step 5: Configure Flux Sync for Your Applications

```yaml
# clusters/autopilot-cluster/apps/my-app/gitrepository.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 1m
  url: https://github.com/your-org/my-app
  ref:
    branch: main
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: my-app
  path: ./deploy/autopilot    # Autopilot-specific overlay with correct resources
  prune: true
  # Autopilot requires all pods to have resource requests
  postBuild:
    substitute:
      ENVIRONMENT: "autopilot"
```

## Step 6: Ensure Application Manifests are Autopilot-Compliant

```yaml
# deploy/autopilot/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: gcr.io/my-project/my-app:1.0.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "250m"     # Must be >= Autopilot minimum
              memory: "512Mi"
            limits:
              cpu: "500m"
              memory: "1Gi"
          # No privileged: true, no hostPath volumes
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
```

## Best Practices

- Use Kustomize overlays to maintain separate resource configurations for Autopilot and Standard cluster environments in the same repository.
- Set up Flux alerts (`Alert` and `Provider` resources) to notify your team when a controller pod fails to start due to resource admission rejection.
- GKE Autopilot automatically applies `cluster-autoscaler.kubernetes.io/safe-to-evict: "true"` annotations — account for this in pod disruption budgets.
- Avoid using `hostNetwork`, `hostPID`, `hostIPC`, or `privileged` in any manifest synced by Flux on Autopilot, as Autopilot's admission webhook will reject them.
- Use Workload Identity for all GCP API access from pods running on Autopilot rather than node service accounts.

## Conclusion

Running Flux CD on GKE Autopilot requires a small but important resource patch to satisfy Autopilot's minimum thresholds. Once the controllers are correctly sized, the GitOps workflow behaves identically to a Standard GKE cluster. You get the operational simplicity of Autopilot combined with the change management and auditability of Flux CD.
