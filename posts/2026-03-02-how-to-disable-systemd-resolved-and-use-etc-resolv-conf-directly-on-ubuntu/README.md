# How to Disable systemd-resolved and Use /etc/resolv.conf Directly on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, DNS, systemd-resolved, Networking, resolv.conf

Description: Step-by-step instructions for disabling systemd-resolved on Ubuntu and configuring /etc/resolv.conf for direct DNS resolution without the stub resolver.

---

`systemd-resolved` works well for most scenarios, but there are legitimate reasons to bypass it. Some environments require static DNS configuration that does not change based on network state. Certain software expects `/etc/resolv.conf` to contain actual DNS server IPs, not the `127.0.0.53` stub. And debugging DNS issues is sometimes easier when the resolver stack is simpler.

This guide covers how to disable `systemd-resolved` and manage `/etc/resolv.conf` directly.

## Why You Might Do This

- Your application requires real DNS server IPs in `/etc/resolv.conf`, not a loopback stub
- You are running your own caching resolver (like `unbound` or `dnsmasq`) and want to eliminate the extra hop
- Corporate or government compliance rules prohibit the stub resolver
- You are troubleshooting a DNS issue and want to eliminate `systemd-resolved` as a variable

If none of these apply to you, keep `systemd-resolved` running - it provides useful features like per-interface routing, DNS caching, and DNS-over-TLS.

## Understanding the Current State

Before making changes, understand what you have:

```bash
# Check if systemd-resolved is running
systemctl status systemd-resolved

# Check what resolv.conf currently is
ls -la /etc/resolv.conf

# See what it contains
cat /etc/resolv.conf
```

On a default Ubuntu installation, `resolv.conf` is a symlink:

```text
/etc/resolv.conf -> ../run/systemd/resolve/stub-resolv.conf
```

The file it points to contains the stub address:

```text
nameserver 127.0.0.53
options edns0 trust-ad
```

## Option 1: Disable Stub Resolver but Keep systemd-resolved Running

This is a middle ground. `systemd-resolved` continues to function (providing DNSSEC, per-interface routing, caching) but `/etc/resolv.conf` contains real DNS server IPs rather than the stub address.

Change the `resolv.conf` symlink to point to the resolved file without the stub:

```bash
# Remove the current symlink
sudo rm /etc/resolv.conf

# Link to the non-stub version generated by systemd-resolved
sudo ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf
```

The file at `/run/systemd/resolve/resolv.conf` contains the actual upstream DNS server IPs that `systemd-resolved` is using, not the loopback stub.

Check the result:

```bash
cat /etc/resolv.conf
# Should now show real DNS server IPs like 9.9.9.9 or 192.168.1.1
```

This approach keeps `systemd-resolved` running but puts real DNS IPs in `resolv.conf`. Applications that cannot work with `127.0.0.53` will now work correctly.

## Option 2: Fully Disable systemd-resolved

If you want to remove `systemd-resolved` from the picture entirely:

### Step 1: Stop and Disable the Service

```bash
# Stop systemd-resolved
sudo systemctl stop systemd-resolved

# Disable it so it does not start on boot
sudo systemctl disable systemd-resolved

# Verify it is stopped
systemctl status systemd-resolved
```

### Step 2: Remove the resolv.conf Symlink

```bash
# The current symlink points to a file managed by systemd-resolved
# Remove it
sudo rm /etc/resolv.conf
```

### Step 3: Create a Static resolv.conf

Write your DNS configuration directly:

```bash
sudo tee /etc/resolv.conf <<'EOF'
# DNS configuration managed manually - systemd-resolved is disabled
# Primary and secondary nameservers
nameserver 9.9.9.9
nameserver 149.112.112.112

# Search domains (append to bare hostnames)
search example.com corp.example.com

# Options
options edns0
EOF
```

Verify DNS resolution works:

```bash
# Test basic resolution
ping -c 2 google.com

# Use dig for more detail
dig google.com
```

### Step 4: Protect resolv.conf from Being Overwritten

Without `systemd-resolved`, other network management tools may try to overwrite `resolv.conf`. Make it immutable to prevent this:

```bash
# Make resolv.conf immutable (cannot be modified even by root)
sudo chattr +i /etc/resolv.conf

# Verify the immutable flag is set
lsattr /etc/resolv.conf
# Output: ----i---------e------- /etc/resolv.conf
```

To temporarily allow changes (for updates):

```bash
# Remove immutable flag
sudo chattr -i /etc/resolv.conf

# Make your changes
sudo nano /etc/resolv.conf

# Re-apply immutable flag
sudo chattr +i /etc/resolv.conf
```

## Preventing NetworkManager from Overwriting resolv.conf

If NetworkManager is installed, it may try to manage `resolv.conf`. Tell it not to:

```bash
# Check if NetworkManager is present
systemctl status NetworkManager

# If present, configure it to leave resolv.conf alone
sudo tee /etc/NetworkManager/conf.d/no-dns.conf <<'EOF'
[main]
dns=none
EOF

# Restart NetworkManager
sudo systemctl restart NetworkManager
```

The `dns=none` setting tells NetworkManager to not touch `/etc/resolv.conf`.

## Handling DHCP

If you use DHCP, the DHCP client may also update `resolv.conf` with DNS servers from the DHCP lease. Configure `dhclient` to not update it:

```bash
# Add to dhclient configuration
sudo tee -a /etc/dhcp/dhclient.conf <<'EOF'
# Do not update resolv.conf from DHCP
supersede domain-name-servers 9.9.9.9, 149.112.112.112;
supersede domain-search "example.com";
EOF
```

The `supersede` directive overrides the DHCP-provided values with your static configuration.

## Using Netplan Without systemd-resolved

If you use Netplan with the `networkd` renderer, note that Netplan's networkd backend uses `systemd-resolved` for DNS. If you disable `systemd-resolved`, configure DNS at the `networkd` level via a network drop-in:

```bash
# Create a systemd-networkd drop-in that configures DNS without resolved
sudo mkdir -p /etc/systemd/network/
sudo tee /etc/systemd/network/20-eth0.network <<'EOF'
[Match]
Name=eth0

[Network]
Address=192.168.1.100/24
Gateway=192.168.1.1
# DNS for this interface
DNS=9.9.9.9 149.112.112.112
Domains=example.com
EOF
```

Then let networkd write `resolv.conf` directly:

```bash
# Configure networkd to write resolv.conf
sudo tee /etc/systemd/network/10-resolv.conf <<'EOF'
# This is handled by networkd directly since resolved is disabled
EOF

# Ensure networkd is configured as the resolver
sudo ln -sf /run/systemd/network/resolv.conf /etc/resolv.conf
```

## Reverting the Changes

If you need to re-enable `systemd-resolved`:

```bash
# Remove immutable flag if set
sudo chattr -i /etc/resolv.conf

# Re-enable systemd-resolved
sudo systemctl enable --now systemd-resolved

# Remove the manual resolv.conf and restore the symlink
sudo rm /etc/resolv.conf
sudo ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf

# Verify
cat /etc/resolv.conf
systemctl status systemd-resolved
```

## Validation

After any DNS configuration change:

```bash
# Basic resolution test
ping -c 2 google.com

# More detailed test
dig example.com @9.9.9.9  # bypasses local config, queries directly
dig example.com             # uses /etc/resolv.conf

# Check what server is being used
dig +stats example.com | grep "SERVER:"

# Test search domain
ping -c 2 webserver          # should append search domains and resolve
```

The direct `resolv.conf` approach is simpler but gives up the benefits of `systemd-resolved`. Make the choice based on your actual requirements rather than avoiding complexity for its own sake.
