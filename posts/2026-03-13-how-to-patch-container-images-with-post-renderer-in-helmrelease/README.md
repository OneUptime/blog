# How to Patch Container Images with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderers, Container Images, Kustomize

Description: Learn how to use Flux post-renderers to override container images in Helm chart resources for custom registries and image versions.

---

## Introduction

Container image management is a critical part of Kubernetes deployments. Organizations often need to pull images from private registries, use internally mirrored images, or override specific image tags for security compliance. While many Helm charts expose image configuration through values, some charts hardcode images for sidecars, init containers, or helper pods that cannot be overridden through values alone.

Flux post-renderers allow you to patch container images in any resource generated by a Helm chart, regardless of whether the chart exposes those images as configurable values. Using Kustomize image transformers or strategic merge patches, you can replace image repositories, tags, and digests without modifying the chart itself.

This guide covers multiple techniques for patching container images using Flux post-renderers, from simple image overrides to complex multi-container scenarios.

## Prerequisites

Before proceeding, ensure you have:

- A Kubernetes cluster with Flux CD installed
- A HelmRelease deployed through Flux
- kubectl access to the cluster
- A private container registry (if mirroring images)

## Using Kustomize Images Transformer

The most straightforward way to override images is using the Kustomize `images` transformer in the post-renderer:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        images:
          - name: docker.io/library/nginx
            newName: registry.internal.company.com/nginx
            newTag: "1.25-alpine"
```

This replaces any occurrence of `docker.io/library/nginx` with `registry.internal.company.com/nginx:1.25-alpine` across all resources rendered by the chart.

## Overriding Multiple Images

Charts often use multiple images. You can override all of them in a single post-renderer:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "3.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        images:
          - name: my-app
            newName: registry.internal.company.com/my-app
            newTag: "3.0.0-patched"
          - name: redis
            newName: registry.internal.company.com/redis
            newTag: "7.2-alpine"
          - name: busybox
            newName: registry.internal.company.com/busybox
            newTag: "1.36"
```

Each entry in the `images` list matches the original image name and replaces it with the new name and tag.

## Using Image Digests for Security

For security-sensitive environments, you can pin images to specific digests instead of tags:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        images:
          - name: my-app
            newName: registry.internal.company.com/my-app
            digest: "sha256:a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2"
```

Using digests ensures that the exact image binary is used, preventing tag mutation attacks.

## Patching Images with Strategic Merge Patch

For more targeted image overrides, use strategic merge patches to modify specific containers:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    containers:
                      - name: my-app
                        image: registry.internal.company.com/my-app:2.0.0-hardened
                      - name: sidecar
                        image: registry.internal.company.com/sidecar:latest
```

This approach gives you precise control over which containers in which resources get their images overridden.

## Redirecting to a Mirror Registry

A common enterprise requirement is to redirect all images to an internal mirror registry. You can combine the images transformer with a consistent naming convention:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: prometheus
  namespace: monitoring
spec:
  interval: 5m
  chart:
    spec:
      chart: kube-prometheus-stack
      version: "56.0.0"
      sourceRef:
        kind: HelmRepository
        name: prometheus-community
        namespace: flux-system
  postRenderers:
    - kustomize:
        images:
          - name: quay.io/prometheus/prometheus
            newName: mirror.internal.company.com/prometheus/prometheus
          - name: quay.io/prometheus/alertmanager
            newName: mirror.internal.company.com/prometheus/alertmanager
          - name: grafana/grafana
            newName: mirror.internal.company.com/grafana/grafana
          - name: quay.io/prometheus-operator/prometheus-operator
            newName: mirror.internal.company.com/prometheus-operator/prometheus-operator
```

## Combining with Flux Image Automation

Flux Image Automation can automatically update image tags in your Git repository. To combine this with post-renderers, set up image policies:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        images:
          - name: my-app
            newName: registry.internal.company.com/my-app
            newTag: "2.0.0" # {"$imagepolicy": "production:my-app:tag"}
```

The image policy marker allows Flux Image Automation to update the tag automatically when new images are pushed to the registry.

## Verifying Image Overrides

After deploying, verify that images were correctly overridden:

```bash
kubectl get deployment my-app -n production -o jsonpath='{.spec.template.spec.containers[*].image}'
```

For all pods in the namespace:

```bash
kubectl get pods -n production -o jsonpath='{range .items[*]}{.metadata.name}: {range .spec.containers[*]}{.image} {end}{"\n"}{end}'
```

## Conclusion

Flux post-renderers provide a flexible mechanism for overriding container images in Helm chart resources. Whether you need to redirect images to a private registry, pin images to specific digests for security, or override hardcoded images that charts do not expose as configurable values, post-renderers handle all these scenarios without requiring chart modifications. Use the Kustomize images transformer for broad image replacements and strategic merge patches for targeted container-level overrides.
