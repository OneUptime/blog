# How to Understand Kernel Ring Buffer and dmesg on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, Linux, Kernel, Debugging, System Administration

Description: Learn how to read and interpret the kernel ring buffer using dmesg on Ubuntu, understand message formats, filter by priority, and use journald for persistent kernel logs.

---

When hardware misbehaves, a driver crashes, a filesystem reports errors, or an OOM kill happens, the kernel records these events in its ring buffer. The `dmesg` command is your primary tool for reading this log. Understanding what's in it - and how to filter and interpret it - is an essential skill for diagnosing Linux system problems.

## What is the Kernel Ring Buffer?

The kernel ring buffer is a circular buffer in kernel memory that stores log messages generated by the kernel itself - including messages from device drivers, filesystem code, network stack, and other kernel subsystems. It's called a "ring buffer" because once it fills up, new messages overwrite the oldest ones.

The buffer size is set at compile time (typically 512KB to 4MB on modern kernels):

```bash
# Check ring buffer size
dmesg -S
# or
cat /proc/sys/kernel/printk_devkmsg
```

Because it's stored in memory, the kernel ring buffer is lost on reboot - unless you configure a persistent log.

## Using dmesg

```bash
# View all kernel messages
dmesg

# With human-readable timestamps (preferred)
dmesg -T
# Output: [Mon Mar  2 10:23:45 2026] kernel message here

# With relative timestamps (seconds since boot)
dmesg -t
# Output: [    0.000000] Kernel version 6.8.0...

# Continuous output (like tail -f)
dmesg -W
# or
dmesg --follow

# Clear the ring buffer (requires root)
sudo dmesg -C
# or just clear (but still shows new messages)
sudo dmesg -c
```

## Filtering by Log Level

The kernel uses numeric log levels (0-7). Lower numbers are more severe:

| Level | Name | Meaning |
|-------|------|---------|
| 0 | emerg | System is unusable |
| 1 | alert | Action must be taken immediately |
| 2 | crit | Critical conditions |
| 3 | err | Error conditions |
| 4 | warn | Warning conditions |
| 5 | notice | Normal but significant |
| 6 | info | Informational |
| 7 | debug | Debug-level messages |

```bash
# Show only errors and above (levels 0-3)
dmesg --level=emerg,alert,crit,err

# Shorter form
dmesg -l err,warn

# Show warnings and errors
dmesg --level=err,warn

# Show only critical and above
dmesg -l emerg,alert,crit

# Combine with timestamps
dmesg -T -l err,warn
```

## Filtering by Facility

Messages come from different kernel subsystems (facilities):

```bash
# Common facilities: kern, user, mail, daemon, syslog, lpr, news
dmesg -f kern    # kernel messages only
dmesg -f daemon  # daemon messages

# Combine level and facility
dmesg -l err -f kern
```

## Reading Specific Types of Messages

### Hardware Detection at Boot

```bash
# USB devices detected
dmesg -T | grep -i usb

# PCI devices
dmesg -T | grep -i pci | head -20

# SATA/NVMe drives
dmesg -T | grep -E "ata[0-9]|nvme|sd[a-z]"

# GPU/display
dmesg -T | grep -iE "drm|gpu|radeon|nvidia|i915"

# Network interfaces
dmesg -T | grep -iE "eth[0-9]|ens[0-9]|enp[0-9]|wlan[0-9]"
```

### Errors and Warnings

```bash
# All errors from this boot
sudo dmesg -T -l err

# Segfaults and kernel oops
dmesg -T | grep -iE "segfault|oops|panic|bug:"

# I/O errors (disk problems)
dmesg -T | grep -iE "i/o error|ata.*error|blk_update_request"

# Memory errors
dmesg -T | grep -iE "mce|memory error|edac|ecc"

# OOM kills
dmesg -T | grep -iE "oom|killed process|out of memory"

# Filesystem errors
dmesg -T | grep -iE "ext4|btrfs|xfs.*error|corrupt"
```

### Driver Issues

```bash
# Failed module loads
dmesg -T | grep -iE "failed|error|cannot"

# Firmware loading
dmesg -T | grep -i firmware

# Device resets (often indicates hardware instability)
dmesg -T | grep -iE "reset|timeout|hung"
```

## Understanding the Message Format

A typical dmesg line looks like:

```
[    3.452901] EXT4-fs (sda1): mounted filesystem without journal. Opts: (null)
```

- `[    3.452901]` - Seconds since kernel start (or a timestamp with `-T`)
- `EXT4-fs (sda1)` - Subsystem and context (filesystem, device)
- The rest is the message itself

Messages in brackets are often prefixed with the kernel module name or subsystem:
- `[drm]` - Direct Rendering Manager (GPU)
- `[UFW BLOCK]` - Firewall (iptables)
- `EXT4-fs` - ext4 filesystem driver

## Persistent Kernel Logs with journald

By default on Ubuntu, journald captures kernel messages and persists them across reboots. This is far more useful than the volatile ring buffer.

```bash
# View kernel messages via journald
journalctl -k

# Kernel messages with timestamps (default)
journalctl -k -o short-precise

# Kernel messages from previous boot
journalctl -k -b -1

# Kernel messages from two boots ago
journalctl -k -b -2

# List available boot logs
journalctl --list-boots

# Kernel errors from last boot
journalctl -k -b -1 -p err

# Kernel messages since specific time
journalctl -k --since "2026-03-02 09:00:00"

# Follow kernel messages in real time
journalctl -k -f
```

### Configuring journald Persistence

```bash
# Check if journal is persistent
ls /var/log/journal/

# If directory doesn't exist, create it to enable persistence
sudo mkdir -p /var/log/journal
sudo systemd-tmpfiles --create --prefix /var/log/journal
sudo systemctl restart systemd-journald

# Configure journal size
sudo nano /etc/systemd/journald.conf
```

```ini
[Journal]
Storage=persistent
SystemMaxUse=500M
SystemKeepFree=1G
MaxRetentionSec=30day
```

## Configuring the Ring Buffer Size

For systems with heavy logging (high I/O, many devices), you might want a larger ring buffer:

```bash
# Current ring buffer size
dmesg -S 2>/dev/null || cat /sys/module/printk/parameters/printk_devkmsg

# Change size at boot via kernel parameter
sudo nano /etc/default/grub
# Add log_buf_len=16M to GRUB_CMDLINE_LINUX_DEFAULT
sudo update-grub
```

## Controlling Kernel Log Level

```bash
# Current printk levels
cat /proc/sys/kernel/printk
# Output: 4  4  1  7
# Fields: current  default  minimum  boot-time

# First field is the console log level:
# Messages at this level and below appear on the console
# 4 = warnings and above show on console

# Show debug messages on console (noisy!)
echo 7 | sudo tee /proc/sys/kernel/printk

# Show only critical errors on console
echo 3 | sudo tee /proc/sys/kernel/printk

# Permanent setting
echo "kernel.printk = 4 4 1 7" | sudo tee /etc/sysctl.d/99-printk.conf
```

## Practical Diagnostic Examples

### Diagnosing a Disk Failure

```bash
# Look for I/O errors
dmesg -T | grep -E "sd[a-z]|ata[0-9]" | grep -i "error\|failed\|reset\|timeout"

# Count error types
dmesg | grep -i "ata.*error" | sort | uniq -c | sort -rn

# Check SMART status alongside dmesg
sudo smartctl -H /dev/sda
sudo dmesg -T | grep sda
```

### Diagnosing a Memory Error

```bash
# Hardware memory errors (requires ECC RAM and edac driver)
dmesg -T | grep -iE "edac|mce|memory error"

# OOM kills
dmesg -T | grep -A5 "Out of memory"
# Shows the OOM killer's selection process and which process was killed

# High memory pressure
dmesg -T | grep -i "lowmem\|swapper"
```

### Diagnosing USB Device Issues

```bash
# USB disconnect/reconnect loop (indicates bad cable or power)
dmesg -T | grep -i "usb.*disconnect\|usb.*new"

# USB errors
dmesg -T | grep -i "usb" | grep -i "error\|failed"

# USB power issues
dmesg -T | grep -iE "usb.*power|over.current"
```

### Script: Boot Health Check

```bash
#!/bin/bash
# Quick kernel health check from ring buffer

echo "=== Kernel Health Check ==="
echo ""

echo "Boot time:"
uptime -p

echo ""
echo "--- Critical/Error Messages ---"
dmesg -T -l emerg,alert,crit,err | tail -20

echo ""
echo "--- Recent Warnings ---"
dmesg -T -l warn | tail -10

echo ""
echo "--- OOM Events ---"
dmesg | grep -i "oom\|killed process" | tail -5

echo ""
echo "--- Disk Errors ---"
dmesg -T | grep -iE "i/o error|blk_update_request" | tail -10

echo ""
echo "--- Hardware Errors ---"
dmesg -T | grep -iE "mce|edac|hardware error" | tail -5
```

The kernel ring buffer and dmesg are your first stops when diagnosing hardware problems, driver failures, or kernel panics. Combined with journald's persistent kernel logs, you have a complete history of what the kernel has been doing since your last reboot - or across multiple reboots with proper configuration. Reading these logs regularly is how experienced sysadmins catch hardware problems before they become outages.
