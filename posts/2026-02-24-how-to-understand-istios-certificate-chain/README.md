# How to Understand Istio's Certificate Chain

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Certificate, TLS, mTLS, Security

Description: A practical guide to understanding the certificate chain in Istio including root CAs, intermediate CAs, and workload certificates.

---

Every mTLS connection in Istio relies on a chain of certificates. When two services communicate, each side presents its certificate, and the other side validates it by walking up the chain to a trusted root. Understanding this chain - how it's structured, how each link is created, and how validation works - is essential for debugging TLS errors and setting up custom CA integrations.

## What Is a Certificate Chain

A certificate chain (also called a chain of trust) is a sequence of certificates where each certificate is signed by the next one in the chain, ending at a trusted root:

```text
Workload Certificate (leaf)
  └── Signed by: Intermediate CA Certificate
        └── Signed by: Root CA Certificate (self-signed, trusted)
```

When Envoy presents its certificate during a TLS handshake, it sends the full chain (minus the root, which the verifier already has). The verifier checks:

1. The leaf certificate is signed by the intermediate CA
2. The intermediate CA certificate is signed by the root CA
3. The root CA is in its trusted store
4. None of the certificates are expired
5. None of the certificates are revoked (if CRL/OCSP is configured)

## Istio's Default Chain

By default, Istio creates a two-level chain:

```text
Self-signed Root CA (generated by Istiod)
  └── Workload Certificate (generated per-pod)
```

There's no intermediate CA in the default setup. Istiod generates a self-signed root certificate and uses it directly to sign workload certificates.

You can inspect this chain:

```bash
# View the root certificate
kubectl get configmap istio-ca-root-cert -n default -o jsonpath='{.data.root-cert\.pem}' | \
  openssl x509 -text -noout

# View a workload's certificate
istioctl proxy-config secret my-pod -o json | \
  python3 -c "
import json, sys, base64, subprocess
data = json.load(sys.stdin)
for s in data.get('dynamicActiveSecrets', []):
    if s['name'] == 'default':
        cert_data = s['secret']['tlsCertificate']['certificateChain']['inlineBytes']
        cert_pem = base64.b64decode(cert_data)
        proc = subprocess.run(['openssl', 'x509', '-text', '-noout'],
                            input=cert_pem, capture_output=True)
        print(proc.stdout.decode())
"
```

## Adding an Intermediate CA

For production, you should use a three-level chain with your organization's root CA at the top:

```text
Organization Root CA (offline, highly secured)
  └── Istio Intermediate CA (used by Istiod to sign workload certs)
        └── Workload Certificate (per-pod)
```

This is better because:
- The root CA key stays offline and safe
- The intermediate CA can be rotated without changing the root trust
- Certificate revocation is simpler at the intermediate level

### Setting Up the Chain

First, create the intermediate CA certificate signed by your root CA:

```bash
# Generate intermediate CA key
openssl genrsa -out ca-key.pem 4096

# Create CSR for intermediate CA
openssl req -new -key ca-key.pem -out ca-csr.pem \
  -subj "/O=my-org/CN=Istio Intermediate CA"

# Sign with root CA
openssl x509 -req -in ca-csr.pem \
  -CA root-cert.pem -CAkey root-key.pem \
  -CAcreateserial \
  -out ca-cert.pem \
  -days 3650 \
  -extfile <(printf "basicConstraints=critical,CA:TRUE\nkeyUsage=critical,keyCertSign,cRLSign")
```

Create the cert chain file (intermediate + root):

```bash
cat ca-cert.pem root-cert.pem > cert-chain.pem
```

Load the certificates into Kubernetes:

```bash
kubectl create secret generic cacerts -n istio-system \
  --from-file=ca-cert.pem \
  --from-file=ca-key.pem \
  --from-file=root-cert.pem \
  --from-file=cert-chain.pem
```

Restart Istiod to pick up the new CA:

```bash
kubectl rollout restart deployment istiod -n istio-system
```

### Verifying the Chain

After setup, verify that workload certificates are signed by your intermediate CA:

```bash
# Get the workload certificate
istioctl proxy-config secret my-pod -o json | \
  python3 -c "
import json, sys, base64
data = json.load(sys.stdin)
for s in data.get('dynamicActiveSecrets', []):
    if s['name'] == 'default':
        cert_data = s['secret']['tlsCertificate']['certificateChain']['inlineBytes']
        print(base64.b64decode(cert_data).decode())
" > /tmp/workload-chain.pem

# Verify the chain
openssl verify -CAfile root-cert.pem -untrusted ca-cert.pem /tmp/workload-chain.pem
```

You should see `OK` if the chain is valid.

## Certificate Fields

Let's look at what's inside a typical Istio workload certificate:

```bash
istioctl proxy-config secret my-pod -o json | \
  python3 -c "
import json, sys, base64, subprocess
data = json.load(sys.stdin)
for s in data.get('dynamicActiveSecrets', []):
    if s['name'] == 'default':
        cert_data = s['secret']['tlsCertificate']['certificateChain']['inlineBytes']
        cert_pem = base64.b64decode(cert_data)
        proc = subprocess.run(['openssl', 'x509', '-text', '-noout'],
                            input=cert_pem, capture_output=True)
        print(proc.stdout.decode())
"
```

Key fields:

- **Subject** - Empty (Istio uses SAN instead)
- **Issuer** - The CA that signed the certificate (Istiod or your intermediate CA)
- **Subject Alternative Name (SAN)** - Contains the SPIFFE URI: `spiffe://cluster.local/ns/default/sa/my-service`
- **Key Usage** - Digital Signature, Key Encipherment
- **Extended Key Usage** - TLS Web Server Authentication, TLS Web Client Authentication
- **Validity** - Not Before and Not After (default 24 hours)

The SAN is the most important field. It carries the workload's SPIFFE identity, which authorization policies use for access control.

## How Chain Validation Works During mTLS

When pod A connects to pod B with mTLS:

1. **A initiates TLS handshake** - Sends ClientHello
2. **B responds** - Sends ServerHello and its certificate chain
3. **A validates B's chain** - Checks the chain against its trusted root (from istio-ca-root-cert ConfigMap)
4. **A sends its certificate** - Client certificate for mutual authentication
5. **B validates A's chain** - Same process in reverse
6. **Both extract SPIFFE identity** - Read the SAN from the peer's certificate
7. **Connection established** - Both sides know each other's identity

If any validation step fails, the connection is rejected with a TLS error.

## Debugging Certificate Chain Issues

Common chain problems and how to diagnose them:

**Chain is incomplete:**

```bash
# Check that the certificate chain includes all intermediate certs
istioctl proxy-config secret my-pod -o json | \
  python3 -c "
import json, sys, base64
data = json.load(sys.stdin)
for s in data.get('dynamicActiveSecrets', []):
    if s['name'] == 'default':
        cert_data = s['secret']['tlsCertificate']['certificateChain']['inlineBytes']
        certs = base64.b64decode(cert_data).decode()
        count = certs.count('BEGIN CERTIFICATE')
        print(f'Certificates in chain: {count}')
"
```

If you see only 1 certificate and you have an intermediate CA, the chain is incomplete.

**Root certificate mismatch:**

```bash
# Compare root certs across namespaces
diff <(kubectl get cm istio-ca-root-cert -n default -o jsonpath='{.data.root-cert\.pem}') \
     <(kubectl get cm istio-ca-root-cert -n other-ns -o jsonpath='{.data.root-cert\.pem}')
```

If they differ, some namespaces might have stale root certificates.

**Expired certificates:**

```bash
# Check expiration across all pods
for pod in $(kubectl get pods -o name); do
  echo "=== $pod ==="
  istioctl proxy-config secret ${pod#pod/} 2>/dev/null | grep -v NAME
done
```

**TLS handshake failures:**

```bash
# Enable TLS debug logging
istioctl proxy-config log my-pod --level tls:debug

# Check for handshake errors
kubectl logs my-pod -c istio-proxy | grep -i "tls\|handshake\|certificate" | tail -20

# Reset log level
istioctl proxy-config log my-pod --level warning
```

## Rotating the CA Certificate

When your intermediate CA certificate is approaching expiration, you need to rotate it. This is a multi-step process:

1. Generate a new intermediate CA certificate signed by the same root
2. Update the `cacerts` secret with the new certificate
3. Restart Istiod to load the new CA
4. Wait for all workload certificates to be re-issued (happens automatically within 12 hours for 24-hour certs)

```bash
# Update the secret
kubectl create secret generic cacerts -n istio-system \
  --from-file=ca-cert.pem=new-ca-cert.pem \
  --from-file=ca-key.pem=new-ca-key.pem \
  --from-file=root-cert.pem=root-cert.pem \
  --from-file=cert-chain.pem=new-cert-chain.pem \
  --dry-run=client -o yaml | kubectl apply -f -

# Restart Istiod
kubectl rollout restart deployment istiod -n istio-system
```

As long as both the old and new intermediate CAs are signed by the same root, the rotation is seamless - workloads with old certificates can still communicate with workloads that have new certificates.

The certificate chain is the foundation of trust in your mesh. Get it right, monitor it consistently, and plan for rotations well in advance.
