# How to Back Up and Restore GRUB Configuration on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, GRUB, Backup, System Recovery

Description: Learn how to back up and restore GRUB bootloader configuration on Ubuntu to recover from boot failures, disk replacements, and system migrations.

---

GRUB (Grand Unified Bootloader) is the first substantial piece of software that runs when your Ubuntu server powers on. If GRUB is misconfigured or its data on disk is corrupted, the server will not boot. This is particularly painful on remote servers where you might not have easy console access. Keeping a backup of your GRUB configuration means you can restore a working bootloader quickly, whether you are recovering from a failed disk, migrating to new hardware, or undoing a bad configuration change.

GRUB has two distinct pieces: the configuration file at `/boot/grub/grub.cfg` and the binary data written to the Master Boot Record (MBR) or the EFI System Partition. Backing up both is important.

## Understanding GRUB's Components

### GRUB Configuration File

`/boot/grub/grub.cfg` is the main GRUB configuration. It contains the boot menu entries, kernel parameters, and display settings. This file is auto-generated by `update-grub` from templates in `/etc/grub.d/` and settings in `/etc/default/grub`. You should back up both the generated file and its sources.

### GRUB Stage Binaries

On BIOS systems, GRUB writes binary data into:
- The MBR (first 446 bytes of the disk)
- The post-MBR gap (sectors between MBR and first partition)
- The `/boot/grub/` directory on the filesystem

On UEFI systems, GRUB installs to:
- The EFI System Partition (usually `/boot/efi`)
- Typically at `/boot/efi/EFI/ubuntu/grubx64.efi`

## Identifying Your Boot Mode

Before backing up, determine whether you are on a BIOS or UEFI system:

```bash
# UEFI systems have this directory; BIOS systems do not
ls /sys/firmware/efi 2>/dev/null && echo "UEFI" || echo "BIOS"

# Check if EFI partition is mounted
df -h /boot/efi 2>/dev/null

# Check efibootmgr for UEFI boot entries
which efibootmgr && sudo efibootmgr -v 2>/dev/null | head -20
```

## Backing Up GRUB Configuration Files

### Backup Sources (the templates)

The templates that generate `grub.cfg` are more important to back up than the generated file, since the generated file can always be recreated from the templates:

```bash
# Create a backup directory with a timestamp
BACKUP_DIR="/root/grub-backup-$(date '+%Y%m%d-%H%M%S')"
sudo mkdir -p "$BACKUP_DIR"

# Back up the GRUB configuration directory
sudo cp -a /etc/default/grub "$BACKUP_DIR/"
sudo cp -a /etc/grub.d/ "$BACKUP_DIR/grub.d/"

# Back up the generated grub.cfg
sudo cp /boot/grub/grub.cfg "$BACKUP_DIR/"

# Save the list of installed kernels (useful for reference)
dpkg -l 'linux-image-*' | sudo tee "$BACKUP_DIR/installed-kernels.txt" > /dev/null

# Record the disk layout
lsblk -f | sudo tee "$BACKUP_DIR/disk-layout.txt" > /dev/null
sudo fdisk -l | sudo tee "$BACKUP_DIR/partition-table.txt" > /dev/null

echo "GRUB config backed up to $BACKUP_DIR"
ls -la "$BACKUP_DIR/"
```

### Backing Up for BIOS Systems (MBR)

On BIOS/MBR systems, back up the MBR and the GRUB stage data:

```bash
# Identify the boot disk
lsblk -d -o NAME,SIZE,MODEL | head -10
# Typically /dev/sda or /dev/nvme0n1

BOOT_DISK="/dev/sda"  # Adjust for your system

# Back up just the MBR (first 512 bytes)
sudo dd if=$BOOT_DISK of="$BACKUP_DIR/mbr-backup.bin" bs=512 count=1
echo "MBR backed up: $(ls -lh $BACKUP_DIR/mbr-backup.bin)"

# Back up a larger chunk including GRUB stage 1.5 data
# This covers the MBR + the post-MBR gap typically used by GRUB
sudo dd if=$BOOT_DISK of="$BACKUP_DIR/mbr-extended-backup.bin" bs=512 count=2048
echo "Extended MBR backup: $(ls -lh $BACKUP_DIR/mbr-extended-backup.bin)"
```

### Backing Up for UEFI Systems

```bash
# Back up the entire EFI System Partition
sudo mkdir -p "$BACKUP_DIR/efi"
sudo cp -a /boot/efi/EFI/ "$BACKUP_DIR/efi/"

# Record EFI boot entries
sudo efibootmgr -v > "$BACKUP_DIR/efi-boot-entries.txt"

echo "EFI backup contents:"
ls -la "$BACKUP_DIR/efi/EFI/"
```

## Creating a Comprehensive Backup Script

```bash
sudo nano /usr/local/bin/backup-grub.sh
```

```bash
#!/bin/bash
# GRUB backup script
# Creates a timestamped backup of all GRUB configuration and boot data

set -e

BACKUP_BASE="/root/grub-backups"
TIMESTAMP=$(date '+%Y%m%d-%H%M%S')
BACKUP_DIR="${BACKUP_BASE}/${TIMESTAMP}"
LOG="${BACKUP_BASE}/backup.log"

# Detect boot mode
if ls /sys/firmware/efi &>/dev/null; then
    BOOT_MODE="uefi"
else
    BOOT_MODE="bios"
fi

# Find the primary boot disk
BOOT_DISK=$(lsblk -ndo PKNAME $(df /boot | tail -1 | awk '{print $1}') 2>/dev/null)
if [ -z "$BOOT_DISK" ]; then
    BOOT_DISK=$(lsblk -d -no NAME | head -1)
fi

mkdir -p "$BACKUP_DIR"

echo "$(date): Starting GRUB backup to $BACKUP_DIR" | tee -a "$LOG"
echo "  Boot mode: $BOOT_MODE" | tee -a "$LOG"
echo "  Boot disk: /dev/$BOOT_DISK" | tee -a "$LOG"

# Always back up configuration sources
cp /etc/default/grub "$BACKUP_DIR/"
cp -a /etc/grub.d/ "$BACKUP_DIR/grub.d/"
cp /boot/grub/grub.cfg "$BACKUP_DIR/" 2>/dev/null || true

# System information
lsblk -f > "$BACKUP_DIR/block-devices.txt"
cat /proc/cmdline > "$BACKUP_DIR/current-kernel-cmdline.txt"
dpkg -l 'linux-image-*' > "$BACKUP_DIR/installed-kernels.txt"

if [ "$BOOT_MODE" = "bios" ]; then
    # Back up MBR and GRUB stage data
    dd if=/dev/$BOOT_DISK of="$BACKUP_DIR/mbr.bin" bs=512 count=1 2>/dev/null
    dd if=/dev/$BOOT_DISK of="$BACKUP_DIR/mbr-full.bin" bs=512 count=2048 2>/dev/null
    fdisk -l /dev/$BOOT_DISK > "$BACKUP_DIR/partition-table.txt"
elif [ "$BOOT_MODE" = "uefi" ]; then
    # Back up EFI partition
    mkdir -p "$BACKUP_DIR/efi"
    cp -a /boot/efi/EFI/ "$BACKUP_DIR/efi/" 2>/dev/null || true
    efibootmgr -v > "$BACKUP_DIR/efi-entries.txt" 2>/dev/null || true
fi

# Create a compressed archive
tar -czf "${BACKUP_BASE}/${TIMESTAMP}-grub-backup.tar.gz" -C "$BACKUP_BASE" "$TIMESTAMP"
rm -rf "$BACKUP_DIR"

echo "$(date): Backup complete: ${BACKUP_BASE}/${TIMESTAMP}-grub-backup.tar.gz" | tee -a "$LOG"

# Retain only the last 10 backups
ls -t "${BACKUP_BASE}"/*.tar.gz 2>/dev/null | tail -n +11 | xargs rm -f 2>/dev/null || true
```

```bash
sudo chmod +x /usr/local/bin/backup-grub.sh

# Test the backup
sudo /usr/local/bin/backup-grub.sh

# Schedule weekly backups via cron
echo "0 3 * * 0 root /usr/local/bin/backup-grub.sh" | sudo tee /etc/cron.d/grub-backup
```

## Restoring GRUB Configuration

### Restoring Configuration Files

```bash
# Extract backup
BACKUP_FILE="/root/grub-backups/20260302-030000-grub-backup.tar.gz"
sudo tar -xzf "$BACKUP_FILE" -C /tmp/

RESTORE_DIR=$(ls -d /tmp/2026*/ | head -1)

# Restore GRUB default settings
sudo cp "$RESTORE_DIR/grub" /etc/default/grub

# Restore GRUB scripts
sudo cp -a "$RESTORE_DIR/grub.d/" /etc/grub.d/

# Regenerate grub.cfg from the restored templates
sudo update-grub
```

### Restoring the MBR on BIOS Systems

If the MBR is corrupted and the system will not boot, restore from a live USB:

```bash
# Boot from Ubuntu live USB, open a terminal

# Mount the system partition
sudo mount /dev/sda1 /mnt

# Restore the MBR backup (if you have it)
sudo dd if=/mnt/root/grub-backups/mbr.bin of=/dev/sda bs=512 count=1

# Or reinstall GRUB from the running live environment
sudo mount --bind /dev /mnt/dev
sudo mount --bind /proc /mnt/proc
sudo mount --bind /sys /mnt/sys
sudo chroot /mnt grub-install /dev/sda
sudo chroot /mnt update-grub
```

### Restoring GRUB on UEFI Systems from a Live Environment

```bash
# Boot from Ubuntu live USB

# Mount system root and EFI partition
sudo mount /dev/sda2 /mnt          # Root partition - adjust as needed
sudo mount /dev/sda1 /mnt/boot/efi # EFI partition

# Bind system filesystems
for dir in dev proc sys; do
    sudo mount --bind /$dir /mnt/$dir
done

# Chroot and reinstall GRUB
sudo chroot /mnt
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=ubuntu
update-grub
exit

# Unmount
for dir in dev proc sys boot/efi; do
    sudo umount /mnt/$dir
done
sudo umount /mnt
```

## Protecting Against Accidental grub.cfg Deletion

Keep a copy of the working `grub.cfg` in multiple locations:

```bash
# Copy grub.cfg to a safe location after every update-grub
sudo cp /boot/grub/grub.cfg /boot/grub/grub.cfg.backup

# Create a hook that runs after kernel updates
sudo nano /etc/kernel/postinst.d/backup-grub-cfg
```

```bash
#!/bin/sh
# Backup grub.cfg after kernel installation
cp /boot/grub/grub.cfg /boot/grub/grub.cfg.backup-$(date +%Y%m%d)
# Keep only last 5 backups
ls -t /boot/grub/grub.cfg.backup-* 2>/dev/null | tail -n +6 | xargs rm -f
```

```bash
sudo chmod +x /etc/kernel/postinst.d/backup-grub-cfg
```

## Copying GRUB Backups Off-Server

Local backups on the same disk that GRUB lives on do not help when the disk fails. Copy backups to a remote location:

```bash
# Copy to a remote backup server via scp
BACKUP_FILE="/root/grub-backups/$(ls -t /root/grub-backups/*.tar.gz | head -1 | xargs basename)"
scp /root/grub-backups/"$BACKUP_FILE" backup-server:/backups/$(hostname)/grub/

# Or sync the entire backup directory
rsync -az /root/grub-backups/ backup-server:/backups/$(hostname)/grub/
```

## Summary

GRUB backups on Ubuntu cover two categories: the configuration files (`/etc/default/grub`, `/etc/grub.d/`, `/boot/grub/grub.cfg`) which control boot behavior, and the binary data in the MBR or EFI System Partition which gets the bootloader off the disk. Back up both categories regularly, store backups off the boot disk, and keep a live USB available for recovery situations. Combined with uptime monitoring through [OneUptime](https://oneuptime.com) to detect boot failures and alert on-call responders, you can minimize recovery time when bootloader issues occur.
