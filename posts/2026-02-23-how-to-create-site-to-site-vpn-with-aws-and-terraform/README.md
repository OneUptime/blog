# How to Create Site-to-Site VPN with AWS and Terraform

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, VPN, AWS, Site-to-Site, IPsec, Networking, Infrastructure as Code

Description: A step-by-step guide to creating a Site-to-Site VPN connection between AWS and your on-premises network using Terraform with IPsec tunnels and BGP routing.

---

A Site-to-Site VPN connects your on-premises network to your AWS VPC through encrypted IPsec tunnels. This gives your office or data center private, secure access to AWS resources without exposing them to the public internet. It is often the first step organizations take when extending their network into the cloud.

In this guide, we will walk through setting up a complete Site-to-Site VPN connection between AWS and an on-premises network using Terraform. We will cover the Virtual Private Gateway, Customer Gateway, VPN connection, routing, and monitoring.

## How Site-to-Site VPN Works

A Site-to-Site VPN connection has three main components:

1. **Virtual Private Gateway (VGW)**: The AWS side of the VPN, attached to your VPC
2. **Customer Gateway (CGW)**: Represents your on-premises VPN device
3. **VPN Connection**: Two IPsec tunnels between the VGW and CGW for redundancy

AWS always creates two tunnels for high availability. Each tunnel terminates on a different AWS endpoint, so if one fails, traffic automatically routes through the other.

## Prerequisites

Before we start, you need:

- A VPC in AWS with private subnets
- The public IP address of your on-premises VPN device
- The BGP ASN of your on-premises network (or choose to use static routing)

## Step 1: Create the Virtual Private Gateway

The Virtual Private Gateway is the AWS side of the VPN tunnel.

```hcl
# vpn-gateway.tf - AWS side of the VPN
resource "aws_vpn_gateway" "main" {
  vpc_id          = aws_vpc.main.id
  amazon_side_asn = 64512 # Default AWS ASN, change if needed

  tags = {
    Name        = "${var.project_name}-vpn-gateway"
    Environment = var.environment
  }
}

# Attach the VPN gateway to the VPC
# (This is implicit when you set vpc_id above, but you can also do it separately)

# Enable route propagation on private route tables
# This automatically adds routes for on-premises CIDRs learned via BGP
resource "aws_vpn_gateway_route_propagation" "private" {
  count = length(var.private_route_table_ids)

  vpn_gateway_id = aws_vpn_gateway.main.id
  route_table_id = var.private_route_table_ids[count.index]
}
```

## Step 2: Create the Customer Gateway

The Customer Gateway represents your on-premises VPN device in AWS. You need the public IP of your device and its BGP ASN.

```hcl
# customer-gateway.tf - On-premises side representation
resource "aws_customer_gateway" "main" {
  bgp_asn    = var.onprem_bgp_asn # e.g., 65000
  ip_address = var.onprem_vpn_ip   # Public IP of your VPN device
  type       = "ipsec.1"

  tags = {
    Name     = "${var.project_name}-customer-gateway"
    Location = var.office_location
  }
}

variable "onprem_bgp_asn" {
  type        = number
  description = "BGP ASN for on-premises network"
  default     = 65000
}

variable "onprem_vpn_ip" {
  type        = string
  description = "Public IP address of on-premises VPN device"
}

variable "office_location" {
  type        = string
  description = "Location of the on-premises site"
  default     = "Main Office"
}
```

## Step 3: Create the VPN Connection

This is where the two IPsec tunnels are created. You can customize encryption algorithms, pre-shared keys, and tunnel options.

```hcl
# vpn-connection.tf - The actual VPN tunnels
resource "aws_vpn_connection" "main" {
  vpn_gateway_id      = aws_vpn_gateway.main.id
  customer_gateway_id = aws_customer_gateway.main.id
  type                = "ipsec.1"

  # Use BGP for dynamic routing (recommended)
  static_routes_only = false

  # Tunnel 1 configuration
  tunnel1_ike_versions                 = ["ikev2"]
  tunnel1_phase1_dh_group_numbers      = [14, 15, 16, 17, 18]
  tunnel1_phase1_encryption_algorithms = ["AES256", "AES256-GCM-16"]
  tunnel1_phase1_integrity_algorithms  = ["SHA2-256", "SHA2-384", "SHA2-512"]
  tunnel1_phase1_lifetime_seconds      = 28800

  tunnel1_phase2_dh_group_numbers      = [14, 15, 16, 17, 18]
  tunnel1_phase2_encryption_algorithms = ["AES256", "AES256-GCM-16"]
  tunnel1_phase2_integrity_algorithms  = ["SHA2-256", "SHA2-384", "SHA2-512"]
  tunnel1_phase2_lifetime_seconds      = 3600

  # Optional: Set inside CIDR for tunnel 1
  tunnel1_inside_cidr = "169.254.10.0/30"

  # Optional: Pre-shared key (generated by AWS if not specified)
  # tunnel1_preshared_key = var.tunnel1_psk

  # Tunnel 2 configuration (same settings for consistency)
  tunnel2_ike_versions                 = ["ikev2"]
  tunnel2_phase1_dh_group_numbers      = [14, 15, 16, 17, 18]
  tunnel2_phase1_encryption_algorithms = ["AES256", "AES256-GCM-16"]
  tunnel2_phase1_integrity_algorithms  = ["SHA2-256", "SHA2-384", "SHA2-512"]
  tunnel2_phase1_lifetime_seconds      = 28800

  tunnel2_phase2_dh_group_numbers      = [14, 15, 16, 17, 18]
  tunnel2_phase2_encryption_algorithms = ["AES256", "AES256-GCM-16"]
  tunnel2_phase2_integrity_algorithms  = ["SHA2-256", "SHA2-384", "SHA2-512"]
  tunnel2_phase2_lifetime_seconds      = 3600

  tunnel2_inside_cidr = "169.254.11.0/30"

  # Enable tunnel lifecycle control
  tunnel_inside_ip_version = "ipv4"

  tags = {
    Name        = "${var.project_name}-vpn-connection"
    Environment = var.environment
    Location    = var.office_location
  }
}
```

## Step 4: Static Routes (If Not Using BGP)

If you are using static routing instead of BGP, you need to define the on-premises CIDR blocks.

```hcl
# static-routes.tf - Only needed if static_routes_only = true
resource "aws_vpn_connection_route" "onprem" {
  count = var.static_routes_only ? length(var.onprem_cidrs) : 0

  destination_cidr_block = var.onprem_cidrs[count.index]
  vpn_connection_id      = aws_vpn_connection.main.id
}

variable "onprem_cidrs" {
  type        = list(string)
  description = "On-premises CIDR blocks to route through VPN"
  default     = ["192.168.0.0/16", "172.16.0.0/12"]
}
```

## Step 5: Output Configuration for On-Premises Device

The VPN connection creates configuration details that your on-premises device needs.

```hcl
# outputs.tf - Configuration for on-premises VPN device
output "vpn_connection_id" {
  value       = aws_vpn_connection.main.id
  description = "VPN Connection ID"
}

output "tunnel1_address" {
  value       = aws_vpn_connection.main.tunnel1_address
  description = "Public IP of VPN Tunnel 1 (AWS side)"
}

output "tunnel2_address" {
  value       = aws_vpn_connection.main.tunnel2_address
  description = "Public IP of VPN Tunnel 2 (AWS side)"
}

output "tunnel1_bgp_asn" {
  value       = aws_vpn_connection.main.tunnel1_bgp_asn
  description = "BGP ASN for Tunnel 1 (AWS side)"
}

output "tunnel1_bgp_holdtime" {
  value       = aws_vpn_connection.main.tunnel1_bgp_holdtime
  description = "BGP hold time for Tunnel 1"
}

output "tunnel1_inside_cidr" {
  value       = aws_vpn_connection.main.tunnel1_inside_cidr
  description = "Inside IP CIDR for Tunnel 1"
}

output "tunnel1_preshared_key" {
  value       = aws_vpn_connection.main.tunnel1_preshared_key
  description = "Pre-shared key for Tunnel 1"
  sensitive   = true
}

output "tunnel2_preshared_key" {
  value       = aws_vpn_connection.main.tunnel2_preshared_key
  description = "Pre-shared key for Tunnel 2"
  sensitive   = true
}

# You can also download the full configuration from the AWS Console
# for specific devices (Cisco, Juniper, Palo Alto, etc.)
```

## Step 6: Monitoring and Alerting

Monitor both tunnels to detect connectivity issues quickly.

```hcl
# monitoring.tf - VPN health monitoring

# Alert when Tunnel 1 goes down
resource "aws_cloudwatch_metric_alarm" "tunnel1_down" {
  alarm_name          = "${var.project_name}-vpn-tunnel1-down"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 2
  metric_name         = "TunnelState"
  namespace           = "AWS/VPN"
  period              = 300
  statistic           = "Maximum"
  threshold           = 1

  dimensions = {
    VpnId         = aws_vpn_connection.main.id
    TunnelIpAddress = aws_vpn_connection.main.tunnel1_address
  }

  alarm_actions = [var.alert_sns_topic_arn]
  ok_actions    = [var.alert_sns_topic_arn]

  alarm_description = "VPN Tunnel 1 is DOWN"
}

# Alert when Tunnel 2 goes down
resource "aws_cloudwatch_metric_alarm" "tunnel2_down" {
  alarm_name          = "${var.project_name}-vpn-tunnel2-down"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 2
  metric_name         = "TunnelState"
  namespace           = "AWS/VPN"
  period              = 300
  statistic           = "Maximum"
  threshold           = 1

  dimensions = {
    VpnId         = aws_vpn_connection.main.id
    TunnelIpAddress = aws_vpn_connection.main.tunnel2_address
  }

  alarm_actions = [var.alert_sns_topic_arn]
  ok_actions    = [var.alert_sns_topic_arn]

  alarm_description = "VPN Tunnel 2 is DOWN"
}

# Alert when BOTH tunnels are down (critical)
resource "aws_cloudwatch_metric_alarm" "both_tunnels_down" {
  alarm_name          = "${var.project_name}-vpn-all-tunnels-down"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 1
  metric_name         = "TunnelState"
  namespace           = "AWS/VPN"
  period              = 60
  statistic           = "Maximum"
  threshold           = 1

  dimensions = {
    VpnId = aws_vpn_connection.main.id
  }

  alarm_actions = [var.critical_alert_sns_topic_arn]

  alarm_description = "ALL VPN tunnels are DOWN - on-premises connectivity lost"
}

# Monitor data transfer
resource "aws_cloudwatch_metric_alarm" "high_tunnel_traffic" {
  alarm_name          = "${var.project_name}-vpn-high-traffic"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 3
  metric_name         = "TunnelDataOut"
  namespace           = "AWS/VPN"
  period              = 300
  statistic           = "Sum"
  threshold           = 1073741824 # 1 GB in 5 minutes

  dimensions = {
    VpnId = aws_vpn_connection.main.id
  }

  alarm_actions = [var.alert_sns_topic_arn]
  alarm_description = "VPN traffic is unusually high"
}
```

## Multiple Office Connections

If you have multiple offices, create a module and reuse it.

```hcl
# multi-site.tf - Connect multiple offices
module "vpn_main_office" {
  source = "./modules/site-to-site-vpn"

  project_name    = var.project_name
  vpc_id          = aws_vpc.main.id
  vpn_gateway_id  = aws_vpn_gateway.main.id
```

## Troubleshooting Tips

Here are the most common issues with Site-to-Site VPN:

1. **Tunnel not coming up**: Check that your on-premises device's public IP matches the Customer Gateway IP, and verify IKE and IPsec settings match on both sides.
2. **BGP session not establishing**: Verify the ASN numbers, inside tunnel CIDRs, and that your device is configured to peer with the AWS BGP IP.
3. **Traffic not flowing**: Check route tables, security groups, and NACLs. Make sure VPN route propagation is enabled.
4. **One tunnel keeps flapping**: This can happen when your on-premises device only establishes one tunnel. Configure it to keep both tunnels active.

## Summary

Setting up a Site-to-Site VPN with Terraform gives you a reproducible, version-controlled connection between your on-premises network and AWS. The key components are the Virtual Private Gateway (AWS side), Customer Gateway (your side), and the VPN Connection with its two IPsec tunnels.

Always configure both tunnels for redundancy and enable BGP for dynamic routing when possible. Monitor tunnel state with CloudWatch alarms so you know immediately when connectivity is degraded.

For monitoring your VPN connectivity alongside the services that depend on it, [OneUptime](https://oneuptime.com/blog/post/2026-02-23-how-to-build-an-observability-platform-with-terraform/view) provides unified monitoring that can correlate VPN issues with application-level impacts.
