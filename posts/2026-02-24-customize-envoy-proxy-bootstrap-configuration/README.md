# How to Customize Envoy Proxy Bootstrap Configuration

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Envoy, Bootstrap, Configuration, Kubernetes

Description: How to customize the Envoy proxy bootstrap configuration in Istio for advanced tuning and feature enablement.

---

Every Envoy sidecar in Istio starts with a bootstrap configuration. This configuration is generated by the pilot-agent process before Envoy launches and contains the initial settings that Envoy needs to connect to the control plane, set up its admin interface, and configure core behavior. Most of the time you never need to touch it, but there are situations where customizing the bootstrap is the only way to enable certain features.

## What is in the Bootstrap Configuration

The bootstrap config includes:

- **Node metadata** - Information about the sidecar (pod name, namespace, service account, labels)
- **Static clusters** - The control plane endpoint (istiod) and other static backends
- **Admin interface** - Address and port for the Envoy admin API
- **Stats sinks** - Where to send metrics
- **Runtime configuration** - Feature flags and runtime settings
- **Tracing configuration** - Initial tracing setup

View the current bootstrap configuration of any sidecar:

```bash
istioctl proxy-config bootstrap <pod-name> -o json
```

Or directly from the Envoy admin API:

```bash
kubectl exec <pod-name> -c istio-proxy -- curl -s localhost:15000/config_dump?resource=bootstrap
```

## Customizing via Proxy Config Annotations

The simplest way to customize the bootstrap is through pod annotations. Istio provides the `proxy.istio.io/config` annotation:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    metadata:
      annotations:
        proxy.istio.io/config: |
          concurrency: 4
          drainDuration: 30s
          parentShutdownDuration: 35s
          terminationDrainDuration: 20s
          tracing:
            zipkin:
              address: zipkin.istio-system:9411
            sampling: 50.0
    spec:
      containers:
      - name: my-app
        image: my-app:latest
```

## Customizing via ProxyConfig Resource

For cluster-wide or namespace-wide customization, use the ProxyConfig resource:

```yaml
apiVersion: networking.istio.io/v1beta1
kind: ProxyConfig
metadata:
  name: custom-proxy-config
  namespace: default
spec:
  concurrency: 2
  environmentVariables:
    ISTIO_META_DNS_CAPTURE: "true"
    ISTIO_META_DNS_AUTO_ALLOCATE: "true"
  image:
    imageType: distroless
```

For mesh-wide settings, put it in the mesh config:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      concurrency: 2
      drainDuration: 45s
      holdApplicationUntilProxyStarts: true
      proxyStatsMatcher:
        inclusionRegexps:
        - ".*circuit_breakers.*"
        - ".*outlier_detection.*"
        - ".*upstream_rq_retry.*"
```

## Adding Custom Bootstrap Configuration

For changes that are not covered by the proxy config API, you can inject custom bootstrap configuration using an annotation:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    metadata:
      annotations:
        sidecar.istio.io/bootstrapOverride: "custom-bootstrap"
    spec:
      containers:
      - name: my-app
        image: my-app:latest
```

Create the ConfigMap with the custom bootstrap:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: custom-bootstrap
  namespace: default
data:
  custom_bootstrap.json: |
    {
      "stats_config": {
        "use_all_default_tags": true,
        "stats_tags": [
          {
            "tag_name": "custom_env",
            "fixed_value": "production"
          }
        ]
      },
      "overload_manager": {
        "refresh_interval": "0.25s",
        "resource_monitors": [
          {
            "name": "envoy.resource_monitors.fixed_heap",
            "typed_config": {
              "@type": "type.googleapis.com/envoy.extensions.resource_monitors.fixed_heap.v3.FixedHeapConfig",
              "max_heap_size_bytes": 536870912
            }
          }
        ],
        "actions": [
          {
            "name": "envoy.overload_actions.shrink_heap",
            "triggers": [
              {
                "name": "envoy.resource_monitors.fixed_heap",
                "threshold": {
                  "value": 0.90
                }
              }
            ]
          },
          {
            "name": "envoy.overload_actions.stop_accepting_requests",
            "triggers": [
              {
                "name": "envoy.resource_monitors.fixed_heap",
                "threshold": {
                  "value": 0.95
                }
              }
            ]
          }
        ]
      }
    }
```

This example adds an overload manager that protects Envoy from running out of memory.

## Configuring Stats Tags

Custom stats tags let you add metadata to all Envoy metrics. This is useful for grouping and filtering metrics in your monitoring system:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      extraStatTags:
      - "request_operation"
      proxyStatsMatcher:
        inclusionPrefixes:
        - "cluster.outbound"
        - "listener"
        - "http.inbound"
        inclusionRegexps:
        - ".*circuit_breakers.*"
```

## Enabling the Overload Manager

The overload manager is an important production hardening feature. It prevents Envoy from crashing when it runs out of resources:

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: overload-manager
  namespace: istio-system
spec:
  configPatches:
  - applyTo: BOOTSTRAP
    patch:
      operation: MERGE
      value:
        overload_manager:
          refresh_interval: 0.25s
          resource_monitors:
          - name: envoy.resource_monitors.global_downstream_max_connections
            typed_config:
              "@type": type.googleapis.com/envoy.extensions.resource_monitors.downstream_connections.v3.DownstreamConnectionsConfig
              max_active_downstream_connections: 50000
```

## Runtime Feature Flags

Envoy has runtime feature flags that can enable or disable features. Set them through the bootstrap:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: custom-bootstrap
  namespace: default
data:
  custom_bootstrap.json: |
    {
      "layered_runtime": {
        "layers": [
          {
            "name": "static_layer",
            "static_layer": {
              "envoy.reloadable_features.http2_decode_metadata_header": true,
              "re2.max_program_size.error_level": 1000
            }
          }
        ]
      }
    }
```

## Environment Variables

Some Envoy and pilot-agent behaviors are controlled by environment variables. Set them through the IstioOperator or pod annotations:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      proxyMetadata:
        ISTIO_META_DNS_CAPTURE: "true"
        ISTIO_META_DNS_AUTO_ALLOCATE: "true"
        BOOTSTRAP_XDS_AGENT: "true"
```

Per-pod:

```yaml
annotations:
  proxy.istio.io/config: |
    proxyMetadata:
      ISTIO_META_DNS_CAPTURE: "true"
```

## Debugging Bootstrap Issues

If a sidecar is not starting correctly after bootstrap customization:

```bash
# Check pilot-agent logs
kubectl logs <pod-name> -c istio-proxy

# Check for bootstrap generation errors
kubectl logs <pod-name> -c istio-init

# View the generated bootstrap file
kubectl exec <pod-name> -c istio-proxy -- cat /etc/istio/proxy/envoy-rev.json
```

Common issues include invalid JSON in the bootstrap override, referencing unknown Envoy extensions, or version mismatches between the bootstrap config format and the Envoy version.

## Verifying Changes

After customizing the bootstrap, verify the changes are applied:

```bash
istioctl proxy-config bootstrap <pod-name> -o json | jq '.bootstrap.statsConfig'
istioctl proxy-config bootstrap <pod-name> -o json | jq '.bootstrap.overloadManager'
```

Bootstrap customization is one of those advanced Istio topics that you usually do not need until you do. The most common legitimate uses are enabling the overload manager for production stability, adding custom stats tags for better metric organization, and tweaking runtime feature flags. Always test bootstrap changes in a staging environment first, because a bad bootstrap can prevent sidecars from starting entirely.
