# How to Set Up Warm Standby Servers on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, High Availability, Database Replication, Disaster Recovery, Server Administration

Description: Learn how to configure warm standby servers on Ubuntu for high availability and disaster recovery, including setup, monitoring, and failover procedures.

---

A warm standby server sits ready to take over from a primary server with minimal delay. Unlike cold standby (which requires manual startup and configuration) or hot standby (which actively serves read queries), a warm standby receives replicated data continuously but stays offline to client connections until promotion. This makes it a practical middle ground for organizations that need fast failover without the complexity of full active-active setups.

This guide covers setting up a warm standby for PostgreSQL on Ubuntu, which is the most common use case, along with the monitoring and failover procedures you need to make it work reliably.

## Prerequisites

- Two Ubuntu servers (22.04 LTS or later recommended)
- PostgreSQL installed on both servers
- Network connectivity between primary and standby
- A shared understanding of your recovery time objective (RTO)

For this guide, assume:
- Primary server: `192.168.1.10` (primary)
- Standby server: `192.168.1.20` (standby)

## Configuring the Primary Server

### Setting Up Replication User

First, create a dedicated replication user on the primary server.

```sql
-- Connect to PostgreSQL as superuser
-- Create a replication-only user with a strong password
CREATE USER replicator REPLICATION LOGIN ENCRYPTED PASSWORD 'strong_password_here';
```

### Configuring postgresql.conf

Edit the PostgreSQL configuration file on the primary:

```bash
# Open the main config file
sudo nano /etc/postgresql/14/main/postgresql.conf
```

Add or modify these directives:

```ini
# Enable WAL archiving - required for standby to catch up after network interruptions
wal_level = replica

# Number of concurrent replication connections to allow
max_wal_senders = 5

# Keep enough WAL segments for the standby to catch up
wal_keep_size = 256MB

# Archive WAL files to a shared location or S3
archive_mode = on
archive_command = 'cp %p /var/lib/postgresql/wal_archive/%f'

# Timeout for replication connections
wal_sender_timeout = 60s
```

### Configuring pg_hba.conf

Allow the standby to connect for replication:

```bash
sudo nano /etc/postgresql/14/main/pg_hba.conf
```

Add this line, replacing the standby IP:

```text
# Allow standby server to connect for streaming replication
host    replication     replicator      192.168.1.20/32         md5
```

Restart PostgreSQL on the primary:

```bash
sudo systemctl restart postgresql
```

### Creating the WAL Archive Directory

```bash
# Create archive directory and set permissions
sudo mkdir -p /var/lib/postgresql/wal_archive
sudo chown postgres:postgres /var/lib/postgresql/wal_archive
sudo chmod 700 /var/lib/postgresql/wal_archive
```

## Setting Up the Standby Server

### Taking a Base Backup

On the standby server, stop PostgreSQL and take a base backup from the primary:

```bash
# Stop PostgreSQL on standby if running
sudo systemctl stop postgresql

# Remove existing data directory (be careful - this wipes the standby's data)
sudo rm -rf /var/lib/postgresql/14/main

# Take base backup as postgres user
sudo -u postgres pg_basebackup \
    -h 192.168.1.10 \
    -U replicator \
    -D /var/lib/postgresql/14/main \
    -P \
    -R \
    --wal-method=stream
```

The `-R` flag is important - it automatically creates the `standby.signal` file and configures `postgresql.auto.conf` with the primary connection info.

### Configuring the Standby

Edit the auto-configuration file that `pg_basebackup` created:

```bash
sudo nano /var/lib/postgresql/14/main/postgresql.auto.conf
```

Verify it contains the primary connection string:

```ini
# This was auto-generated by pg_basebackup -R
primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator password=strong_password_here'
```

You can also add a recovery configuration:

```bash
# Create a recovery configuration file
sudo -u postgres cat > /var/lib/postgresql/14/main/postgresql.auto.conf << 'EOF'
primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator password=strong_password_here application_name=standby1'
restore_command = 'cp /var/lib/postgresql/wal_archive/%f %p'
recovery_target_timeline = 'latest'
EOF
```

Verify the `standby.signal` file exists:

```bash
ls -la /var/lib/postgresql/14/main/standby.signal
```

### Starting the Standby

```bash
sudo systemctl start postgresql
```

## Verifying Replication

### Check Replication Status on Primary

```sql
-- Run this on the primary server
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    sync_state
FROM pg_stat_replication;
```

You should see the standby listed with `state = streaming`.

### Check Standby Status

```sql
-- Run this on the standby server
SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag,
    pg_is_in_recovery() AS is_standby,
    pg_last_wal_receive_lsn() AS received_lsn,
    pg_last_wal_replay_lsn() AS replayed_lsn;
```

## Monitoring Replication Lag

Create a simple monitoring script:

```bash
#!/bin/bash
# /usr/local/bin/check-replication-lag.sh
# Monitors replication lag and alerts if it exceeds threshold

PRIMARY_HOST="192.168.1.10"
STANDBY_HOST="192.168.1.20"
DB_USER="postgres"
LAG_THRESHOLD_SECONDS=300  # Alert if lag exceeds 5 minutes

LAG=$(psql -h "$STANDBY_HOST" -U "$DB_USER" -t -c \
    "SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))::int;")

if [ "$LAG" -gt "$LAG_THRESHOLD_SECONDS" ]; then
    echo "WARNING: Replication lag is ${LAG} seconds - exceeds threshold of ${LAG_THRESHOLD_SECONDS}s"
    # Add alerting here (email, PagerDuty, OneUptime webhook, etc.)
    exit 1
fi

echo "Replication lag: ${LAG} seconds - OK"
```

Make it executable and add to cron:

```bash
chmod +x /usr/local/bin/check-replication-lag.sh

# Check every 5 minutes
echo "*/5 * * * * postgres /usr/local/bin/check-replication-lag.sh" | sudo tee -a /etc/cron.d/pg-replication-monitor
```

## Performing a Failover

When the primary fails, promote the standby to primary:

```bash
# Method 1: Use pg_ctl to promote (preferred)
sudo -u postgres pg_ctl promote -D /var/lib/postgresql/14/main

# Method 2: Create a trigger file (if configured)
sudo -u postgres touch /tmp/postgresql.trigger.5432
```

After promotion, verify the former standby is now accepting writes:

```sql
-- Should return false after promotion
SELECT pg_is_in_recovery();

-- Test a write operation
CREATE TABLE failover_test (id serial, ts timestamptz default now());
INSERT INTO failover_test DEFAULT VALUES;
```

### Post-Failover Steps

After promoting the standby:

1. Update application connection strings to point to the new primary (former standby IP)
2. Rebuild the old primary as a new standby once it comes back online
3. Investigate the root cause of the primary failure
4. Update monitoring to reflect the new topology

```bash
# On the old primary (now repaired), rebuild as a new standby
sudo systemctl stop postgresql
sudo -u postgres pg_basebackup \
    -h 192.168.1.20 \
    -U replicator \
    -D /var/lib/postgresql/14/main \
    -P \
    -R \
    --wal-method=stream
sudo systemctl start postgresql
```

## Automating Failover with Patroni

For production systems, manual failover is error-prone. Patroni is a Python-based high-availability solution that automates PostgreSQL failover.

```bash
# Install Patroni and etcd (distributed consensus store)
sudo apt-get install -y python3-pip etcd
sudo pip3 install patroni[etcd]
```

Patroni watches cluster health continuously and promotes the best standby automatically when the primary goes down, handling split-brain scenarios via etcd quorum.

## Summary

Warm standby on Ubuntu with PostgreSQL streaming replication gives you:

- Continuous data synchronization with low lag
- Fast promotion (seconds to minutes, not hours)
- Point-in-time recovery capability via WAL archiving
- A clear runbook for failover operations

The key operational discipline is testing your failover procedure regularly. A standby you've never promoted is a standby you can't trust when you actually need it.
