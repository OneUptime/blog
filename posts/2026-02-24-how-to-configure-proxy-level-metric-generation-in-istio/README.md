# How to Configure Proxy-Level Metric Generation in Istio

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Envoy, Metrics, Proxy, Observability, Kubernetes

Description: Step-by-step guide to configuring proxy-level metric generation in Istio for granular observability into your service mesh traffic.

---

When people talk about Istio metrics, they usually mean the standard service-level metrics like request counts and latencies. But there is a whole other layer of metrics generated directly by the Envoy proxy itself. These proxy-level metrics give you insight into things like connection pool behavior, circuit breaker trips, TLS handshakes, and memory usage that the standard metrics do not cover.

Knowing how to configure and tune proxy-level metric generation is critical for running Istio in production. Here is how to do it.

## How Proxy Metrics Work in Istio

Istio uses a stats filter built into Envoy to generate the standard service metrics (the `istio_*` ones). These are generated by a WebAssembly (Wasm) or native stats plugin that runs inside each sidecar.

Separately, Envoy itself produces internal operational metrics about its own behavior. These include things like:

- How many active connections exist on each listener
- How many times a circuit breaker tripped
- How much memory the proxy is using
- DNS resolution stats
- Load balancer metrics per upstream cluster

By default, Istio only exposes a subset of these through Prometheus. The rest are available on the admin interface but not scraped.

## Checking Current Proxy Metrics

First, see what metrics are currently being exposed by a proxy:

```bash
kubectl exec deploy/my-service -c istio-proxy -- curl -s localhost:15000/stats/prometheus | wc -l
```

That gives you a line count. For a typical service, you will see a few hundred to a few thousand metric lines. Compare that with the full stats:

```bash
kubectl exec deploy/my-service -c istio-proxy -- curl -s localhost:15000/stats | wc -l
```

The full stats output is usually much larger. The difference is all the internal Envoy stats that are not exposed to Prometheus by default.

## Configuring Which Proxy Metrics to Generate

The `proxyStatsMatcher` in the proxy configuration controls which additional Envoy metrics get included in the Prometheus output.

### Mesh-Wide Configuration

Apply this through your Istio installation:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      proxyStatsMatcher:
        inclusionPrefixes:
          - "cluster_manager"
          - "listener_manager"
          - "server"
          - "runtime"
          - "http.inbound_0.0.0.0_80"
        inclusionSuffixes:
          - "upstream_cx_active"
          - "upstream_rq_active"
          - "membership_healthy"
          - "membership_total"
        inclusionRegexps:
          - ".*circuit_breakers.*"
```

The three matching options are:

- **inclusionPrefixes** - Match stats that start with the given string
- **inclusionSuffixes** - Match stats that end with the given string
- **inclusionRegexps** - Match stats against a regular expression

### Per-Pod Configuration

For specific workloads, use the proxy config annotation:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
spec:
  template:
    metadata:
      annotations:
        proxy.istio.io/config: |
          proxyStatsMatcher:
            inclusionPrefixes:
              - "cluster.outbound|443||payment-gateway"
              - "cluster.outbound|443||bank-api"
            inclusionSuffixes:
              - "upstream_cx_connect_timeout"
              - "upstream_cx_destroy_with_active_rq"
              - "upstream_rq_retry"
              - "upstream_rq_retry_success"
            inclusionRegexps:
              - ".*outlier_detection.*"
    spec:
      containers:
        - name: payment-service
          image: payment-service:v2
```

This is great for debugging specific services without changing the global config. When you are troubleshooting connection issues to an upstream dependency, adding those specific cluster metrics temporarily can save you hours.

## Configuring the Stats Plugin

The stats plugin (sometimes called the stats filter) in Istio generates the standard `istio_*` metrics. You can configure it using the Telemetry API.

### Controlling Metric Generation Mode

Istio can generate metrics on both the client side and server side of a request. You can control this:

```yaml
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: metric-mode
  namespace: istio-system
spec:
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - match:
            metric: REQUEST_COUNT
            mode: CLIENT
          disabled: false
        - match:
            metric: REQUEST_COUNT
            mode: SERVER
          disabled: false
        - match:
            metric: REQUEST_DURATION
            mode: CLIENT
          disabled: true
        - match:
            metric: REQUEST_DURATION
            mode: SERVER
          disabled: false
```

In this example, request duration is only recorded on the server side, which cuts the number of histogram time series in half. For latency monitoring, server-side metrics are usually more accurate anyway since they do not include network transit time.

### Custom Metric Definitions

You can define entirely custom metrics using the stats plugin configuration through EnvoyFilter:

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: custom-metric
  namespace: istio-system
spec:
  configPatches:
    - applyTo: HTTP_FILTER
      match:
        context: SIDECAR_INBOUND
        listener:
          filterChain:
            filter:
              name: envoy.filters.network.http_connection_manager
              subFilter:
                name: envoy.filters.http.router
      patch:
        operation: INSERT_BEFORE
        value:
          name: istio.stats
          typed_config:
            "@type": type.googleapis.com/udpa.type.v1.TypedStruct
            type_url: type.googleapis.com/stats.PluginConfig
            value:
              metrics:
                - name: "custom_request_count"
                  dimensions:
                    source_app: "source.workload.name"
                    destination_service: "destination.service.name"
                    api_version: "request.headers['x-api-version']"
                  type: COUNTER
```

## Configuring Histogram Buckets

The default histogram buckets for request duration might not match your application's latency profile. You can customize them:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      proxyHeaders:
        metadataExchangeHeaders:
          mode: IN_MESH
  values:
    telemetry:
      v2:
        prometheus:
          configOverride:
            inboundSidecar:
              metrics:
                - name: request_duration_milliseconds
                  type: HISTOGRAM
                  histogram_buckets_override:
                    - 1
                    - 5
                    - 10
                    - 25
                    - 50
                    - 100
                    - 250
                    - 500
                    - 1000
                    - 5000
                    - 10000
```

If your service responds in single-digit milliseconds, the default buckets waste resolution. Custom buckets let you see the distribution where it matters.

## Monitoring Proxy Resource Usage

Proxy-level metrics are especially useful for monitoring the health of the proxy itself. Key metrics to watch:

```bash
# Memory usage
kubectl exec deploy/my-service -c istio-proxy -- curl -s localhost:15000/stats | grep server.memory

# Active connections
kubectl exec deploy/my-service -c istio-proxy -- curl -s localhost:15000/stats | grep "cx_active"

# Worker thread stats
kubectl exec deploy/my-service -c istio-proxy -- curl -s localhost:15000/stats | grep "worker"
```

Include these in your mesh-wide config so they appear in Prometheus:

```yaml
proxyStatsMatcher:
  inclusionPrefixes:
    - "server.memory"
    - "server.total_connections"
    - "server.live"
```

## Debugging Metric Generation

When metrics are not showing up as expected, here is a systematic debugging approach:

1. Check if the proxy config was applied:

```bash
istioctl proxy-config bootstrap deploy/my-service -o json | jq '.bootstrap.statsConfig'
```

2. Check the Envoy admin stats directly:

```bash
kubectl exec deploy/my-service -c istio-proxy -- curl -s localhost:15000/stats | grep "your_metric_prefix"
```

3. Check if Prometheus is scraping the pod:

```bash
kubectl get pod -l app=my-service -o jsonpath='{.items[0].metadata.annotations}' | jq .
```

Look for `prometheus.io/scrape: "true"` and the correct port annotation.

4. Look at the proxy logs for errors:

```bash
kubectl logs deploy/my-service -c istio-proxy --tail=100
```

## Performance Impact

Every metric you add costs memory and CPU. In a mesh with thousands of pods, this adds up. Here are some real numbers from production:

- Default Istio metrics: roughly 2-5 MB per proxy
- Adding circuit breaker stats: adds about 0.5-1 MB per proxy
- Adding all cluster stats: can add 5-20 MB per proxy depending on the number of services

For a cluster with 1000 pods, going from default to full cluster stats means an extra 5-20 GB of memory across the mesh. That is not free.

Be intentional about what you include. Add metrics when you have a specific use case for them, and remove them when you are done debugging.
