# How to Monitor Flux CD with Kubernetes Dashboard

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, GitOps, Kubernetes, Monitoring, Kubernetes Dashboard, Observability

Description: Learn how to use the Kubernetes Dashboard to monitor Flux CD custom resources, track reconciliation status, and troubleshoot deployment issues visually.

---

The Kubernetes Dashboard provides a web-based interface for managing and inspecting resources running in your cluster. While Flux CD operates through custom resources and controllers, these resources are fully visible in the Kubernetes Dashboard, making it a convenient tool for monitoring your GitOps pipelines without relying solely on the command line.

This guide covers how to set up the Kubernetes Dashboard to work effectively with Flux CD resources, what to look for when monitoring reconciliation, and how to use the dashboard for basic troubleshooting.

## Prerequisites

Before you begin, ensure you have:

- A Kubernetes cluster with Flux CD installed
- `kubectl` access to the cluster
- The Kubernetes Dashboard deployed in your cluster
- Appropriate RBAC permissions to view Flux custom resources

## Step 1: Install the Kubernetes Dashboard

If you do not already have the Kubernetes Dashboard installed, deploy it using the official manifest:

```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes/dashboard/v2.7.0/aio/deploy/recommended.yaml
```

To access the dashboard, create a proxy:

```bash
kubectl proxy
```

Then open your browser and navigate to:

```text
http://localhost:8001/api/v1/namespaces/kubernetes-dashboard/services/https:kubernetes-dashboard:/proxy/
```

## Step 2: Configure RBAC for Flux Resources

The default dashboard service account may not have permissions to view Flux custom resources. Create a ClusterRole that grants read access to Flux CRDs:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: flux-dashboard-viewer
rules:
  - apiGroups:
      - source.toolkit.fluxcd.io
      - kustomize.toolkit.fluxcd.io
      - helm.toolkit.fluxcd.io
      - notification.toolkit.fluxcd.io
      - image.toolkit.fluxcd.io
    resources:
      - "*"
    verbs:
      - get
      - list
      - watch
```

Bind this role to the dashboard service account:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: flux-dashboard-viewer-binding
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: flux-dashboard-viewer
subjects:
  - kind: ServiceAccount
    name: kubernetes-dashboard
    namespace: kubernetes-dashboard
```

Apply both manifests:

```bash
kubectl apply -f flux-dashboard-rbac.yaml
```

## Step 3: Navigate Flux Resources in the Dashboard

Once RBAC is configured, you can view Flux custom resources through the dashboard. Navigate to **Custom Resource Definitions** in the left sidebar. You will see entries for Flux resource types including:

- `gitrepositories.source.toolkit.fluxcd.io`
- `kustomizations.kustomize.toolkit.fluxcd.io`
- `helmreleases.helm.toolkit.fluxcd.io`
- `helmrepositories.source.toolkit.fluxcd.io`

Click on any CRD to see its instances. Each instance shows metadata, spec, and status fields. The status section is particularly important because it contains the reconciliation state.

## Step 4: Monitor Reconciliation Status

For each Flux resource, the status conditions tell you whether reconciliation succeeded or failed. In the dashboard, open a specific resource and scroll to the status section. Look for the `Ready` condition:

```yaml
status:
  conditions:
    - type: Ready
      status: "True"
      reason: ReconciliationSucceeded
      message: "Applied revision: main@sha1:abc123"
      lastTransitionTime: "2026-03-05T10:30:00Z"
```

A `Ready` status of `False` indicates a problem. The `message` field usually contains details about what went wrong, such as validation errors, missing secrets, or network issues fetching source artifacts.

## Step 5: Check Events for Troubleshooting

Kubernetes events generated by Flux controllers appear in the dashboard under the **Events** tab for each namespace. These events provide a timeline of what happened during reconciliation:

```text
Type     Reason                  Message
Normal   ReconciliationSucceeded Applied revision: main@sha1:abc123
Warning  ReconciliationFailed    kustomize build failed: ...
Normal   ArtifactUpToDate        artifact up-to-date with remote
```

You can filter events by namespace to focus on specific applications managed by Flux. Events older than one hour are typically garbage collected by Kubernetes, so check them promptly when investigating issues.

## Step 6: Monitor Source Artifacts

Source resources like GitRepository and HelmRepository show artifact information in their status. In the dashboard, check the `artifact` field to verify that Flux has successfully fetched the latest source:

```yaml
status:
  artifact:
    revision: "main@sha1:abc123def456"
    lastUpdateTime: "2026-03-05T10:25:00Z"
    path: "gitrepository/flux-system/my-app/abc123.tar.gz"
```

If the artifact revision is stale or the `lastUpdateTime` is old, it may indicate connectivity issues with your Git or Helm repository.

## Step 7: Set Up Dashboard Refresh for Real-Time Monitoring

The Kubernetes Dashboard does not auto-refresh by default. For continuous monitoring, enable auto-refresh in the dashboard settings or use the browser's auto-refresh feature. A refresh interval of 30 seconds works well for monitoring Flux reconciliation cycles, which typically run on intervals of one to ten minutes.

## Limitations of the Kubernetes Dashboard for Flux Monitoring

While the Kubernetes Dashboard is useful for visual inspection, it has some limitations for Flux monitoring:

- It does not render Flux-specific visualizations like dependency graphs
- Custom resource views show raw YAML without Flux-aware formatting
- There is no built-in alerting capability
- Historical data is limited to what Kubernetes events retain

For production environments, consider supplementing the dashboard with Prometheus metrics and Grafana dashboards that are specifically designed for Flux CD monitoring. The Kubernetes Dashboard works best as a quick inspection tool rather than a primary monitoring solution.

## Summary

The Kubernetes Dashboard provides a straightforward way to visually inspect Flux CD resources and their reconciliation status. By configuring appropriate RBAC permissions, you can browse GitRepository, Kustomization, and HelmRelease resources directly in the web interface. Check the status conditions and events for each resource to understand whether reconciliation is succeeding and to diagnose failures when they occur. For teams that prefer graphical interfaces over the command line, the dashboard serves as a useful complement to the `flux` CLI and more specialized monitoring tools.
