# How to Configure Tracing Using MeshConfig in Istio

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, MeshConfig, Tracing, Configuration, Service Mesh

Description: A detailed guide to configuring distributed tracing in Istio through MeshConfig, covering all available options and their effects.

---

MeshConfig is the central configuration for Istio's control plane, and it's one of the primary ways to set up tracing for the entire mesh. While the newer Telemetry API provides more granular control, MeshConfig remains the foundation for global tracing settings, extension provider definitions, and default proxy behavior. Understanding how MeshConfig tracing options work gives you a solid base for any tracing setup.

## Where MeshConfig Lives

MeshConfig can be set in several places:

1. **IstioOperator resource** - Used during installation with `istioctl`
2. **ConfigMap `istio` in `istio-system`** - The runtime configuration
3. **Helm values** - When installing via Helm charts

All of these ultimately update the same `istio` ConfigMap that istiod reads.

## Viewing Current MeshConfig

```bash
# View the full mesh config
kubectl get configmap istio -n istio-system -o jsonpath='{.data.mesh}' | yq .

# Or with just the tracing-related fields
kubectl get configmap istio -n istio-system -o yaml | grep -A30 "enableTracing\|tracing\|extensionProviders"
```

## Core Tracing Settings

### enableTracing

The master switch for tracing:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    enableTracing: true
```

When `false`, Envoy sidecars don't generate trace spans regardless of other settings. Default is `true` in most Istio versions, but it's good practice to set it explicitly.

### Extension Providers

Extension providers define where traces are sent. This is the most important tracing configuration in MeshConfig:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    enableTracing: true
    extensionProviders:
      - name: zipkin
        zipkin:
          service: zipkin.observability.svc.cluster.local
          port: 9411
      - name: jaeger
        zipkin:
          service: jaeger-collector.observability.svc.cluster.local
          port: 9411
      - name: otel
        opentelemetry:
          service: otel-collector.observability.svc.cluster.local
          port: 4317
      - name: skywalking
        skywalking:
          service: skywalking-oap.observability.svc.cluster.local
          port: 11800
```

You can define multiple providers and reference them from Telemetry resources. The provider type determines the protocol used:

| Provider Type | Protocol | Header Format |
|--------------|----------|---------------|
| `zipkin` | Zipkin HTTP/JSON | B3 |
| `opentelemetry` | OTLP gRPC | W3C Trace Context |
| `skywalking` | SkyWalking gRPC | SkyWalking native |

### defaultConfig.tracing

The `defaultConfig` section sets proxy-level defaults that apply to all sidecars:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    enableTracing: true
    defaultConfig:
      tracing:
        sampling: 10.0
        customTags:
          environment:
            literal:
              value: production
          cluster:
            literal:
              value: us-east-1
```

The `sampling` field sets the default sampling percentage (0-100). Custom tags defined here are added to every span generated by every sidecar.

## Complete MeshConfig Tracing Example

Here's a comprehensive example that covers all the tracing-related options:

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    # Enable tracing globally
    enableTracing: true

    # Define tracing backends
    extensionProviders:
      - name: primary-otel
        opentelemetry:
          service: otel-collector.observability.svc.cluster.local
          port: 4317
          resourceDetectors:
            environment: {}
      - name: backup-zipkin
        zipkin:
          service: zipkin.observability.svc.cluster.local
          port: 9411

    # Default proxy configuration
    defaultConfig:
      tracing:
        sampling: 5.0
        customTags:
          mesh.name:
            literal:
              value: production-mesh
          node.name:
            environment:
              name: NODE_NAME
              defaultValue: unknown
```

## Editing MeshConfig at Runtime

You can update MeshConfig without reinstalling Istio:

```bash
kubectl edit configmap istio -n istio-system
```

The ConfigMap's `data.mesh` field contains the YAML configuration:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: istio
  namespace: istio-system
data:
  mesh: |-
    enableTracing: true
    extensionProviders:
      - name: otel
        opentelemetry:
          service: otel-collector.observability.svc.cluster.local
          port: 4317
    defaultConfig:
      tracing:
        sampling: 10.0
```

After editing, istiod picks up the changes and pushes updated configuration to all sidecars. You don't need to restart istiod for ConfigMap changes, but proxy changes typically require a pod restart:

```bash
# Restart workloads to pick up new config
kubectl rollout restart deployment -n my-app
```

## MeshConfig vs Telemetry API

Both configure tracing, but they serve different purposes:

**MeshConfig is for:**
- Defining extension providers (tracing backends)
- Setting global defaults (sampling rate, custom tags)
- Enabling/disabling tracing mesh-wide
- Configuring proxy bootstrap settings

**Telemetry API is for:**
- Selecting which provider to use
- Setting per-namespace or per-workload sampling rates
- Defining per-namespace or per-workload custom tags
- Filtering which requests to trace

A typical setup uses both:

```yaml
# MeshConfig: Define the providers and global defaults
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    enableTracing: true
    extensionProviders:
      - name: otel
        opentelemetry:
          service: otel-collector.observability.svc.cluster.local
          port: 4317
    defaultConfig:
      tracing:
        sampling: 1.0
```

```yaml
# Telemetry API: Activate the provider and customize per-namespace
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: mesh-tracing
  namespace: istio-system
spec:
  tracing:
    - providers:
        - name: otel
      randomSamplingPercentage: 1.0
---
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: staging-tracing
  namespace: staging
spec:
  tracing:
    - providers:
        - name: otel
      randomSamplingPercentage: 100
```

## How MeshConfig Propagates to Proxies

When you update MeshConfig, here's what happens:

1. The `istio` ConfigMap is updated
2. istiod watches the ConfigMap and detects the change
3. istiod regenerates the proxy configuration
4. istiod pushes the new configuration to all connected sidecars via xDS

Some changes take effect immediately (like sampling rate when using the Telemetry API), while others require a proxy restart (like changing the extension provider endpoint).

Verify that a sidecar has the updated configuration:

```bash
# Check the bootstrap config (requires proxy restart for changes)
istioctl proxy-config bootstrap deploy/my-service -o json | python3 -c "
import json, sys
data = json.load(sys.stdin)
print(json.dumps(data.get('bootstrap', {}).get('tracing', {}), indent=2))
"

# Check the dynamic config (updated without restart)
istioctl proxy-config all deploy/my-service -o json | grep -i tracing
```

## Configuring via Helm

If you install Istio with Helm:

```yaml
# values.yaml
meshConfig:
  enableTracing: true
  extensionProviders:
    - name: otel
      opentelemetry:
        service: otel-collector.observability.svc.cluster.local
        port: 4317
  defaultConfig:
    tracing:
      sampling: 5.0
```

```bash
helm upgrade istio-base istio/base -n istio-system -f values.yaml
helm upgrade istiod istio/istiod -n istio-system -f values.yaml
```

## Validating MeshConfig

Before applying changes, validate the configuration:

```bash
# Check for syntax errors
istioctl analyze

# Verify the mesh config is valid
kubectl get configmap istio -n istio-system -o jsonpath='{.data.mesh}' | istioctl validate -f -
```

If there's a syntax error in the mesh config, istiod might reject it silently and continue using the previous configuration. Always check istiod logs after making changes:

```bash
kubectl logs deploy/istiod -n istio-system --tail=20 | grep -i "error\|warn\|mesh"
```

## Common MeshConfig Tracing Mistakes

**Defining providers without activating them:**
Extension providers in MeshConfig just define where traces can be sent. You also need a Telemetry resource to actually activate tracing with that provider.

**Setting sampling in two places:**
If you set `defaultConfig.tracing.sampling` in MeshConfig and also `randomSamplingPercentage` in a Telemetry resource, the Telemetry API value takes precedence. This can be confusing if you're only checking one of them.

**Wrong service names:**
The service name in the extension provider must be a fully qualified Kubernetes service name reachable from all sidecars. Use the full `service.namespace.svc.cluster.local` format to be safe.

**Missing port:**
Each provider type requires a specific port. Zipkin uses 9411, OpenTelemetry uses 4317, SkyWalking uses 11800. A wrong port means traces are sent but never arrive.

## Summary

MeshConfig is the foundation for Istio tracing configuration. It defines your extension providers (where traces go), sets global defaults for sampling and custom tags, and controls whether tracing is enabled. Use it alongside the Telemetry API for a complete tracing setup - MeshConfig for the global infrastructure, Telemetry API for the per-namespace and per-workload fine-tuning. Always verify your changes by checking both the istiod logs and the proxy bootstrap configuration.
