# Validate Cilium on k0s with k0sctl

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Cilium, Kubernetes, K0s, eBPF

Description: A guide to validating Cilium on k0s clusters deployed with k0sctl, including checking k0sctl cluster configuration, Cilium installation via addons, and end-to-end connectivity testing.

---

## Introduction

k0sctl is the cluster lifecycle management tool for k0s, enabling declarative multi-node cluster creation, upgrades, and configuration management. When deploying k0s clusters with Cilium via k0sctl, the entire cluster configuration-including Cilium's Helm chart settings-is defined in the `k0sctl.yaml` file. This declarative approach makes it straightforward to version-control your network configuration but requires specific validation steps to confirm the deployment matched the configuration.

Validating a k0sctl-deployed k0s cluster with Cilium involves checking that the k0sctl configuration was correctly applied, that all k0s nodes are in the expected state, and that Cilium is functioning correctly across all control plane and worker nodes.

## Prerequisites

- k0sctl installed on your local machine
- k0sctl.yaml file used to deploy the cluster
- `kubectl` configured with the cluster's kubeconfig
- `cilium` CLI installed

## Step 1: Review the k0sctl Configuration

Inspect the k0sctl.yaml to confirm Cilium is correctly specified.

```yaml
# k0sctl.yaml - Example with Cilium as CNI
apiVersion: k0sctl.k0sproject.io/v1beta1
kind: Cluster
metadata:
  name: my-k0s-cluster
spec:
  hosts:
    - ssh:
        address: 192.168.1.10
        user: ubuntu
        keyPath: ~/.ssh/id_rsa
      role: controller
    - ssh:
        address: 192.168.1.11
        user: ubuntu
        keyPath: ~/.ssh/id_rsa
      role: worker
  k0s:
    version: "1.29.3+k0s.0"
    config:
      apiVersion: k0s.k0sproject.io/v1beta1
      kind: ClusterConfig
      spec:
        network:
          provider: custom    # Required for Cilium CNI
          podCIDR: 10.244.0.0/16
          serviceCIDR: 10.96.0.0/12
        extensions:
          helm:
            repositories:
              - name: cilium
                url: https://helm.cilium.io
            charts:
              - name: cilium
                chartname: cilium/cilium
                version: "1.15.5"
                namespace: kube-system
                values: |
                  kubeProxyReplacement: true
                  k8sServiceHost: "192.168.1.10"
                  k8sServicePort: "6443"
```

## Step 2: Validate Cluster Deployment Status

Use k0sctl to confirm the cluster was correctly deployed.

```bash
# Check the cluster status via k0sctl
k0sctl kubeconfig --config k0sctl.yaml > /tmp/k0s-kubeconfig.yaml
export KUBECONFIG=/tmp/k0s-kubeconfig.yaml

# Verify all nodes are Ready
kubectl get nodes -o wide

# Check that node count matches k0sctl.yaml spec
kubectl get nodes --no-headers | wc -l
```

## Step 3: Confirm Cilium Addon Deployment

Verify that the Cilium Helm extension was correctly applied by k0sctl.

```bash
# Check if the Cilium Helm chart was deployed as a HelmChart resource
kubectl -n kube-system get helmcharts | grep cilium

# Alternatively, check via Helm
helm list -n kube-system

# Review the deployed Helm values versus the k0sctl.yaml spec
helm get values cilium -n kube-system
```

## Step 4: Validate Cilium Health

```bash
# Check Cilium DaemonSet rollout
kubectl -n kube-system rollout status daemonset/cilium

# Use cilium CLI for full status
cilium status --wait

# Verify kube-proxy is not running (if kubeProxyReplacement: true)
kubectl -n kube-system get daemonset kube-proxy 2>/dev/null && \
  echo "WARNING: kube-proxy still running" || \
  echo "OK: kube-proxy absent"
```

## Step 5: Run Full Connectivity Validation

```bash
# Run the Cilium connectivity test suite
cilium connectivity test --test-namespace cilium-test

# Verify service discovery works (requires kube-proxy replacement to be functional)
kubectl run svc-test --image=busybox:1.36 --restart=Never -- \
  nslookup kubernetes.default.svc.cluster.local
kubectl logs svc-test && kubectl delete pod svc-test
```

## Best Practices

- Version-control your k0sctl.yaml and track Cilium version updates separately
- Use k0sctl's `--dry-run` flag to preview changes before applying
- Test Cilium updates in a staging cluster deployed with the same k0sctl.yaml before production
- Run `cilium connectivity test` as a post-deploy step in your k0sctl automation
- Store the kubeconfig generated by k0sctl securely and rotate it when needed

## Conclusion

Validating a k0sctl-deployed k0s cluster with Cilium combines k0sctl-specific checks (cluster state, addon deployment) with standard Cilium validation. The declarative nature of k0sctl makes it straightforward to ensure the deployed configuration matches intent, while the Cilium connectivity test suite provides comprehensive functional validation of the resulting network configuration.
