# How to Configure Flux CD with Amazon S3 Bucket Source

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: flux cd, amazon s3, bucket source, gitops, kubernetes, aws, irsa

Description: A practical guide to configuring Flux CD to use Amazon S3 buckets as a source for Kubernetes manifests and Helm charts.

---

## Introduction

Flux CD supports multiple source types for fetching Kubernetes manifests, including Git repositories, Helm repositories, and S3-compatible buckets. Using Amazon S3 as a source is useful when your deployment artifacts are generated by CI pipelines and stored in S3 rather than committed to Git.

This guide walks you through configuring Flux CD to pull manifests from an Amazon S3 bucket using IAM Roles for Service Accounts (IRSA) for secure, credential-free access.

## Prerequisites

Before starting, ensure you have:

- An Amazon EKS cluster running Kubernetes 1.25 or later
- Flux CD installed on the cluster
- AWS CLI configured with appropriate permissions
- kubectl configured to communicate with your cluster
- An OIDC provider associated with your EKS cluster

## Step 1: Create the S3 Bucket

First, create an S3 bucket to store your Kubernetes manifests.

```bash
# Create the S3 bucket in your desired region
aws s3api create-bucket \
  --bucket flux-manifests-production \
  --region us-east-1

# Enable versioning so Flux can detect changes
aws s3api put-bucket-versioning \
  --bucket flux-manifests-production \
  --versioning-configuration Status=Enabled
```

## Step 2: Upload Manifests to S3

Organize your manifests in the bucket with a clear directory structure.

```bash
# Example directory structure
# flux-manifests-production/
#   apps/
#     deployment.yaml
#     service.yaml
#   infrastructure/
#     namespace.yaml
#     configmap.yaml

# Upload manifests to the bucket
aws s3 sync ./manifests/ s3://flux-manifests-production/ \
  --delete
```

## Step 3: Create an IAM Policy for S3 Access

Create an IAM policy that grants read-only access to the S3 bucket.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "FluxS3BucketAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket",
        "s3:GetBucketLocation"
      ],
      "Resource": [
        "arn:aws:s3:::flux-manifests-production",
        "arn:aws:s3:::flux-manifests-production/*"
      ]
    }
  ]
}
```

Save this as `flux-s3-policy.json` and create the policy:

```bash
# Create the IAM policy
aws iam create-policy \
  --policy-name FluxS3ReadAccess \
  --policy-document file://flux-s3-policy.json
```

## Step 4: Configure IRSA for the Flux Source Controller

IRSA allows the Flux source controller pod to assume an IAM role without needing static credentials.

```bash
# Get your EKS cluster's OIDC provider URL
OIDC_PROVIDER=$(aws eks describe-cluster \
  --name my-cluster \
  --query "cluster.identity.oidc.issuer" \
  --output text | sed 's|https://||')

# Get your AWS account ID
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)

# Create the trust policy for the IAM role
cat > trust-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::${ACCOUNT_ID}:oidc-provider/${OIDC_PROVIDER}"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "${OIDC_PROVIDER}:sub": "system:serviceaccount:flux-system:source-controller",
          "${OIDC_PROVIDER}:aud": "sts.amazonaws.com"
        }
      }
    }
  ]
}
EOF

# Create the IAM role
aws iam create-role \
  --role-name flux-source-controller-s3 \
  --assume-role-policy-document file://trust-policy.json

# Attach the S3 policy to the role
aws iam attach-role-policy \
  --role-name flux-source-controller-s3 \
  --policy-arn arn:aws:iam::${ACCOUNT_ID}:policy/FluxS3ReadAccess
```

## Step 5: Annotate the Source Controller Service Account

Annotate the Flux source controller service account with the IAM role ARN so that IRSA can inject the credentials.

```yaml
# source-controller-patch.yaml
# This patch adds the IRSA annotation to the source-controller service account
apiVersion: v1
kind: ServiceAccount
metadata:
  name: source-controller
  namespace: flux-system
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/flux-source-controller-s3
```

Apply the patch:

```bash
kubectl apply -f source-controller-patch.yaml
# Restart the source controller to pick up the new credentials
kubectl rollout restart deployment/source-controller -n flux-system
```

## Step 6: Create the Flux Bucket Source

Now define the Flux Bucket resource that points to your S3 bucket.

```yaml
# s3-bucket-source.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: Bucket
metadata:
  name: s3-manifests
  namespace: flux-system
spec:
  # How often to check for changes in the bucket
  interval: 5m
  # The name of the S3 bucket
  bucketName: flux-manifests-production
  # The S3-compatible endpoint (for AWS S3)
  endpoint: s3.amazonaws.com
  # The AWS region where the bucket is located
  region: us-east-1
  # The provider type - use "aws" for native AWS S3
  provider: aws
  # Optional: only sync a specific prefix/directory
  prefix: apps/
  # Ignore certain files during sync
  ignore: |
    # Exclude non-manifest files
    *.md
    *.txt
    .gitignore
```

Apply the Bucket resource:

```bash
kubectl apply -f s3-bucket-source.yaml
```

## Step 7: Create a Kustomization to Deploy from the Bucket

Create a Kustomization resource that references the Bucket source.

```yaml
# s3-kustomization.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: s3-apps
  namespace: flux-system
spec:
  # How often to reconcile the manifests
  interval: 10m
  # Reference the Bucket source created above
  sourceRef:
    kind: Bucket
    name: s3-manifests
  # Path within the bucket to apply
  path: ./
  # Automatically apply changes
  prune: true
  # Wait for resources to become ready
  wait: true
  # Timeout for waiting on resources
  timeout: 5m
  # Health checks for deployed resources
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: my-app
      namespace: default
```

Apply the Kustomization:

```bash
kubectl apply -f s3-kustomization.yaml
```

## Step 8: Verify the Configuration

Check that Flux is successfully pulling from S3.

```bash
# Check the Bucket source status
kubectl get bucket -n flux-system
# Expected output shows the bucket as ready with a revision

# Check detailed status
kubectl describe bucket s3-manifests -n flux-system

# Check the Kustomization status
kubectl get kustomization -n flux-system

# View Flux events for troubleshooting
kubectl get events -n flux-system --sort-by='.lastTimestamp'
```

## Step 9: Configure S3 Bucket Notifications (Optional)

For faster reconciliation, configure S3 event notifications to trigger Flux via a webhook receiver.

```yaml
# webhook-receiver.yaml
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Receiver
metadata:
  name: s3-receiver
  namespace: flux-system
spec:
  type: generic
  # Secret containing the webhook token
  secretRef:
    name: webhook-token
  resources:
    - kind: Bucket
      name: s3-manifests
---
# Secret for the webhook receiver token
apiVersion: v1
kind: Secret
metadata:
  name: webhook-token
  namespace: flux-system
type: Opaque
stringData:
  token: "your-secure-webhook-token-here"
```

## Step 10: Using Static Credentials (Alternative to IRSA)

If IRSA is not available, you can use static AWS credentials stored in a Kubernetes secret.

```yaml
# aws-credentials-secret.yaml
# Note: IRSA is preferred over static credentials for security
apiVersion: v1
kind: Secret
metadata:
  name: s3-credentials
  namespace: flux-system
type: Opaque
stringData:
  accesskey: "AKIAIOSFODNN7EXAMPLE"
  secretkey: "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
```

Then reference the secret in the Bucket resource:

```yaml
# s3-bucket-source-static.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: Bucket
metadata:
  name: s3-manifests
  namespace: flux-system
spec:
  interval: 5m
  bucketName: flux-manifests-production
  endpoint: s3.amazonaws.com
  region: us-east-1
  provider: aws
  # Reference static credentials instead of using IRSA
  secretRef:
    name: s3-credentials
```

## Troubleshooting

Common issues and their solutions:

```bash
# Issue: Bucket source shows "not found" error
# Solution: Verify bucket name and region are correct
aws s3 ls s3://flux-manifests-production/

# Issue: Access denied errors
# Solution: Verify IRSA is configured correctly
kubectl describe sa source-controller -n flux-system | grep eks.amazonaws.com

# Issue: Source controller not picking up IRSA credentials
# Solution: Restart the source controller pod
kubectl rollout restart deployment/source-controller -n flux-system

# Issue: Manifests not applying
# Solution: Check the Kustomization path matches bucket structure
flux get kustomizations
```

## Conclusion

Using Amazon S3 as a Flux CD source provides flexibility for CI/CD pipelines that produce artifacts stored in S3. By leveraging IRSA, you maintain a secure, credential-free setup that follows AWS best practices. The Bucket source controller polls S3 at configurable intervals, and you can enhance responsiveness with webhook-based notifications.
