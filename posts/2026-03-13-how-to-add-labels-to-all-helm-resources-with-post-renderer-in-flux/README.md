# How to Add Labels to All Helm Resources with Post-Renderer in Flux

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderers, Labels, Kustomize

Description: Learn how to use Flux post-renderers to add custom labels to all resources generated by a Helm chart without modifying the chart itself.

---

## Introduction

Adding consistent labels to all Kubernetes resources is a common requirement for organization, cost tracking, compliance, and tooling integration. While many Helm charts support custom labels through values, not all charts expose every labeling option, and some resources within a chart may not inherit the labels you configure.

Flux CD solves this problem with post-renderers. A post-renderer is a processing step that runs after Helm renders the chart templates but before the manifests are applied to the cluster. Flux includes built-in support for Kustomize-based post-renderers, allowing you to add labels to every resource in a Helm release without modifying the chart or maintaining a fork.

This guide shows you how to configure post-renderers to add labels to all resources in a HelmRelease, with practical examples for common labeling scenarios.

## Prerequisites

Before you begin, ensure you have:

- A Kubernetes cluster with Flux CD installed
- A working HelmRelease in your cluster
- kubectl access to the cluster
- Basic familiarity with Kustomize

## How Post-Renderers Work

When Flux processes a HelmRelease, it follows these steps:

1. Fetch the chart from the source
2. Render the chart templates with the configured values
3. Pass the rendered manifests through any configured post-renderers
4. Apply the final manifests to the cluster

Post-renderers receive the full set of rendered Kubernetes manifests and can modify them using Kustomize operations. This includes adding labels, annotations, patches, and other transformations.

## Adding Labels with commonLabels

The simplest way to add labels to all resources is using the Kustomize `commonLabels` feature in a post-renderer:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches: []
        commonLabels:
          team: platform
          cost-center: engineering
          environment: production
```

This adds the labels `team: platform`, `cost-center: engineering`, and `environment: production` to every resource rendered by the chart. Kustomize applies these labels to all resources including Deployments, Services, ConfigMaps, and any other resource types in the chart.

## Understanding Label Propagation

When using `commonLabels`, Kustomize adds the labels to the `metadata.labels` of every resource. For certain resource types, it also propagates labels to nested selectors. Specifically:

- For Deployments, labels are added to `metadata.labels`, `spec.selector.matchLabels`, and `spec.template.metadata.labels`
- For Services, labels are added to `metadata.labels` and `spec.selector`
- For other resources, labels are added to `metadata.labels` only

This propagation behavior is important to understand because it can affect how Services select Pods and how Deployments manage ReplicaSets.

## Adding Labels Without Selector Propagation

If you want to add labels only to the metadata without affecting selectors, use patches instead of `commonLabels`:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ".*"
            patch: |
              - op: add
                path: /metadata/labels/team
                value: platform
              - op: add
                path: /metadata/labels/cost-center
                value: engineering
```

This JSON 6902 patch adds labels only to the metadata of all resources without touching selectors.

## Adding Labels to Specific Resource Types

Sometimes you only want to label certain resource types. Use the target selector to narrow the scope:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: placeholder
                labels:
                  monitoring: enabled
                  log-aggregation: enabled
          - target:
              kind: Service
            patch: |
              apiVersion: v1
              kind: Service
              metadata:
                name: placeholder
                labels:
                  service-mesh: enabled
```

This adds monitoring-related labels to Deployments and service mesh labels to Services.

## Multiple Label Sets for Different Purposes

You can organize labels by purpose using multiple post-renderer entries:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        commonLabels:
          app.kubernetes.io/managed-by: flux
          app.kubernetes.io/part-of: my-platform
          team: backend
          environment: production
          cost-center: eng-backend
```

## Verifying Labels

After applying your HelmRelease, verify that labels were added correctly:

```bash
kubectl get all -n default -l team=platform -o name
```

You can also inspect a specific resource to see all labels:

```bash
kubectl get deployment my-app -n default -o jsonpath='{.metadata.labels}' | jq .
```

## Conclusion

Post-renderers in Flux provide a powerful way to add labels to all resources generated by a Helm chart without modifying the chart itself. Use `commonLabels` when you want labels propagated to selectors, or use patches when you need metadata-only labels. This approach keeps your labeling strategy consistent across all HelmReleases and decoupled from chart-specific configuration, making it easy to enforce organizational labeling standards across your entire cluster.
