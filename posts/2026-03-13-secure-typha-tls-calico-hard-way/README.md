# How to Secure Typha TLS in a Calico Hard Way Installation

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Calico, Typha, Kubernetes, Networking, TLS, Security, Hardening, Hard Way

Description: A guide to hardening Typha TLS security beyond the default configuration including CA protection, CN enforcement, cipher restrictions, and certificate auditing.

---

## Introduction

Securing Typha TLS in a hard way installation goes beyond generating certificates and enabling mTLS. A hardened TLS configuration restricts which cipher suites are allowed, enforces minimum TLS versions, protects the CA private key from unauthorized access, and includes certificate auditing to detect unexpected certificates in the trust chain. These controls are particularly important when the cluster is subject to security compliance requirements.

## Step 1: Protect the CA Private Key

The CA private key is the most sensitive artifact in the Typha TLS setup. If compromised, an attacker can issue valid Felix client certificates and connect to Typha.

```bash
# Verify CA key permissions
ls -la /etc/calico/pki/typha-ca.key
# Should show: -rw------- (600) owned by root
chmod 600 /etc/calico/pki/typha-ca.key
chown root:root /etc/calico/pki/typha-ca.key
```

Do not store the CA private key in a Kubernetes Secret. If the control plane's etcd is compromised, Secrets are accessible - but files with restrictive permissions on the control plane host require OS-level access.

## Step 2: Enforce CN-Based Client Verification

Require Felix to present a specific CN to be accepted by Typha.

```bash
kubectl set env deployment/calico-typha -n calico-system \
  TYPHA_CLIENTCN=calico-felix
```

Test enforcement.

```bash
# Generate a certificate with a different CN
openssl req -x509 -newkey rsa:2048 -keyout /tmp/wrong.key -out /tmp/wrong.crt \
  -days 1 -nodes -subj "/CN=unauthorized-client"

# Attempt connection with wrong CN certificate (should be rejected)
echo | openssl s_client \
  -connect calico-typha.calico-system.svc.cluster.local:5473 \
  -cert /tmp/wrong.crt -key /tmp/wrong.key \
  -CAfile /etc/calico/pki/typha-ca.crt \
  2>&1 | grep -i "alert\|error\|rejected"
```

## Step 3: Restrict TLS Version

```bash
kubectl set env deployment/calico-typha -n calico-system \
  TYPHA_MINTLSVERSION=VersionTLS13
```

TLS 1.3 eliminates known vulnerabilities in older protocol versions and removes weak cipher suites from negotiation.

## Step 4: Audit Secret Access

Track who is accessing the Typha TLS Secrets.

```bash
# Enable audit logging for Secret access in the audit policy
# In /etc/kubernetes/audit-policy.yaml, add:
# - level: Metadata
#   resources:
#   - group: ""
#     resources: ["secrets"]
#   namespaces: ["calico-system"]
#   verbs: ["get", "list", "watch"]
```

```bash
# Check who has access to Typha TLS secrets
kubectl auth can-i get secret/calico-typha-tls -n calico-system --list
```

## Step 5: Verify No Default or Weak Certificates

Ensure the Typha TLS setup does not fall back to self-signed certificates generated by Typha itself.

```bash
# Typha generates a self-signed cert if no cert is configured - verify cert is from our CA
kubectl get secret calico-typha-tls -n calico-system -o jsonpath='{.data.tls\.crt}' | \
  base64 -d | openssl x509 -noout -issuer | grep "calico-typha-ca"
```

If the issuer is not the cluster CA, Typha has fallen back to a self-generated certificate - this means the Secret is misconfigured.

## Step 6: Set Certificate Validity to 90 Days

Short-lived certificates limit the exposure window if a certificate is compromised.

When regenerating certificates, use 90-day validity with automated renewal (cert-manager or Ansible).

```bash
# Regenerate with 90-day validity
openssl x509 -req -in /etc/calico/pki/typha-server.csr \
  -CA /etc/calico/pki/typha-ca.crt \
  -CAkey /etc/calico/pki/typha-ca.key \
  -CAcreateserial \
  -out /etc/calico/pki/typha-server.crt \
  -days 90
```

## Step 7: Rotate Compromised Certificates Immediately

If a Typha or Felix certificate is potentially compromised:

```bash
# Revoke by rotating to new certificates immediately
# 1. Generate new certs (same CA)
# 2. Update secrets
kubectl create secret generic calico-typha-tls \
  --from-file=ca.crt=/etc/calico/pki/typha-ca.crt \
  --from-file=tls.crt=/etc/calico/pki/typha-server-new.crt \
  --from-file=tls.key=/etc/calico/pki/typha-server-new.key \
  -n calico-system --dry-run=client -o yaml | kubectl apply -f -

# 3. Restart Typha immediately
kubectl rollout restart deployment/calico-typha -n calico-system
```

Old certificates with the same CA are still valid until expiry - rotate the CA if the CA key is compromised.

## Conclusion

Hardening Typha TLS involves protecting the CA private key from Kubernetes-accessible storage, enforcing CN-based client verification to reject unauthorized certificates, requiring TLS 1.3, auditing Secret access, verifying that the cluster-issued CA signed all active certificates, using short 90-day certificate validity, and having a tested rotation procedure for compromised certificates. Together these controls create a defense-in-depth TLS security posture for the Calico fan-out layer.
