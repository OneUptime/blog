# How to Use OpenTofu for_each with Count Results

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: OpenTofu, Terraform, for_each, Count, Infrastructure as Code, HCL

Description: Learn how to use OpenTofu's feature that allows for_each to reference resources created with count, solving a common limitation in Terraform HCL configuration patterns.

---

In Terraform, you cannot use `for_each` on a resource that references another resource created with `count`, because the number of items is not known until apply time. This is one of the most frustrating limitations in HCL. OpenTofu addresses this with improved `for_each` handling that works with count results. This guide explains the problem, the solution, and practical patterns.

## The Problem in Terraform

Consider this common scenario: you create several subnets using `count`, then want to create route table associations for each subnet using `for_each`:

```hcl
# Create subnets with count
resource "aws_subnet" "private" {
  count = 3

  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(aws_vpc.main.cidr_block, 8, count.index)
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "private-${count.index}"
  }
}

# THIS FAILS IN TERRAFORM
# "The for_each value depends on resource attributes that cannot
# be determined until apply"
resource "aws_route_table_association" "private" {
  for_each = { for idx, subnet in aws_subnet.private : idx => subnet.id }

  subnet_id      = each.value
  route_table_id = aws_route_table.private.id
}
```

Terraform rejects this because `aws_subnet.private` is a list whose contents are not fully known during planning (the subnet IDs are generated by AWS). The `for_each` meta-argument requires all keys to be known at plan time.

## The Traditional Workarounds

Before OpenTofu fixed this, teams used several workarounds.

### Workaround 1: Use count on both resources

```hcl
# Both resources use count
resource "aws_subnet" "private" {
  count = 3
  # ...
}

resource "aws_route_table_association" "private" {
  count = 3

  subnet_id      = aws_subnet.private[count.index].id
  route_table_id = aws_route_table.private.id
}
```

This works but has a problem: if you remove a subnet from the middle of the list (say, index 1), both subnet 1 and 2 get recreated because the indices shift. With `for_each` and proper keys, only the target resource would be removed.

### Workaround 2: Use for_each on both resources

```hcl
variable "subnets" {
  type = map(object({
    cidr_block        = string
    availability_zone = string
  }))
}

resource "aws_subnet" "private" {
  for_each = var.subnets

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value.cidr_block
  availability_zone = each.value.availability_zone
}

resource "aws_route_table_association" "private" {
  for_each = var.subnets

  subnet_id      = aws_subnet.private[each.key].id
  route_table_id = aws_route_table.private.id
}
```

This works but requires restructuring your configuration to use maps instead of counts, which is not always practical.

## OpenTofu's Solution

OpenTofu improves `for_each` to handle references to count-based resources more gracefully. The key insight is that even though the resource attributes (like IDs) are unknown during planning, the number of resources and their indices are known.

```hcl
# In OpenTofu, this works
resource "aws_subnet" "private" {
  count = 3

  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(aws_vpc.main.cidr_block, 8, count.index)
  availability_zone = data.aws_availability_zones.available.names[count.index]
}

resource "aws_route_table_association" "private" {
  for_each = { for idx, subnet in aws_subnet.private : tostring(idx) => subnet }

  subnet_id      = each.value.id
  route_table_id = aws_route_table.private.id
}
```

OpenTofu can determine that the `for_each` map will have keys "0", "1", and "2" during planning, even though the subnet IDs within each value are not yet known. It plans the correct number of resources with the correct keys.

## Practical Patterns

### Pattern 1: Subnets with Route Table Associations

```hcl
variable "az_count" {
  type    = number
  default = 3
}

data "aws_availability_zones" "available" {
  state = "available"
}

resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "public" {
  count = var.az_count

  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(aws_vpc.main.cidr_block, 8, count.index)
  availability_zone = data.aws_availability_zones.available.names[count.index]

  map_public_ip_on_launch = true

  tags = {
    Name = "public-${count.index}"
    Tier = "public"
  }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
}

# Use for_each referencing count-based subnets
resource "aws_route_table_association" "public" {
  for_each = {
    for idx, subnet in aws_subnet.public :
    tostring(idx) => subnet
  }

  subnet_id      = each.value.id
  route_table_id = aws_route_table.public.id
}
```

### Pattern 2: Security Group Rules from Instance List

```hcl
resource "aws_instance" "workers" {
  count = var.worker_count

  ami           = var.ami_id
  instance_type = var.instance_type
  subnet_id     = aws_subnet.private[count.index % length(aws_subnet.private)].id

  tags = {
    Name = "worker-${count.index}"
  }
}

# Create security group rules for each worker
resource "aws_security_group_rule" "worker_ingress" {
  for_each = {
    for idx, instance in aws_instance.workers :
    "worker-${idx}" => instance
  }

  type              = "ingress"
  from_port         = 8080
  to_port           = 8080
  protocol          = "tcp"
  cidr_blocks       = ["${each.value.private_ip}/32"]
  security_group_id = aws_security_group.app.id
}
```

### Pattern 3: DNS Records for Load-Balanced Instances

```hcl
resource "aws_instance" "app" {
  count = var.instance_count

  ami           = var.ami_id
  instance_type = "t3.medium"
  subnet_id     = aws_subnet.private[count.index % 3].id
}

# Create DNS records for each instance
resource "aws_route53_record" "app" {
  for_each = {
    for idx, instance in aws_instance.app :
    "app-${idx}" => instance
  }

  zone_id = data.aws_route53_zone.main.zone_id
  name    = "app-${each.key}.internal.example.com"
  type    = "A"
  ttl     = 300
  records = [each.value.private_ip]
}
```

### Pattern 4: EBS Volumes for EC2 Instances

```hcl
resource "aws_instance" "database" {
  count = var.db_node_count

  ami           = var.ami_id
  instance_type = "r6g.xlarge"
  subnet_id     = aws_subnet.private[count.index].id
}

# Attach additional EBS volumes to each instance
resource "aws_ebs_volume" "data" {
  for_each = {
    for idx, instance in aws_instance.database :
    "db-${idx}" => {
      instance  = instance
      az        = instance.availability_zone
    }
  }

  availability_zone = each.value.az
  size              = 500
  type              = "gp3"
  iops              = 3000

  tags = {
    Name = "data-${each.key}"
  }
}

resource "aws_volume_attachment" "data" {
  for_each = aws_ebs_volume.data

  device_name = "/dev/xvdf"
  volume_id   = each.value.id
  instance_id = aws_instance.database[tonumber(split("-", each.key)[1])].id
}
```

## Best Practices for for_each Keys

When creating the `for_each` map from count resources, choose stable keys:

```hcl
# Good: string index as key (stable)
for_each = {
  for idx, item in aws_instance.workers :
  tostring(idx) => item
}

# Good: descriptive key with index
for_each = {
  for idx, item in aws_instance.workers :
  "worker-${idx}" => item
}

# Bad: using a computed attribute as key (may change)
for_each = {
  for idx, item in aws_instance.workers :
  item.id => item  # ID changes if instance is recreated
}
```

The key should be something that does not change when the resource is recreated. Index-based keys work well because they are determined by the configuration, not by AWS.

## Combining with Dynamic Blocks

You can also use count results inside dynamic blocks:

```hcl
resource "aws_instance" "app" {
  count = var.app_count

  ami           = var.ami_id
  instance_type = "t3.medium"
  subnet_id     = aws_subnet.private[count.index].id
}

resource "aws_lb_target_group" "app" {
  name     = "app-tg"
  port     = 8080
  protocol = "HTTP"
  vpc_id   = aws_vpc.main.id
}

# Register all instances with the target group
resource "aws_lb_target_group_attachment" "app" {
  for_each = {
    for idx, instance in aws_instance.app :
    tostring(idx) => instance
  }

  target_group_arn = aws_lb_target_group.app.arn
  target_id        = each.value.id
  port             = 8080
}
```

## Migrating Existing Count-to-Count Patterns

If you have existing resources using count that you want to convert to for_each:

```hcl
# Before (both use count)
resource "aws_route_table_association" "private" {
  count = 3
  subnet_id      = aws_subnet.private[count.index].id
  route_table_id = aws_route_table.private.id
}

# After (second resource uses for_each)
resource "aws_route_table_association" "private" {
  for_each = {
    for idx, subnet in aws_subnet.private :
    tostring(idx) => subnet
  }
  subnet_id      = each.value.id
  route_table_id = aws_route_table.private.id
}
```

When making this change, you need to move state to avoid resource recreation:

```bash
# Move state from count indices to for_each keys
tofu state mv \
  'aws_route_table_association.private[0]' \
  'aws_route_table_association.private["0"]'

tofu state mv \
  'aws_route_table_association.private[1]' \
  'aws_route_table_association.private["1"]'

tofu state mv \
  'aws_route_table_association.private[2]' \
  'aws_route_table_association.private["2"]'
```

## When to Use Count vs for_each

Even with this improvement, the general guidance remains:

- Use `for_each` when resources are identified by a meaningful key (like name, region, or role)
- Use `count` when resources are numbered or identical
- Use the new `for_each` with count results when you need the stability of `for_each` keys but your source data comes from a count-based resource

The ability to use `for_each` with count results removes one of the most annoying limitations in HCL. It lets you mix count and for_each in the same configuration without restructuring everything into maps upfront. This is a practical improvement that simplifies real-world infrastructure code.
