# How to Use netplan apply vs netplan try on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, Networking, Netplan, Configuration

Description: Understand the difference between netplan apply and netplan try on Ubuntu, when to use each command, and how netplan try protects you from locking yourself out of remote servers.

---

Netplan is Ubuntu's network configuration layer that sits above NetworkManager and systemd-networkd. It reads YAML files from `/etc/netplan/` and generates the appropriate backend configuration. When you change netplan configuration, you have two ways to apply it: `netplan apply` and `netplan try`. They do similar things but with an important safety difference.

## The Basic Difference

`netplan apply` applies your configuration changes immediately and permanently, with no rollback mechanism.

`netplan try` applies your configuration changes temporarily and rolls back automatically after a timeout (120 seconds by default) unless you confirm the changes are working.

That timeout-based rollback is what makes `netplan try` valuable, particularly when working on remote servers where a bad network configuration would lock you out.

## When Each Command Makes Sense

Use `netplan apply`:
- In scripts or automation where interactive confirmation isn't possible
- When you're physically in front of the machine and can fix things if they go wrong
- After already testing with `netplan try` and confirming the config works

Use `netplan try`:
- When editing network config on a remote server
- When making complex changes you're not 100% sure about
- When testing new configurations before committing them
- Essentially, any interactive use case

## Netplan Try in Practice

The workflow looks like this:

```bash
# Edit your netplan configuration
sudo nano /etc/netplan/01-netcfg.yaml
```

Make your changes, save the file, then:

```bash
# Apply temporarily
sudo netplan try
```

You'll see output like:

```
Do you want to keep these settings?

Press ENTER before the timeout to accept the new configuration

Changes will revert in 120 seconds
```

If your network connection is still working (you can still SSH into the server, for example), press Enter to make the changes permanent. If something broke and you can't press Enter, the system automatically rolls back to the previous configuration after 120 seconds.

You can also specify a custom timeout:

```bash
# Give yourself 300 seconds instead of 120
sudo netplan try --timeout 300
```

## Netplan Apply in Practice

```bash
# Apply configuration immediately and permanently
sudo netplan apply
```

No confirmation prompt. Changes are live instantly. If the configuration breaks networking, you'll need physical console access or out-of-band management to fix it.

For scripting:

```bash
#!/bin/bash
# Script to update network config and apply it

# Write new config
cat > /etc/netplan/01-netcfg.yaml << 'EOF'
network:
  version: 2
  ethernets:
    eth0:
      dhcp4: true
      dhcp6: false
EOF

# Apply without interactive prompt
netplan apply
```

## Validating Before Applying

Before either apply or try, validate your YAML syntax:

```bash
# Check for syntax errors in netplan config
sudo netplan generate

# More verbose output
sudo netplan --debug generate
```

`netplan generate` creates the backend configuration files in `/run/systemd/network/` without activating them. If there's a YAML syntax error, it reports it. Fix syntax errors before trying to apply.

```bash
# Full workflow for safe remote changes
sudo netplan generate    # validate syntax
sudo netplan try         # apply temporarily
# verify networking still works
# press Enter to confirm
```

## Understanding What Netplan Actually Does

Netplan generates configuration for either systemd-networkd or NetworkManager as a backend:

```bash
# See which backend is configured
grep renderer /etc/netplan/*.yaml

# Check what files were generated for systemd-networkd
ls /run/systemd/network/

# Check NetworkManager profiles generated by netplan
ls /run/NetworkManager/system-connections/
```

When you run `netplan apply`, it:
1. Generates backend config files
2. Asks the backend (systemd-networkd or NetworkManager) to reload its configuration
3. The changes take effect

When you run `netplan try`, it does the same but also:
1. Saves the previous configuration state
2. Starts a timeout countdown
3. On Enter: marks changes permanent
4. On timeout: reverts to previous state and asks the backend to reload

## Common Netplan Configuration Examples

A typical static IP configuration:

```yaml
# /etc/netplan/01-netcfg.yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    ens3:
      addresses:
        - 192.168.1.100/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses: [8.8.8.8, 1.1.1.1]
      dhcp4: false
```

A DHCP configuration:

```yaml
network:
  version: 2
  ethernets:
    ens3:
      dhcp4: true
      dhcp6: false
```

A bonded interface configuration:

```yaml
network:
  version: 2
  ethernets:
    eth0:
      dhcp4: false
    eth1:
      dhcp4: false
  bonds:
    bond0:
      interfaces: [eth0, eth1]
      parameters:
        mode: active-backup
        primary: eth0
      dhcp4: true
```

Apply any of these safely:

```bash
# Validate first
sudo netplan generate

# Try it on a remote server
sudo netplan try

# Confirm with Enter if working
```

## Troubleshooting Apply Issues

If `netplan apply` breaks networking and you have console access:

```bash
# View what netplan generated
cat /run/systemd/network/*.network

# Manually restart networkd
sudo systemctl restart systemd-networkd

# Or restart NetworkManager
sudo systemctl restart NetworkManager

# Check networkd status
networkctl status

# View logs
journalctl -u systemd-networkd -n 50
```

If you need to revert manually after `netplan apply`:

```bash
# Restore from backup if you made one
sudo cp /etc/netplan/01-netcfg.yaml.bak /etc/netplan/01-netcfg.yaml
sudo netplan apply
```

This is why making a backup before changes is good practice:

```bash
# Always back up before editing
sudo cp /etc/netplan/01-netcfg.yaml /etc/netplan/01-netcfg.yaml.bak.$(date +%Y%m%d%H%M%S)

# Then edit
sudo nano /etc/netplan/01-netcfg.yaml

# Test safely
sudo netplan try
```

The distinction between `netplan apply` and `netplan try` is simple but consequential. For remote server management, always default to `netplan try`. The 120-second rollback window has saved many sysadmins from an emergency trip to the datacenter.
