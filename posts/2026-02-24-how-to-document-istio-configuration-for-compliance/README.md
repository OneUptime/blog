# How to Document Istio Configuration for Compliance

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Compliance, Documentation, Security, Audit

Description: Prepare Istio configuration documentation that satisfies compliance audits for frameworks like SOC 2, PCI-DSS, HIPAA, and ISO 27001.

---

Compliance auditors want evidence that your security controls are in place, properly configured, and reviewed regularly. Istio provides several controls that map to compliance requirements: mTLS for encryption in transit, authorization policies for access control, and telemetry for audit logging. But auditors don't look at YAML files. They need structured documentation that maps technical controls to compliance requirements.

## Mapping Istio Controls to Compliance Frameworks

Before generating documentation, understand which Istio features map to which compliance requirements:

**SOC 2 (Trust Services Criteria):**
- CC6.1 (Logical access security) -> AuthorizationPolicy
- CC6.6 (Encryption in transit) -> PeerAuthentication (mTLS)
- CC7.2 (System monitoring) -> Istio telemetry and access logs

**PCI-DSS:**
- Requirement 4.1 (Encrypt transmission) -> PeerAuthentication (STRICT mTLS)
- Requirement 7.1 (Limit access by business need) -> AuthorizationPolicy
- Requirement 10.1 (Audit trail) -> Istio access logging

**HIPAA:**
- 164.312(a)(1) (Access control) -> AuthorizationPolicy
- 164.312(e)(1) (Transmission security) -> PeerAuthentication (mTLS)

**ISO 27001:**
- A.13.1.1 (Network controls) -> NetworkPolicy + AuthorizationPolicy
- A.13.2.1 (Information transfer) -> PeerAuthentication (mTLS)
- A.9.4.1 (Information access restriction) -> AuthorizationPolicy

## Generating a Compliance Evidence Report

Build a script that generates a compliance-ready report:

```bash
#!/bin/bash
# compliance-report.sh

REPORT_DATE=$(date -u +%FT%TZ)
CLUSTER_NAME=${CLUSTER_NAME:-"production"}

cat << EOF
# Istio Security Controls - Compliance Evidence Report

**Report Date:** ${REPORT_DATE}
**Cluster:** ${CLUSTER_NAME}
**Generated By:** Automated compliance reporter
**Review Status:** Pending review

---

## 1. Encryption in Transit (mTLS)

### 1.1 Mesh-Wide mTLS Configuration
EOF

# Check mesh-wide mTLS
MESH_MTLS=$(kubectl get peerauthentication -n istio-system -o json 2>/dev/null | \
  jq -r '.items[] | select(.metadata.name == "default") | .spec.mtls.mode // "NOT SET"')

echo "**Mesh-wide mTLS mode:** ${MESH_MTLS:-NOT CONFIGURED}"
echo ""

echo "### 1.2 Namespace-Level mTLS Policies"
echo ""
echo "| Namespace | Policy Name | mTLS Mode | Port Exceptions |"
echo "|-----------|-------------|-----------|-----------------|"

kubectl get peerauthentications -A -o json | jq -r '
  .items[] |
  "| " + .metadata.namespace +
  " | " + .metadata.name +
  " | " + (.spec.mtls.mode // "UNSET") +
  " | " + (if .spec.portLevelMtls then (.spec.portLevelMtls | to_entries | map(.key + ":" + .value.mode) | join(", ")) else "None" end) +
  " |"
'

cat << 'EOF'

### 1.3 mTLS Compliance Assessment

EOF

# Count namespaces
TOTAL=$(kubectl get ns -l istio-injection=enabled -o name 2>/dev/null | wc -l | tr -d ' ')
STRICT=$(kubectl get peerauthentications -A -o json 2>/dev/null | \
  jq '[.items[] | select(.spec.mtls.mode == "STRICT")] | length')

echo "- Istio-enabled namespaces: $TOTAL"
echo "- Namespaces with STRICT mTLS: $STRICT"
if [ "$TOTAL" = "$STRICT" ]; then
  echo "- **Status: COMPLIANT** - All Istio namespaces enforce strict mTLS"
else
  echo "- **Status: FINDING** - Some namespaces do not enforce strict mTLS"
fi

cat << 'EOF'

## 2. Access Control (Authorization Policies)

### 2.1 Authorization Policy Inventory

EOF

echo "| Namespace | Policy | Action | Target | Rules |"
echo "|-----------|--------|--------|--------|-------|"

kubectl get authorizationpolicies -A -o json | jq -r '
  .items[] |
  "| " + .metadata.namespace +
  " | " + .metadata.name +
  " | " + (.spec.action // "ALLOW") +
  " | " + (.spec.selector.matchLabels | to_entries | map(.value) | join(",")) +
  " | " + ((.spec.rules // []) | length | tostring) + " rules |"
'

cat << 'EOF'

### 2.2 Default Deny Assessment

EOF

# Check for default deny policies
DENY_ALL=$(kubectl get authorizationpolicies -A -o json | \
  jq '[.items[] | select(.metadata.name == "deny-all" or .metadata.name == "default-deny")] | length')

echo "- Default deny policies found: $DENY_ALL"
if [ "$DENY_ALL" -gt 0 ]; then
  echo "- **Status: COMPLIANT** - Default deny is configured"
else
  echo "- **Status: FINDING** - No default deny policy detected"
fi
```

## Documenting mTLS Evidence

Auditors want proof that encryption is actually happening, not just configured. Generate evidence:

```bash
#!/bin/bash
# mtls-evidence.sh

echo "## mTLS Verification Evidence"
echo ""
echo "### Active mTLS Connections"
echo ""

# Sample connections to verify mTLS is active
for deploy in $(kubectl get deploy -n production -o name | head -5); do
  NAME=$(echo $deploy | cut -d/ -f2)
  echo "#### $NAME"

  # Check TLS version on active connections
  kubectl exec -n production deploy/$NAME -c istio-proxy -- \
    pilot-agent request GET /stats | grep "ssl.versions" 2>/dev/null

  echo ""
done

echo "### Certificate Details"
echo ""

# Show certificate info for a sample workload
SAMPLE_POD=$(kubectl get pods -n production -o name | head -1)
if [ -n "$SAMPLE_POD" ]; then
  kubectl exec -n production $SAMPLE_POD -c istio-proxy -- \
    openssl x509 -text -noout -in /var/run/secrets/istio/cert-chain.pem 2>/dev/null | \
    grep -E "Issuer|Subject|Not Before|Not After"
fi
```

## Documenting Authorization Policy Review Schedule

Compliance frameworks require regular review of access controls. Track review dates:

```yaml
# Example policy with review tracking
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: payment-access
  namespace: production
  annotations:
    compliance/created: "2025-06-15"
    compliance/last-reviewed: "2026-01-15"
    compliance/next-review: "2026-07-15"
    compliance/reviewer: "security-team"
    compliance/framework: "PCI-DSS 7.1"
    compliance/ticket: "COMP-2026-015"
spec:
  selector:
    matchLabels:
      app: payment-service
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/production/sa/checkout"
    to:
    - operation:
        methods: ["POST"]
        paths: ["/api/v1/charge"]
```

Generate a review schedule report:

```bash
#!/bin/bash
# review-schedule.sh

echo "# Authorization Policy Review Schedule"
echo ""
echo "| Policy | Namespace | Last Reviewed | Next Review | Reviewer | Status |"
echo "|--------|-----------|---------------|-------------|----------|--------|"

TODAY=$(date +%Y-%m-%d)

kubectl get authorizationpolicies -A -o json | jq -r --arg today "$TODAY" '
  .items[] |
  . as $item |
  ($item.metadata.annotations["compliance/next-review"] // "NOT SET") as $next |
  (if $next < $today and $next != "NOT SET" then "OVERDUE" elif $next == "NOT SET" then "NO SCHEDULE" else "OK" end) as $status |
  "| " + $item.metadata.name +
  " | " + $item.metadata.namespace +
  " | " + ($item.metadata.annotations["compliance/last-reviewed"] // "Never") +
  " | " + $next +
  " | " + ($item.metadata.annotations["compliance/reviewer"] // "Unassigned") +
  " | " + $status + " |"
'
```

## Audit Log Configuration Documentation

Document how Istio access logs are configured, where they're stored, and for how long:

```yaml
# Document the access log configuration
apiVersion: v1
kind: ConfigMap
metadata:
  name: compliance-documentation
  namespace: istio-system
data:
  access-logging.md: |
    ## Istio Access Logging Configuration

    ### Log Format
    Istio access logs capture the following fields:
    - Source identity (service account, namespace)
    - Destination identity (service, pod, port)
    - Request details (method, path, protocol)
    - Response details (status code, response time)
    - TLS information (mTLS version, cipher)

    ### Log Storage
    - Short-term: Fluentd -> Elasticsearch (30 days)
    - Long-term: Fluentd -> S3 (1 year, encrypted at rest)
    - Access: Security team has read access, SOC team has read access

    ### Log Integrity
    - Logs are immutable in S3 (object lock enabled)
    - SHA-256 checksums generated daily
    - Tamper detection alerts configured
```

Verify access logging is enabled:

```bash
# Check mesh config for access logging
kubectl get configmap istio -n istio-system -o json | \
  jq -r '.data.mesh' | grep -A10 "accessLog"

# Verify logs are actually being generated
kubectl logs -n production deploy/payment-service -c istio-proxy --tail=5 | head -3
```

## Automated Compliance Checks

Run these checks before every audit to identify gaps:

```bash
#!/bin/bash
# pre-audit-check.sh

FINDINGS=0

echo "# Pre-Audit Compliance Check"
echo ""

# Check 1: mTLS is STRICT everywhere
echo "## Check 1: mTLS Enforcement"
NON_STRICT=$(kubectl get peerauthentications -A -o json | \
  jq '[.items[] | select(.spec.mtls.mode != "STRICT")] | length')
if [ "$NON_STRICT" -gt 0 ]; then
  echo "FINDING: $NON_STRICT namespaces without STRICT mTLS"
  FINDINGS=$((FINDINGS + 1))
else
  echo "PASS: All namespaces enforce STRICT mTLS"
fi

# Check 2: No wildcard authorization policies
echo ""
echo "## Check 2: No Wildcard Access"
WILDCARDS=$(kubectl get authorizationpolicies -A -o json | \
  jq '[.items[] | select(.spec.rules[]?.from == null)] | length')
if [ "$WILDCARDS" -gt 0 ]; then
  echo "FINDING: $WILDCARDS policies with no source restriction"
  FINDINGS=$((FINDINGS + 1))
else
  echo "PASS: All policies have explicit source restrictions"
fi

# Check 3: All policies have compliance annotations
echo ""
echo "## Check 3: Policy Documentation"
UNDOCUMENTED=$(kubectl get authorizationpolicies -A -o json | \
  jq '[.items[] | select(.metadata.annotations["compliance/framework"] == null)] | length')
if [ "$UNDOCUMENTED" -gt 0 ]; then
  echo "FINDING: $UNDOCUMENTED policies without compliance framework annotation"
  FINDINGS=$((FINDINGS + 1))
else
  echo "PASS: All policies have compliance framework annotation"
fi

echo ""
echo "## Summary"
echo "Total findings: $FINDINGS"
```

Treat compliance documentation as code. Store it in version control, generate it from live configuration, review it regularly, and automate the checks. This approach scales much better than manually maintaining spreadsheets, and it gives auditors confidence that the documentation reflects reality.
