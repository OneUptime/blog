# How to Verify Flux CD SLSA Provenance

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, GitOps, Kubernetes, Security, SLSA, Supply Chain, Provenance

Description: Learn how to verify Flux CD SLSA provenance attestations to ensure controller images were built from trusted sources using trusted build systems.

---

SLSA (Supply-chain Levels for Software Artifacts) is a framework for ensuring the integrity of software artifacts throughout the supply chain. Flux CD provides SLSA provenance attestations for its controller images, proving they were built from a specific source commit using a specific build system. This guide shows you how to verify these attestations.

## What is SLSA Provenance

SLSA provenance is a signed attestation that describes:

- **Who** built the artifact (the build system identity).
- **What** source code was used (the Git commit and repository).
- **How** it was built (the build configuration and steps).

Flux CD achieves SLSA Level 3 compliance, meaning builds are performed on a hosted, hardened build service (GitHub Actions) with provenance generated by a trusted third party (the SLSA GitHub Generator).

## Prerequisites

Install the required tools:

```bash
# Install Cosign for signature and attestation verification
brew install cosign

# Install slsa-verifier for SLSA provenance verification
brew install slsa-framework/slsa-verifier/slsa-verifier

# Install Flux CLI
brew install fluxcd/tap/flux

# Verify installations
cosign version
slsa-verifier version
```

## Step 1: Verify SLSA Provenance for Flux CLI Binary

The Flux CLI binary includes SLSA provenance that can be verified with slsa-verifier:

```bash
# Download the Flux CLI binary and its provenance
FLUX_VERSION="2.4.0"
curl -sLO "https://github.com/fluxcd/flux2/releases/download/v${FLUX_VERSION}/flux_${FLUX_VERSION}_linux_amd64.tar.gz"
curl -sLO "https://github.com/fluxcd/flux2/releases/download/v${FLUX_VERSION}/flux_${FLUX_VERSION}_linux_amd64.tar.gz.prov"

# Verify the SLSA provenance of the Flux CLI binary
slsa-verifier verify-artifact \
  "flux_${FLUX_VERSION}_linux_amd64.tar.gz" \
  --provenance-path "flux_${FLUX_VERSION}_linux_amd64.tar.gz.prov" \
  --source-uri github.com/fluxcd/flux2 \
  --source-tag "v${FLUX_VERSION}"
```

## Step 2: Verify SLSA Provenance for Controller Images

Flux controller images have SLSA provenance attestations attached as OCI artifacts. Verify them using Cosign:

```bash
# Verify source-controller SLSA provenance
cosign verify-attestation ghcr.io/fluxcd/source-controller:v1.4.1 \
  --type slsaprovenance \
  --certificate-identity-regexp="^https://github.com/slsa-framework/slsa-github-generator/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com"

# Verify kustomize-controller SLSA provenance
cosign verify-attestation ghcr.io/fluxcd/kustomize-controller:v1.4.0 \
  --type slsaprovenance \
  --certificate-identity-regexp="^https://github.com/slsa-framework/slsa-github-generator/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com"

# Verify helm-controller SLSA provenance
cosign verify-attestation ghcr.io/fluxcd/helm-controller:v1.1.0 \
  --type slsaprovenance \
  --certificate-identity-regexp="^https://github.com/slsa-framework/slsa-github-generator/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com"
```

## Step 3: Inspect the Provenance Attestation

Extract and examine the provenance details to understand the build:

```bash
# Download and decode the provenance attestation
cosign verify-attestation ghcr.io/fluxcd/source-controller:v1.4.1 \
  --type slsaprovenance \
  --certificate-identity-regexp="^https://github.com/slsa-framework/slsa-github-generator/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  --output json | jq -r '.payload' | base64 -d | jq .

# Extract specific provenance fields
cosign verify-attestation ghcr.io/fluxcd/source-controller:v1.4.1 \
  --type slsaprovenance \
  --certificate-identity-regexp="^https://github.com/slsa-framework/slsa-github-generator/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  --output json | jq -r '.payload' | base64 -d | jq '{
    builder: .predicate.builder.id,
    sourceRepo: .predicate.invocation.configSource.uri,
    sourceDigest: .predicate.invocation.configSource.digest,
    buildType: .predicate.buildType
  }'
```

## Step 4: Verify All Flux Controller Images

Create a script to verify all controller images at once:

```bash
#!/bin/bash
# verify-flux-slsa.sh
# Verifies SLSA provenance for all Flux CD controller images

CONTROLLERS=(
  "ghcr.io/fluxcd/source-controller:v1.4.1"
  "ghcr.io/fluxcd/kustomize-controller:v1.4.0"
  "ghcr.io/fluxcd/helm-controller:v1.1.0"
  "ghcr.io/fluxcd/notification-controller:v1.4.0"
)

ISSUER="https://token.actions.githubusercontent.com"
IDENTITY="^https://github.com/slsa-framework/slsa-github-generator/.*"

PASS=0
FAIL=0

for IMAGE in "${CONTROLLERS[@]}"; do
  echo "Verifying SLSA provenance: $IMAGE"
  if cosign verify-attestation "$IMAGE" \
    --type slsaprovenance \
    --certificate-identity-regexp="$IDENTITY" \
    --certificate-oidc-issuer="$ISSUER" > /dev/null 2>&1; then
    echo "  PASSED: SLSA provenance verified"
    PASS=$((PASS + 1))
  else
    echo "  FAILED: SLSA provenance verification failed"
    FAIL=$((FAIL + 1))
  fi
done

echo ""
echo "Results: $PASS passed, $FAIL failed"
exit $FAIL
```

## Step 5: Integrate into CI/CD Pipeline

Add SLSA verification to your deployment pipeline:

```yaml
# .github/workflows/verify-flux-slsa.yaml
# Verify Flux CD SLSA provenance before deploying upgrades
name: Verify Flux SLSA Provenance
on:
  pull_request:
    paths:
      - 'clusters/**/flux-system/gotk-components.yaml'

jobs:
  verify-slsa:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Extract and verify Flux images
        run: |
          ISSUER="https://token.actions.githubusercontent.com"
          IDENTITY="^https://github.com/slsa-framework/slsa-github-generator/.*"

          # Extract images from gotk-components.yaml
          IMAGES=$(grep "image:" clusters/production/flux-system/gotk-components.yaml \
            | awk '{print $2}' | sort -u)

          for IMAGE in $IMAGES; do
            echo "Verifying SLSA provenance for: $IMAGE"
            cosign verify-attestation "$IMAGE" \
              --type slsaprovenance \
              --certificate-identity-regexp="$IDENTITY" \
              --certificate-oidc-issuer="$ISSUER"
          done
```

## Understanding SLSA Levels

Flux CD achieves SLSA Level 3, which provides:

| Requirement | SLSA L3 | Flux CD |
|---|---|---|
| Source versioned | Yes | Git tags on GitHub |
| Build service | Hosted | GitHub Actions |
| Build as code | Yes | Workflow files in repo |
| Provenance generated | By build service | SLSA GitHub Generator |
| Non-falsifiable provenance | Yes | Signed by Sigstore |

## Best Practices

1. **Verify before every upgrade**: Always verify SLSA provenance before deploying new Flux versions.
2. **Automate verification**: Include provenance checks in your CI/CD pipeline as a required gate.
3. **Pin to digests**: Use image digests instead of tags to ensure the exact verified image is deployed.
4. **Combine with image signatures**: Verify both Cosign signatures and SLSA provenance for defense in depth.
5. **Check source repository**: Inspect the provenance to confirm the source URI matches the expected Flux GitHub repository.

Verifying SLSA provenance gives you confidence that Flux CD controller images were built from the correct source code using a trusted, tamper-resistant build process. This is a key component of supply chain security for your GitOps infrastructure.
