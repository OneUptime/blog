# How to Add Node Selectors with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderer, Node Selector, Scheduling

Description: Learn how to use Flux post-renderers to add node selectors to Helm chart workloads for controlling pod placement on specific nodes.

---

## Introduction

Node selectors are the simplest way to constrain pods to run on specific nodes in a Kubernetes cluster. They allow you to schedule workloads on nodes with particular characteristics, such as specific hardware, geographic location, or dedicated node pools. While many Helm charts expose node selector configuration through values, some charts do not, and even those that do may not apply node selectors consistently to all resources.

Flux post-renderers allow you to add node selectors to any workload resource generated by a Helm chart. Using Kustomize patches, you can ensure all pods land on the correct nodes regardless of the chart's built-in configuration options.

This guide shows you how to add node selectors to Helm chart workloads using Flux post-renderers, with examples for common scheduling scenarios.

## Prerequisites

Before you begin, ensure you have:

- A Kubernetes cluster with Flux CD installed and nodes labeled appropriately
- A HelmRelease deployed through Flux
- kubectl access to the cluster
- Nodes with labels for scheduling (e.g., `node-type=compute`)

## Adding a Basic Node Selector

To add a node selector to a Deployment, use a strategic merge patch:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    nodeSelector:
                      node-type: compute
                      region: us-east-1
```

This ensures pods from this Deployment are scheduled only on nodes with both the `node-type: compute` and `region: us-east-1` labels.

## Adding Node Selectors to All Workloads

If a chart produces multiple Deployments, StatefulSets, or DaemonSets, you can target all of them:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-platform
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-platform
      version: "3.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    nodeSelector:
                      workload-type: application
          - target:
              kind: StatefulSet
            patch: |
              apiVersion: apps/v1
              kind: StatefulSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    nodeSelector:
                      workload-type: stateful
                      storage-type: ssd
```

This places all Deployments on application nodes and all StatefulSets on nodes with SSD storage.

## Different Node Selectors for Different Components

Complex charts may have components that should run on different node types:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: monitoring-stack
  namespace: monitoring
spec:
  interval: 5m
  chart:
    spec:
      chart: kube-prometheus-stack
      version: "56.0.0"
      sourceRef:
        kind: HelmRepository
        name: prometheus-community
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: .*grafana.*
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    nodeSelector:
                      node-pool: monitoring
          - target:
              kind: StatefulSet
              name: .*prometheus.*
            patch: |
              apiVersion: apps/v1
              kind: StatefulSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    nodeSelector:
                      node-pool: monitoring
                      storage-class: fast-ssd
          - target:
              kind: StatefulSet
              name: .*alertmanager.*
            patch: |
              apiVersion: apps/v1
              kind: StatefulSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    nodeSelector:
                      node-pool: monitoring
```

This places Grafana on monitoring nodes, Prometheus on monitoring nodes with fast SSD storage, and Alertmanager on general monitoring nodes.

## Combining Node Selectors with Other Scheduling Options

Node selectors work alongside tolerations and affinity rules. Here is a comprehensive scheduling configuration:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    nodeSelector:
                      node-pool: application
                      topology.kubernetes.io/zone: us-east-1a
                    tolerations:
                      - key: dedicated
                        operator: Equal
                        value: application
                        effect: NoSchedule
```

This schedules pods on application pool nodes in a specific availability zone and tolerates the dedicated taint on those nodes.

## Verifying Node Placement

After deploying, verify that pods are scheduled on the correct nodes:

```bash
kubectl get pods -n production -l app.kubernetes.io/name=my-app -o wide
```

Check node labels:

```bash
kubectl get nodes --show-labels | grep node-pool
```

Verify the node selector is applied:

```bash
kubectl get deployment my-app -n production -o jsonpath='{.spec.template.spec.nodeSelector}' | jq .
```

## Conclusion

Flux post-renderers provide a straightforward way to add node selectors to Helm chart workloads without modifying the chart. Whether you need to schedule workloads on specific node pools, target GPU nodes, or separate components across different node types, strategic merge patches handle all these scenarios cleanly. Combine node selectors with tolerations and affinity rules for more sophisticated scheduling requirements, and always verify pod placement after deployment to confirm the scheduling constraints are working as expected.
