# How to Limit ApplicationSet Generated Applications in ArgoCD

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: ArgoCD, GitOps, Kubernetes, ApplicationSets, Scaling

Description: Learn how to limit the number of applications generated by ArgoCD ApplicationSets to prevent resource exhaustion and maintain cluster stability.

---

ApplicationSets can generate a massive number of applications from dynamic generators. A cluster generator combined with a matrix generator and a Git directory generator can easily produce hundreds or thousands of Application resources. Without limits, this can overwhelm the ArgoCD controller, exhaust API server resources, and make your ArgoCD installation unusable.

This guide covers the mechanisms available to limit ApplicationSet generation, from built-in controls to architectural patterns.

## Why Limiting Matters

Consider a matrix generator combining 50 clusters with 30 microservices: that is 1,500 Application resources. Each Application needs periodic reconciliation, health checks, and Git operations. Without limits, you might face:

- ArgoCD controller running out of memory
- API server rate limiting
- Slow UI response times
- Git rate limiting from too many fetch operations
- Redis cache exhaustion

## Global ApplicationSet Policy

ArgoCD provides a global configuration to limit how many applications a single ApplicationSet can create.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cmd-params-cm
  namespace: argocd
data:
  # Maximum number of applications per ApplicationSet
  applicationsetcontroller.policy: |
    maxApplications: 100
```

When an ApplicationSet attempts to generate more applications than this limit, the excess applications are not created and a warning appears in the controller logs.

## Post Selectors for Filtering

Post selectors are the primary mechanism to reduce the number of generated applications from a generator that produces too many results.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: filtered-apps
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/myorg/services.git
        revision: HEAD
        directories:
          - path: 'services/*'
        # Only generate apps for services with deploy=true
        selector:
          matchLabels:
            deploy: "true"
  template:
    metadata:
      name: '{{path.basename}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/services.git
        targetRevision: HEAD
        path: '{{path}}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{path.basename}}'
```

Each service directory would need a metadata file (like `config.json`) containing `"deploy": "true"` for the selector to match.

## Cluster Generator with Label Selectors

Instead of generating applications for all clusters, use label selectors to target specific subsets.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: limited-cluster-apps
  namespace: argocd
spec:
  generators:
    # Instead of: clusters: {} (all clusters)
    - clusters:
        selector:
          matchLabels:
            # Only target clusters explicitly opted in
            monitoring-enabled: "true"
          matchExpressions:
            - key: environment
              operator: In
              values:
                - production
                - staging
  template:
    metadata:
      name: 'monitoring-{{name}}'
    spec:
      project: monitoring
      source:
        repoURL: https://github.com/myorg/monitoring.git
        targetRevision: HEAD
        path: stack
      destination:
        server: '{{server}}'
        namespace: monitoring
```

## Git Directory Generator with Exclude Patterns

The Git directory generator supports exclude patterns to skip directories you do not want to generate applications for.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: selective-directory-apps
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/myorg/services.git
        revision: HEAD
        directories:
          # Include all services
          - path: 'services/*'
          # Exclude specific directories
          - path: 'services/deprecated-*'
            exclude: true
          - path: 'services/_template'
            exclude: true
          - path: 'services/test-*'
            exclude: true
  template:
    metadata:
      name: '{{path.basename}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/services.git
        targetRevision: HEAD
        path: '{{path}}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{path.basename}}'
```

## Matrix Generator Output Control

When using matrix generators, the output is multiplicative. Control it by limiting each dimension.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: controlled-matrix
  namespace: argocd
spec:
  generators:
    - matrix:
        generators:
          # Dimension 1: Limit to production clusters only
          - clusters:
              selector:
                matchLabels:
                  environment: production
                  # Further limit with an opt-in label
                  full-stack-enabled: "true"
          # Dimension 2: Only core services
          - list:
              elements:
                - service: api-gateway
                - service: auth-service
                # Keep this list short and intentional
        # Post selector for additional filtering
        selector:
          matchExpressions:
            - key: service
              operator: NotIn
              values:
                - deprecated-service
  template:
    metadata:
      name: '{{service}}-{{name}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/services.git
        targetRevision: HEAD
        path: '{{service}}'
      destination:
        server: '{{server}}'
        namespace: '{{service}}'
```

## Controller Resource Limits

Ensure the ApplicationSet controller has appropriate resource limits to prevent it from consuming excessive cluster resources.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: argocd-applicationset-controller
  namespace: argocd
spec:
  template:
    spec:
      containers:
        - name: argocd-applicationset-controller
          resources:
            requests:
              cpu: 200m
              memory: 256Mi
            limits:
              cpu: "1"
              memory: 1Gi
```

## Monitoring Application Count

Set up monitoring to track the number of generated applications and alert when thresholds are crossed.

```bash
# Count applications managed by ApplicationSets
kubectl get applications -n argocd \
  -l app.kubernetes.io/managed-by=applicationset-controller \
  --no-headers | wc -l

# Count per ApplicationSet
for appset in $(kubectl get applicationsets -n argocd -o name); do
  name=$(echo "$appset" | cut -d/ -f2)
  count=$(kubectl get applicationset "$name" -n argocd -o json | \
    jq '.status.resources | length')
  echo "$name: $count applications"
done

# Monitor controller memory usage
kubectl top pod -n argocd \
  -l app.kubernetes.io/name=argocd-applicationset-controller
```

## Gradual Rollout for New ApplicationSets

When introducing a new ApplicationSet that will generate many applications, use a staged approach.

```yaml
# Phase 1: Start with a very specific selector
generators:
  - clusters:
      selector:
        matchLabels:
          environment: production
          region: us-east-1
          pilot-group: "true"

# Phase 2: Expand to more clusters
generators:
  - clusters:
      selector:
        matchLabels:
          environment: production
          region: us-east-1

# Phase 3: Expand to all regions
generators:
  - clusters:
      selector:
        matchLabels:
          environment: production

# Phase 4: Include all environments
generators:
  - clusters: {}
```

Limiting ApplicationSet generation is about being intentional with what you deploy. Start small, validate, and expand gradually.
