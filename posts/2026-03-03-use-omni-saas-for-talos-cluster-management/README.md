# How to Use Omni SaaS for Talos Cluster Management

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, Sidero Omni, SaaS, Cluster Management, Kubernetes

Description: A complete guide to using Sidero Omni SaaS for managing Talos Linux clusters including setup, cluster operations, and day-two management

---

Sidero Omni SaaS is a hosted management platform for Talos Linux infrastructure. Instead of running your own management tools, you connect your Talos nodes to Omni's cloud service and manage everything through its web interface and CLI. This removes the operational burden of maintaining a management plane while giving you full control over your Talos clusters.

This guide covers how to get started with Omni SaaS, create and manage clusters, and handle day-to-day operations.

## Getting Started with Omni SaaS

Setting up Omni SaaS takes just a few minutes:

```bash
# Step 1: Sign up at https://omni.siderolabs.com
# You will receive an account and a unique Omni endpoint

# Step 2: Install the omnictl CLI
curl -sL https://omni.siderolabs.com/install | sh

# Step 3: Authenticate
omnictl auth login --url https://your-account.omni.siderolabs.com

# Step 4: Verify your connection
omnictl version
omnictl get machines
```

Your Omni SaaS endpoint is your central management point. All nodes you connect will report to this endpoint through SideroLink VPN tunnels.

## Registering Machines

Before you can create clusters, you need to register Talos machines with Omni. There are several ways to do this:

### Download an Omni-Enabled Image

The simplest approach is to boot machines with an image generated by Omni:

```bash
# Download ISO for bare metal
omnictl download installation-media --output talos-omni.iso

# Download for specific platforms
omnictl download installation-media --format aws --output talos-omni-aws.raw
omnictl download installation-media --format gcp --output talos-omni-gcp.raw.tar.gz
omnictl download installation-media --format azure --output talos-omni-azure.vhd
```

Machines booted with this image automatically establish a SideroLink connection to your Omni account and appear in the Machines list.

### Register Existing Talos Nodes

For nodes already running Talos Linux, add the SideroLink connection:

```bash
# Get the join configuration from Omni
omnictl get joinconfig -o yaml

# Apply the join configuration to existing nodes
talosctl apply-config --nodes 10.0.0.1 \
  --config-patch-control-plane @join-config.yaml
```

### Docker-Based Registration for Testing

For testing, you can register Docker-based Talos containers:

```bash
# Create Talos containers that connect to Omni
omnictl cluster template create test-cluster \
  --control-planes 1 \
  --workers 2 \
  --provider docker
```

## Creating Clusters Through the Web UI

The Omni web UI provides a visual interface for cluster creation:

1. Navigate to Clusters in the sidebar
2. Click "Create Cluster"
3. Select machines for control plane nodes (minimum 1, recommended 3)
4. Select machines for worker nodes
5. Configure cluster settings (Kubernetes version, CNI, etc.)
6. Click Create

The UI shows real-time progress as Omni bootstraps the cluster, installs components, and brings nodes online.

## Creating Clusters Through the CLI

For automation, use omnictl to create clusters programmatically:

```bash
# Create a cluster with specific machines
omnictl cluster create production-cluster \
  --control-planes machine-id-1,machine-id-2,machine-id-3 \
  --workers machine-id-4,machine-id-5

# Or use machine classes for dynamic selection
omnictl cluster create production-cluster \
  --control-planes 3 \
  --control-plane-machine-class high-mem \
  --workers 5 \
  --worker-machine-class general

# Specify Kubernetes version
omnictl cluster create production-cluster \
  --kubernetes-version 1.29.2 \
  --control-planes 3 \
  --workers 5
```

## Using Cluster Templates

For repeatable cluster deployments, use cluster templates:

```yaml
# cluster-template.yaml
kind: Cluster
name: production-cluster
kubernetes:
  version: "1.29.2"
controlPlane:
  machineCount: 3
  machineClass:
    name: control-plane
workers:
  machineCount: 5
  machineClass:
    name: worker
patches:
  - name: custom-sysctls
    inline:
      machine:
        sysctls:
          vm.max_map_count: "262144"
          net.core.somaxconn: "65535"
```

```bash
# Apply the template
omnictl cluster template apply -f cluster-template.yaml

# List clusters created from templates
omnictl cluster template list
```

## Managing Cluster Configuration

Omni allows you to apply configuration patches to clusters and individual machines:

```bash
# Apply a machine config patch to all workers in a cluster
omnictl machineconfig patch production-cluster \
  --role worker \
  --patch @worker-tuning.yaml
```

```yaml
# worker-tuning.yaml
machine:
  sysctls:
    net.core.somaxconn: "65535"
    vm.max_map_count: "262144"
  kubelet:
    extraConfig:
      maxPods: 200
      serializeImagePulls: false
```

Configuration patches are version-controlled in Omni, so you can track changes and roll back if needed.

## Upgrading Clusters

Rolling upgrades are one of Omni's strongest features. It handles the complexities of upgrading Talos Linux and Kubernetes versions across your cluster:

```bash
# Upgrade Talos Linux version
omnictl cluster upgrade production-cluster \
  --talos-version v1.7.0

# Upgrade Kubernetes version
omnictl cluster kubernetes-upgrade production-cluster \
  --to 1.30.0

# Check upgrade status
omnictl cluster status production-cluster
```

Omni performs upgrades in a rolling fashion:
1. Drain a node of workloads
2. Upgrade Talos Linux on the node
3. Wait for the node to rejoin the cluster
4. Move to the next node

Control plane nodes are upgraded one at a time to maintain etcd quorum. Worker nodes can be upgraded in parallel (configurable).

## Accessing Clusters

Get kubeconfig files for your clusters directly from Omni:

```bash
# Download kubeconfig for a cluster
omnictl kubeconfig --cluster production-cluster > kubeconfig.yaml

# Use kubectl with the downloaded kubeconfig
kubectl --kubeconfig kubeconfig.yaml get nodes
kubectl --kubeconfig kubeconfig.yaml get pods -A

# You can also use the Omni proxy
# This routes kubectl traffic through the SideroLink tunnel
omnictl kubeconfig --cluster production-cluster --service-account
```

## Monitoring and Observability

Omni provides built-in monitoring for your clusters:

```bash
# View cluster status
omnictl cluster status production-cluster

# View machine health
omnictl get machines -o table

# View cluster events
omnictl get events --cluster production-cluster

# View Talos service status on a specific machine
omnictl talosctl --nodes machine-id -- services

# View logs from a specific machine
omnictl talosctl --nodes machine-id -- logs kubelet --tail 100
```

The web UI provides dashboards showing:
- Node health and resource utilization
- Kubernetes cluster status
- Recent events and alerts
- Configuration drift detection

## Machine Lifecycle Management

Manage the lifecycle of machines through Omni:

```bash
# Reboot a machine
omnictl machine reboot machine-id

# Wipe a machine (remove from cluster and reset)
omnictl machine wipe machine-id

# Remove a machine from Omni
omnictl machine remove machine-id

# Add labels for organization
omnictl machine label machine-id --label environment=production
omnictl machine label machine-id --label rack=r42
```

## Access Control

Omni provides role-based access control for team management:

```bash
# Invite a team member
omnictl user invite user@example.com --role admin

# Available roles:
# - admin: Full access to all clusters and machines
# - operator: Can manage clusters but not Omni settings
# - reader: Read-only access to all resources

# List users
omnictl user list

# Remove a user
omnictl user remove user@example.com
```

## Scaling Clusters

Add or remove nodes from existing clusters:

```bash
# Add workers to a cluster
omnictl cluster scale production-cluster \
  --workers 8 \
  --worker-machine-class general

# Add control plane nodes (for HA)
omnictl cluster scale production-cluster \
  --control-planes 3

# Remove specific machines from a cluster
omnictl cluster machine-remove production-cluster machine-id-5
```

## Backup and Disaster Recovery

Omni maintains the cluster state and can recreate clusters if needed:

```bash
# Take an etcd snapshot
omnictl talosctl --cluster production-cluster -- \
  etcd snapshot /tmp/backup.snapshot

# Export cluster configuration
omnictl cluster template export production-cluster > cluster-backup.yaml

# The cluster template includes all patches and machine assignments
# Use it to recreate the cluster if needed
```

Track your resource usage across all connected machines:

```bash
# View resource summary
omnictl get machines -o table

# The output shows:
# - CPU cores per machine
# - Memory per machine
# - Disk capacity
# - Cluster assignment

```

## Conclusion

Sidero Omni SaaS provides a production-ready management platform for Talos Linux without the operational overhead of self-hosting management tools. From initial machine registration through cluster creation, day-to-day operations, and rolling upgrades, everything is handled through a single interface. The SideroLink VPN ensures secure connectivity regardless of where your machines are located, and the role-based access control makes it suitable for team environments. Start with a few nodes to evaluate, and scale up as your confidence grows.
