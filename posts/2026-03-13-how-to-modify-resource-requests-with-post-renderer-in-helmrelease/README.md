# How to Modify Resource Requests with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderer, Resource Requests, Resource Limits

Description: Learn how to override CPU and memory resource requests and limits in Helm chart containers using Flux post-renderers.

---

## Introduction

Proper resource requests and limits are essential for Kubernetes cluster stability and efficient resource utilization. While most Helm charts provide values to configure resources for the main application container, many charts include additional containers, init containers, or helper pods with hardcoded resource values that may not suit your environment.

Flux post-renderers allow you to modify resource requests and limits for any container in any resource generated by a Helm chart. This gives you complete control over resource allocation without maintaining chart forks or waiting for upstream changes.

This guide demonstrates how to use post-renderers to adjust resource requests and limits across different container types and deployment scenarios.

## Prerequisites

Before you begin, ensure you have:

- A Kubernetes cluster with Flux CD installed
- A HelmRelease deployed through Flux
- kubectl access to the cluster
- Understanding of Kubernetes resource management (requests vs limits)

## Overriding Resources for the Main Container

To override resource requests and limits for a specific container in a Deployment:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    containers:
                      - name: my-app
                        resources:
                          requests:
                            cpu: 500m
                            memory: 512Mi
                          limits:
                            cpu: "2"
                            memory: 1Gi
```

This overrides any resource values defined in the chart template or default values with your environment-specific requirements.

## Modifying Resources for Multiple Containers

When a Deployment has multiple containers and you need to adjust resources for each:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    containers:
                      - name: my-app
                        resources:
                          requests:
                            cpu: "1"
                            memory: 1Gi
                          limits:
                            cpu: "4"
                            memory: 4Gi
                      - name: sidecar-proxy
                        resources:
                          requests:
                            cpu: 100m
                            memory: 128Mi
                          limits:
                            cpu: 200m
                            memory: 256Mi
```

Each container is identified by name, and the strategic merge patch updates only the resource fields without affecting other container settings.

## Adjusting Init Container Resources

Init containers often have different resource requirements than the main containers. Override them the same way:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    initContainers:
                      - name: db-migration
                        resources:
                          requests:
                            cpu: 250m
                            memory: 256Mi
                          limits:
                            cpu: 500m
                            memory: 512Mi
                    containers:
                      - name: my-app
                        resources:
                          requests:
                            cpu: 500m
                            memory: 512Mi
                          limits:
                            cpu: "2"
                            memory: 2Gi
```

## Modifying Resources for StatefulSets and DaemonSets

Post-renderers work with any workload type, not just Deployments:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: elasticsearch
  namespace: logging
spec:
  interval: 5m
  chart:
    spec:
      chart: elasticsearch
      version: "8.0.0"
      sourceRef:
        kind: HelmRepository
        name: elastic
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: StatefulSet
            patch: |
              apiVersion: apps/v1
              kind: StatefulSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    containers:
                      - name: elasticsearch
                        resources:
                          requests:
                            cpu: "2"
                            memory: 4Gi
                          limits:
                            cpu: "4"
                            memory: 8Gi
```

## Verifying Resource Overrides

After deployment, verify the resource settings:

```bash
kubectl get deployment my-app -n production -o jsonpath='{range .spec.template.spec.containers[*]}{.name}: requests={.resources.requests} limits={.resources.limits}{"\n"}{end}'
```

Check that pods are scheduled correctly with the new resource requirements:

```bash
kubectl describe pod -n production -l app.kubernetes.io/name=my-app | grep -A 4 "Requests\|Limits"
```

## Conclusion

Flux post-renderers give you complete control over resource requests and limits for any container in a Helm chart, whether it is a main container, sidecar, or init container. This capability is essential for right-sizing workloads across different environments without maintaining chart forks. Use targeted patches to adjust resources per container and per workload type, and always verify the applied values after deployment to ensure your pods are properly resourced.
