# How to Use the local_file Data Source in Terraform

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Data Sources, Local Files, Infrastructure as Code, Configuration

Description: Learn how to use the local_file data source in Terraform to read local files from disk and use their content in your infrastructure configuration dynamically.

---

The `local_file` data source reads a file from the local filesystem where Terraform is running and makes its content available as an attribute. This is useful for loading configuration files, certificates, scripts, and other artifacts that live alongside your Terraform code or are generated by external processes.

## Basic Usage

```hcl
data "local_file" "config" {
  filename = "${path.module}/config/app.json"
}

output "config_content" {
  value = data.local_file.config.content
}
```

The data source reads the file at the specified path and exposes its content as a string.

## Available Attributes

After reading the file, you can access:

- `content` - The raw content of the file as a string
- `content_base64` - The content encoded as base64
- `content_md5` - MD5 hash of the content
- `content_sha1` - SHA1 hash of the content
- `content_sha256` - SHA256 hash of the content
- `id` - The file path (same as `filename`)

```hcl
data "local_file" "certificate" {
  filename = "${path.module}/certs/server.crt"
}

output "cert_hash" {
  value = data.local_file.certificate.content_sha256
}

output "cert_base64" {
  value = data.local_file.certificate.content_base64
}
```

## Practical Use Cases

### Loading a JSON Configuration File

```hcl
data "local_file" "app_config" {
  filename = "${path.module}/config/${var.environment}.json"
}

locals {
  config = jsondecode(data.local_file.app_config.content)
}

resource "aws_instance" "app" {
  ami           = local.config.ami_id
  instance_type = local.config.instance_type
  subnet_id     = local.config.subnet_id

  tags = local.config.tags
}
```

This pattern lets you keep environment-specific configurations in JSON files:

```json
{
  "ami_id": "ami-0abcdef1234567890",
  "instance_type": "t3.large",
  "subnet_id": "subnet-abc123",
  "tags": {
    "Environment": "production",
    "Team": "platform"
  }
}
```

### Loading SSH Public Keys

```hcl
data "local_file" "ssh_public_key" {
  filename = pathexpand("~/.ssh/id_rsa.pub")
}

resource "aws_key_pair" "deployer" {
  key_name   = "deployer-key"
  public_key = trimspace(data.local_file.ssh_public_key.content)
}
```

The `pathexpand()` function resolves `~` to the user's home directory. `trimspace()` removes any trailing newline from the file content.

### Loading TLS Certificates and Keys

```hcl
data "local_file" "tls_cert" {
  filename = "${path.module}/certs/server.crt"
}

data "local_file" "tls_key" {
  filename = "${path.module}/certs/server.key"
}

resource "aws_iam_server_certificate" "app" {
  name             = "app-cert"
  certificate_body = data.local_file.tls_cert.content
  private_key      = data.local_file.tls_key.content
}
```

### Loading IAM Policy Documents from Files

```hcl
data "local_file" "iam_policy" {
  filename = "${path.module}/policies/s3-read-access.json"
}

resource "aws_iam_policy" "s3_read" {
  name   = "s3-read-access"
  policy = data.local_file.iam_policy.content
}
```

### Loading Scripts for User Data

```hcl
data "local_file" "user_data_script" {
  filename = "${path.module}/scripts/bootstrap.sh"
}

resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = "t3.medium"

  user_data = data.local_file.user_data_script.content

  tags = {
    Name = "web-server"
  }
}
```

### Using Base64 Content for Binary Files

When you need to upload binary content, use the base64 encoding:

```hcl
data "local_file" "lambda_zip" {
  filename = "${path.module}/dist/function.zip"
}

resource "aws_lambda_function" "app" {
  function_name = "my-function"
  role          = aws_iam_role.lambda.arn
  handler       = "index.handler"
  runtime       = "nodejs18.x"

  # Use the content hash for change detection
  source_code_hash = data.local_file.lambda_zip.content_base64

  filename = data.local_file.lambda_zip.filename
}
```

### Loading YAML Configuration

```hcl
data "local_file" "k8s_config" {
  filename = "${path.module}/manifests/deployment.yaml"
}

# Parse YAML (requires Terraform 1.0+)
locals {
  deployment = yamldecode(data.local_file.k8s_config.content)
}

output "deployment_name" {
  value = local.deployment.metadata.name
}
```

### Environment-Specific Configuration Loading

```hcl
variable "environment" {
  type = string
}

data "local_file" "env_config" {
  filename = "${path.module}/environments/${var.environment}/config.json"
}

data "local_file" "env_secrets_policy" {
  filename = "${path.module}/environments/${var.environment}/secrets-policy.json"
}

locals {
  env_config = jsondecode(data.local_file.env_config.content)
}

resource "aws_instance" "app" {
  ami           = local.env_config.ami
  instance_type = local.env_config.instance_type

  tags = merge(
    local.env_config.default_tags,
    { Name = "app-${var.environment}" }
  )
}
```

## local_file Data Source vs. Built-in Functions

Terraform has built-in functions that can also read files:

- `file()` - Reads a file as a string
- `filebase64()` - Reads a file and returns base64-encoded content
- `filemd5()` - Returns the MD5 hash of a file
- `filesha256()` - Returns the SHA256 hash of a file
- `templatefile()` - Reads a file and processes it as a template

So when should you use `local_file` instead of `file()`?

### When to Use file()

Use `file()` when you just need the content as a string in an expression:

```hcl
resource "aws_key_pair" "deployer" {
  key_name   = "deployer"
  public_key = file("${path.module}/keys/deployer.pub")
}
```

### When to Use local_file

Use `local_file` when you need multiple attributes (content, hash, base64) from the same file, or when you want to use the data source with `depends_on`, `lifecycle`, or in module compositions.

```hcl
# Need both content and hash
data "local_file" "config" {
  filename = "${path.module}/config/app.json"
}

resource "aws_s3_object" "config" {
  bucket  = aws_s3_bucket.config.id
  key     = "app.json"
  content = data.local_file.config.content
  etag    = data.local_file.config.content_md5
}
```

## Combining with other Data Sources

### Read a File, then Use HTTP to Validate It

```hcl
data "local_file" "api_spec" {
  filename = "${path.module}/openapi/spec.yaml"
}

resource "aws_api_gateway_rest_api" "main" {
  name = "my-api"
  body = data.local_file.api_spec.content
}
```

### Read Multiple Files with for_each

```hcl
locals {
  policy_files = fileset("${path.module}/policies", "*.json")
}

data "local_file" "policies" {
  for_each = local.policy_files
  filename = "${path.module}/policies/${each.value}"
}

resource "aws_iam_policy" "custom" {
  for_each = data.local_file.policies
  name     = trimsuffix(basename(each.key), ".json")
  policy   = each.value.content
}
```

This reads all JSON files from a `policies/` directory and creates an IAM policy for each one.

## Error Handling

If the file does not exist, Terraform fails during the plan phase with a clear error:

```
Error: Read local file data source error

  on main.tf line 1, in data "local_file" "missing":
   1: data "local_file" "missing" {

The file at given path cannot be read.
  Path: config/nonexistent.json
```

To handle optional files, use a conditional:

```hcl
variable "custom_config_path" {
  type    = string
  default = ""
}

locals {
  has_custom_config = var.custom_config_path != "" && fileexists(var.custom_config_path)
}

data "local_file" "custom_config" {
  count    = local.has_custom_config ? 1 : 0
  filename = var.custom_config_path
}

locals {
  config = local.has_custom_config ? jsondecode(data.local_file.custom_config[0].content) : {
    instance_type = "t3.medium"
    ami_id        = var.default_ami
  }
}
```

## Security Notes

1. **File content goes into state.** Whatever the file contains ends up in your Terraform state. If the file has secrets, your state backend must be encrypted.

2. **Relative paths.** Always use `${path.module}` for files relative to the module. Bare relative paths resolve from the working directory, which may not be what you expect.

3. **File permissions.** The Terraform process must have read access to the file. In CI/CD environments, make sure the files are available in the runner's workspace.

## Summary

The `local_file` data source reads files from the local filesystem and makes their content available in your Terraform configuration. It is useful for loading configuration files, certificates, policies, scripts, and any artifact that lives on disk. For simple file reading, the built-in `file()` function is often sufficient. Use `local_file` when you need multiple attributes like content and hash, or when you need data source features like `for_each`.

For fetching data from remote sources instead, see our posts on the [http data source](https://oneuptime.com/blog/post/2026-02-23-terraform-http-data-source/view) and [external data source](https://oneuptime.com/blog/post/2026-02-23-terraform-external-data-source/view).
