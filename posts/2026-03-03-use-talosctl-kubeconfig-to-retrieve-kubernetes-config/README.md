# How to Use talosctl kubeconfig to Retrieve Kubernetes Config

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, talosctl, Kubernetes, kubeconfig, Cluster Access

Description: Learn how to use talosctl kubeconfig to retrieve and manage Kubernetes configuration files for accessing your Talos Linux cluster

---

After bootstrapping a Talos Linux cluster, the first thing you need is a kubeconfig file so you can interact with Kubernetes using `kubectl`. The `talosctl kubeconfig` command retrieves this configuration directly from the cluster, and this guide covers all the ways you can use it.

## How Kubernetes Configuration Works in Talos

In traditional Kubernetes setups, the kubeconfig file is usually generated during cluster installation and placed in a well-known location. Talos Linux works differently. Since there is no SSH access and no traditional filesystem you can browse, you retrieve the kubeconfig through the Talos API using `talosctl kubeconfig`.

The command connects to a control plane node, retrieves the certificate authority data and generates a client certificate, and packages everything into a standard kubeconfig file that `kubectl` can use.

## Basic Usage

The simplest way to retrieve a kubeconfig is:

```bash
# Retrieve kubeconfig and merge it into the default location
talosctl kubeconfig --nodes 192.168.1.10
```

This command retrieves the kubeconfig from the control plane node at 192.168.1.10 and merges it into your default kubeconfig file (usually `~/.kube/config`). If the file does not exist, it creates it.

## Saving to a Specific File

If you want to save the kubeconfig to a specific file instead of merging it into your default config:

```bash
# Save kubeconfig to a specific file
talosctl kubeconfig /path/to/my-cluster-kubeconfig --nodes 192.168.1.10
```

This is useful when you manage multiple clusters and want to keep their configurations separate:

```bash
# Save configs for different clusters
talosctl kubeconfig ~/.kube/production-config --nodes 10.0.1.10
talosctl kubeconfig ~/.kube/staging-config --nodes 10.0.2.10
talosctl kubeconfig ~/.kube/development-config --nodes 10.0.3.10
```

You can then switch between clusters using the `KUBECONFIG` environment variable:

```bash
# Use a specific kubeconfig
export KUBECONFIG=~/.kube/production-config
kubectl get nodes

# Or specify it per command
kubectl --kubeconfig ~/.kube/staging-config get nodes
```

## Merging Into Default Kubeconfig

When you run `talosctl kubeconfig` without specifying a file path, it merges the new configuration into your default kubeconfig. If you already have other clusters configured there, they will be preserved. The new cluster context will be added alongside your existing ones.

```bash
# Merge into default kubeconfig
talosctl kubeconfig --nodes 192.168.1.10

# List all available contexts
kubectl config get-contexts

# Switch to the new cluster context
kubectl config use-context admin@my-talos-cluster
```

The context name is derived from the cluster name you specified in your Talos machine configuration.

## Specifying the Endpoint

The kubeconfig generated by `talosctl` uses the control plane endpoint as the server URL. If your control plane is behind a load balancer, make sure the kubeconfig points to the load balancer and not individual control plane nodes:

```bash
# Force a specific Kubernetes API server URL in the kubeconfig
talosctl kubeconfig --nodes 192.168.1.10 --force
```

If the generated kubeconfig has the wrong endpoint, you can manually edit it:

```bash
# View the current server URL
kubectl config view --minify | grep server

# Update the server URL if needed
kubectl config set-cluster my-talos-cluster --server=https://api.mycluster.example.com:6443
```

## Using the Force Flag

If you need to overwrite an existing kubeconfig entry (for example, after rotating certificates), use the `--force` flag:

```bash
# Force overwrite existing kubeconfig entry
talosctl kubeconfig --nodes 192.168.1.10 --force
```

Without `--force`, the command will not overwrite an existing configuration for the same cluster. This is a safety feature to prevent accidentally replacing a working configuration.

## Generating Admin vs. User Kubeconfigs

By default, `talosctl kubeconfig` generates an admin-level kubeconfig with full cluster access. This is appropriate for cluster administrators but not for regular users or service accounts.

For non-admin access, you should create Kubernetes RBAC roles and generate service account tokens:

```bash
# First, get the admin kubeconfig
talosctl kubeconfig --nodes 192.168.1.10

# Create a namespace and service account for a team
kubectl create namespace team-frontend
kubectl create serviceaccount frontend-deployer -n team-frontend

# Create a role with limited permissions
kubectl apply -f - <<EOF
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: team-frontend
  name: deployer
rules:
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
- apiGroups: [""]
  resources: ["pods", "services", "configmaps"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
EOF

# Bind the role to the service account
kubectl create rolebinding frontend-deployer-binding \
  --role=deployer \
  --serviceaccount=team-frontend:frontend-deployer \
  -n team-frontend
```

## Setting Up CI/CD Access

For CI/CD pipelines, you typically want a kubeconfig with limited permissions. Here is how to set that up:

```bash
# Retrieve the admin kubeconfig first
talosctl kubeconfig /tmp/admin-kubeconfig --nodes 192.168.1.10

# Use it to create a service account for CI/CD
export KUBECONFIG=/tmp/admin-kubeconfig

kubectl create serviceaccount ci-deployer -n default
kubectl create clusterrolebinding ci-deployer-binding \
  --clusterrole=edit \
  --serviceaccount=default:ci-deployer

# Create a long-lived token for the service account
kubectl apply -f - <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: ci-deployer-token
  namespace: default
  annotations:
    kubernetes.io/service-account.name: ci-deployer
type: kubernetes.io/service-account-token
EOF

# Extract the token
CI_TOKEN=$(kubectl get secret ci-deployer-token -o jsonpath='{.data.token}' | base64 -d)

# Create a kubeconfig for CI/CD
kubectl config set-credentials ci-deployer --token="$CI_TOKEN"
kubectl config set-context ci-context \
  --cluster=my-talos-cluster \
  --user=ci-deployer \
  --namespace=default
```

## Refreshing an Expired Kubeconfig

The client certificates in the kubeconfig have an expiration date. When they expire, you will get authentication errors from `kubectl`. To fix this, simply regenerate the kubeconfig:

```bash
# Check if your current kubeconfig works
kubectl get nodes

# If you get authentication errors, regenerate
talosctl kubeconfig --nodes 192.168.1.10 --force

# Verify it works now
kubectl get nodes
```

## Troubleshooting

If `talosctl kubeconfig` fails, here are common issues to investigate:

```bash
# Make sure you can reach the Talos API
talosctl version --nodes 192.168.1.10

# Verify your talosctl config is pointing to the right cluster
talosctl config info

# Check if the Kubernetes API server is running
talosctl services --nodes 192.168.1.10 | grep kube-apiserver

# Check API server health
talosctl logs kube-apiserver --nodes 192.168.1.10 | tail -20
```

If the Kubernetes API server is not running, you might need to bootstrap the cluster first:

```bash
# Bootstrap the cluster if not done yet
talosctl bootstrap --nodes 192.168.1.10
```

## Best Practices

- Store admin kubeconfig files securely and limit access to them.
- Use separate kubeconfig files for different clusters to avoid accidental cross-cluster operations.
- Create service accounts with limited RBAC permissions for CI/CD and team members.
- Regenerate kubeconfigs before certificates expire.
- Never commit kubeconfig files to version control repositories.
- Use the `--force` flag carefully and only when you intentionally want to overwrite existing configuration.
- Back up your kubeconfig files as part of your disaster recovery process.

The `talosctl kubeconfig` command is your gateway to managing Kubernetes on Talos Linux. Getting it set up correctly from the start saves a lot of troubleshooting later.
