# How to Use the generate Block in Terragrunt

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Terragrunt, Infrastructure as Code, DevOps, Code Generation, Configuration

Description: Learn how to use the generate block in Terragrunt to automatically create Terraform files like provider configs, backend configs, and shared locals.

---

One of the annoying parts of managing multiple Terraform modules is repeating the same boilerplate files across every module directory. Provider configurations, backend blocks, common variables - they all tend to be identical or nearly identical across modules. The `generate` block in Terragrunt solves this by automatically creating Terraform files before Terraform runs.

## What the generate Block Does

The `generate` block writes a file into the Terraform working directory (the `.terragrunt-cache` directory where Terragrunt copies your module source). You specify the file name, the content, and what to do if the file already exists.

Here is the simplest example:

```hcl
# terragrunt.hcl

# Generate a provider configuration file
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "us-east-1"

  default_tags {
    tags = {
      ManagedBy = "terraform"
      Project   = "my-project"
    }
  }
}
EOF
}
```

When Terragrunt runs, it creates `provider.tf` in the working directory with the specified contents. Your Terraform module does not need to define a provider - Terragrunt handles it.

## The if_exists Parameter

The `if_exists` parameter controls what happens when the target file already exists (for example, if your Terraform module source already has a `provider.tf`). There are four options:

### overwrite

Unconditionally overwrites the file:

```hcl
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite"
  contents  = "..."
}
```

### overwrite_terragrunt

Overwrites the file only if it was previously generated by Terragrunt. If a human-written file with the same name exists in the module, Terragrunt leaves it alone:

```hcl
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"  # safe default
  contents  = "..."
}
```

This is the safest option for most situations. It prevents Terragrunt from clobbering files that are part of the original module.

### skip

If the file exists (regardless of who created it), do nothing:

```hcl
generate "provider" {
  path      = "provider.tf"
  if_exists = "skip"
  contents  = "..."
}
```

### error

If the file exists, throw an error:

```hcl
generate "provider" {
  path      = "provider.tf"
  if_exists = "error"
  contents  = "..."
}
```

## Common Use Cases

### 1. Provider Configuration

The most common use case is generating provider blocks:

```hcl
# Root terragrunt.hcl

generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
# AWS provider configuration
provider "aws" {
  region = "us-east-1"

  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Environment = "dev"
    }
  }
}

# Secondary provider for cross-region resources
provider "aws" {
  alias  = "us_west_2"
  region = "us-west-2"

  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Environment = "dev"
    }
  }
}
EOF
}
```

Every child module that includes this root configuration gets both providers without needing to define them.

### 2. Backend Configuration

Terragrunt's `remote_state` block can also generate backend configuration, but you can do it manually with `generate` for more control:

```hcl
generate "backend" {
  path      = "backend.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "${path_relative_to_include()}/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-lock"
  }
}
EOF
}
```

### 3. Terraform Version Constraints

Ensure all modules use the same Terraform version:

```hcl
generate "versions" {
  path      = "versions.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
terraform {
  required_version = ">= 1.5.0, < 2.0.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
EOF
}
```

This is powerful for enforcing consistency across dozens of modules.

### 4. Common Variables

Generate a shared variables file that all modules can use:

```hcl
generate "common_variables" {
  path      = "common_variables.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
# Common variables provided by Terragrunt
variable "environment" {
  description = "The deployment environment"
  type        = string
}

variable "project" {
  description = "The project name"
  type        = string
}

variable "aws_region" {
  description = "AWS region for resources"
  type        = string
}
EOF
}
```

### 5. Common Data Sources

Some data sources are used by nearly every module:

```hcl
generate "common_data" {
  path      = "common_data.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
# Commonly needed data sources
data "aws_caller_identity" "current" {}
data "aws_region" "current" {}

# Make the account ID and region available as locals
locals {
  account_id = data.aws_caller_identity.current.account_id
  region     = data.aws_region.current.name
}
EOF
}
```

## Dynamic Content with Terragrunt Functions

The `contents` field supports Terragrunt functions, which lets you make the generated content dynamic:

```hcl
# Read environment-specific locals
locals {
  env_config = read_terragrunt_config(find_in_parent_folders("env.hcl"))
  environment = local.env_config.locals.environment
  aws_region  = local.env_config.locals.aws_region
  account_id  = local.env_config.locals.account_id
}

generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "${local.aws_region}"

  # Assume role into the target account
  assume_role {
    role_arn = "arn:aws:iam::${local.account_id}:role/TerraformDeployRole"
  }

  default_tags {
    tags = {
      Environment = "${local.environment}"
      ManagedBy   = "terraform"
    }
  }
}
EOF
}
```

This generates a different provider configuration depending on which environment directory you are in.

## Multiple generate Blocks

You can have as many `generate` blocks as you need:

```hcl
# Root terragrunt.hcl

# Generate provider config
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "us-east-1"
}
EOF
}

# Generate version constraints
generate "versions" {
  path      = "versions.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
terraform {
  required_version = ">= 1.5.0"
}
EOF
}

# Generate common local values
generate "project_locals" {
  path      = "project_locals.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
locals {
  project_name = "my-project"
  cost_center  = "engineering"
}
EOF
}
```

Each block generates a separate file. This keeps things organized - providers in one file, versions in another, locals in a third.

## generate in Child Configurations

Child modules can also have `generate` blocks. If a child generates a file with the same name as the parent, the behavior depends on the `if_exists` setting:

```hcl
# Root terragrunt.hcl
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "us-east-1"
}
EOF
}
```

```hcl
# live/dev/special-module/terragrunt.hcl

include "root" {
  path = find_in_parent_folders()
}

# Override the provider for this specific module
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "aws" {
  region = "eu-west-1"  # this module deploys to EU
}
EOF
}
```

The child's `generate` block overrides the parent's because they target the same file name and the child is processed after the parent.

## Debugging Generated Files

To see what Terragrunt generates, look inside the `.terragrunt-cache` directory after running any Terragrunt command:

```bash
# Run init to trigger file generation
terragrunt init

# Check what was generated
ls -la .terragrunt-cache/*/

# View the generated provider file
cat .terragrunt-cache/*/provider.tf
```

You can also use `terragrunt render-json` to see the fully resolved configuration:

```bash
# See the full rendered configuration as JSON
terragrunt render-json
```

## Best Practices

**Use overwrite_terragrunt as the default**: It is the safest option because it will not overwrite files that are part of the Terraform module source.

**Keep generated files focused**: Generate one concern per file. Do not put providers, versions, and variables all in one generated file.

**Do not generate resource definitions**: The `generate` block is for boilerplate like providers, backends, and version constraints. Your actual infrastructure resources should live in Terraform modules.

**Name generate blocks descriptively**: The block label (like `"provider"` or `"versions"`) is just for documentation, but it should clearly indicate what the block generates.

## Conclusion

The `generate` block eliminates boilerplate Terraform files across your modules. Provider configurations, version constraints, backend settings, and common data sources can all be defined once in a root `terragrunt.hcl` and automatically generated for every child module.

This is one of the simplest Terragrunt features to adopt and one of the most immediately useful. Even if you are not using other Terragrunt features like `dependency` or `run-all`, the `generate` block alone can save you from maintaining dozens of identical files.

For more on structuring your root configuration, see [How to Organize Terragrunt for Multi-Environment Projects](https://oneuptime.com/blog/post/2026-02-23-how-to-organize-terragrunt-for-multi-environment-projects/view).
