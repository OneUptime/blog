# How to Configure Alert Exclusion Rules in Flux

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, GitOps, Kubernetes, Notifications, Alerts, Exclusion Rules

Description: Learn how to use the exclusionList field in Flux alerts to suppress unwanted notifications using regex patterns.

---

As your Flux CD deployment grows, the volume of events generated by reconciliations can become overwhelming. The `spec.exclusionList` field in the Flux Alert resource allows you to define regex patterns that suppress matching events before they reach your notification provider. This guide covers how to configure exclusion rules effectively.

## Prerequisites

- A Kubernetes cluster with Flux CD installed
- The notification controller running
- An existing Alert resource that you want to refine
- Basic familiarity with regular expressions

## How Exclusion Rules Work

The `spec.exclusionList` field accepts an array of regex pattern strings. When an event is generated, the notification controller checks the event message against each pattern in the list. If any pattern matches, the event is excluded and no notification is sent. The patterns use Go regex syntax.

## Step 1: Add a Simple Exclusion Rule

Start by excluding the most common noisy event, which is a reconciliation that finds no changes.

```yaml
# Alert with a single exclusion rule
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Alert
metadata:
  name: alert-with-exclusions
  namespace: flux-system
spec:
  providerRef:
    name: slack-provider
  eventSeverity: info
  eventSources:
    - kind: Kustomization
      name: "*"
      namespace: flux-system
  # Suppress no-change reconciliation events
  exclusionList:
    - "^Reconciliation finished.*no changes$"
```

Apply it.

```bash
# Apply the alert with exclusion rules
kubectl apply -f alert-exclusion.yaml
```

## Step 2: Build a Comprehensive Exclusion List

Add multiple patterns to filter out various types of routine events.

```yaml
# Alert with a comprehensive set of exclusion rules
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Alert
metadata:
  name: comprehensive-exclusions
  namespace: flux-system
spec:
  providerRef:
    name: slack-provider
  eventSeverity: info
  eventSources:
    - kind: Kustomization
      name: "*"
      namespace: flux-system
    - kind: HelmRelease
      name: "*"
      namespace: flux-system
    - kind: GitRepository
      name: "*"
      namespace: flux-system
  exclusionList:
    # No-change reconciliation events
    - "^Reconciliation finished.*no changes$"
    # Source artifact unchanged
    - "^stored artifact.*same revision$"
    - "^artifact up-to-date.*"
    # Progress and waiting messages
    - ".*is not ready$"
    - ".*waiting for.*"
    - ".*dependency.*not ready.*"
    # No updates applied
    - "^no updates made$"
```

## Step 3: Exclude Events by Resource Name

Use regex to exclude events that mention specific resource names.

```yaml
# Alert excluding events related to specific resources
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Alert
metadata:
  name: resource-filtered-alert
  namespace: flux-system
spec:
  providerRef:
    name: slack-provider
  eventSeverity: info
  eventSources:
    - kind: HelmRelease
      name: "*"
      namespace: flux-system
  exclusionList:
    # Exclude events mentioning the monitoring stack
    - ".*prometheus.*"
    - ".*grafana.*"
    # Exclude events from test releases
    - ".*-test-.*"
    - ".*canary.*"
```

## Step 4: Exclude Events by Namespace Reference

Filter out events that reference resources in certain namespaces within their messages.

```yaml
# Alert excluding events referencing specific namespaces
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Alert
metadata:
  name: namespace-filtered-alert
  namespace: flux-system
spec:
  providerRef:
    name: slack-provider
  eventSeverity: info
  eventSources:
    - kind: Kustomization
      name: "*"
      namespace: flux-system
  exclusionList:
    # Exclude events mentioning dev namespace resources
    - ".*namespace/dev.*"
    # Exclude events mentioning test namespace resources
    - ".*namespace/test.*"
```

## Step 5: Use Case-Insensitive Matching

Go regex supports flags for case-insensitive matching using the `(?i)` prefix.

```yaml
# Alert with case-insensitive exclusion patterns
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Alert
metadata:
  name: case-insensitive-exclusions
  namespace: flux-system
spec:
  providerRef:
    name: slack-provider
  eventSeverity: info
  eventSources:
    - kind: Kustomization
      name: "*"
      namespace: flux-system
  exclusionList:
    # Case-insensitive match for "no changes"
    - "(?i).*no changes.*"
    # Case-insensitive match for "up to date"
    - "(?i).*up.to.date.*"
```

## Step 6: Test Your Exclusion Rules

Verify that your exclusion rules are working as expected.

```bash
# Force a reconciliation to generate events
flux reconcile kustomization flux-system --with-source

# Watch the notification controller logs to see which events are excluded
kubectl logs -n flux-system deploy/notification-controller -f

# View the full alert spec to confirm exclusion rules
kubectl get alert comprehensive-exclusions -n flux-system -o jsonpath='{.spec.exclusionList}'
```

You can also test regex patterns locally before applying them.

```bash
# Test a regex pattern against a sample event message
echo "Reconciliation finished, no changes" | grep -P "^Reconciliation finished.*no changes$"

# Test multiple messages against your patterns
for msg in "Reconciliation finished, no changes" "Reconciliation failed" "stored artifact for revision main@sha1:abc123, same revision"; do
  if echo "$msg" | grep -qP "^Reconciliation finished.*no changes$|^stored artifact.*same revision$"; then
    echo "EXCLUDED: $msg"
  else
    echo "INCLUDED: $msg"
  fi
done
```

## Step 7: Iterative Refinement

The best approach to building exclusion rules is iterative. Start with no exclusions, observe the events, then add patterns one at a time.

```bash
# Step 1: View recent events to identify noisy patterns
kubectl get events -n flux-system --sort-by='.lastTimestamp' -o custom-columns=MESSAGE:.message,REASON:.reason,KIND:.involvedObject.kind | head -30

# Step 2: Identify repetitive messages
kubectl get events -n flux-system -o json | jq -r '.items[].message' | sort | uniq -c | sort -rn | head -20
```

## Common Exclusion Patterns Reference

| Pattern | Purpose |
|---|---|
| `^Reconciliation finished.*no changes$` | No-change reconciliations |
| `^stored artifact.*same revision$` | Unchanged source artifacts |
| `^artifact up-to-date.*` | Source already current |
| `.*is not ready$` | Resources not yet ready |
| `.*waiting for.*` | Waiting on dependencies |
| `.*dependency.*not ready.*` | Dependencies pending |
| `^no updates made$` | No updates applied |
| `(?i).*retry.*` | Retry messages |

## Important Considerations

- Exclusion patterns match against the full event message string
- Each pattern is evaluated independently; if any pattern matches, the event is excluded
- An empty exclusion list or omitting the field means no events are excluded
- Be careful not to create overly broad patterns that might exclude important error messages
- Test patterns in a development environment before applying them to production alerts

## Summary

The `spec.exclusionList` field in Flux alerts is your primary tool for reducing notification noise. By crafting regex patterns that match routine or unimportant event messages, you can ensure that only meaningful notifications reach your team. Build your exclusion list iteratively by observing event patterns and adding filters as needed. Always test your patterns to avoid accidentally suppressing important alerts.
