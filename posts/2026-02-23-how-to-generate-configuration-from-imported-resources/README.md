# How to Generate Configuration from Imported Resources

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Import, Configuration Generation, Infrastructure as Code, Migration

Description: Learn how to automatically generate Terraform configuration from imported resources using terraform plan -generate-config-out for fast infrastructure-as-code adoption.

---

One of the most tedious parts of importing existing infrastructure into Terraform is writing the resource configuration to match the current state. Terraform 1.5 introduced the ability to automatically generate configuration files from imported resources using the -generate-config-out flag. This dramatically speeds up the adoption of existing infrastructure into Terraform by creating configuration that matches your real resources.

In this guide, we will walk through the complete workflow of generating configuration from imported resources, cleaning up the generated code, and integrating it into your Terraform projects.

## Understanding Configuration Generation

When you run terraform plan with the -generate-config-out flag and have import blocks for resources without corresponding configuration, Terraform will create a configuration file with all the attributes of the imported resources. This generated file provides a starting point that you then clean up and customize.

## Prerequisites

```hcl
# main.tf
terraform {
  required_version = ">= 1.5.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}
```

## Step 1: Define Import Blocks Without Configuration

The key is to define import blocks without the corresponding resource blocks:

```hcl
# imports.tf - Only import blocks, no resource blocks
import {
  to = aws_instance.app_server
  id = "i-1234567890abcdef0"
}

import {
  to = aws_s3_bucket.data
  id = "my-company-data-bucket"
}

import {
  to = aws_security_group.web
  id = "sg-0abcdef1234567890"
}

import {
  to = aws_db_instance.production
  id = "production-database"
}

import {
  to = aws_iam_role.app
  id = "app-production-role"
}
```

## Step 2: Generate Configuration

Run terraform plan with the generation flag:

```bash
# Generate configuration into a file
terraform plan -generate-config-out=generated_resources.tf
```

## Step 3: Clean Up Generated Configuration

The generated configuration includes every attribute, including computed values and defaults. Clean it up by removing unnecessary attributes:

```hcl
# cleaned_resources.tf - Cleaned up version
resource "aws_instance" "app_server" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.large"
  key_name      = "deploy-key"
  subnet_id     = "subnet-0aaaa1111"
  monitoring    = true

  vpc_security_group_ids = ["sg-0abcdef1234567890"]

  root_block_device {
    encrypted   = true
    volume_size = 50
    volume_type = "gp3"
  }

  tags = {
    Environment = "production"
    Name        = "app-server"
  }

  lifecycle {
    ignore_changes = [ami]
  }
}

resource "aws_s3_bucket" "data" {
  bucket = "my-company-data-bucket"

  tags = {
    Environment = "production"
  }
}
```

## Step 4: Verify and Apply

```bash
# Plan should show the import with no changes
terraform plan

# Apply the import
terraform apply

# Verify no drift
terraform plan
# Expected: No changes. Your infrastructure matches the configuration.
```

## Configuration Generation with for_each

```hcl
# for-each-import.tf - Generate config for dynamic resources
variable "existing_instances" {
  type = map(string)
  default = {
    "web-1" = "i-aaaa1111"
    "web-2" = "i-bbbb2222"
    "web-3" = "i-cccc3333"
  }
}

import {
  for_each = var.existing_instances
  to       = aws_instance.web[each.key]
  id       = each.value
}
```

## Cleaning Up Guidelines

When cleaning generated configuration, follow these guidelines:

1. Remove tags_all (it is computed from tags)
2. Remove attributes that match provider defaults
3. Remove computed-only attributes like ARNs and IDs
4. Replace hardcoded IDs with resource references where possible
5. Add lifecycle rules for attributes you do not want Terraform to manage
6. Organize attributes in a logical order
7. Add comments explaining the purpose of each resource

```hcl
# Before cleanup (generated):
resource "aws_security_group" "web" {
  arn                    = "arn:aws:ec2:us-east-1:123456789:security-group/sg-123"
  description            = "Web server security group"
  egress                 = []
  id                     = "sg-0abcdef1234567890"
  ingress                = []
  name                   = "web-sg"
  name_prefix            = null
  owner_id               = "123456789012"
  revoke_rules_on_delete = false
  tags                   = { Name = "web-sg" }
  tags_all               = { Name = "web-sg" }
  vpc_id                 = "vpc-0abcd1234"
}

# After cleanup:
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Web server security group"
  vpc_id      = aws_vpc.main.id  # Reference instead of hardcoded ID

  tags = {
    Name = "web-sg"
  }
}
```

## Automating the Import Workflow

Create a script to automate the import process:

```bash
#!/bin/bash
# import-workflow.sh - Automate the import workflow

set -euo pipefail

echo "Step 1: Initializing Terraform..."
terraform init

echo "Step 2: Generating configuration..."
terraform plan -generate-config-out=generated.tf

echo "Step 3: Generated configuration saved to generated.tf"
echo "Please review and clean up the generated configuration."
echo "Then run: terraform apply"
```

## Conclusion

Generating configuration from imported resources is the fastest path to adopting existing infrastructure into Terraform. Instead of manually writing configuration for each resource, you let Terraform inspect the real resource and create a matching configuration. The key is to always clean up the generated code afterward, removing computed attributes and replacing hardcoded IDs with proper references.
