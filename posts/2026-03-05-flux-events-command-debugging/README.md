# How to Use flux events Command for Debugging

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux CD, GitOps, Kubernetes, Monitoring, Debugging, CLI, Events

Description: Learn how to use the flux events command to inspect Kubernetes events related to Flux CD resources for fast and effective debugging of reconciliation issues.

---

The `flux events` command is purpose-built for viewing Kubernetes events that are relevant to Flux CD resources. Unlike `kubectl get events`, which shows all events in a namespace, `flux events` filters events to show only those generated by Flux controllers. It also supports filtering by specific resources, making it a precise debugging tool when a reconciliation fails or behaves unexpectedly.

## Basic Usage

View all Flux-related events in the flux-system namespace:

```bash
flux events -n flux-system
```

The output includes the event timestamp, type (Normal/Warning), reason, object, and message, similar to `kubectl get events` but filtered to Flux resources only.

## View Events for a Specific Resource

The `--for` flag lets you target events for a specific Flux resource:

```bash
flux events --for GitRepository/flux-system -n flux-system
```

View events for a specific Kustomization:

```bash
flux events --for Kustomization/infrastructure -n flux-system
```

View events for a specific HelmRelease:

```bash
flux events --for HelmRelease/my-app -n my-namespace
```

View events for a HelmRepository:

```bash
flux events --for HelmRepository/bitnami -n flux-system
```

This is the most common use case -- when you know which resource is failing and want to see exactly what events it has generated.

## Watch Events in Real Time

Use the `--watch` flag to stream events as they occur:

```bash
flux events -n flux-system --watch
```

Watch events for a specific resource in real time:

```bash
flux events --for Kustomization/apps -n flux-system --watch
```

This is useful when you have just pushed a change to your Git repository and want to follow the reconciliation process from source fetch through to manifest application.

## Filter Events by Event Type

Focus on warnings to quickly find problems:

```bash
flux events -n flux-system --types=Warning
```

Show only normal events for successful operations:

```bash
flux events -n flux-system --types=Normal
```

## Debug Common Reconciliation Failures

Here are practical examples for debugging common issues using `flux events`.

### Source Fetch Failures

```bash
flux events --for GitRepository/my-repo -n flux-system
```

Example output for an authentication failure:

```text
LAST SEEN  TYPE      REASON                OBJECT                   MESSAGE
2m         Warning   GitOperationFailed    GitRepository/my-repo    failed to checkout: authentication required
```

### Kustomization Apply Failures

```bash
flux events --for Kustomization/apps -n flux-system
```

Example output for a validation error:

```text
LAST SEEN  TYPE      REASON                 OBJECT                  MESSAGE
5m         Warning   ReconciliationFailed   Kustomization/apps      Deployment/my-app validation error: spec.replicas must be non-negative
```

### HelmRelease Upgrade Failures

```bash
flux events --for HelmRelease/ingress-nginx -n ingress
```

Example output for a failed Helm upgrade:

```text
LAST SEEN  TYPE      REASON           OBJECT                       MESSAGE
3m         Warning   UpgradeFailed    HelmRelease/ingress-nginx    Helm upgrade failed: timed out waiting for the condition
```

## Correlate Events Across Resources

A common debugging pattern is to check events across the dependency chain. For example, if a Kustomization is not ready, the source it depends on may also have issues:

```bash
# Check the Kustomization events
flux events --for Kustomization/apps -n flux-system

# If the Kustomization reports "source not ready", check the source
flux events --for GitRepository/flux-system -n flux-system

# Use flux trace to see the full dependency chain
flux trace kustomization apps -n flux-system
```

## View Events Across All Namespaces

For a cluster-wide view of Flux events:

```bash
flux events --all-namespaces
```

Watch all Flux events across all namespaces:

```bash
flux events --all-namespaces --watch
```

Filter cluster-wide events for warnings only:

```bash
flux events --all-namespaces --types=Warning
```

## Combine with Other Flux Commands

Use `flux events` alongside other Flux CLI commands for a comprehensive debugging workflow:

```bash
# 1. Get an overview of all resources
flux get all --all-namespaces

# 2. Identify failing resources
flux get kustomizations --all-namespaces --status-selector ready=false

# 3. Check events for the failing resource
flux events --for Kustomization/my-failing-app -n flux-system

# 4. Check logs for more detail
flux logs --kind=Kustomization --name=my-failing-app --level=error --since=30m

# 5. Force a reconciliation after fixing the issue
flux reconcile kustomization my-failing-app -n flux-system --with-source
```

## Comparing flux events with kubectl get events

| Feature | kubectl get events | flux events |
|---------|-------------------|-------------|
| Shows all K8s events | Yes | No, Flux only |
| Filter by Flux resource | Manual field selectors | --for flag |
| Watch mode | --watch | --watch |
| Cross-namespace | --all-namespaces | --all-namespaces |
| Event type filter | --field-selector type=Warning | --types=Warning |
| Requires Flux CLI | No | Yes |

## Event Retention

Kubernetes events have a default retention of 1 hour. If you need to review events from further back, consider:

1. Using the Flux notification controller to forward events to external systems.
2. Setting up a log aggregation system to capture events.
3. Using a monitoring platform that persists event data.

## Summary

The `flux events` command is a focused debugging tool that filters Kubernetes events to show only Flux-relevant activity. Use `--for` to target specific resources, `--watch` for real-time monitoring, and `--types=Warning` to quickly identify failures. Combined with `flux logs` and `flux trace`, it forms a powerful debugging toolkit for diagnosing issues in your GitOps pipeline.
