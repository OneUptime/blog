# How to Handle Terraform State Access Control

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Security, State Management, AWS, Access Control

Description: Learn how to secure Terraform state files with proper access controls, encryption, locking, and audit logging to protect sensitive infrastructure data.

---

The Terraform state file is one of the most sensitive artifacts in your infrastructure. It contains the full configuration of every resource Terraform manages, including sensitive values like database passwords, access keys, and private keys. If someone gains access to your state file, they effectively have a map of your entire infrastructure along with the keys to access it. Protecting state access is not optional.

This guide covers practical approaches to locking down Terraform state with proper access controls, encryption, and monitoring.

## Why State Access Matters

The state file contains things you would not want exposed:

- Resource IDs and ARNs that map your infrastructure
- Database passwords and connection strings
- Private keys generated by the `tls_private_key` resource
- API keys passed to resources via variables
- Output values, including any marked as sensitive

Anyone with read access to the state can extract this information. Anyone with write access can potentially corrupt the state and cause Terraform to make destructive changes.

## S3 Backend with Full Security

The most common backend for AWS teams is S3 with DynamoDB locking. Here is how to set it up securely:

```hcl
# State bucket with all security controls
resource "aws_s3_bucket" "terraform_state" {
  bucket = "my-org-terraform-state"

  lifecycle {
    prevent_destroy = true
  }

  tags = {
    Name    = "Terraform State"
    Purpose = "terraform-state-storage"
  }
}

# Enable versioning for state recovery
resource "aws_s3_bucket_versioning" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  versioning_configuration {
    status = "Enabled"
  }
}

# Encrypt state at rest with CMK
resource "aws_s3_bucket_server_side_encryption_configuration" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.terraform_state.arn
    }
    bucket_key_enabled = true
  }
}

# Block all public access
resource "aws_s3_bucket_public_access_block" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable access logging
resource "aws_s3_bucket_logging" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  target_bucket = aws_s3_bucket.access_logs.id
  target_prefix = "terraform-state/"
}

# DynamoDB table for state locking
resource "aws_dynamodb_table" "terraform_locks" {
  name         = "terraform-state-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }

  server_side_encryption {
    enabled     = true
    kms_key_arn = aws_kms_key.terraform_state.arn
  }

  point_in_time_recovery {
    enabled = true
  }

  tags = {
    Name    = "Terraform State Locks"
    Purpose = "terraform-state-locking"
  }
}
```

## KMS Key for State Encryption

Create a dedicated KMS key with a carefully controlled policy:

```hcl
resource "aws_kms_key" "terraform_state" {
  description             = "KMS key for Terraform state encryption"
  deletion_window_in_days = 30
  enable_key_rotation     = true

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "EnableRootAccountFullAccess"
        Effect = "Allow"
        Principal = {
          AWS = "arn:aws:iam::${data.aws_caller_identity.current.account_id}:root"
        }
        Action   = "kms:*"
        Resource = "*"
      },
      {
        Sid    = "AllowStateAccessRoles"
        Effect = "Allow"
        Principal = {
          AWS = [
            aws_iam_role.terraform_ci.arn,
            aws_iam_role.terraform_admin.arn
          ]
        }
        Action = [
          "kms:Encrypt",
          "kms:Decrypt",
          "kms:ReEncrypt*",
          "kms:GenerateDataKey*",
          "kms:DescribeKey"
        ]
        Resource = "*"
      }
    ]
  })

  tags = {
    Name = "terraform-state-key"
  }
}

resource "aws_kms_alias" "terraform_state" {
  name          = "alias/terraform-state"
  target_key_id = aws_kms_key.terraform_state.key_id
}
```

## IAM Policies for State Access

Define separate policies for different levels of access:

```hcl
# Read-only access (for terraform plan)
resource "aws_iam_policy" "terraform_state_read" {
  name        = "terraform-state-read"
  description = "Read-only access to Terraform state"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "AllowStateRead"
        Effect = "Allow"
        Action = [
          "s3:GetObject",
          "s3:ListBucket"
        ]
        Resource = [
          aws_s3_bucket.terraform_state.arn,
          "${aws_s3_bucket.terraform_state.arn}/*"
        ]
      },
      {
        Sid    = "AllowKMSDecrypt"
        Effect = "Allow"
        Action = [
          "kms:Decrypt",
          "kms:DescribeKey"
        ]
        Resource = aws_kms_key.terraform_state.arn
      }
    ]
  })
}

# Read-write access (for terraform apply)
resource "aws_iam_policy" "terraform_state_write" {
  name        = "terraform-state-write"
  description = "Read-write access to Terraform state"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "AllowStateReadWrite"
        Effect = "Allow"
        Action = [
          "s3:GetObject",
          "s3:PutObject",
          "s3:ListBucket",
          "s3:DeleteObject"
        ]
        Resource = [
          aws_s3_bucket.terraform_state.arn,
          "${aws_s3_bucket.terraform_state.arn}/*"
        ]
      },
      {
        Sid    = "AllowStateLocking"
        Effect = "Allow"
        Action = [
          "dynamodb:GetItem",
          "dynamodb:PutItem",
          "dynamodb:DeleteItem"
        ]
        Resource = aws_dynamodb_table.terraform_locks.arn
      },
      {
        Sid    = "AllowKMSEncryptDecrypt"
        Effect = "Allow"
        Action = [
          "kms:Encrypt",
          "kms:Decrypt",
          "kms:ReEncrypt*",
          "kms:GenerateDataKey*",
          "kms:DescribeKey"
        ]
        Resource = aws_kms_key.terraform_state.arn
      }
    ]
  })
}
```

## Scope State Access by Environment

Use S3 key prefixes to separate state files by environment, and scope IAM permissions accordingly:

```hcl
# Backend configuration per environment
# environments/production/backend.tf
terraform {
  backend "s3" {
    bucket         = "my-org-terraform-state"
    key            = "production/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    kms_key_id     = "alias/terraform-state"
    dynamodb_table = "terraform-state-locks"
  }
}

# environments/staging/backend.tf
terraform {
  backend "s3" {
    bucket         = "my-org-terraform-state"
    key            = "staging/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    kms_key_id     = "alias/terraform-state"
    dynamodb_table = "terraform-state-locks"
  }
}
```

Then scope IAM access by prefix:

```hcl
# Production access - only platform team
resource "aws_iam_policy" "production_state" {
  name = "terraform-production-state"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = ["s3:GetObject", "s3:PutObject"]
        Resource = "${aws_s3_bucket.terraform_state.arn}/production/*"
      }
    ]
  })
}

# Staging access - wider team
resource "aws_iam_policy" "staging_state" {
  name = "terraform-staging-state"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = ["s3:GetObject", "s3:PutObject"]
        Resource = "${aws_s3_bucket.terraform_state.arn}/staging/*"
      }
    ]
  })
}
```

## Bucket Policy for Defense in Depth

Add a bucket policy that enforces encryption and restricts access:

```hcl
resource "aws_s3_bucket_policy" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "DenyUnencryptedUploads"
        Effect    = "Deny"
        Principal = "*"
        Action    = "s3:PutObject"
        Resource  = "${aws_s3_bucket.terraform_state.arn}/*"
        Condition = {
          StringNotEquals = {
            "s3:x-amz-server-side-encryption" = "aws:kms"
          }
        }
      },
      {
        Sid       = "DenyInsecureTransport"
        Effect    = "Deny"
        Principal = "*"
        Action    = "s3:*"
        Resource = [
          aws_s3_bucket.terraform_state.arn,
          "${aws_s3_bucket.terraform_state.arn}/*"
        ]
        Condition = {
          Bool = {
            "aws:SecureTransport" = "false"
          }
        }
      },
      {
        Sid       = "RestrictToTerraformRoles"
        Effect    = "Deny"
        Principal = "*"
        Action    = "s3:*"
        Resource = [
          aws_s3_bucket.terraform_state.arn,
          "${aws_s3_bucket.terraform_state.arn}/*"
        ]
        Condition = {
          StringNotLike = {
            "aws:PrincipalArn" = [
              aws_iam_role.terraform_ci.arn,
              aws_iam_role.terraform_admin.arn,
              "arn:aws:iam::${data.aws_caller_identity.current.account_id}:root"
            ]
          }
        }
      }
    ]
  })
}
```

## Audit State Access with CloudTrail

Monitor every read and write to the state bucket:

```hcl
resource "aws_cloudtrail" "state_audit" {
  name           = "terraform-state-audit"
  s3_bucket_name = aws_s3_bucket.audit_logs.id
  enable_logging = true

  event_selector {
    read_write_type           = "All"
    include_management_events = false

    data_resource {
      type   = "AWS::S3::Object"
      values = ["${aws_s3_bucket.terraform_state.arn}/"]
    }
  }
}

# Alert on state file access
resource "aws_cloudwatch_log_metric_filter" "state_access" {
  name           = "terraform-state-access"
  pattern        = "{ $.requestParameters.bucketName = \"my-org-terraform-state\" }"
  log_group_name = aws_cloudwatch_log_group.cloudtrail.name

  metric_transformation {
    name      = "TerraformStateAccessCount"
    namespace = "Custom/Terraform"
    value     = "1"
  }
}
```

## Terraform Cloud and Enterprise

If you use Terraform Cloud, state access is managed through workspace permissions:

```hcl
resource "tfe_workspace" "production" {
  name         = "production"
  organization = "my-org"

  # Only specific teams can access this workspace
  # Configured via tfe_team_access resources
}

resource "tfe_team_access" "platform_production" {
  access       = "write"
  team_id      = tfe_team.platform.id
  workspace_id = tfe_workspace.production.id
}

resource "tfe_team_access" "developers_production" {
  access       = "read"
  team_id      = tfe_team.developers.id
  workspace_id = tfe_workspace.production.id
}
```

## Wrapping Up

Terraform state is a high-value target that deserves serious protection. Use encrypted S3 backends with KMS CMKs, scope access by environment using IAM policies, enable versioning for recovery, set up state locking to prevent corruption, and audit every access with CloudTrail. The state file contains your infrastructure secrets, so treat it with the same care you would give to any other secrets store.
