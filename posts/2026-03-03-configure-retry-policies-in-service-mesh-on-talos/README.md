# How to Configure Retry Policies in Service Mesh on Talos

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, Retry Policies, Service Mesh, Kubernetes, Resilience

Description: A practical guide to configuring retry policies in service mesh on Talos Linux to handle transient failures and improve service reliability.

---

Transient failures are a fact of life in distributed systems. A network hiccup, a brief pod restart, or a momentary resource spike can cause a request to fail even when the service is generally healthy. Retry policies let the service mesh automatically retry failed requests, turning many of these transient failures into successful responses from the user's perspective. On Talos Linux, retry policies are configured through service mesh custom resources without any changes to your application code.

This guide covers how to configure retry policies using both Istio and Linkerd on a Talos Linux cluster, including best practices for avoiding common retry pitfalls.

## Why Retries Matter

In a monolithic application running on a single server, failures are relatively rare. But in a microservices architecture where a single user request might touch ten different services, the chance of at least one transient failure is much higher. If each service has a 99.9% success rate individually, ten services in a chain have only a 99% combined success rate. That is ten times more failures than any single service would produce.

Retries at the mesh level handle this transparently:

- The user never sees the transient failure
- The application code does not need retry logic
- The retry behavior is consistent across all services
- You can configure different retry policies for different routes

## Retry Fundamentals

Before configuring retries, understand the key parameters:

- **Max retries**: How many times to retry a failed request
- **Per-try timeout**: How long to wait for each individual attempt
- **Retry conditions**: Which types of failures should trigger a retry (5xx errors, connection failures, etc.)
- **Backoff**: How long to wait between retries (often exponential)
- **Retry budget**: A limit on the total percentage of additional requests generated by retries

## Retries with Istio

Istio provides fine-grained retry configuration through VirtualService resources.

### Basic Retry Configuration

```yaml
# istio-retry-basic.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: api-service
  namespace: default
spec:
  hosts:
  - api-service
  http:
  - route:
    - destination:
        host: api-service
    retries:
      attempts: 3
      perTryTimeout: 5s
      retryOn: "5xx,reset,connect-failure,retriable-4xx"
```

This retries up to 3 times on 5xx errors, connection resets, connection failures, and retriable 4xx errors. Each attempt has a 5-second timeout.

### Per-Route Retry Policies

Different API endpoints may need different retry behavior:

```yaml
# istio-retry-per-route.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: api-service
  namespace: default
spec:
  hosts:
  - api-service
  http:
  # Read operations - safe to retry aggressively
  - match:
    - method:
        exact: GET
    route:
    - destination:
        host: api-service
    retries:
      attempts: 5
      perTryTimeout: 3s
      retryOn: "5xx,reset,connect-failure"

  # Write operations - retry cautiously
  - match:
    - method:
        exact: POST
    route:
    - destination:
        host: api-service
    retries:
      attempts: 1
      perTryTimeout: 10s
      retryOn: "connect-failure,refused-stream"

  # Default - moderate retries
  - route:
    - destination:
        host: api-service
    retries:
      attempts: 2
      perTryTimeout: 5s
```

### Retry Conditions Explained

Istio supports several retry conditions:

```yaml
retries:
  retryOn: "5xx,reset,connect-failure,retriable-4xx,refused-stream,cancelled,deadline-exceeded,unavailable"
```

- `5xx` - Any 5xx HTTP status code
- `reset` - The upstream reset the connection
- `connect-failure` - Connection to the upstream failed
- `retriable-4xx` - 409 Conflict (useful for optimistic locking)
- `refused-stream` - The upstream refused the stream (HTTP/2)
- `cancelled` - gRPC cancelled status
- `deadline-exceeded` - gRPC deadline exceeded
- `unavailable` - gRPC unavailable status

### Controlling Retry Backoff

Istio uses exponential backoff between retries. You can control the base interval:

```yaml
retries:
  attempts: 3
  perTryTimeout: 5s
  retryOn: "5xx"
  retryRemoteLocalities: true
```

The default backoff starts at 25ms and doubles with each attempt, capped at the per-try timeout.

## Retries with Linkerd

Linkerd configures retries through ServiceProfile resources:

```yaml
# linkerd-retry.yaml
apiVersion: linkerd.io/v1alpha2
kind: ServiceProfile
metadata:
  name: api-service.default.svc.cluster.local
  namespace: default
spec:
  routes:
  - name: GET /api/users
    condition:
      method: GET
      pathRegex: /api/users(/.*)?
    isRetryable: true
    timeout: 10s
    responseClasses:
    - condition:
        status:
          min: 500
          max: 599
      isFailure: true

  - name: POST /api/users
    condition:
      method: POST
      pathRegex: /api/users
    isRetryable: false  # Don't retry writes by default
    timeout: 15s
```

Linkerd automatically applies a retry budget to prevent retry storms:

```bash
# Check the effective retry budget
linkerd viz routes deployment/api-service -n default -o wide
```

By default, Linkerd allows retries to add at most 20% additional request volume. This prevents a scenario where retries overwhelm the downstream service.

### Configuring Retry Budgets in Linkerd

```yaml
# You can adjust the retry budget through annotations
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
spec:
  template:
    metadata:
      annotations:
        config.linkerd.io/proxy-retry-budget-ttl: "10s"
```

## Avoiding Retry Storms

Retries can backfire if not configured carefully. Consider this scenario:

1. Service A retries requests to Service B 3 times
2. Service B retries requests to Service C 3 times
3. A single failure at Service C generates up to 9 requests (3 x 3)

This multiplicative effect can turn a small problem into a big one. Here is how to prevent it:

### Set Retry Budgets

Limit the total additional request volume from retries:

```yaml
# Istio - use EnvoyFilter for retry budget
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: retry-budget
  namespace: default
spec:
  configPatches:
  - applyTo: CLUSTER
    match:
      cluster:
        service: api-service.default.svc.cluster.local
    patch:
      operation: MERGE
      value:
        circuit_breakers:
          thresholds:
          - retry_budget:
              budget_percent:
                value: 20.0
              min_retry_concurrency: 3
```

### Only Retry Idempotent Operations

Do not retry requests that could produce duplicate side effects:

```yaml
# Only retry GET requests
- match:
  - method:
      exact: GET
  retries:
    attempts: 3
# No retries for POST, PUT, DELETE unless explicitly safe
- match:
  - method:
      exact: POST
  retries:
    attempts: 0
```

### Set Reasonable Timeouts

Always pair retries with timeouts to prevent requests from hanging:

```yaml
retries:
  attempts: 3
  perTryTimeout: 2s  # Each attempt times out after 2 seconds
timeout: 10s  # Total timeout including all retries
```

## Testing Retry Policies

Deploy a service that fails intermittently:

```yaml
# fault-injection.yaml (Istio)
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: api-service-fault
  namespace: default
spec:
  hosts:
  - api-service
  http:
  - fault:
      abort:
        httpStatus: 503
        percentage:
          value: 50  # 50% of requests fail
    route:
    - destination:
        host: api-service
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: "5xx"
```

With 50% failure rate and 3 retries, the effective failure rate from the client perspective drops to about 6.25% (0.5 ^ 4).

```bash
# Test and observe the retry behavior
for i in $(seq 1 100); do
  curl -s -o /dev/null -w "%{http_code}\n" http://api-service.default.svc.cluster.local
done | sort | uniq -c
```

## Monitoring Retries

Track retry activity to understand how often they are triggered:

```bash
# Istio - check Envoy retry stats
kubectl exec <POD_NAME> -c istio-proxy -- \
  curl -s localhost:15000/stats | grep retry

# Linkerd - check retry stats
linkerd viz routes deployment/api-service -n default -o wide
# The EFFECTIVE_RPS vs ACTUAL_RPS columns show retry amplification
```

## Conclusion

Retry policies in a service mesh on Talos Linux transform transient failures into successful responses, improving the reliability that end users experience. The key is to configure retries thoughtfully - retry idempotent operations aggressively, be cautious with write operations, always set timeouts and retry budgets, and monitor retry rates to detect underlying issues. Both Istio and Linkerd make this straightforward through declarative Kubernetes resources, and the configuration is completely transparent to your application code.
