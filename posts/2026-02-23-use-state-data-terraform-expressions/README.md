# How to Use State Data in Terraform Expressions

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, State Management, Expressions, Data Source, Infrastructure as Code

Description: Learn how to reference and use state data in Terraform expressions, including resource attributes, outputs, remote state, and data sources for dynamic configurations.

---

Terraform expressions are how you wire resources together and build dynamic configurations. At the core of most expressions is state data - the attributes of resources that Terraform has created, along with outputs and data source results. Understanding how to work with this data effectively makes your configurations more flexible and less repetitive.

This guide covers the practical patterns for referencing state data in expressions.

## Resource Attribute References

The most basic form of state data usage is referencing attributes of other resources:

```hcl
# Create a VPC
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"

  tags = {
    Name = "main-vpc"
  }
}

# Reference the VPC's ID in a subnet resource
resource "aws_subnet" "public" {
  vpc_id            = aws_vpc.main.id  # References the VPC ID from state
  cidr_block        = "10.0.1.0/24"
  availability_zone = "us-east-1a"
}

# Reference the subnet ID in an instance
resource "aws_instance" "web" {
  ami           = "ami-0123456789abcdef0"
  instance_type = "t3.micro"
  subnet_id     = aws_subnet.public.id  # References the subnet ID from state

  tags = {
    Name = "web-server"
  }
}
```

When Terraform creates `aws_vpc.main`, it stores all the resource's attributes in state. Other resources can then reference those attributes. Terraform figures out the dependency order automatically.

## Computed Attributes

Some attributes are only known after resource creation. These are called computed attributes:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0123456789abcdef0"
  instance_type = "t3.micro"
}

# These attributes are computed - only available after apply
output "instance_public_ip" {
  value = aws_instance.web.public_ip  # Only known after the instance is created
}

output "instance_arn" {
  value = aws_instance.web.arn  # Generated by AWS, stored in state
}

output "instance_private_dns" {
  value = aws_instance.web.private_dns  # Assigned by the VPC
}
```

During `terraform plan`, computed attributes show as `(known after apply)`. After `terraform apply`, they're populated in state and available for reference.

## Working with count and for_each

When resources use `count` or `for_each`, you reference them with indexing:

```hcl
# Resources created with count
resource "aws_subnet" "private" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index + 10)
  availability_zone = element(["us-east-1a", "us-east-1b", "us-east-1c"], count.index)
}

# Reference a specific instance by index
output "first_subnet_id" {
  value = aws_subnet.private[0].id
}

# Reference all instance IDs as a list
output "all_subnet_ids" {
  value = aws_subnet.private[*].id  # Splat expression - returns a list
}

# Use the list in another resource
resource "aws_db_subnet_group" "main" {
  name       = "main-db-subnet-group"
  subnet_ids = aws_subnet.private[*].id  # All private subnet IDs
}
```

For `for_each` resources:

```hcl
# Resources created with for_each
resource "aws_security_group" "services" {
  for_each = toset(["web", "api", "worker"])

  name   = "${each.key}-sg"
  vpc_id = aws_vpc.main.id

  tags = {
    Service = each.key
  }
}

# Reference a specific instance by key
output "web_sg_id" {
  value = aws_security_group.services["web"].id
}

# Reference all as a map
output "all_sg_ids" {
  value = { for k, v in aws_security_group.services : k => v.id }
}
```

## Using Outputs from Modules

Module outputs are another common source of state data:

```hcl
# Define a module
module "networking" {
  source = "./modules/networking"

  vpc_cidr = "10.0.0.0/16"
}

# Reference module outputs
resource "aws_instance" "web" {
  ami           = "ami-0123456789abcdef0"
  instance_type = "t3.micro"
  subnet_id     = module.networking.public_subnet_id  # Output from the module

  vpc_security_group_ids = [
    module.networking.web_sg_id  # Another module output
  ]
}
```

Inside the module, you define outputs to expose state data:

```hcl
# modules/networking/outputs.tf
output "public_subnet_id" {
  description = "ID of the public subnet"
  value       = aws_subnet.public.id
}

output "web_sg_id" {
  description = "ID of the web security group"
  value       = aws_security_group.web.id
}

output "vpc_id" {
  description = "ID of the VPC"
  value       = aws_vpc.main.id
}
```

## Remote State Data Source

To reference state from a completely separate Terraform configuration:

```hcl
# Read state from another Terraform project
data "terraform_remote_state" "networking" {
  backend = "s3"
  config = {
    bucket = "my-terraform-state"
    key    = "networking/terraform.tfstate"
    region = "us-east-1"
  }
}

# Use the remote state outputs in expressions
resource "aws_instance" "web" {
  ami           = "ami-0123456789abcdef0"
  instance_type = "t3.micro"
  subnet_id     = data.terraform_remote_state.networking.outputs.public_subnet_id

  tags = {
    VPC = data.terraform_remote_state.networking.outputs.vpc_id
  }
}
```

Only `output` values from the remote state are accessible - you can't directly reference resource attributes from another state file.

## Data Sources for Existing Resources

Data sources query your cloud provider for information about existing resources that may or may not be managed by Terraform:

```hcl
# Look up an existing AMI
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]  # Canonical

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }
}

# Look up the current AWS account ID
data "aws_caller_identity" "current" {}

# Look up existing VPC by tag
data "aws_vpc" "existing" {
  tags = {
    Environment = "production"
  }
}

# Use data source results in expressions
resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id  # Latest Ubuntu AMI
  instance_type = "t3.micro"
  subnet_id     = data.aws_vpc.existing.id

  tags = {
    Account = data.aws_caller_identity.current.account_id
  }
}
```

## Advanced Expression Patterns

### Conditional Expressions with State Data

```hcl
# Use a different subnet based on whether an instance needs public access
resource "aws_instance" "app" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.micro"

  # Conditional: use public subnet if the instance needs a public IP
  subnet_id = var.needs_public_ip ? aws_subnet.public.id : aws_subnet.private.id

  # Conditional: only assign a public IP if in the public subnet
  associate_public_ip_address = var.needs_public_ip
}
```

### Dynamic Blocks Using State Data

```hcl
# Create security group rules dynamically from a map of services
locals {
  service_ports = {
    web    = { port = 80,   cidr = "0.0.0.0/0" }
    api    = { port = 8080, cidr = "10.0.0.0/8" }
    health = { port = 8081, cidr = "10.0.0.0/8" }
  }
}

resource "aws_security_group" "app" {
  name   = "app-sg"
  vpc_id = aws_vpc.main.id

  dynamic "ingress" {
    for_each = local.service_ports
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      cidr_blocks = [ingress.value.cidr]
      description = "Allow ${ingress.key} traffic"
    }
  }
}
```

### Transforming State Data with for Expressions

```hcl
# Create a map of instance IDs to private IPs
locals {
  instance_ips = {
    for instance in aws_instance.web :
    instance.id => instance.private_ip
  }
}

# Filter resources based on state data
locals {
  # Get only the instances that have public IPs
  public_instances = [
    for instance in aws_instance.web :
    instance if instance.public_ip != ""
  ]
}

# Flatten nested data from state
locals {
  all_subnet_ids = flatten([
    for az_subnets in aws_subnet.multi_az :
    az_subnets.id
  ])
}
```

### String Interpolation with State Data

```hcl
# Build strings from state data
resource "aws_route53_record" "web" {
  zone_id = data.aws_route53_zone.main.zone_id
  name    = "web.${data.aws_route53_zone.main.name}"
  type    = "A"
  ttl     = 300
  records = [aws_instance.web.public_ip]
}

# Build IAM policy from state data
resource "aws_iam_policy" "bucket_access" {
  name = "bucket-access"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = ["s3:GetObject"]
        Resource = "${aws_s3_bucket.data.arn}/*"  # Interpolate bucket ARN from state
      }
    ]
  })
}
```

## The terraform_data Resource

The `terraform_data` resource (introduced in Terraform 1.4) stores arbitrary values in state:

```hcl
# Store computed values in state for later reference
resource "terraform_data" "deployment_info" {
  input = {
    deployed_at    = timestamp()
    deployed_by    = data.aws_caller_identity.current.arn
    git_commit     = var.git_commit
    instance_count = length(aws_instance.web)
  }
}

# Reference the stored data
output "deployment_info" {
  value = terraform_data.deployment_info.output
}
```

## Wrapping Up

State data is the glue that connects Terraform resources together. Whether you're referencing attributes from a resource in the same configuration, pulling outputs from a module, reading remote state from another project, or querying existing infrastructure with data sources, the patterns are consistent and composable.

The key is understanding what data is available (use `terraform state show` and provider documentation) and how to transform it (using for expressions, conditionals, and built-in functions) to build the configuration you need.
