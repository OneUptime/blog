# How to Generate Config Files with templatefile in Terraform

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Templatefile, Configuration Management, Infrastructure as Code, DevOps

Description: Learn how to use Terraform's templatefile function to generate application configuration files dynamically with variable substitution, loops, and conditionals.

---

Application configuration files are a constant pain point in infrastructure management. Every environment needs slightly different settings, and manually maintaining config files for dev, staging, and production is a recipe for drift and mistakes. Terraform's `templatefile` function lets you generate these files dynamically from templates, keeping a single source of truth for your configuration structure while injecting environment-specific values at deploy time.

## The Problem with Static Config Files

Consider a typical application that reads from a JSON or YAML config file. You might have `config.dev.json`, `config.staging.json`, and `config.prod.json` sitting in your repository. Every time you add a new config key, you have to update all three files. Every time someone changes one but forgets the others, things break in unexpected ways.

With `templatefile`, you write one template and let Terraform fill in the blanks.

## How templatefile Works

The function signature is simple:

```hcl
# templatefile(path, variables)
# path: relative path to the template file
# variables: a map of values to substitute
```

Inside the template, you use `${variable_name}` for simple substitution, `%{ if ... }` for conditionals, and `%{ for ... }` for loops.

## Generating a JSON Configuration File

Let's say your application expects a JSON config file at `/etc/myapp/config.json`. Here is how you would set this up.

Create the template:

```json
// templates/app_config.json.tpl
{
  "app_name": "${app_name}",
  "environment": "${environment}",
  "log_level": "${log_level}",
  "database": {
    "host": "${db_host}",
    "port": ${db_port},
    "name": "${db_name}",
    "max_connections": ${db_max_connections},
    "ssl_enabled": ${db_ssl_enabled}
  },
  "cache": {
    "host": "${cache_host}",
    "port": ${cache_port},
    "ttl_seconds": ${cache_ttl}
  },
  "features": {
%{ for i, feature in features ~}
    "${feature.name}": ${feature.enabled}%{ if i < length(features) - 1 },
%{ endif }
%{ endfor ~}
  }
}
```

Now use it in your Terraform code:

```hcl
# main.tf

locals {
  # Define environment-specific settings
  env_config = {
    dev = {
      log_level          = "debug"
      db_max_connections = 5
      db_ssl_enabled     = false
      cache_ttl          = 60
    }
    staging = {
      log_level          = "info"
      db_max_connections = 20
      db_ssl_enabled     = true
      cache_ttl          = 300
    }
    production = {
      log_level          = "warn"
      db_max_connections = 100
      db_ssl_enabled     = true
      cache_ttl          = 3600
    }
  }

  # Select the right config for the current environment
  current_config = local.env_config[var.environment]
}

# Render the config and write it to the instance
resource "aws_instance" "app" {
  ami           = var.ami_id
  instance_type = var.instance_type

  # Provision the config file
  provisioner "file" {
    content = templatefile("${path.module}/templates/app_config.json.tpl", {
      app_name           = "my-awesome-app"
      environment        = var.environment
      log_level          = local.current_config.log_level
      db_host            = aws_db_instance.main.address
      db_port            = aws_db_instance.main.port
      db_name            = aws_db_instance.main.db_name
      db_max_connections = local.current_config.db_max_connections
      db_ssl_enabled     = local.current_config.db_ssl_enabled
      cache_host         = aws_elasticache_cluster.main.cache_nodes[0].address
      cache_port         = aws_elasticache_cluster.main.cache_nodes[0].port
      cache_ttl          = local.current_config.cache_ttl
      features = [
        { name = "dark_mode", enabled = true },
        { name = "beta_api", enabled = var.environment != "production" },
        { name = "analytics", enabled = true },
      ]
    })
    destination = "/etc/myapp/config.json"
  }
}
```

## Generating Nginx Configuration

Web server configs are another common use case. Here is an Nginx config template that supports multiple virtual hosts:

```nginx
# templates/nginx.conf.tpl

# Auto-generated by Terraform - do not edit manually
worker_processes auto;
pid /run/nginx.pid;

events {
    worker_connections ${worker_connections};
}

http {
    sendfile on;
    tcp_nopush on;
    keepalive_timeout 65;

    # Logging
    access_log /var/log/nginx/access.log;
    error_log /var/log/nginx/error.log ${log_level};

%{ for vhost in virtual_hosts }
    server {
        listen ${vhost.port};
        server_name ${vhost.server_name};

%{ if vhost.ssl_enabled }
        ssl_certificate /etc/ssl/certs/${vhost.server_name}.crt;
        ssl_certificate_key /etc/ssl/private/${vhost.server_name}.key;
%{ endif }

%{ for location in vhost.locations }
        location ${location.path} {
            proxy_pass ${location.backend};
            proxy_set_header Host $host;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
%{ if location.websocket }
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
%{ endif }
        }
%{ endfor }
    }
%{ endfor }
}
```

```hcl
# main.tf

locals {
  # Define virtual hosts for the load balancer
  vhosts = [
    {
      server_name = "api.example.com"
      port        = 443
      ssl_enabled = true
      locations = [
        {
          path      = "/"
          backend   = "http://api-backend:8080"
          websocket = false
        },
        {
          path      = "/ws"
          backend   = "http://api-backend:8080"
          websocket = true
        },
      ]
    },
    {
      server_name = "app.example.com"
      port        = 443
      ssl_enabled = true
      locations = [
        {
          path      = "/"
          backend   = "http://frontend:3000"
          websocket = false
        },
      ]
    },
  ]
}

resource "local_file" "nginx_config" {
  filename = "${path.module}/output/nginx.conf"
  content = templatefile("${path.module}/templates/nginx.conf.tpl", {
    worker_connections = 1024
    log_level          = var.environment == "production" ? "warn" : "info"
    virtual_hosts      = local.vhosts
  })
}
```

## Generating YAML Config Files

YAML configuration files need careful handling because whitespace matters. Here is a template for a Kubernetes-style config:

```yaml
# templates/app.yaml.tpl

# Application configuration
# Generated by Terraform for ${environment}
app:
  name: ${app_name}
  version: "${app_version}"
  port: ${app_port}

logging:
  level: ${log_level}
  format: json
  output: stdout

database:
  primary:
    host: ${db_primary_host}
    port: ${db_primary_port}
%{ if db_replica_host != "" }
  replica:
    host: ${db_replica_host}
    port: ${db_replica_port}
    read_only: true
%{ endif }

%{ if length(allowed_origins) > 0 }
cors:
  allowed_origins:
%{ for origin in allowed_origins ~}
    - "${origin}"
%{ endfor ~}
%{ endif }
```

```hcl
# main.tf

resource "local_file" "app_config" {
  filename = "${path.module}/output/app.yaml"
  content = templatefile("${path.module}/templates/app.yaml.tpl", {
    environment    = var.environment
    app_name       = "myservice"
    app_version    = var.app_version
    app_port       = 8080
    log_level      = var.environment == "production" ? "warn" : "debug"
    db_primary_host = aws_db_instance.primary.address
    db_primary_port = aws_db_instance.primary.port
    db_replica_host = var.environment == "production" ? aws_db_instance.replica[0].address : ""
    db_replica_port = var.environment == "production" ? aws_db_instance.replica[0].port : 0
    allowed_origins = var.environment == "production" ? [
      "https://app.example.com",
      "https://admin.example.com",
    ] : ["http://localhost:3000"]
  })
}
```

## Writing Config Files with local_file

The `local_file` resource is handy during development for inspecting what your templates produce. For production, you will typically use `templatefile` with provisioners or pass the rendered content to resources that accept configuration strings.

```hcl
# Write to disk for inspection
resource "local_file" "config" {
  filename        = "${path.module}/generated/config.json"
  content         = templatefile("${path.module}/templates/config.json.tpl", local.config_vars)
  file_permission = "0644"
}
```

## Using templatefile with Kubernetes ConfigMaps

A practical pattern is generating Kubernetes ConfigMaps from templates:

```hcl
# Generate a ConfigMap with rendered config
resource "kubernetes_config_map" "app" {
  metadata {
    name      = "app-config"
    namespace = var.namespace
  }

  data = {
    "config.json" = templatefile("${path.module}/templates/config.json.tpl", {
      environment = var.environment
      db_host     = var.db_host
      db_port     = var.db_port
    })

    "logging.json" = templatefile("${path.module}/templates/logging.json.tpl", {
      log_level = var.log_level
      log_format = "json"
    })
  }
}
```

## Using jsonencode Inside Templates

For JSON config files, consider using the `jsonencode` function instead of hand-crafting JSON in a template. This avoids issues with trailing commas and escaping:

```hcl
# Instead of a template, generate JSON directly
resource "local_file" "config" {
  filename = "${path.module}/output/config.json"
  content = jsonencode({
    app_name    = "myapp"
    environment = var.environment
    database = {
      host = aws_db_instance.main.address
      port = aws_db_instance.main.port
    }
    features = { for f in var.features : f.name => f.enabled }
  })
}
```

This approach works well for simpler configs, but `templatefile` shines when the config file format is complex or when you need fine-grained control over the output structure.

## Tips for Managing Templates

Keep your templates organized. A common project structure looks like this:

```text
project/
  main.tf
  variables.tf
  outputs.tf
  templates/
    user_data.sh.tpl
    nginx.conf.tpl
    app_config.json.tpl
    logrotate.conf.tpl
```

Name your template files with a `.tpl` extension so they are easy to identify. Some teams use double extensions like `.json.tpl` or `.sh.tpl` to indicate both the output format and the fact that it is a template.

## Summary

The `templatefile` function transforms how you manage application configuration in Terraform. Instead of maintaining separate config files for each environment, you define one template and let Terraform handle the variation. Whether you are generating JSON, YAML, Nginx configs, or any other text-based format, `templatefile` keeps your infrastructure code clean and your configurations consistent across environments. Start small with one config file, prove the pattern, and expand from there.
