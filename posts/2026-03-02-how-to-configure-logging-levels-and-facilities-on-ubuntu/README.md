# How to Configure Logging Levels and Facilities on Ubuntu

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Ubuntu, Logging, rsyslog, Syslog, System Administration

Description: Understand and configure syslog facilities and severity levels on Ubuntu to control what gets logged, where it goes, and how to filter log messages effectively.

---

The syslog system categorizes log messages using two dimensions: facility (what generated the message) and severity level (how important the message is). Understanding this two-dimensional categorization is fundamental to configuring effective logging on Ubuntu, whether you're using rsyslog, syslog-ng, or the systemd journal.

## Syslog Facilities

Facilities identify the source or type of the logging message. The original syslog standard defined these facilities:

| Code | Facility | Used by |
|------|----------|---------|
| 0 | kern | Kernel messages |
| 1 | user | User-level messages |
| 2 | mail | Mail system |
| 3 | daemon | System daemons |
| 4 | auth | Security/authentication |
| 5 | syslog | Messages generated by syslogd |
| 6 | lpr | Line printer subsystem |
| 7 | news | Network news subsystem |
| 8 | uucp | UUCP subsystem |
| 9 | cron | Clock daemon |
| 10 | authpriv | Security messages (private) |
| 11 | ftp | FTP daemon |
| 16-23 | local0-local7 | Local use - custom applications |

The `local0` through `local7` facilities are particularly important for production systems - they're the facilities you assign to custom applications and services.

## Syslog Severity Levels

Severity levels indicate how critical a message is:

| Code | Level | Meaning |
|------|-------|---------|
| 0 | emerg | System is unusable |
| 1 | alert | Action must be taken immediately |
| 2 | crit | Critical conditions |
| 3 | err | Error conditions |
| 4 | warning | Warning conditions |
| 5 | notice | Normal but significant condition |
| 6 | info | Informational messages |
| 7 | debug | Debug-level messages |

In rsyslog configuration, specifying a level means "this level and all more severe levels." So `warning` means warning, err, crit, alert, and emerg.

## rsyslog Configuration Syntax

rsyslog uses a `facility.severity` selector syntax to route messages:

```bash
# View the main rsyslog configuration
cat /etc/rsyslog.conf

# View additional configuration files
ls /etc/rsyslog.d/
```

### Selector Syntax

```bash
# Format: facility.severity  destination
# facility.severity can be:
#   *.*         = all facilities, all severities
#   kern.err    = kernel facility, error severity and above
#   *.crit      = all facilities, critical and above
#   mail.none   = exclude mail facility

# Examples in /etc/rsyslog.conf:

# All kernel messages
kern.*                /var/log/kern.log

# Authentication - all levels
auth,authpriv.*       /var/log/auth.log

# All messages except auth
*.*;auth,authpriv.none    /var/log/syslog

# Emergency messages to all users
*.emerg               :omusrmsg:*

# Critical and above to dedicated file
*.crit                /var/log/critical.log

# Errors from daemon and cron facilities
daemon.err,cron.err   /var/log/daemon-errors.log
```

### The `.none` Selector

`.none` means "exclude messages from this facility":

```bash
# All messages except auth and mail
*.*;auth,authpriv.none;mail.none    /var/log/messages

# All facilities except kernel
*.*;kern.none    /var/log/syslog-no-kernel.log
```

### Exact Level Matching

Use `=` to match exactly one severity level (not "and above"):

```bash
# Log ONLY debug messages (not info, notice, etc.)
*.=debug    /var/log/debug-only.log

# Log ONLY notice level messages
*.=notice   /var/log/notices.log
```

## Configuring Custom Log Files

Create a drop-in configuration to route specific messages to custom files:

```bash
sudo nano /etc/rsyslog.d/40-custom-facilities.conf
```

```bash
# Route local0 facility to application-specific log
local0.*    /var/log/myapp/application.log

# Route local1 to another application
local1.*    /var/log/webapp/access.log

# Only errors and above from local2
local2.err  /var/log/myservice/errors.log

# Debug from local3 (useful during development)
local3.=debug   /var/log/myservice/debug.log

# All local facilities to a combined local log
local0,local1,local2,local3,local4.*    /var/log/local-apps.log
```

After editing:

```bash
# Create log directories if needed
sudo mkdir -p /var/log/myapp /var/log/webapp /var/log/myservice
sudo chown syslog:adm /var/log/myapp /var/log/webapp /var/log/myservice

# Test configuration syntax
rsyslogd -N1

# Restart rsyslog
sudo systemctl restart rsyslog
```

## Sending Application Logs with the Right Facility

Applications can select their syslog facility via the `logger` command for testing:

```bash
# Send a message with specific facility and severity
logger -p local0.info "Application started successfully"
logger -p local0.err "Database connection failed"
logger -p local0.debug "Processing request ID 12345"

# Send to a specific tag
logger -t myapp -p local0.info "Service initialized"

# Verify routing
tail -f /var/log/myapp/application.log
```

For applications written in Python, Node.js, etc., the syslog library allows specifying facility:

```python
# Python example
import syslog

# Open with facility
syslog.openlog("myapp", syslog.LOG_PID, syslog.LOG_LOCAL0)

# Log at different levels
syslog.syslog(syslog.LOG_INFO, "Application started")
syslog.syslog(syslog.LOG_ERR, "Database connection failed")
syslog.syslog(syslog.LOG_DEBUG, "Processing item 42")

syslog.closelog()
```

## Configuring Systemd Journal Filtering

The systemd journal also supports priority-based filtering:

```bash
# Show only errors and above
journalctl -p err

# Show only warnings
journalctl -p warning

# Show a priority range
journalctl -p warning..err

# Priority names: emerg, alert, crit, err, warning, notice, info, debug
# Or numeric: 0-7

# Filter by systemd unit and priority
journalctl -u nginx -p warning

# Real-time error monitoring
journalctl -f -p err
```

## Configuring journald Log Levels

Control what journald stores via `/etc/systemd/journald.conf`:

```bash
sudo nano /etc/systemd/journald.conf
```

```bash
[Journal]
# Maximum log level to store
# Values: emerg, alert, crit, err, warning, notice, info, debug
MaxLevelStore=info

# Level for storing to disk (when volatile storage overflows)
MaxLevelSyslog=warning

# Level to forward to the kernel log buffer (for dmesg)
MaxLevelKMsg=notice

# Level to forward to a connected terminal
MaxLevelConsole=info

# Level to forward to the wall (broadcast to all users)
MaxLevelWall=emerg
```

```bash
# Apply changes
sudo systemctl restart systemd-journald
```

## Configuring Application-Specific Log Levels

Many services have their own logging configuration independent of syslog:

### Apache2

```bash
sudo nano /etc/apache2/apache2.conf
```

```bash
# Apache log levels: trace8...trace1, debug, info, notice, warn, error, crit, alert, emerg
LogLevel warn

# Per-module log levels
LogLevel mod_ssl.c:info mod_auth_basic.c:debug

# Per-virtual-host log levels
<VirtualHost *:443>
    LogLevel error
</VirtualHost>
```

### Nginx

```bash
sudo nano /etc/nginx/nginx.conf
```

```bash
# Nginx error log levels: debug, info, notice, warn, error, crit, alert, emerg
error_log /var/log/nginx/error.log warn;
```

### MySQL/MariaDB

```bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
```

```bash
[mysqld]
# Log errors only
log_error = /var/log/mysql/error.log
log_error_verbosity = 2   # 1=errors, 2=errors+warnings, 3=all

# Enable general query log (careful - very verbose)
# general_log = 1
# general_log_file = /var/log/mysql/general.log

# Slow query log
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2
```

## Creating a Multi-Tier Logging Configuration

A practical multi-tier setup routes messages by importance:

```bash
sudo nano /etc/rsyslog.d/20-multi-tier.conf
```

```bash
# Tier 1: Critical alerts - write immediately and alert
*.alert     /var/log/critical/alerts.log
*.alert     |/usr/local/bin/send-alert.sh   # Named pipe to alert script

# Tier 2: Errors - standard error log
*.err;*.!alert    /var/log/errors.log

# Tier 3: Warnings - warning log (rotated frequently)
*.warning;*.!err    /var/log/warnings.log

# Tier 4: Information - standard syslog
*.info;*.!warning;auth,authpriv.none;mail.none    /var/log/syslog

# Debug - only in development (comment out in production)
# *.debug    /var/log/debug.log

# Authentication - always to its own file
auth,authpriv.*    /var/log/auth.log
```

## Monitoring Log Volume by Facility

Track which facilities are generating the most messages:

```bash
# Count messages by facility from syslog
awk '{print $5}' /var/log/syslog | sort | uniq -c | sort -rn | head -20

# Using journalctl to count by unit
journalctl --since "1 hour ago" -o json | \
    python3 -c "
import sys, json, collections
counts = collections.Counter()
for line in sys.stdin:
    try:
        entry = json.loads(line)
        counts[entry.get('SYSLOG_FACILITY', 'unknown')] += 1
    except:
        pass
for facility, count in counts.most_common():
    print(f'{count:6d}  facility={facility}')
"
```

## Testing Your Configuration

```bash
# Send test messages at various levels and facilities
for level in emerg alert crit err warning notice info debug; do
    logger -p local0.$level "Test message at level: $level"
done

# Verify each went to the right destination
grep "Test message" /var/log/myapp/application.log
grep "Test message" /var/log/errors.log
grep "Test message" /var/log/syslog

# Check rsyslog statistics
cat /proc/$(pgrep rsyslogd)/status | grep VmRSS
```

Getting facility and level configuration right means your log files contain what you expect, at the granularity you need, without being overwhelmed by noise from verbose subsystems. Start with a sensible default configuration and adjust based on what you actually need to monitor.
