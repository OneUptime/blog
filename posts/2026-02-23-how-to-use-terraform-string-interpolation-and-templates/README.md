# How to Use Terraform String Interpolation and Templates

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, HCL, Infrastructure as Code, Templates, String

Description: Learn how to use string interpolation and template expressions in Terraform to build dynamic values, generate configuration files, and create reusable infrastructure code.

---

String interpolation is one of the features you use constantly in Terraform. Any time you need to combine a variable with a static string, reference a resource attribute inside a name, or build a URL dynamically, you are using interpolation. And when you need to generate more complex text like configuration files or user data scripts, Terraform's template features take you the rest of the way.

Let's start with the basics and work up to practical template patterns.

## Basic String Interpolation

String interpolation embeds an expression inside a string using `${ }`:

```hcl
variable "environment" {
  type    = string
  default = "production"
}

variable "project" {
  type    = string
  default = "myapp"
}

resource "aws_s3_bucket" "data" {
  # Interpolate variables into the bucket name
  bucket = "${var.project}-${var.environment}-data"
  # Result: "myapp-production-data"
}
```

You can put any valid Terraform expression inside `${ }`:

```hcl
locals {
  # Interpolate a function call
  timestamp = "backup-${formatdate("YYYY-MM-DD", timestamp())}"

  # Interpolate a conditional
  suffix = "server-${var.is_primary ? "primary" : "secondary"}"

  # Interpolate arithmetic
  port_label = "port-${var.base_port + 1}"

  # Interpolate resource attributes
  connection_string = "postgresql://${aws_db_instance.main.address}:${aws_db_instance.main.port}/mydb"
}
```

## When You Do Not Need Interpolation

A common mistake is wrapping a single expression in unnecessary interpolation:

```hcl
# BAD - unnecessary interpolation
instance_type = "${var.instance_type}"
subnet_id     = "${aws_subnet.public.id}"
count         = "${var.instance_count}"

# GOOD - use the expression directly
instance_type = var.instance_type
subnet_id     = aws_subnet.public.id
count         = var.instance_count
```

Only use `${ }` when you are combining an expression with other text in a string. If the entire value is just one expression, use it directly. Running `terraform fmt` will not fix this, but it is a style issue that makes code harder to read.

## Interpolating Different Types

Terraform automatically converts non-string types to strings during interpolation:

```hcl
variable "port" {
  type    = number
  default = 8080
}

variable "enable_ssl" {
  type    = bool
  default = true
}

locals {
  # Numbers are converted to string representation
  port_label = "port-${var.port}"
  # Result: "port-8080"

  # Booleans become "true" or "false"
  ssl_status = "SSL is ${var.enable_ssl}"
  # Result: "SSL is true"

  # Lists need to be joined first
  cidrs = join(", ", var.allowed_cidrs)
  # Then interpolated
  rule_desc = "Allow traffic from: ${local.cidrs}"
}
```

## The templatefile Function

For anything more than simple string building, use the `templatefile` function. It reads a separate template file and renders it with the variables you provide.

Create a template file:

```bash
# templates/user_data.tftpl
#!/bin/bash
# Server initialization script
# Generated by Terraform - do not edit manually

# Set hostname
hostnamectl set-hostname ${hostname}

# Install packages
apt-get update
apt-get install -y ${join(" ", packages)}

# Configure the application
cat > /etc/myapp/config.json <<APPCONFIG
{
  "port": ${port},
  "environment": "${environment}",
  "database_host": "${db_host}",
  "database_port": ${db_port},
  "features": {
%{ for feature in features ~}
    "${feature}": true,
%{ endfor ~}
    "_end": true
  }
}
APPCONFIG

# Start the service
systemctl enable myapp
systemctl start myapp
```

Then reference it in your Terraform configuration:

```hcl
resource "aws_instance" "app" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = var.instance_type

  # Render the template with variables
  user_data = templatefile("${path.module}/templates/user_data.tftpl", {
    hostname    = "app-${var.environment}-01"
    packages    = ["nginx", "postgresql-client", "jq"]
    port        = 8080
    environment = var.environment
    db_host     = aws_db_instance.main.address
    db_port     = aws_db_instance.main.port
    features    = ["auth", "logging", "metrics"]
  })
}
```

### Template File Naming

By convention, Terraform template files use the `.tftpl` extension. This helps editors provide syntax highlighting and makes it clear the file is a Terraform template. You can use any extension, but `.tftpl` is the standard.

## Template Directives

Templates support directives for loops and conditionals. These use `%{ }` syntax (note: the percent sign, not dollar sign).

### Conditionals in Templates

```bash
# templates/nginx.conf.tftpl

server {
    listen ${port};
    server_name ${server_name};

%{ if enable_ssl }
    listen 443 ssl;
    ssl_certificate     /etc/ssl/certs/${domain}.crt;
    ssl_certificate_key /etc/ssl/private/${domain}.key;
%{ endif }

    location / {
        proxy_pass http://localhost:${app_port};
    }

%{ if enable_logging }
    access_log /var/log/nginx/${server_name}_access.log;
    error_log  /var/log/nginx/${server_name}_error.log;
%{ endif }
}
```

Usage:

```hcl
locals {
  nginx_config = templatefile("${path.module}/templates/nginx.conf.tftpl", {
    port           = 80
    server_name    = "api.example.com"
    domain         = "example.com"
    enable_ssl     = true
    app_port       = 8080
    enable_logging = true
  })
}
```

### Loops in Templates

```bash
# templates/hosts.tftpl
# /etc/hosts file generated by Terraform

127.0.0.1 localhost

# Application servers
%{ for idx, server in servers ~}
${server.ip}  ${server.hostname}  # ${server.role}
%{ endfor ~}

# Database servers
%{ for idx, db in databases ~}
${db.ip}  ${db.hostname}
%{ endfor ~}
```

Usage:

```hcl
locals {
  hosts_file = templatefile("${path.module}/templates/hosts.tftpl", {
    servers = [
      { ip = "10.0.1.10", hostname = "app-01", role = "web" },
      { ip = "10.0.1.11", hostname = "app-02", role = "web" },
      { ip = "10.0.1.12", hostname = "app-03", role = "worker" },
    ]
    databases = [
      { ip = "10.0.2.10", hostname = "db-primary" },
      { ip = "10.0.2.11", hostname = "db-replica" },
    ]
  })
}
```

## Whitespace Control in Templates

Templates often produce unwanted blank lines. The `~` (tilde) character strips whitespace:

```bash
# Without whitespace control - produces blank lines
%{ for name in names }
${name}
%{ endfor }

# With whitespace control - clean output
%{ for name in names ~}
${name}
%{ endfor ~}
```

Place the `~` on the side where you want whitespace removed:

- `%{ ... ~}` removes trailing whitespace (including newline)
- `%{~ ... }` removes leading whitespace (including newline)
- `%{~ ... ~}` removes both leading and trailing whitespace

```bash
# templates/env.tftpl
%{ for key, value in env_vars ~}
${key}=${value}
%{ endfor ~}
```

This produces clean output without extra blank lines:

```text
DATABASE_URL=postgresql://localhost:5432/mydb
REDIS_URL=redis://localhost:6379
LOG_LEVEL=info
```

## Inline Templates with Heredoc

For shorter templates, you can use heredoc strings directly in your Terraform configuration instead of a separate file:

```hcl
resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.micro"

  user_data = <<-SCRIPT
    #!/bin/bash
    hostnamectl set-hostname ${var.hostname}
    echo "Environment: ${var.environment}" > /etc/app-env
    systemctl start nginx
  SCRIPT
}
```

For more on heredoc syntax, see our post on [heredoc strings in Terraform](https://oneuptime.com/blog/post/2026-02-23-how-to-use-heredoc-strings-in-terraform-for-multi-line-text/view).

## Practical Template Examples

### IAM Policy Document

```bash
# templates/s3_policy.json.tftpl
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": [
%{ for idx, bucket in buckets ~}
        "arn:aws:s3:::${bucket}/*"${ idx < length(buckets) - 1 ? "," : "" }
%{ endfor ~}
      ]
    }
%{ if allow_delete }
    ,{
      "Effect": "Allow",
      "Action": "s3:DeleteObject",
      "Resource": [
%{ for idx, bucket in buckets ~}
        "arn:aws:s3:::${bucket}/*"${ idx < length(buckets) - 1 ? "," : "" }
%{ endfor ~}
      ]
    }
%{ endif }
  ]
}
```

### Docker Compose Template

```yaml
# templates/docker-compose.yml.tftpl
version: '3.8'

services:
%{ for service in services ~}
  ${service.name}:
    image: ${service.image}:${service.tag}
    ports:
%{ for port in service.ports ~}
      - "${port}"
%{ endfor ~}
    environment:
      - NODE_ENV=${environment}
      - DATABASE_URL=${database_url}
%{ if service.replicas > 1 }
    deploy:
      replicas: ${service.replicas}
%{ endif ~}

%{ endfor ~}
```

## The format and formatlist Functions

For simpler string formatting, Terraform provides `format` and `formatlist`:

```hcl
locals {
  # format works like printf
  bucket_name = format("%s-%s-%s", var.project, var.environment, "data")
  # Result: "myapp-production-data"

  # Padding and alignment
  log_entry = format("%-20s %s", "instance_type:", var.instance_type)
  # Result: "instance_type:       t3.micro"

  # formatlist applies format to each element in a list
  server_names = formatlist("server-%02d.%s", range(1, 4), var.domain)
  # Result: ["server-01.example.com", "server-02.example.com", "server-03.example.com"]
}
```

## Summary

String interpolation with `${ }` handles most day-to-day string building in Terraform. For complex multi-line content, use `templatefile` with `.tftpl` files that support conditionals and loops via `%{ }` directives. Use whitespace control with `~` to keep template output clean. And remember: if the entire value is a single expression, skip the interpolation and use the expression directly.
