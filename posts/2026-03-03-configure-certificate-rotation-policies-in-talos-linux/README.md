# How to Configure Certificate Rotation Policies in Talos Linux

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, Certificate, TLS, Security, Kubernetes

Description: A detailed guide to configuring and managing certificate rotation policies in Talos Linux for secure Kubernetes cluster operations.

---

Certificates are the backbone of authentication and encryption in a Kubernetes cluster. Both the Talos API and the Kubernetes components rely on TLS certificates to secure communication. Talos Linux handles much of the certificate lifecycle automatically, but understanding how rotation works and how to configure it is essential for maintaining a secure cluster over time.

## The Certificate Landscape in Talos

A Talos-managed Kubernetes cluster uses several sets of certificates:

**Talos API certificates** - Used for mutual TLS authentication between talosctl clients and the Talos API server on each node. These are the certificates that authenticate administrative access to your nodes.

**Kubernetes PKI certificates** - The Kubernetes certificate authority and all the certificates it signs, including those for the API server, kubelet, controller manager, scheduler, and front proxy.

**etcd certificates** - Separate certificates for etcd peer communication and client access. These are critical because etcd stores all cluster state.

**Service account keys** - Used to sign and verify service account tokens.

```bash
# View the current certificates on a Talos node
talosctl get certificates --nodes <node-ip>

# Check certificate expiration dates
talosctl get certificates --nodes <node-ip> -o yaml | grep -A2 "notAfter"
```

## Automatic Certificate Rotation

Talos Linux automatically rotates Kubernetes component certificates before they expire. This is one of the significant advantages of using Talos compared to managing certificates manually with tools like kubeadm.

The automatic rotation process works like this:

1. Talos monitors the expiration dates of all managed certificates
2. When a certificate approaches its expiration (typically at 70-80% of its lifetime), Talos generates a new certificate
3. The new certificate is applied to the relevant component
4. The component is restarted to pick up the new certificate

```bash
# Check the status of certificate rotation
talosctl get machineconfig --nodes <node-ip> -o yaml | grep -A5 "certSANs"

# View the Kubernetes CA certificate details
talosctl read /system/secrets/kubernetes/certs/ca/tls.crt --nodes <node-ip> | \
  openssl x509 -text -noout | grep -E "Not Before|Not After|Subject:"
```

## Configuring Certificate Lifetimes

You can configure the lifetime of certificates generated by Talos through the machine configuration:

```yaml
# machine-config-cert-rotation.yaml
# Configure certificate rotation settings
machine:
  certSANs:
    - "10.0.0.10"
    - "cluster.example.com"
  # Additional machine-level certificate settings
cluster:
  # CA certificate lifetime (default is 10 years)
  # Component certificates are typically 1 year
  apiServer:
    certSANs:
      - "api.cluster.example.com"
      - "10.0.0.10"
```

Apply the configuration:

```bash
# Apply certificate configuration
talosctl apply-config --nodes <node-ip> \
  --config-patch @machine-config-cert-rotation.yaml
```

## Manual Certificate Rotation

In some cases, you may need to trigger certificate rotation manually. This is necessary when you suspect a certificate has been compromised or when you need to change the certificate SANs:

```bash
# Rotate the Talos API certificates
talosctl gen secrets --from-controlplane-config controlplane.yaml > secrets.yaml
talosctl apply-config --nodes <node-ip> --file updated-config.yaml

# Rotate Kubernetes certificates by triggering a config update
# This causes Talos to regenerate the Kubernetes PKI
talosctl apply-config --nodes <node-ip> --file controlplane.yaml --mode no-reboot
```

For a full cluster certificate rotation:

```bash
#!/bin/bash
# rotate-cluster-certs.sh
# Rotate certificates across all cluster nodes

CONTROL_PLANE_NODES="10.0.0.10 10.0.0.11 10.0.0.12"
WORKER_NODES="10.0.0.20 10.0.0.21 10.0.0.22"

echo "Starting certificate rotation for control plane nodes..."
for node in ${CONTROL_PLANE_NODES}; do
  echo "Rotating certificates on ${node}..."
  talosctl apply-config --nodes ${node} --file controlplane.yaml

  # Wait for the node to be ready before proceeding
  echo "Waiting for node ${node} to be ready..."
  sleep 30
  talosctl health --nodes ${node} --wait-timeout 5m
done

echo "Starting certificate rotation for worker nodes..."
for node in ${WORKER_NODES}; do
  echo "Rotating certificates on ${node}..."
  talosctl apply-config --nodes ${node} --file worker.yaml

  sleep 30
  talosctl health --nodes ${node} --wait-timeout 5m
done

echo "Certificate rotation complete."
```

## Monitoring Certificate Expiration

Proactive monitoring of certificate expiration dates prevents unexpected outages. Set up monitoring to alert well before certificates expire:

```yaml
# cert-monitor-deployment.yaml
# Deploy a certificate expiration monitor
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cert-monitor
  namespace: monitoring
spec:
  replicas: 1
  selector:
    matchLabels:
      app: cert-monitor
  template:
    metadata:
      labels:
        app: cert-monitor
    spec:
      containers:
        - name: cert-monitor
          image: enix/x509-certificate-exporter:latest
          args:
            - --watch-kube-secrets
            - --secret-type=kubernetes.io/tls
          ports:
            - containerPort: 9793
              name: metrics
```

Create Prometheus alerting rules for certificate expiration:

```yaml
# cert-expiration-alerts.yaml
# Alert on certificates approaching expiration
groups:
  - name: certificate-expiration
    rules:
      - alert: CertificateExpiringIn30Days
        expr: |
          x509_cert_not_after - time() < 86400 * 30
        for: 1h
        labels:
          severity: warning
        annotations:
          summary: "Certificate {{ $labels.subject_CN }} expires in less than 30 days"

      - alert: CertificateExpiringIn7Days
        expr: |
          x509_cert_not_after - time() < 86400 * 7
        for: 1h
        labels:
          severity: critical
        annotations:
          summary: "Certificate {{ $labels.subject_CN }} expires in less than 7 days"

      - alert: CertificateExpired
        expr: |
          x509_cert_not_after - time() < 0
        labels:
          severity: critical
        annotations:
          summary: "Certificate {{ $labels.subject_CN }} has expired"
```

## Handling the Talos CA Certificate

The Talos CA certificate has a long lifetime (typically 10 years) because rotating it requires regenerating all downstream certificates. However, you should plan for CA rotation:

```bash
# Check the Talos CA certificate expiration
talosctl get certificates --nodes <node-ip> -o yaml | \
  grep -B5 -A5 "talos"

# If you need to rotate the CA, you will need to regenerate
# the entire cluster configuration
talosctl gen secrets -o new-secrets.yaml
talosctl gen config my-cluster https://endpoint:6443 \
  --with-secrets new-secrets.yaml
```

CA rotation is a major operation that requires updating all nodes. Plan for this well in advance of the CA expiration.

## Client Certificate Management

The talosctl client certificates also need management. Each admin who accesses the cluster has their own certificate:

```bash
# View the current talosctl client configuration
talosctl config info

# Generate a new client certificate for a team member
# This derives a new certificate from the existing CA
talosctl gen config my-cluster https://endpoint:6443 \
  --roles os:reader \
  --output-types talosconfig \
  -o team-member-talosconfig.yaml

# The new talosconfig can be distributed to the team member
# It contains their unique client certificate
```

For organizations with many administrators, consider integrating with an external certificate authority or using short-lived certificates:

```bash
# You can configure talosctl to use an external identity provider
# by setting up an OIDC-based authentication flow

# Example: generating a short-lived certificate (1 hour)
# This requires custom tooling around the Talos CA
```

## etcd Certificate Rotation

etcd certificates are especially important because etcd stores all Kubernetes secrets and state. Talos manages etcd certificates separately from other Kubernetes certificates:

```bash
# Check etcd certificate status
talosctl etcd status --nodes <node-ip>

# View etcd member details including certificate info
talosctl etcd members --nodes <node-ip>
```

If you need to force etcd certificate rotation:

```bash
# This is typically handled automatically, but you can force it
# by applying the configuration with updated settings
talosctl apply-config --nodes <control-plane-ip> \
  --file controlplane.yaml --mode staged

# Then reboot the node to apply
talosctl reboot --nodes <control-plane-ip>
```

## Best Practices for Certificate Management

Based on production experience, here are recommendations for managing certificates in Talos:

1. **Document your certificate hierarchy** - Know which CA signs which certificates and what their lifetimes are.

2. **Set up monitoring from day one** - Do not wait until you have an expiration-related outage.

3. **Test rotation procedures** - Regularly practice certificate rotation in a staging environment.

4. **Keep backups of your secrets bundle** - The secrets.yaml file generated during cluster creation is critical for recovery.

```bash
# Back up the cluster secrets
talosctl gen secrets -o secrets-backup.yaml
# Store this file in a secure, encrypted location
```

5. **Use cert-manager for workload certificates** - For application-level TLS, deploy cert-manager on your cluster:

```bash
# Install cert-manager
helm repo add jetstack https://charts.jetstack.io
helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --create-namespace \
  --set installCRDs=true
```

6. **Audit certificate usage** - Regularly review which certificates exist and who has access to them.

## Conclusion

Certificate rotation in Talos Linux is largely automated, which removes one of the biggest operational pain points of running Kubernetes. However, automation does not mean you can ignore certificates entirely. You need to understand the certificate hierarchy, monitor for expirations, plan for CA rotation, and manage client certificates for your team. By setting up proper monitoring and following the practices outlined in this guide, you can keep your cluster's certificates healthy and avoid the downtime that comes from expired credentials.
