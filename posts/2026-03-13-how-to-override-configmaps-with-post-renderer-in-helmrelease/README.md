# How to Override ConfigMaps with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderers, ConfigMap, Kustomize

Description: Learn how to use Flux post-renderers to override and modify ConfigMap contents generated by Helm charts.

---

## Introduction

ConfigMaps are a fundamental way to inject configuration data into containerized applications in Kubernetes. Helm charts typically generate ConfigMaps from templates, and while chart values often provide some customization options, they may not expose every configuration parameter. You might need to add additional configuration keys, override default values, or merge environment-specific settings that the chart does not anticipate.

Flux post-renderers allow you to modify ConfigMaps generated by Helm charts using Kustomize patches. This gives you the ability to adjust any configuration data without forking the chart or waiting for upstream changes.

This guide shows you how to use post-renderers to override, extend, and replace ConfigMap data in HelmRelease resources.

## Prerequisites

Before following this guide, you need:

- A Kubernetes cluster with Flux CD installed
- A HelmRelease that generates ConfigMaps
- kubectl access to the cluster
- Basic knowledge of Kustomize patch syntax

## Overriding ConfigMap Data Fields

To override specific data fields in a ConfigMap generated by a Helm chart, use a strategic merge patch:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ConfigMap
              name: my-app-config
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: my-app-config
              data:
                LOG_LEVEL: "debug"
                MAX_CONNECTIONS: "500"
                CACHE_TTL: "3600"
```

This patch adds or overrides the specified keys in the ConfigMap data. Existing keys not mentioned in the patch remain unchanged.

## Overriding Configuration Files

Many ConfigMaps contain full configuration files as data values. You can override an entire file:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: nginx
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: nginx
      version: "15.0.0"
      sourceRef:
        kind: HelmRepository
        name: bitnami
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ConfigMap
              name: nginx-server-block
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: nginx-server-block
              data:
                server-block.conf: |
                  server {
                      listen 8080;
                      server_name _;

                      location / {
                          proxy_pass http://backend-service:3000;
                          proxy_set_header Host $host;
                          proxy_set_header X-Real-IP $remote_addr;
                      }

                      location /health {
                          return 200 'OK';
                          add_header Content-Type text/plain;
                      }
                  }
```

This replaces the entire server-block configuration file in the ConfigMap with a custom version.

## Adding New Configuration Keys

To add new keys to an existing ConfigMap without disturbing the chart-generated values:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ConfigMap
              name: my-app-config
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: my-app-config
              data:
                FEATURE_FLAG_NEW_UI: "true"
                EXTERNAL_API_URL: "https://api.partner.com/v2"
                CUSTOM_HEADER_NAME: "X-Request-Trace"
```

The strategic merge patch adds new keys while preserving all existing keys generated by the chart.

## Modifying Multiple ConfigMaps

If a chart generates multiple ConfigMaps, target each one separately:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ConfigMap
              name: my-app-config
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: my-app-config
              data:
                DATABASE_POOL_SIZE: "25"
          - target:
              kind: ConfigMap
              name: my-app-nginx
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: my-app-nginx
              data:
                worker_connections: "4096"
          - target:
              kind: ConfigMap
              name: my-app-scripts
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: my-app-scripts
              data:
                startup.sh: |
                  #!/bin/bash
                  echo "Running custom startup script"
                  /app/warmup-cache.sh
                  exec /app/server
```

## Using JSON 6902 Patches for Precise Control

For more precise control, use JSON 6902 patches to add, replace, or remove specific keys:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ConfigMap
              name: my-app-config
            patch: |
              - op: replace
                path: /data/LOG_LEVEL
                value: "warn"
              - op: add
                path: /data/NEW_FEATURE
                value: "enabled"
              - op: remove
                path: /data/DEPRECATED_SETTING
```

JSON 6902 patches give you explicit add, replace, and remove operations, which is useful when you need to remove keys that the chart generates by default.

## Environment-Specific Configuration

A common pattern is to use different ConfigMap overrides per environment:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: ConfigMap
              name: my-app-config
            patch: |
              apiVersion: v1
              kind: ConfigMap
              metadata:
                name: my-app-config
              data:
                LOG_LEVEL: "warn"
                REPLICAS: "5"
                CACHE_SIZE: "10Gi"
                RATE_LIMIT: "1000"
                ENABLE_DEBUG: "false"
```

In staging, you would use different values in the same patch structure, allowing environment-specific configuration without chart modifications.

## Verifying ConfigMap Overrides

After deploying, verify the ConfigMap contents:

```bash
kubectl get configmap my-app-config -n production -o yaml
```

Check specific keys:

```bash
kubectl get configmap my-app-config -n production -o jsonpath='{.data.LOG_LEVEL}'
```

## Conclusion

Flux post-renderers provide a powerful way to override ConfigMap data generated by Helm charts. Whether you need to adjust individual configuration values, replace entire configuration files, or add environment-specific settings, Kustomize patches handle all these scenarios cleanly. Use strategic merge patches for adding and overriding keys, and JSON 6902 patches when you need to remove keys. This approach keeps your configuration management separate from chart maintenance, making it easy to customize any Helm chart for your specific requirements.
