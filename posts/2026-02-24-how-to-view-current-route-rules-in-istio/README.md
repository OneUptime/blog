# How to View Current Route Rules in Istio

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Routing, Debugging, Kubernetes, Envoy, Service Mesh

Description: How to inspect and view the current routing rules that Istio has applied to your services using istioctl and Envoy admin commands.

---

When traffic is not going where you expect in Istio, the first step is to see what routes are actually configured. Istio translates your VirtualService and DestinationRule resources into Envoy proxy configuration, and sometimes what you think is configured and what is actually in the proxy are two different things.

Here are all the ways to inspect your current routing rules.

## Viewing VirtualService and DestinationRule Resources

Start with the Kubernetes resources themselves:

```bash
# List all VirtualServices
kubectl get virtualservices --all-namespaces

# View a specific VirtualService
kubectl get virtualservice my-service -n production -o yaml

# List all DestinationRules
kubectl get destinationrules --all-namespaces

# View a specific DestinationRule
kubectl get destinationrule my-service -n production -o yaml
```

For a quick overview of all routing configuration:

```bash
# Show all VirtualServices with their hosts and gateways
kubectl get virtualservices --all-namespaces -o custom-columns=\
'NAMESPACE:.metadata.namespace,NAME:.metadata.name,HOSTS:.spec.hosts[*],GATEWAYS:.spec.gateways[*]'
```

## Using istioctl proxy-config

The `istioctl proxy-config` command shows you the actual Envoy configuration that is running inside a specific pod's sidecar. This is what matters because it is what the proxy actually uses for routing decisions.

### View Routes

```bash
# View all routes for a specific pod
istioctl proxy-config routes my-pod -n production

# View routes for a specific port
istioctl proxy-config routes my-pod -n production --port 8080

# Get detailed route information in JSON
istioctl proxy-config routes my-pod -n production -o json
```

The output shows route names, domains they match, and the virtual hosts with their route rules.

### View Clusters (Upstream Endpoints)

```bash
# View all upstream clusters
istioctl proxy-config clusters my-pod -n production

# Filter by service name
istioctl proxy-config clusters my-pod -n production | grep my-service

# Get detailed cluster info
istioctl proxy-config clusters my-pod -n production --fqdn my-service.production.svc.cluster.local -o json
```

### View Listeners

```bash
# View all listeners
istioctl proxy-config listeners my-pod -n production

# View listeners on a specific port
istioctl proxy-config listeners my-pod -n production --port 8080

# Detailed listener configuration
istioctl proxy-config listeners my-pod -n production -o json
```

### View Endpoints

```bash
# View the actual endpoints (pod IPs) for a service
istioctl proxy-config endpoints my-pod -n production

# Filter by cluster name
istioctl proxy-config endpoints my-pod -n production --cluster "outbound|8080||my-service.production.svc.cluster.local"
```

## Reading Route Configuration Output

When you run `istioctl proxy-config routes`, the output looks something like this:

```
NAME          DOMAINS                           MATCH     VIRTUAL SERVICE
8080          my-service.production             /*        my-service.production
8080          other-service.production          /*        other-service.production
              *                                 /healthz
```

The key columns are:
- **NAME**: The port number
- **DOMAINS**: What host headers this route matches
- **MATCH**: The path prefix or exact match
- **VIRTUAL SERVICE**: Which VirtualService resource generated this route

If the VIRTUAL SERVICE column is empty, the route was auto-generated by Istio from the Kubernetes Service definition.

For more detail, use the JSON output:

```bash
istioctl proxy-config routes my-pod -n production -o json | \
  jq '.[].virtualHosts[] | select(.name | contains("my-service")) | .routes'
```

This shows the full route configuration including:
- Match conditions (path, headers, query parameters)
- Route actions (destination cluster, weight)
- Retry policies
- Timeout settings
- Header manipulation

## Using the Envoy Admin Interface

Every sidecar runs an Envoy admin interface on port 15000. You can access it directly for the most detailed view:

```bash
# Port-forward to the Envoy admin interface
kubectl port-forward my-pod -n production 15000:15000

# Then in another terminal:
# View all routes
curl http://localhost:15000/config_dump?resource=dynamic_route_configs

# View all clusters
curl http://localhost:15000/config_dump?resource=dynamic_active_clusters

# View all listeners
curl http://localhost:15000/config_dump?resource=dynamic_listeners
```

Or do it without port-forwarding:

```bash
# Dump full Envoy configuration
kubectl exec my-pod -n production -c istio-proxy -- \
  pilot-agent request GET config_dump > envoy-config.json

# Search for specific routes
cat envoy-config.json | jq '.configs[] | select(."@type" | contains("RoutesConfigDump"))'
```

## Checking Route Sync Status

Sometimes the routes in the proxy do not match what you defined in your VirtualService. Check the sync status:

```bash
# Check proxy sync status
istioctl proxy-status

# Output shows:
# NAME           CDS    LDS    EDS    RDS    ECDS   ISTIOD
# my-pod         SYNCED SYNCED SYNCED SYNCED        istiod-abc
```

If any column shows `STALE` instead of `SYNCED`, the proxy has not received the latest configuration from istiod. Check istiod logs for errors:

```bash
kubectl logs -n istio-system -l app=istiod | grep -i "error\|push\|reject"
```

## Validating Route Configuration

Use `istioctl analyze` to check for common routing mistakes:

```bash
# Analyze a specific namespace
istioctl analyze -n production

# Analyze specific files before applying
istioctl analyze -f my-virtualservice.yaml
```

Common issues it catches:
- VirtualService references a Gateway that does not exist
- VirtualService references a host that has no matching Service
- DestinationRule references a subset that does not exist
- Conflicting VirtualServices for the same host

## Tracing a Request Through Routes

To understand how a specific request would be routed, use `istioctl proxy-config` to trace the path:

```bash
# Step 1: Find the listener that matches the request
istioctl proxy-config listeners my-pod -n production --port 8080

# Step 2: Find the route configuration for that listener
istioctl proxy-config routes my-pod -n production --name "8080"

# Step 3: Find the cluster that the route points to
istioctl proxy-config clusters my-pod -n production \
  --fqdn "my-service.production.svc.cluster.local"

# Step 4: Find the endpoints in that cluster
istioctl proxy-config endpoints my-pod -n production \
  --cluster "outbound|8080||my-service.production.svc.cluster.local"
```

## Viewing Traffic Splits

If you have traffic splitting configured (canary deployments, A/B testing), view the weights:

```bash
# View the VirtualService with traffic weights
kubectl get virtualservice my-service -n production -o yaml
```

```yaml
# Example output showing traffic split
spec:
  hosts:
  - my-service
  http:
  - route:
    - destination:
        host: my-service
        subset: v1
      weight: 90
    - destination:
        host: my-service
        subset: v2
      weight: 10
```

Verify the split is configured in the proxy:

```bash
istioctl proxy-config routes my-pod -n production -o json | \
  jq '.[].virtualHosts[] | select(.name | contains("my-service")) | .routes[].route.weightedClusters'
```

## Quick Reference Commands

Here is a cheat sheet for the most commonly needed commands:

```bash
# All VirtualServices
kubectl get vs -A

# All DestinationRules
kubectl get dr -A

# Routes for a pod
istioctl pc routes <pod> -n <namespace>

# Clusters for a pod
istioctl pc clusters <pod> -n <namespace>

# Endpoints for a pod
istioctl pc endpoints <pod> -n <namespace>

# Full proxy status
istioctl ps

# Analyze for issues
istioctl analyze -A

# Compare two pods' route configs
diff <(istioctl pc routes pod-a -o json) <(istioctl pc routes pod-b -o json)
```

Understanding how to inspect routing rules is one of the most important debugging skills for operating Istio. When something is not routing correctly, these commands get you to the answer faster than reading logs or guessing at configuration.
