# How to Deploy Consul Template for Dynamic Configuration on RHEL

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: RHEL, Consul, Consul Template, Configuration Management, Service Discovery, Linux

Description: Learn how to deploy and configure Consul Template on RHEL to dynamically generate configuration files from Consul service catalog and KV store data, with automatic service reload on changes.

---

Consul Template watches Consul for changes in services, key-value data, and certificate information, then renders templates and optionally runs commands when changes are detected. This lets you keep configuration files like Nginx configs, HAProxy backends, and application settings in sync with your Consul cluster without manual intervention. This guide covers deploying Consul Template on RHEL.

## How Consul Template Works

Consul Template runs as a daemon that:

1. Connects to a Consul cluster
2. Watches for changes in specified services, KV paths, or Vault secrets
3. Renders Go templates using the current data
4. Writes the rendered output to files
5. Optionally executes a command (like reloading a service)

## Prerequisites

- RHEL with root or sudo access
- A running Consul cluster (or single agent)
- Services registered in Consul that you want to template

## Installing Consul Template

Download the binary:

```bash
# Download Consul Template
CT_VERSION="0.37.4"
curl -LO https://releases.hashicorp.com/consul-template/${CT_VERSION}/consul-template_${CT_VERSION}_linux_amd64.zip
sudo dnf install -y unzip
unzip consul-template_${CT_VERSION}_linux_amd64.zip
sudo mv consul-template /usr/local/bin/
```

Verify installation:

```bash
# Check the version
consul-template --version
```

## Basic Configuration

Create a configuration file for Consul Template:

```bash
# Create the configuration directory
sudo mkdir -p /etc/consul-template.d
```

```bash
# Create the main configuration
sudo tee /etc/consul-template.d/config.hcl > /dev/null << 'EOF'
consul {
  address = "127.0.0.1:8500"
}

log_level = "info"

template {
  source      = "/etc/consul-template.d/templates/nginx-upstreams.tpl"
  destination = "/etc/nginx/conf.d/upstreams.conf"
  command     = "systemctl reload nginx"
  perms       = 0644
}
EOF
```

## Creating Templates

### Dynamic Nginx Upstream Configuration

Create a template that generates Nginx upstream blocks from Consul services:

```bash
# Create the templates directory
sudo mkdir -p /etc/consul-template.d/templates
```

```bash
# Create the Nginx upstream template
sudo tee /etc/consul-template.d/templates/nginx-upstreams.tpl > /dev/null << 'TEMPLATE'
{{range services}}{{if .Tags | contains "web"}}
upstream {{.Name}} {
  {{range service .Name}}
  server {{.Address}}:{{.Port}} weight=1;{{end}}
}
{{end}}{{end}}

server {
    listen 80;
    {{range services}}{{if .Tags | contains "web"}}
    location /{{.Name}}/ {
        proxy_pass http://{{.Name}}/;
    }
    {{end}}{{end}}
}
TEMPLATE
```

### Dynamic HAProxy Backend Configuration

```bash
# Create an HAProxy backend template
sudo tee /etc/consul-template.d/templates/haproxy-backends.tpl > /dev/null << 'TEMPLATE'
global
    daemon
    maxconn 4096

defaults
    mode http
    timeout connect 5s
    timeout client 30s
    timeout server 30s

frontend http_front
    bind *:80
    default_backend http_back

backend http_back
    balance roundrobin
    {{range service "webapp"}}
    server {{.Node}} {{.Address}}:{{.Port}} check inter 5s fall 3 rise 2
    {{end}}
TEMPLATE
```

### Application Configuration from KV Store

```bash
# Create a template that pulls from the KV store
sudo tee /etc/consul-template.d/templates/app-config.tpl > /dev/null << 'TEMPLATE'
# Application Configuration
# Auto-generated by Consul Template - do not edit manually

database_host={{key "config/database/host"}}
database_port={{keyOrDefault "config/database/port" "5432"}}
database_name={{key "config/database/name"}}
cache_ttl={{keyOrDefault "config/cache/ttl" "300"}}
log_level={{keyOrDefault "config/log_level" "info"}}
{{range ls "config/features"}}
feature_{{.Key}}={{.Value}}
{{end}}
TEMPLATE
```

## Running Consul Template

Test the template rendering first:

```bash
# Dry run to see what would be generated
consul-template -config /etc/consul-template.d/config.hcl -once -dry
```

Run once to generate the initial configuration:

```bash
# Generate the config once
consul-template -config /etc/consul-template.d/config.hcl -once
```

## Creating a Systemd Service

Set up Consul Template as a managed service:

```bash
# Create the systemd unit file
sudo tee /etc/systemd/system/consul-template.service > /dev/null << 'EOF'
[Unit]
Description=Consul Template
After=network-online.target consul.service
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/local/bin/consul-template -config /etc/consul-template.d/config.hcl
ExecReload=/bin/kill -HUP $MAINPID
KillSignal=SIGINT
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF
```

```bash
# Enable and start the service
sudo systemctl daemon-reload
sudo systemctl enable --now consul-template
```

## Multiple Template Blocks

You can render multiple templates in a single configuration:

```bash
# Configuration with multiple templates
sudo tee /etc/consul-template.d/config.hcl > /dev/null << 'EOF'
consul {
  address = "127.0.0.1:8500"
}

log_level = "info"

template {
  source      = "/etc/consul-template.d/templates/nginx-upstreams.tpl"
  destination = "/etc/nginx/conf.d/upstreams.conf"
  command     = "systemctl reload nginx"
  perms       = 0644
}

template {
  source      = "/etc/consul-template.d/templates/app-config.tpl"
  destination = "/opt/myapp/config.env"
  command     = "systemctl restart myapp"
  perms       = 0640
  left_delimiter  = "{{"
  right_delimiter = "}}"
}

template {
  source      = "/etc/consul-template.d/templates/haproxy-backends.tpl"
  destination = "/etc/haproxy/haproxy.cfg"
  command     = "systemctl reload haproxy"
  perms       = 0644
}
EOF
```

## Template Functions

Consul Template provides many useful functions:

```bash
# Get a service and filter by tag
{{range service "webapp|production"}}
server {{.Address}}:{{.Port}}
{{end}}

# Use a default value for missing keys
database_url={{keyOrDefault "config/db_url" "postgres://localhost/mydb"}}

# Check if a key exists
{{if keyExists "config/maintenance_mode"}}
maintenance=true
{{else}}
maintenance=false
{{end}}

# Loop over all key-value pairs under a prefix
{{range ls "config/env"}}
export {{.Key}}="{{.Value}}"
{{end}}

# Use the tree function for recursive listing
{{range tree "config"}}
{{.Key}}={{.Value}}
{{end}}
```

## Handling Template Errors

Configure error handling for cases where Consul data is not yet available:

```bash
# Add error handling to the configuration
sudo tee /etc/consul-template.d/config.hcl > /dev/null << 'EOF'
consul {
  address = "127.0.0.1:8500"
  retry {
    enabled  = true
    attempts = 12
    backoff  = "250ms"
    max_backoff = "1m"
  }
}

wait {
  min = "5s"
  max = "10s"
}

template {
  source      = "/etc/consul-template.d/templates/nginx-upstreams.tpl"
  destination = "/etc/nginx/conf.d/upstreams.conf"
  command     = "systemctl reload nginx"
  command_timeout = "30s"
  error_on_missing_key = false
  wait {
    min = "2s"
    max = "10s"
  }
}
EOF
```

The `wait` block adds a small delay to batch multiple changes together rather than regenerating the template on every individual change.

## Conclusion

Consul Template on RHEL bridges the gap between dynamic service discovery and static configuration files. By watching Consul for changes in services and KV data, it automatically regenerates configuration files and reloads services, keeping your infrastructure in sync with your service catalog without any manual steps.
