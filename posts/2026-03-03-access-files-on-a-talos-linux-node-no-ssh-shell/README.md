# How to Access Files on a Talos Linux Node (No SSH/Shell)

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, File Access, talosctl, Security, Kubernetes, System Administration

Description: Learn how to access and manage files on Talos Linux nodes without SSH or shell access using talosctl and other techniques.

---

One of the first things people notice about Talos Linux is that there is no SSH access. There is no shell. You cannot just log in and browse the filesystem like you would on Ubuntu or CentOS. This is by design - Talos is built to be an immutable, API-driven operating system for running Kubernetes, and removing shell access is a core part of its security model.

But sometimes you genuinely need to see what is on the filesystem. Maybe you are debugging a mount issue, checking if a configuration file landed correctly, or investigating why a volume is not behaving as expected. Here is how to do it.

## Using talosctl read

The `talosctl read` command lets you read the contents of a file on a Talos node:

```bash
# Read a file from a Talos node
talosctl read --nodes 192.168.1.10 /etc/os-release
```

This prints the contents of the file to your terminal. It works for text files, configuration files, and other readable files on the node.

```bash
# Read the machine configuration currently applied
talosctl read --nodes 192.168.1.10 /system/state/config.yaml
```

You can redirect the output to save the file locally:

```bash
# Save a remote file to your local machine
talosctl read --nodes 192.168.1.10 /etc/os-release > /tmp/os-release-node1.txt
```

## Using talosctl list

To see what files exist in a directory, use `talosctl list`:

```bash
# List files in a directory
talosctl list --nodes 192.168.1.10 /etc/

# List files with details (size, permissions, timestamps)
talosctl list --nodes 192.168.1.10 /etc/ -l

# List files recursively
talosctl list --nodes 192.168.1.10 /etc/ -r
```

The output format is similar to `ls -la` on a traditional Linux system:

```
NODE           MODE        UID   GID   SIZE    LASTMOD              NAME
192.168.1.10   -rw-r--r--  0     0     234     2026-03-03 10:00:00  os-release
192.168.1.10   -rw-r--r--  0     0     1234    2026-03-03 10:00:00  resolv.conf
192.168.1.10   drwxr-xr-x  0     0     4096    2026-03-03 10:00:00  kubernetes
```

## Using talosctl copy

If you need to copy an entire file or directory from the node to your local machine, use `talosctl copy`:

```bash
# Copy a file from the node to your local machine
talosctl copy --nodes 192.168.1.10 /var/log/containers/ /tmp/container-logs/
```

This is particularly useful for gathering log files or configuration snapshots for offline analysis.

## Exploring Key Directories

Talos Linux has a different filesystem layout than traditional distributions. Here are the important directories:

### /system/state

This contains the machine state, including the applied configuration:

```bash
# List the state directory
talosctl list --nodes 192.168.1.10 /system/state/
```

### /etc

The /etc directory contains runtime configuration files, but many of them are generated by Talos from the machine configuration rather than being directly editable:

```bash
# Check the hosts file
talosctl read --nodes 192.168.1.10 /etc/hosts

# Check DNS resolver configuration
talosctl read --nodes 192.168.1.10 /etc/resolv.conf
```

### /var

The /var directory contains variable data, including container logs and kubelet data:

```bash
# List kubelet directory
talosctl list --nodes 192.168.1.10 /var/lib/kubelet/

# Check container log files
talosctl list --nodes 192.168.1.10 /var/log/ -r
```

### /proc and /sys

These virtual filesystems provide kernel and hardware information:

```bash
# Check CPU information
talosctl read --nodes 192.168.1.10 /proc/cpuinfo

# Check memory information
talosctl read --nodes 192.168.1.10 /proc/meminfo

# Check disk partitions
talosctl read --nodes 192.168.1.10 /proc/partitions
```

## Checking Mounted Volumes

If you need to verify that a Persistent Volume (PV) is correctly mounted or check files on a mounted volume:

```bash
# List mount points
talosctl read --nodes 192.168.1.10 /proc/mounts

# List files on a specific mount point
talosctl list --nodes 192.168.1.10 /var/lib/kubelet/pods/<pod-uid>/volumes/
```

Finding the right path requires knowing the pod UID. You can get this from kubectl:

```bash
# Get the pod UID
kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.metadata.uid}'

# Then use that UID to navigate to the volume
talosctl list --nodes 192.168.1.10 /var/lib/kubelet/pods/<uid>/volumes/kubernetes.io~csi/
```

## Using Debug Containers

For more interactive file exploration, you can run a debug container on the node:

```bash
# Run a debug container with host filesystem access
kubectl debug node/talos-node-1 -it --image=busybox -- sh
```

Inside the debug container, the host filesystem is available at `/host`:

```bash
# Inside the debug container
ls /host/etc/
cat /host/proc/meminfo
```

This approach gives you a shell environment for exploring files, though it runs in a container context rather than directly on the host.

## Using talosctl for Binary Files

For binary files, redirecting to a local file is the best approach:

```bash
# Copy a binary file from the node
talosctl read --nodes 192.168.1.10 /path/to/binary > /tmp/binary-copy

# Verify the file
file /tmp/binary-copy
md5sum /tmp/binary-copy
```

## Checking Machine Configuration

The most common file people need to access is the current machine configuration:

```bash
# Get the full machine configuration as YAML
talosctl get machineconfig --nodes 192.168.1.10 -o yaml

# This is more reliable than reading the config file directly
# because it shows the configuration as Talos understands it
```

This is better than reading the raw config file because it shows any defaults or transformations that Talos applied.

## Monitoring File Changes

If you need to monitor a file for changes, you can use a watch loop:

```bash
# Poll a file every 5 seconds (run locally on your workstation)
while true; do
  echo "--- $(date) ---"
  talosctl read --nodes 192.168.1.10 /etc/resolv.conf
  sleep 5
done
```

This is a crude approach, but it works for debugging transient issues.

## Writing Files to Talos Nodes

Talos Linux intentionally limits the ability to write arbitrary files to the filesystem. The operating system is immutable, and changes should be made through the machine configuration.

However, there are a few scenarios where you can place files:

### Through Machine Configuration

Extra files can be included in the machine configuration:

```yaml
machine:
  files:
    - content: |
        custom content here
      permissions: 0644
      path: /etc/custom-file
      op: create
```

### Through Kubernetes Volumes

Applications running in pods can write to their own volumes. If you need to place files for an application, use ConfigMaps, Secrets, or PersistentVolumes:

```yaml
# Using a ConfigMap to place files
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  config.yaml: |
    key: value
    setting: enabled
```

## Security Considerations

The inability to freely access and modify files on Talos Linux is a feature, not a limitation. Traditional Linux servers are often compromised through shell access, where attackers can modify system files, install backdoors, or exfiltrate data.

By requiring all file access to go through the authenticated talosctl API, Talos ensures that every file operation is authenticated, authorized, and auditable. This is a significant security improvement over SSH-based management.

## Conclusion

While Talos Linux removes the traditional shell-based approach to file access, it provides robust alternatives through talosctl commands like `read`, `list`, and `copy`. These tools, combined with the Kubernetes debug container approach, give you full visibility into the node filesystem when you need it. The key shift is that file access becomes an explicit, authenticated operation rather than something anyone with SSH credentials can do freely.
