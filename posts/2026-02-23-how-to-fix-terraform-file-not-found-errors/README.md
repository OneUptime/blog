# How to Fix Terraform File Not Found Errors

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Terraform, Troubleshooting, Infrastructure as Code, File

Description: Resolve Terraform file not found errors caused by incorrect paths, missing files, path references in modules, and working directory confusion.

---

Terraform reads files from disk during planning for templates, configuration files, certificates, and other data. When it cannot find a file, the error halts your plan. The frustrating part is that the file often exists on disk but Terraform is looking in the wrong place. This guide covers why files go missing and how to fix the path.

## The Error

```
Error: Error in function call

  on main.tf line 5, in resource "aws_instance" "web":
   5:   user_data = file("scripts/init.sh")

Call to function "file" failed: no file exists at "scripts/init.sh";
this function works only with files that are distributed as part of
the configuration.
```

Or with `templatefile()`:

```
Error: Error in function call

  on main.tf line 3, in resource "aws_instance" "web":
   3:   user_data = templatefile("templates/init.sh", {})

Call to function "templatefile" failed: no file exists at
"templates/init.sh".
```

## Understanding Path References

Terraform has three path references:

- `path.module` - The filesystem path of the module where the expression is defined
- `path.root` - The filesystem path of the root module
- `path.cwd` - The filesystem path of the current working directory (where you ran `terraform`)

These can all be different:

```
/project/               <- path.cwd (if you run terraform from here)
/project/               <- path.root (root module)
/project/modules/vpc/   <- path.module (inside a child module)
```

## Fix 1: Use path.module for Module-Relative Files

The most common fix. When a file is next to the `.tf` file that references it, use `path.module`:

```hcl
# Wrong - relative to working directory, not the module
resource "aws_instance" "web" {
  user_data = file("scripts/init.sh")
}

# Right - relative to the current module
resource "aws_instance" "web" {
  user_data = file("${path.module}/scripts/init.sh")
}
```

This is especially important in child modules. Without `path.module`, Terraform looks for the file relative to the root module or working directory.

## Fix 2: Use path.root for Root Module Files

If the file is in the root module and you need to reference it from a child module:

```hcl
# Inside a child module, referencing a root-level file
resource "aws_instance" "web" {
  user_data = file("${path.root}/shared/scripts/init.sh")
}
```

But this creates a tight coupling between the module and the root layout. It is usually better to pass file contents as a variable:

```hcl
# Root module
module "web" {
  source    = "./modules/web"
  user_data = file("${path.module}/shared/scripts/init.sh")
}

# Child module
variable "user_data" {
  type = string
}

resource "aws_instance" "web" {
  user_data = var.user_data
}
```

## Fix 3: Files Generated During Apply

If the file is generated by another process during `terraform apply`, the `file()` function will not find it during `terraform plan` because it runs at plan time:

```hcl
# This fails on the first run because the file does not exist yet
resource "null_resource" "generate_cert" {
  provisioner "local-exec" {
    command = "openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 365 -nodes -subj '/CN=example.com'"
  }
}

resource "aws_iam_server_certificate" "cert" {
  certificate_body = file("cert.pem")  # File does not exist during plan
  private_key      = file("key.pem")
}
```

**Fix:** Use Terraform resources instead of external file generation:

```hcl
resource "tls_private_key" "cert" {
  algorithm = "RSA"
  rsa_bits  = 2048
}

resource "tls_self_signed_cert" "cert" {
  private_key_pem = tls_private_key.cert.private_key_pem

  subject {
    common_name = "example.com"
  }

  validity_period_hours = 8760

  allowed_uses = [
    "key_encipherment",
    "digital_signature",
    "server_auth",
  ]
}

resource "aws_iam_server_certificate" "cert" {
  certificate_body = tls_self_signed_cert.cert.cert_pem
  private_key      = tls_private_key.cert.private_key_pem
}
```

## Fix 4: Working Directory Mismatch

If you run Terraform from a different directory than expected, relative paths break:

```bash
# From /project - this works
cd /project
terraform plan

# From /project/environments/prod - relative paths are different
cd /project/environments/prod
terraform plan
# Files relative to /project are now at ../../
```

**Fix:** Always use `path.module` or `path.root` instead of bare relative paths. Or run Terraform from a consistent directory.

## Fix 5: fileset() and fileexists() Helpers

Before trying to read a file, check if it exists:

```hcl
locals {
  # Check if a file exists before reading
  has_custom_config = fileexists("${path.module}/custom-config.json")
  config = local.has_custom_config ? file("${path.module}/custom-config.json") : "{}"
}
```

Use `fileset()` to find files matching a pattern:

```hcl
locals {
  # Find all .json files in the configs directory
  config_files = fileset("${path.module}/configs", "*.json")
}

# Process each config file
resource "aws_ssm_parameter" "configs" {
  for_each = local.config_files

  name  = "/app/config/${trimsuffix(each.value, ".json")}"
  type  = "String"
  value = file("${path.module}/configs/${each.value}")
}
```

## Fix 6: Terraform Cloud and Remote Execution

When using Terraform Cloud or remote backends, only files within the working directory are uploaded:

```
Error: Error in function call

Call to function "file" failed: no file exists at "/home/user/certs/ca.pem"
```

Absolute paths to local files do not work in remote execution because the file is on your laptop, not on the Terraform Cloud runner.

**Fix:** Keep all referenced files within the Terraform working directory:

```
project/
  main.tf
  certs/          # Include certs in the project
    ca.pem
  scripts/        # Include scripts in the project
    init.sh
```

And reference them with `path.module`:

```hcl
locals {
  ca_cert = file("${path.module}/certs/ca.pem")
}
```

For sensitive files, use variables or a secrets manager instead of file references.

## Fix 7: Symlinks and Git Issues

Symbolic links in your project might not work as expected, especially in CI/CD:

```bash
# Symlink might not be followed
ls -la scripts/init.sh
# lrwxrwxrwx 1 user user 25 Jan 1 00:00 scripts/init.sh -> ../../shared/init.sh
```

If the symlink target is outside the working directory, Terraform Cloud will not include it.

**Fix:** Copy files instead of symlinking, or include the symlink target in your working directory.

Also, check your `.gitignore` - files excluded from Git will not be in your Terraform Cloud workspace:

```bash
# Check if the file is tracked by Git
git ls-files scripts/init.sh
```

## Fix 8: Case Sensitivity

On macOS (case-insensitive filesystem), `file("Config.json")` finds `config.json`. But on Linux (case-sensitive), it does not:

```hcl
# This works on macOS but fails on Linux if the file is config.json
resource "aws_instance" "web" {
  user_data = file("${path.module}/Config.json")
}
```

**Fix:** Always use the exact case that matches the filename. This avoids surprises when moving between development (macOS) and CI/CD (Linux).

## Fix 9: Dynamic File Paths

Building file paths dynamically can result in paths that do not exist:

```hcl
variable "environment" {
  type = string
}

# This fails if there is no file for the given environment
locals {
  config = file("${path.module}/configs/${var.environment}.json")
}
```

**Fix:** Validate the file exists or provide a fallback:

```hcl
locals {
  config_path = "${path.module}/configs/${var.environment}.json"
  config = fileexists(local.config_path) ? file(local.config_path) : file("${path.module}/configs/default.json")
}
```

## Debugging File Path Issues

When a file is not found, trace the actual path Terraform is looking for:

```hcl
# Output the path to see what Terraform resolves
output "debug_path" {
  value = "${path.module}/scripts/init.sh"
}

output "debug_root" {
  value = path.root
}

output "debug_cwd" {
  value = path.cwd
}
```

Run `terraform plan` and check the output values.

Also verify the file exists on disk:

```bash
ls -la $(terraform output -raw debug_path)
```

## Conclusion

Terraform file-not-found errors are almost always a path issue. The file exists but Terraform is looking in the wrong place. Use `path.module` for files next to your Terraform code, `path.root` for root-level files, and never use bare relative paths. For files that might not exist, use `fileexists()` to check first. And when running remotely (Terraform Cloud, CI/CD), make sure all referenced files are within the working directory and tracked by version control.
