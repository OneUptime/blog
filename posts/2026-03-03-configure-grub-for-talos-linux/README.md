# How to Configure GRUB for Talos Linux

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Talos Linux, GRUB, Boot Loader, Kubernetes, Legacy BIOS

Description: Learn how GRUB works in Talos Linux, when to use it, and how to configure it for both BIOS and UEFI boot scenarios.

---

GRUB (GRand Unified Bootloader) has been the standard Linux boot loader for decades, and Talos Linux continues to support it for systems that need it. While newer Talos installations on UEFI hardware default to systemd-boot, GRUB remains important for legacy BIOS systems, certain hardware configurations, and existing installations that have not migrated.

This guide covers how GRUB works within Talos Linux, how to configure it, and when it is the right choice.

## When Does Talos Linux Use GRUB?

Talos Linux uses GRUB in these situations:

- **Legacy BIOS systems**: Machines that do not support UEFI must use GRUB since systemd-boot is UEFI-only
- **Older Talos installations**: Systems installed with older Talos versions that defaulted to GRUB
- **Specific hardware compatibility**: Some UEFI implementations work better with GRUB than systemd-boot
- **PXE boot chains**: Some PXE setups chainload GRUB for the final boot stage

If you are setting up a new system with UEFI, you will likely get systemd-boot by default. But understanding GRUB is valuable for managing existing infrastructure.

## How GRUB Is Installed in Talos

When Talos Linux installs with GRUB, it sets up the boot loader on the target disk. The layout differs between BIOS and UEFI:

### BIOS Layout

```text
Disk:
  ├── MBR (GRUB stage 1)
  ├── Post-MBR gap (GRUB stage 1.5)
  ├── BOOT partition (GRUB stage 2, kernel, initramfs)
  ├── META partition
  ├── STATE partition
  └── EPHEMERAL partition
```

### UEFI Layout with GRUB

```text
Disk:
  ├── EFI System Partition
  │   └── EFI/BOOT/BOOTX64.EFI (GRUB EFI binary)
  ├── BOOT partition (kernel, initramfs, grub.cfg)
  ├── META partition
  ├── STATE partition
  └── EPHEMERAL partition
```

## Viewing GRUB Configuration

On a running Talos system using GRUB, you can inspect the configuration:

```bash
# Check the boot loader type
talosctl dmesg | grep -i grub

# View the current kernel command line (set by GRUB)
talosctl read /proc/cmdline

# List files on the boot partition
talosctl ls /boot/
```

The GRUB configuration file is generated by Talos during installation and upgrades. You do not edit it directly - instead, you configure Talos, and it generates the appropriate GRUB configuration.

## Configuring GRUB Through Talos

All GRUB configuration is managed through the Talos machine configuration. Here are the key settings:

### Extra Kernel Arguments

Pass additional kernel parameters through the machine config:

```yaml
# controlplane.yaml or worker.yaml
machine:
  install:
    extraKernelArgs:
      - net.ifnames=0
      - console=ttyS0,115200
      - nosmt
      - mitigations=auto
```

These parameters appear in the GRUB configuration and are passed to the kernel at every boot.

### Serial Console Configuration

For servers managed through serial consoles or IPMI/BMC:

```yaml
machine:
  install:
    extraKernelArgs:
      - console=ttyS0,115200n8
      - console=tty0
```

This tells GRUB and the kernel to output to both the serial port and the local display. The serial port configuration (115200 baud, 8N1) should match your IPMI/BMC settings.

### Install Disk Configuration

```yaml
machine:
  install:
    disk: /dev/sda        # Target disk for installation
    image: ghcr.io/siderolabs/installer:v1.9.0
    wipe: false           # Set true for clean install
    bootloader: true      # Install/update the boot loader
```

The `bootloader: true` setting ensures GRUB is installed or updated during the installation process. If you set this to false, the existing boot loader configuration is preserved.

## The A/B Boot Scheme with GRUB

Like systemd-boot, GRUB in Talos supports the A/B partition scheme for safe upgrades:

```text
BOOT partition:
  ├── A/
  │   ├── vmlinuz    # Kernel (slot A)
  │   └── initramfs  # Initramfs (slot A)
  ├── B/
  │   ├── vmlinuz    # Kernel (slot B)
  │   └── initramfs  # Initramfs (slot B)
  └── grub/
      └── grub.cfg   # GRUB configuration
```

During an upgrade:

```bash
# Trigger an upgrade
talosctl upgrade --image ghcr.io/siderolabs/installer:v1.10.0

# What happens:
# 1. New kernel and initramfs are written to the inactive slot
# 2. grub.cfg is updated to boot from the new slot
# 3. Machine reboots into the new version
# 4. If boot fails, GRUB falls back to the previous slot
```

The GRUB configuration includes fallback logic:

```text
# Simplified view of what Talos generates in grub.cfg
set default=0
set timeout=3

menuentry "Talos (A)" {
    linux /A/vmlinuz talos.platform=metal ...
    initrd /A/initramfs
}

menuentry "Talos (B)" {
    linux /B/vmlinuz talos.platform=metal ...
    initrd /B/initramfs
}
```

## GRUB and Secure Boot

Using GRUB with UEFI Secure Boot requires a shim - a first-stage boot loader signed by Microsoft that then loads GRUB. Talos Linux can work with this setup, but it is more complex than systemd-boot's native Secure Boot support.

If Secure Boot is a requirement and you are using UEFI, consider switching to systemd-boot for a cleaner implementation.

## Performance Comparison

GRUB adds a small amount of overhead compared to systemd-boot:

- GRUB typically adds 1-3 seconds to boot time due to its module loading and configuration parsing
- systemd-boot adds less than 1 second
- For most use cases, this difference is negligible
- On systems that boot frequently (e.g., auto-scaling nodes), it can add up

## Troubleshooting GRUB Issues

### Boot Hangs at GRUB

If the system stops at the GRUB menu or a GRUB prompt:

```bash
# From the GRUB prompt, try booting manually
grub> ls
grub> set root=(hd0,2)
grub> linux /A/vmlinuz talos.platform=metal
grub> initrd /A/initramfs
grub> boot
```

This manual boot helps identify whether the issue is with the GRUB configuration or the kernel/initramfs themselves.

### GRUB Rescue Mode

If you see `grub rescue>`, it means GRUB cannot find its configuration or modules:

```bash
# From grub rescue, try to find the boot partition
grub rescue> ls
grub rescue> set prefix=(hd0,2)/grub
grub rescue> insmod normal
grub rescue> normal
```

If this does not work, boot from a Talos USB and reinstall:

```bash
# Boot from USB, then reinstall Talos
talosctl apply-config --insecure \
  --nodes <NODE_IP> \
  --file controlplane.yaml
```

### Corrupt GRUB Installation

If GRUB is corrupted, the easiest fix is to reinstall Talos to the same disk:

```bash
# Boot from USB into maintenance mode
# Apply the configuration - this reinstalls GRUB
talosctl apply-config --insecure \
  --nodes <NODE_IP> \
  --file controlplane.yaml
```

The STATE partition (which holds your cluster configuration) is preserved during reinstallation, so you will not lose your cluster membership.

### Checking GRUB Version

```bash
# View GRUB-related messages in the kernel log
talosctl dmesg | grep -i grub

# Check the boot partition contents
talosctl ls /boot/grub/
```

## Migrating from GRUB to systemd-boot

If you are running GRUB on a UEFI system and want to switch to systemd-boot, you can do so during a Talos upgrade. The upgrade process will detect UEFI and install systemd-boot if the Talos version supports it.

```bash
# Upgrade to a version that defaults to systemd-boot
talosctl upgrade --image ghcr.io/siderolabs/installer:v1.9.0

# After the upgrade, verify the boot loader
talosctl dmesg | grep -i "boot\|systemd\|grub"
```

Note that this migration requires careful planning. Test on a non-critical node first to make sure your hardware works well with systemd-boot.

## Best Practices

1. **Let Talos manage GRUB**: Do not attempt to modify GRUB configuration files directly. Use the machine configuration instead.
2. **Keep a recovery USB ready**: GRUB issues can leave systems unbootable, and a USB stick with Talos is your fastest recovery option.
3. **Use serial console parameters**: If your servers have IPMI/BMC, always configure serial console output so you can debug boot issues remotely.
4. **Plan for migration**: If you are running GRUB on UEFI, plan a migration path to systemd-boot for better security and performance.

## Conclusion

GRUB remains an important part of the Talos Linux ecosystem, especially for legacy BIOS systems and existing installations. While Talos handles most GRUB configuration automatically, understanding how it works gives you the ability to diagnose boot issues and make informed decisions about your infrastructure. For new UEFI installations, systemd-boot is the recommended path forward, but GRUB continues to serve reliably where it is needed.
