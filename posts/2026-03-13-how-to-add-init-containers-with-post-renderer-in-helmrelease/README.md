# How to Add Init Containers with Post-Renderer in HelmRelease

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Flux, HelmRelease, Kubernetes, GitOps, Helm, Post-Renderer, Init Containers, Kustomize

Description: Learn how to inject init containers into Helm chart deployments using Flux post-renderers for setup tasks and dependency checks.

---

## Introduction

Init containers run before the main application containers in a pod and are used for setup tasks such as waiting for dependencies, running database migrations, downloading configuration files, or setting up permissions on shared volumes. While some Helm charts expose init container configuration through values, many do not, leaving you unable to add these essential startup tasks.

Flux post-renderers provide a clean way to inject init containers into any workload resource generated by a Helm chart. Using strategic merge patches, you can add one or more init containers without modifying the chart itself, keeping your deployment configuration separate from the chart source.

This guide covers common init container patterns and shows you how to implement them using Flux post-renderers.

## Prerequisites

Before you begin, you need:

- A Kubernetes cluster with Flux CD installed
- A HelmRelease deployed through Flux
- kubectl access to the cluster
- Familiarity with Kubernetes init container behavior

## Adding a Simple Init Container

To add an init container that waits for a database to be ready before starting the application:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: default
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    initContainers:
                      - name: wait-for-db
                        image: busybox:1.28
                        command:
                          - sh
                          - -c
                          - |
                            until nc -z postgres-service 5432; do
                              echo "Waiting for database..."
                              sleep 2
                            done
                            echo "Database is ready"
                        resources:
                          requests:
                            cpu: 10m
                            memory: 16Mi
                          limits:
                            cpu: 50m
                            memory: 32Mi
```

This init container checks for database connectivity before the main application container starts.

## Adding a Database Migration Init Container

Running database migrations as an init container ensures the schema is up to date before the application starts:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    initContainers:
                      - name: run-migrations
                        image: my-app:2.0.0
                        command: ["./migrate", "--direction", "up"]
                        env:
                          - name: DATABASE_URL
                            valueFrom:
                              secretKeyRef:
                                name: db-credentials
                                key: url
                        resources:
                          requests:
                            cpu: 100m
                            memory: 128Mi
                          limits:
                            cpu: 500m
                            memory: 256Mi
```

The migration container uses the same application image but runs the migration command instead of the application server.

## Adding Multiple Init Containers

Init containers run sequentially in the order they are defined. You can chain multiple setup tasks:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    initContainers:
                      - name: wait-for-db
                        image: busybox:1.28
                        command:
                          - sh
                          - -c
                          - "until nc -z postgres-service 5432; do sleep 2; done"
                        resources:
                          requests:
                            cpu: 10m
                            memory: 16Mi
                      - name: run-migrations
                        image: my-app:2.0.0
                        command: ["./migrate", "--direction", "up"]
                        env:
                          - name: DATABASE_URL
                            valueFrom:
                              secretKeyRef:
                                name: db-credentials
                                key: url
                        resources:
                          requests:
                            cpu: 100m
                            memory: 128Mi
                      - name: seed-cache
                        image: curlimages/curl:8.5.0
                        command:
                          - sh
                          - -c
                          - "curl -sf http://config-service/warmup || true"
                        resources:
                          requests:
                            cpu: 10m
                            memory: 16Mi
```

These init containers run in order: first waiting for the database, then running migrations, and finally warming up a cache.

## Volume Setup Init Container

Init containers can prepare volumes for the main container, such as setting permissions or downloading configuration:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: production
spec:
  interval: 5m
  chart:
    spec:
      chart: my-app
      version: "2.0.0"
      sourceRef:
        kind: HelmRepository
        name: my-repo
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: my-app
            patch: |
              apiVersion: apps/v1
              kind: Deployment
              metadata:
                name: my-app
              spec:
                template:
                  spec:
                    initContainers:
                      - name: fix-permissions
                        image: busybox:1.28
                        command:
                          - sh
                          - -c
                          - "chown -R 1000:1000 /data && chmod 750 /data"
                        volumeMounts:
                          - name: data-volume
                            mountPath: /data
                        securityContext:
                          runAsUser: 0
                        resources:
                          requests:
                            cpu: 10m
                            memory: 16Mi
                    volumes:
                      - name: data-volume
                        persistentVolumeClaim:
                          claimName: my-app-data
```

This init container runs as root to fix volume permissions before the main container starts with a non-root user.

## Adding Init Containers to StatefulSets

The same approach works for StatefulSets and other workload types:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: elasticsearch
  namespace: logging
spec:
  interval: 5m
  chart:
    spec:
      chart: elasticsearch
      version: "8.0.0"
      sourceRef:
        kind: HelmRepository
        name: elastic
        namespace: flux-system
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: StatefulSet
            patch: |
              apiVersion: apps/v1
              kind: StatefulSet
              metadata:
                name: placeholder
              spec:
                template:
                  spec:
                    initContainers:
                      - name: sysctl-tuning
                        image: busybox:1.28
                        command:
                          - sysctl
                          - -w
                          - vm.max_map_count=262144
                        securityContext:
                          privileged: true
                        resources:
                          requests:
                            cpu: 10m
                            memory: 16Mi
```

This adds a required sysctl tuning init container that Elasticsearch needs for proper operation.

## Verifying Init Containers

After deployment, verify init containers are running correctly:

```bash
kubectl get pods -n production -l app.kubernetes.io/name=my-app -o jsonpath='{range .items[*]}{.metadata.name}: {range .spec.initContainers[*]}{.name} {end}{"\n"}{end}'
```

Check init container completion status:

```bash
kubectl describe pod -n production -l app.kubernetes.io/name=my-app | grep -A 5 "Init Containers"
```

## Conclusion

Flux post-renderers provide a reliable way to inject init containers into Helm chart workloads without modifying the chart source. Whether you need dependency checks, database migrations, volume preparation, or system tuning, init containers added through post-renderers integrate seamlessly with the Helm release lifecycle. Always include resource requests and limits for init containers, and chain them in the correct order when multiple setup steps are required.
