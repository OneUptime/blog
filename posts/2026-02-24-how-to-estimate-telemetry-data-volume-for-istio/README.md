# How to Estimate Telemetry Data Volume for Istio

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Istio, Telemetry, Observability, Monitoring, Tracing

Description: Learn how to estimate the volume of metrics, traces, and access logs generated by Istio and plan storage and infrastructure accordingly.

---

Istio ships with a full observability stack built into every sidecar proxy. Metrics, distributed traces, and access logs are all generated at the proxy level for every request that flows through the mesh. This is incredibly powerful for debugging and monitoring, but it also means you can end up generating terabytes of telemetry data per day if you are not careful about what you collect and how you store it.

Before you turn on all the knobs, take the time to estimate how much data you will actually produce.

## The Three Pillars of Istio Telemetry

Istio generates three types of telemetry:

1. **Metrics**: Numeric measurements exposed in Prometheus format (counters, histograms, gauges)
2. **Distributed traces**: Span data showing request flow across services
3. **Access logs**: Detailed per-request log entries

Each has different volume characteristics and storage requirements.

## Estimating Metrics Volume

Metrics are the most storage-efficient form of telemetry, but their cardinality (number of unique time series) can still get out of hand.

### Counting Time Series

```bash
# Check how many metric lines a single sidecar exposes
kubectl exec <pod-name> -c istio-proxy -- curl -s localhost:15090/stats/prometheus | wc -l

# Check specific Istio metrics
kubectl exec <pod-name> -c istio-proxy -- curl -s localhost:15090/stats/prometheus | grep "^istio_" | wc -l
```

Typical numbers per sidecar:
- Total metric lines: 500-2,000
- Istio-specific metric lines: 100-500
- Unique time series (after label expansion): varies by connectivity pattern

### Calculating Total Metrics Data

```
Metrics data rate = Time_Series x Sample_Size / Scrape_Interval

Where:
  Time_Series = varies (see cardinality estimation)
  Sample_Size = 1.5 bytes (compressed in Prometheus TSDB)
  Scrape_Interval = 15 seconds (typical default)

Example for 200,000 time series, 15s scrape:
  Ingestion rate: 200,000 / 15 = 13,333 samples/second
  Storage rate: 13,333 x 1.5 = 20 KB/s = 1.7 GB/day

With 15-day retention: 1.7 GB/day x 15 = 25.5 GB
```

The raw scrape data is much larger than the compressed storage. Each Prometheus scrape for a sidecar transfers 50-200 KB:

```
Scrape bandwidth = Num_Sidecars x Scrape_Size / Scrape_Interval
Example: 500 x 100 KB / 15s = 3.3 MB/s network bandwidth for scraping
```

## Estimating Trace Volume

Distributed traces are the telemetry type that can generate the most data, because you produce spans for every sampled request across every service in the call chain.

### Trace Data Model

Each trace consists of multiple spans. A single HTTP request might generate 3-10 spans as it flows through the mesh:

- Client sidecar span (outbound)
- Server sidecar span (inbound)
- Application span (if instrumented)
- Repeated for each hop in the call chain

### Volume Calculation

```
Trace data rate = RPS x Sampling_Rate x Spans_Per_Request x Avg_Span_Size

Where:
  RPS = requests per second (at the ingress point)
  Sampling_Rate = fraction of requests sampled (0.01 = 1%)
  Spans_Per_Request = average number of spans per traced request
  Avg_Span_Size = 500-1000 bytes per span (JSON/protobuf)
```

**Example calculations for 10,000 RPS at the edge:**

At 1% sampling:
```
10,000 x 0.01 x 6 spans x 750 bytes = 450 KB/s = 39 GB/day
```

At 10% sampling:
```
10,000 x 0.10 x 6 spans x 750 bytes = 4.5 MB/s = 389 GB/day
```

At 100% sampling:
```
10,000 x 1.00 x 6 spans x 750 bytes = 45 MB/s = 3.9 TB/day
```

The difference between 1% and 100% sampling is enormous. This is why choosing the right sampling strategy matters.

### Configuring Trace Sampling

```yaml
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    defaultConfig:
      tracing:
        sampling: 1.0
    extensionProviders:
      - name: zipkin
        zipkin:
          service: zipkin.istio-system.svc.cluster.local
          port: 9411
```

The `sampling` value is a percentage (1.0 = 1%, 100.0 = 100%).

For production, start with 1% and adjust based on your debugging needs and storage budget.

## Estimating Access Log Volume

Access logs are the most verbose telemetry type. Every request through every sidecar generates a log entry.

### Default Access Log Format

Istio's default access log includes information like:

```
[2026-02-24T10:15:30.123Z] "GET /api/users HTTP/1.1" 200 - via_upstream - "-" 0 1234 45 43 "-" "curl/7.68.0" "abc-123-def-456" "users.default.svc.cluster.local:8080" "10.0.1.15:8080" inbound|8080|| 10.0.1.15:42312 10.0.1.15:8080 10.0.2.30:51234 outbound_.8080_._.users.default.svc.cluster.local default
```

A typical access log entry is 300-600 bytes.

### Volume Calculation

```
Access log data rate = Total_RPS_Across_Mesh x Avg_Log_Entry_Size x Log_Points

Where:
  Total_RPS = sum of RPS across all sidecars (both inbound and outbound)
  Avg_Log_Entry_Size = 400 bytes
  Log_Points = 2 (one at source sidecar, one at destination sidecar)
```

**Example for 10,000 edge RPS with average 3 hops per request:**

```
Total proxy-level RPS = 10,000 x 3 hops x 2 (inbound + outbound) = 60,000 log entries/sec
Data rate = 60,000 x 400 bytes = 24 MB/s = 2.07 TB/day
```

That is a lot of data. Access logs should be used selectively.

### Enabling Access Logs Selectively

Instead of enabling access logs mesh-wide, target specific namespaces or workloads:

```yaml
apiVersion: networking.istio.io/v1
kind: Telemetry
metadata:
  name: access-logs
  namespace: frontend
spec:
  accessLogging:
    - providers:
        - name: envoy
      filter:
        expression: "response.code >= 400"
```

This only logs errors, which dramatically reduces volume while keeping the most useful debugging information.

## Total Telemetry Volume Summary

For a mesh with 500 pods and 10,000 edge RPS:

| Telemetry Type | Configuration | Data Rate | Daily Volume |
|---|---|---|---|
| Metrics | Default, 15s scrape | 20 KB/s | 1.7 GB |
| Traces (1% sampling) | 1% random | 450 KB/s | 39 GB |
| Traces (100% sampling) | Full sampling | 45 MB/s | 3.9 TB |
| Access logs (all) | All requests | 24 MB/s | 2.07 TB |
| Access logs (errors only) | >= 400 status | 240 KB/s | 20 GB |

**Recommended production setup:**

```
Metrics (default): 1.7 GB/day
Traces (1% sampling): 39 GB/day
Access logs (errors only): 20 GB/day
Total: ~61 GB/day
```

Compare that to "everything on":

```
Metrics + 100% traces + all access logs: ~6 TB/day
```

## Infrastructure Sizing for Telemetry Backends

Based on your volume estimates, size your telemetry infrastructure:

### Prometheus/Thanos for Metrics

```yaml
# Prometheus storage
storage: 50 GB (for 15-day retention of 200K time series)
CPU: 2 cores
Memory: 8 GB
```

### Jaeger/Tempo for Traces

```yaml
# At 1% sampling, 39 GB/day with 7-day retention
storage: 300 GB
CPU: 4 cores
Memory: 16 GB
```

### Loki/Elasticsearch for Logs

```yaml
# Errors-only access logs, 20 GB/day with 7-day retention
storage: 200 GB (with compression)
CPU: 4 cores
Memory: 16 GB
```

## Monitoring Your Telemetry Pipeline

Keep tabs on the telemetry infrastructure itself:

```promql
# Prometheus ingestion rate
rate(prometheus_tsdb_head_samples_appended_total[5m])

# Dropped samples (indicates overload)
rate(prometheus_target_scrapes_exceeded_sample_limit_total[5m])

# Trace collector queue depth (Jaeger)
jaeger_collector_queue_length
```

The key takeaway is to start conservative with your telemetry configuration. Enable metrics by default, use low trace sampling rates, and only turn on access logs for specific debugging needs. You can always dial things up when you need to investigate an issue, but you cannot get back the storage and compute costs of collecting data you never look at.
