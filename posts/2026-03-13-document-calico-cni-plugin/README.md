# Document Calico CNI Plugin for Operators

Author: [nawazdhandala](https://github.com/nawazdhandala)

Tags: Calico, Kubernetes, Networking, CNI, Plugin, Documentation, Operations

Description: How to document the Calico CNI plugin configuration, IPAM design, and operational procedures to support engineers managing pod networking in Kubernetes clusters.

---

## Introduction

CNI plugin documentation often receives less attention than higher-level Kubernetes or Calico documentation, yet it is critical for understanding how pod networking is initialized and why pods sometimes fail to start. Good CNI documentation explains the IPAM design, describes the CNI configuration file and why each parameter is set the way it is, and provides runbooks for the most common CNI-related failures.

## Prerequisites

- Calico CNI operational in production
- Documentation system accessible to the team

## Documentation Component 2: IPAM Design Reference

```markdown
## Calico IPAM Design - Cluster: prod-k8s-01

### IP Pool Design
| Pool Name | CIDR | Namespace | Block Size | Encapsulation |
|-----------|------|-----------|-----------|---------------|
| prod-pool | 192.168.0.0/20 | production | /24 | CrossSubnet |
| dev-pool | 192.168.16.0/20 | development | /24 | CrossSubnet |
| default-pool | 192.168.32.0/16 | all others | /24 | CrossSubnet |

### Capacity Planning
- Prod pool: 4,096 IPs, max 16 nodes × 256 pods/node
- Dev pool: 4,096 IPs, max 16 nodes × 256 pods/node
- Default pool: 65,536 IPs, remaining cluster capacity

### IPAM Block Allocation Strategy
Each node receives /24 blocks on demand.
Nodes never share blocks (isolation prevents IPAM conflicts).
```

## Documentation Component 3: CNI Configuration Reference

```markdown
## CNI Configuration File Reference (/etc/cni/net.d/10-calico.conflist)

### Current Configuration (auto-generated by calico-node DaemonSet)
Key settings and their purpose:

| Parameter | Value | Why |
|-----------|-------|-----|
| `datastore_type` | `kubernetes` | Uses K8s API for better caching |
| `mtu` | `0` | Auto-detect from Felix config |
| `log_level` | `warning` | Minimal I/O overhead |
| `ipam.type` | `calico-ipam` | Calico-native IPAM |
| `ipam.ipv4_pools` | `["default-pool"]` | Default pool for all pods |

To change: Edit calico-config ConfigMap `cni_network_config` field.
```

## Documentation Component 4: CNI Operational Runbooks

```markdown
## Runbook: Pods Stuck in ContainerCreating

### Symptoms
- Pods never leave ContainerCreating status
- kubectl describe shows CNI error events

### Diagnosis
1. Get pod's node: kubectl get pod <pod> -o jsonpath='{.spec.nodeName}'
2. Check CNI logs on that node:
   kubectl debug node/<NODE> -it --image=ubuntu -- tail -50 /host/var/log/calico/cni/cni.log
3. Check IPAM capacity: calicoctl ipam show
4. Check calico-node pod is running: kubectl get pods -n calico-system | grep <NODE>

### Common Fixes
- IPAM exhausted: calicoctl create -f additional-pool.yaml
- CNI config missing: kubectl rollout restart ds/calico-node -n calico-system
- Leaked handles: calicoctl ipam check && calicoctl ipam gc
```

## Documentation Component 5: Capacity Planning

```markdown
## CNI Capacity Planning Guide

### How to Calculate Max Pods Per Cluster
1. Total IPs in all pools: calicoctl ipam show | grep total
2. Subtract 5% for IPAM overhead
3. Divide by average pods per IP (typically 1:1)

### Warning Thresholds
- 70% pool utilization: Plan expansion
- 85% pool utilization: Start expansion immediately
- 95% pool utilization: Emergency - new pods will fail

### How to Expand an IP Pool
calicoctl create -f new-pool.yaml  # Add additional pool
# Note: You CANNOT shrink a pool that has active allocations
```

## Conclusion

Documenting the Calico CNI plugin provides operators with the context needed to understand IPAM design decisions, the CNI configuration file parameters and their rationale, and runbooks for the most common failure modes. The IPAM capacity planning section is particularly valuable for avoiding the most common production CNI failure — IP address pool exhaustion that prevents new pods from starting.
